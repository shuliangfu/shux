# 全量测试结果说明

## 当前 run-all.sh 行为

- **无 SHU（默认）**：`export RUN_ALL_USE_C=1`，仅 `make -C compiler all` 得到 C 版 `compiler/shux`；**run-hello.sh 不执行 bootstrap-driver-seed**，整次全量**全程使用 C 流水线**。
- **有 SHU**：用指定的 `$SHUX`（如 `./tests/run-all-x.sh` 设 `SHUX=compiler/shux_x`）跑所有 run-*.sh，走 .x 流水线（run-hello 等会加 `-x`）。

## .x 是否有测试、是否通过

- **有**：入口为 **`./tests/run-all-x.sh`**。该脚本会先构建 `compiler/shux_x`（bootstrap-pipeline、shux-x-pipeline、bootstrap-driver-seed 等），再执行 `SHUX=compiler/shux_x ./tests/run-all.sh`，即用 .x 流水线跑与 run-all 相同的全量用例。
- **通过情况**：在修复「run-all 默认 C / run-all-x 用 .x」及 C 端单参调用约定（print_i32/ok_i32/err_i32 等）后，**C 与 .x 全量均已通过**：
  - **C**：`./tests/run-all.sh` → `all tests OK`
  - **.x**：`./tests/run-all-x.sh` → `run-all-x: all tests OK (shux_x)`

## 打印 C 与 asm SKIP

- **生成 C 不打印到终端**：`driver_run_compiler_full` 在无 `out_path` 时会再扫 argv 补取 `-o` 下一参，避免误向 stdout 写 C；run-hello 在用 `shux_x -x ... -o ...` 时对编译命令加 `1>/dev/null`，仅丢弃 stdout，stderr 仍保留。
- **asm 构建（macOS arm64，2026-05-20）**：`SHUX=./shux ./scripts/build_shux_asm.sh` 后 `check_asm_o_quality.sh` 为 **`summary bad=0 total=24`**，`build_asm/.asm_text_quality=1`（`asm_build_list` 所列 24 个 `.o` 的 `__text` 均非空）。非 Linux 宿主拓扑仍为 **`pipeline_x`（B-hybrid）**：`shux_asm` 由 `-E` 的 `pipeline_x.o` + C seed 链出（非 full_asm）；完整 LSP 诊断仍用 `make bootstrap-driver-seed` 的 `shux` / `shux-x`。Linux 在同等质检通过时可自动选 `full_asm` 并优先 crt0 路径（见 `docs/SELFHOST.md` §4）。

## 建议

- **日常/CI 全量（默认 C）**：`./tests/run-all.sh`
- **验证 .x 流水线全量**：`./tests/run-all-x.sh`
- **显式 C 全量**：`./tests/run-all-c.sh`（`SHUX=compiler/shux-c`）

结论：**C 与 .x 均有全量测试；当前两者均通过。**
