# 全量测试结果说明

## 当前 run-all.sh 行为

- **无 SHU（默认）**：`export RUN_ALL_USE_C=1`，仅 `make -C compiler all` 得到 C 版 `compiler/shu`；**run-hello.sh 不执行 bootstrap-driver-seed**，整次全量**全程使用 C 流水线**。
- **有 SHU**：用指定的 `$SHU`（如 `./tests/run-all-su.sh` 设 `SHU=compiler/shu_su`）跑所有 run-*.sh，走 .su 流水线（run-hello 等会加 `-su`）。

## SU 是否有测试、是否通过

- **有**：入口为 **`./tests/run-all-su.sh`**。该脚本会先构建 `compiler/shu_su`（bootstrap-pipeline、shu-su-pipeline、bootstrap-driver-seed 等），再执行 `SHU=compiler/shu_su ./tests/run-all.sh`，即用 .su 流水线跑与 run-all 相同的全量用例。
- **通过情况**：在修复「run-all 默认 C / run-all-su 用 SU」及 C 端单参调用约定（print_i32/ok_i32/err_i32 等）后，**C 与 SU 全量均已通过**：
  - **C**：`./tests/run-all.sh` → `all tests OK`
  - **SU**：`./tests/run-all-su.sh` → `run-all-su: all tests OK (shu_su)`

## 打印 C 与 asm SKIP

- **生成 C 不打印到终端**：`driver_run_compiler_full` 在无 `out_path` 时会再扫 argv 补取 `-o` 下一参，避免误向 stdout 写 C；run-hello 在用 `shu_su -su ... -o ...` 时对编译命令加 `1>/dev/null`，仅丢弃 stdout，stderr 仍保留。
- **asm 构建**：`-backend asm` 已走**真正 asm 后端**（`driver_run_asm_backend`）；ELF 侧将 `syms` 上限由 16 提至 256、`relocs` 由 32 提至 128（`compiler/src/asm/platform/elf.su`），typeck/parser 等大模块不再因符号表满而 SKIP。当前各模块 asm 编译均为 OK；最终链接 shu_asm 若报 missing symbols 属 Goal 2 已知问题，见 src/asm/README.md。

## 建议

- **日常/CI 全量（默认 C）**：`./tests/run-all.sh`
- **验证 .su 流水线全量**：`./tests/run-all-su.sh`
- **显式 C 全量**：`./tests/run-all-c.sh`（`SHU=compiler/shu-c`）

结论：**C 与 SU 均有全量测试；当前两者均通过。**
