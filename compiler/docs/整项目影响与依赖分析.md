# 整项目影响与依赖分析

本文档描述「改哪里会牵涉哪里」、run-* 与各组件的依赖关系，以及已做修改的影响半径，避免修一处、崩另一处。与 [run-hello 整体分析与最小修复](run-hello整体分析与最小修复.md) 配合：后者只讲 run-hello 的因果与单点修复，本文档讲全项目拓扑与修改前必查清单。

---

## 一、编译与测试拓扑

### 1.1 谁调谁、谁生成谁

```
run-all.sh / run-hello / run-fmt / … 
  → 都用 compiler/shu（即 $(TARGET)，多为 bootstrap-driver-seed 产出的 shu）
  → shu 读入用户 .su，走 runtime.c → driver_run_compiler_full
  → 入口与 dep 的解析/typeck/codegen 在 pipeline 内完成
  → pipeline 实现在 pipeline_su.o（由 pipeline_gen.c 编出）
  → pipeline_gen.c = shu-c 对 pipeline.su 做 -E 展开（会内联 parser/typeck/codegen/ast/lexer 等 .su）
```

- **bootstrap-driver-seed**：用 **shu-c**（纯 C 编译器）把 pipeline.su **-E 展开**得到 pipeline_gen.c，再编成 pipeline_su.o，与 main_driver、**runtime_driver**、**parser.o**、**typeck.o**、**codegen.o** 等一起链接成 shu。
- 用户用 shu 编 hello.su 时：
  - **runtime.c** 负责：读源、组 dep、调 **pipeline_run_su_pipeline(module, arena, source_data, source_len, ...)**，写 C 到文件、调 cc 链接。
  - **pipeline_su.o** 里：入口由 **parse_into_with_init_buf(arena, module, source_data, source_len)** 解析（pipeline.su），内部调 **parser.parse_into_buf**（parser.su 的代码被展开进 pipeline_gen.c）。
  - **typeck / codegen** 由 **C 的 typeck.o / codegen.o** 提供（seed 不链 typeck_su.o / codegen_su.o），故生成用户 C 的代码在 **codegen.c**。

因此：**run-hello / run-fmt / run-string / run-fs 等所有用 compiler/shu 的 run-***，都受 **runtime.c、pipeline.su、codegen.c** 影响；**parser.su** 通过「被展开进 pipeline_gen.c」影响 seed 行为，但 seed 不链 parser_su.o。

### 1.2 两套 parser / codegen 的区分

| 组件 | C 版（.c） | .su 版（.su → -E → 进 pipeline_gen.c 或单独编） | run-* 用的谁（seed） |
|------|------------|--------------------------------------------------|------------------------|
| parser | parser.c → parser.o | parser.su 被 -E 内联进 pipeline_gen.c | 解析入口：**parser.su 内联版**（在 pipeline_su.o 里） |
| typeck | typeck.c → typeck.o | typeck.su → typeck_su.o（seed 不链） | **typeck.o** |
| codegen | codegen.c → codegen.o | codegen.su → codegen_su.o（seed 不链） | **codegen.o** |

- **LSP 诊断**：lsp_diag.c 用 **parser.o**（C 解析器）做错误收集，与 pipeline 内解析无关。
- 修改 **codegen.su** 只影响链了 codegen_su.o 的构建（如 shu_su、run-all-su 等）；**run-all 默认的 compiler/shu（seed）只看 codegen.c**。

## 二、run-* 与组件的对应关系

### 2.1 所有 run-* 共同依赖

- **runtime.c**：入口与 dep 的调度、**pipeline_run_su_pipeline(..., source_data, source_len)**、memset(arena,0); memset(module,0)；不改 entry_already_parsed。
- **pipeline.su**：**parse_into_with_init_buf(arena, module, source_data, source_len)**，入口在 pipeline 内解析。
- **codegen.c**：生成用户 C；若生成错（如 extern 返回 []u8 的 let 初值），会间接导致入口 num_funcs=0、无 main（见 run-hello 文档）。

### 2.2 按测试分类的额外依赖

| 测试 | 入口/用例特点 | 额外依赖 |
|------|----------------|----------|
| **run-hello** | examples/hello.su，需生成 main | 入口解析正确 + **codegen 对「let x: []u8 = extern 调用」的生成**（parser fallback 用） |
| **run-fmt** | tests/fmt/main.su，core.fmt | 需 _fmt_i32 等符号（preamble/std 链接）、入口有 main |
| **run-string** | tests/string/*.su，std.string | String 类型与链接、入口有 main |
| **run-fs** | tests/fs/*.su，std.fs | fs_* 符号与链接、入口有 main |
| **run-return-value** | main 最后一表达式为返回值 | 入口解析 + codegen 正确 |
| 其余 run-* | 各用例不同 | 共同依赖 + 各用例的 import/符号 |

结论：**改 runtime.c 的入口/ pipeline 调用、改 pipeline.su 的解析入口、改 codegen.c 的 let/extern 生成，都会影响所有 run-*；** 修 run-hello 的「extern 返回 slice」只动 codegen.c 一处，可避免牵涉 parser/pipeline/runtime 其它逻辑。

## 三、已做修改的影响半径（不要回退）

| 修改 | 位置 | 影响范围 | 回退后果 |
|------|------|----------|----------|
| 入口在 pipeline 内解析，memset(arena,0); memset(module,0)，不设 entry_already_parsed | runtime.c | 所有用 shu 编译的 run-* | C 解析与 .su Module 混用 → num_funcs 错位，无 main |
| 用 (data, len) 调 parse_into_with_init_buf | pipeline.su | 所有 run-*（入口解析） | []u8 布局 C/.su 不一致，解析或传参错误 |
| codegen 跳过 is_extern 函数定义 | codegen.su / codegen.c（若已同步） | 含 std.io 等 extern 的用例；bootstrap 生成 pipeline_gen.c 时 | 生成 io_read() 等定义与 preamble 的 extern 冲突，链接错误 |

这三项是「按下葫芦」时已稳定的前提；**再修 run-hello 只应动 codegen.c 里「extern 返回 slice」的赋值生成**，否则容易「起来瓢」。

## 四、修改前必查清单（避免起来瓢）

### 4.1 若你要改 runtime.c（入口 / pipeline 调用）

- [ ] 是否仍保证 **入口在 pipeline 内用 .su 解析**（parse_into_with_init_buf(data, len)），且 **memset(arena,0); memset(module,0)**？
- [ ] 是否影响 **pipeline_run_su_pipeline** 的 (module, arena, source_data, source_len) 约定？pipeline_glue.c / pipeline.su 是否仍一致？
- [ ] 跑 **run-hello、run-fmt、run-return-value** 至少三种，确认仍有 main、链接通过。

### 4.2 若你要改 pipeline.su（解析入口 / 依赖顺序）

- [ ] **parse_into_with_init_buf** 是否仍接收 (arena, module, source_data, source_len)，且内部只调 parser.parse_into_buf(arena, module, data, len)？
- [ ] 是否动到 dep 与 entry 的解析顺序或 Module 写回？跑 **run-hello、run-import、run-multi-file**。
- [ ] 改完是否需 **重新生成 pipeline_gen.c**（`make bootstrap-pipeline` 或 `make pipeline_su.o`；依赖 `PIPELINE_SU_DEPS`）？

### 4.2b 若你要改 main.su / preprocess.su（驱动 / 预处理）

- [ ] **main.su** 变更后须 **`make driver_su.o`**（或 `make gen-su-driver-objs` / `make bootstrap-driver-seed`）；`make shu-su` 已依赖 `driver_su.o`。
- [ ] **preprocess.su** 变更后须 **`make preprocess_su.o`**；`bootstrap-driver-seed` 与 `shu-su` 均链 `preprocess_su.o`。

### 4.3 若你要改 codegen.c（let/const/函数体/extern）

- [ ] **let x: []u8 = extern 调用(...)** 是否仍生成 `struct shulang_slice_uint8_t x = ...;` 形式？（否则 run-hello 的 parser fallback 失败 → num_funcs=0。）
- [ ] **extern 函数** 是否仍不生成定义（与 preamble 一致）？否则 run-hello / 含 std.io 的用例可能链接重定义。
- [ ] 改完 **make bootstrap-driver-seed** 再跑 **run-hello、run-fmt**，确认生成 C 含 main 且链接通过。

### 4.4 若你要改 parser.su（parse_into_buf / slice_for_impl）

- [ ] 行为变更会通过 **pipeline_gen.c 的重新生成** 进入 seed；生成 pipeline_gen.c 的是 **shu-c**，用的是 **codegen.c**。
- [ ] 若在 parser.su 里新增「用 slice 的路径」或 extern 调用，需确保 **codegen.c** 对「extern 返回 []u8」的生成正确，否则同 run-hello 问题。

### 4.5 若你要改 codegen.su

- [ ] 只影响链了 **codegen_su.o** 的构建（如 run-all-su、shu_su）；**run-all 默认的 seed 用 codegen.o**。
- [ ] 若某逻辑需在 seed 路径也生效（如跳过 extern 定义），**codegen.c 须同步**。
