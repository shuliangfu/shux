## 当前阻塞问题：std/string/mod.x init_globals 重复定义

### 现象

`run-all-bstrict.sh` 在 std 模块编译阶段失败：

```
shux_compile_std_module.sh: cc -c failed for ../std/string/mod.x
/tmp/tmp.C67ames5Q7/mod_0.c:503:13: error: redefinition of 'init_globals'
```

`./shux-c -x -E -L .. ../std/string/mod.x` 生成的 C 代码中 `init_globals` 出现两次（L102 和 L503）。

### 根因方向

codegen 对含 import binding call 的模块可能重复 emit `init_globals`。需要检查 codegen 的 top-level let hoist 逻辑。

## 已完成工作（2026-07-13）

### import binding call typeck XT001 修复（commit `179071ac` + `0e26f19e`）

**根因**：seed `typeck_gen.linux.x86_64.c` 的 `typeck_check_expr_method_call` 是旧的内联实现，只检查参数不解析 callee。.x 源码已委托 `pipeline_typeck_check_expr_method_call_c`（C glue，支持 import binding 解析 + unsafe 边界），但 seed 从未同步。

**调试过程**：
1. 在 `typeck_check_expr` 和 `check_expr_impl_mega` 加 `SHUX_DEBUG_CALL` 调试 print
2. 发现 `heap_ops.heap_mem_set_c(...)` 被解析为 EXPR_METHOD_CALL (kind=49)，不是 EXPR_CALL (kind=48)
3. seed 的 `typeck_check_expr_method_call` 只检查参数不解析 callee，返回 -1
4. `pipeline_typeck_check_expr_method_call_c` 未被调用（无调试输出）

**修复**：patch 脚本增加 `METHOD_CALL_BODY` 模板，将 seed 的 `typeck_check_expr_method_call` 替换为委托 `pipeline_typeck_check_expr_method_call_c`（与 CALL/DEREF 相同模式）。

**验证**：
- 最小测试用例 `heap_ops.heap_mem_set_c(ptr, byte, n)` typeck 通过 ✅
- G-FFI-5 release CI gate 全绿 ✅
- `run-all-bstrict.sh` 不再报 XT001，前进到 std/string/mod.x 编译阶段 ✅
