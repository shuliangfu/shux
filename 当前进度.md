### 本轮真正修掉的根因

`simd_loop.x` 仍残留一组旧的 `.x` 移植尾巴，具体表现为：

1. 多个 `extern "C"` 调用仍裸露在 `unsafe` 外部  
2. `glue_emit_f32_soa_sum_strip_c()` 中存在已损坏的条件分支形态，导致 typeck 无法稳定推导返回类型  
3. `glue_emit_runtime_strip_loop_c()` 中仍有同类裸 `extern` 调用与标签生成调用

本轮修复点：

- `glue_simd_local_var_stack_off_c()`
  - 将 `pipeline_expr_var_name_len/into` 与 `asm_ctx_local_find_offset*` 全部收进单一 `unsafe` 块
  - 去掉跨 `unsafe` 传递临时局部的旧形态
- `glue_simd_loop_emit_chunk_binop_c()`
  - 将 `simd_enc_try_hw_vector_*` 返回路径统一收进 `unsafe`
- `glue_emit_full_const_peel_c()`
  - 将 `getenv()` 调用收进 `unsafe`
- `glue_emit_f32_soa_sum_strip_c()`
  - 依据 seed C 权威逻辑重写损坏段
  - 修正 `off_n >= 0` 分支、`movups` 严格模式分支、标签生成与 epilogue 逻辑
- `glue_emit_runtime_strip_loop_c()`
  - 修正标签生成与 `simd_enc_try_hw_vector_binop_rbp_at_idx()` 的 `unsafe` 包裹

### 对 `unsafe parent_block_ref` 判断的更新

`unsafe { r = ...; }` 这类写法确实曾经触发过 typeck / parent 链相关问题；但**截至本轮复测，它已经不是当前这组 `-E` 文件的活跃前排**。  
当前更精确的说法应是：

> `simd_loop.x` 的失败根因是 `.x` 移植过程中残留的 `extern/unsafe` 不闭合和损坏分支形态，而不是继续卡在统一的 `parent_block_ref` 总问题上。

| 模式 | 现象 | 状态 |
|------|------|------|
| `if(A){if(B){}}` | 嵌套 if → parser skip 整个函数 | 已定位 |

## 关键约束

- `shux -E` 只能对 **0 imports** 的 .x 文件运行
- `shux -E` 会丢弃含 `&&`/`||` 操作符的函数体
- `shux -E` 在 32-bit 平台跳过含 `u64` 类型的函数
- 修复必须遵循**根源治理**原则
- macOS 开发 → git push → ssh ubuntu-server → Linux 验证
- `codegen_gen.c` / `parser_gen.c` / `pipeline_gen.c` 等生成文件在 `.gitignore` 中，不在 git 跟踪
- Ubuntu 修改 `pipeline_glue.c` 或 `ast_pool.c` 后需重建 `pipeline_x.o`
