## 当前焦点：struct `let` 字段 layout 登记失败（已修）

### 现象（原记录为 codegen `.`/`->` 与 typeck heap 失败）

1. 形参 `p: *S` / `s: S` 上 `p.x` / `s.x` 报 `found ?`
2. `-x -E` 对指针形参误用 `.`（codegen 侧已有 `field_access_base_is_pointer_param`，commit 1c449472）
3. `std/heap` 等大量使用 `struct { let field: T; }` 的模块 typeck 失败

### 真根因（2026-07-10 定位）

**不是** codegen / typeck 形参解析主路径坏了，而是 **parser 登记 struct layout 时不接受 `let` 字段前缀**：

| 语法 | layout 登记 | 形参 `s.x` typeck |
|------|-------------|-------------------|
| `struct S { x: i32; }` | ✅ | ✅ |
| `struct S { let x: i32; }` | ❌ 失败后 skip | ❌ `found ?`（无 STRUCT_LIT 先建 layout 时） |

- `parser_asm_parse_struct_record_layout_into_slice_c` 字段循环把 `TOKEN_LET` 当非法字段名 → return -1
- `parse_into` 失败后 `skip_one_struct`，**不写** `module.struct_layouts`
- typeck `get_field_type_ref_from_layout` 查不到字段 → 表达式类型 `?`
- 仅当同文件先有 `S { x: … }` 字面量走 `ensure_struct_layout_from_struct_lit` 时才「碰巧」绿

### 修复

| 文件 | 改动 |
|------|------|
| `compiler/src/asm/parser_asm_struct_layout_slice.c` | 字段循环消费可选 `let`/`const` 后再读字段名 |
| `compiler/src/asm/parser_asm_thin_c.c` | `field_continues` 允许 `TOKEN_LET`/`TOKEN_CONST`（多字段 `;` 后） |
| `compiler/Makefile` `relink-shux-lexer` | 对齐 `relink-shux`：补 `RELINK_SHUX_GLUE_SUFFIX` + filtered glue prefix |

## G-03 ASM 后端 bug 修复链（已完成）

### Bug 修复链（按修复顺序）

#### 1. bug ③ dep 前缀模型（commit c28d83e9 + db201393）

- **根因**：codegen.x line 6898/6927 的 `dep_index < 0 &&` 条件阻止 dep 模块 emit 顶层 const/let
- **修复**：去掉 dep_index<0 条件，dep 模块 const 以裸名 emit

#### 2. INDEX 步长 bug（commit fae16e5b）

- **根因**：typeck skip 时返回 0，默认 `return 4`（i32 步长）
- **修复**：typeck skip 时回退到 VAR 声明的 type_ref 推断元素大小

#### 3. shux_sys_open 指针截断 bug（2026-07-09）

- **根因**：`mov %edi,%edi` 将 path 指针高 32 位清零
- **修复**：仅 shux_sys_open 删除该指令

#### 4. 栈布局 bug —— dep struct slot size（commit b310bf0c）

- **根因**：`asm_local_slot_bytes_mod` 对 dep 模块 struct 查不到 layout
- **修复**：添加 dep 模块 struct layout 遍历

#### 5. 栈布局 bug —— 名称前缀不匹配（commit 45f97d71）

- **根因**：type name 带模块前缀 "heap.PageMmapHeap"，struct layout name 是裸名 "PageMmapHeap"
- **修复**：比较前取 type name 最后一个 "." 之后的部分

#### 6. u16/i16 字面量窄化 bug（commit 91ee0732）

- **根因**：AST 枚举无 TYPE_U16/TYPE_I16，typeck 字面量窄化和 codegen emit_type 缺 u16/i16 处理
- **修复**：typeck name 范围检查 + codegen TYPE_NAMED 分支映射

#### 7. 负数字面量 bug（2026-07-09）

- **根因**：`-1` 解析为 EXPR_NEG(lit(1))，typeck 的 init_kind 允许列表缺 EXPR_NEG
- **修复**：添加 EXPR_NEG 到 init_kind 检查 + codegen i8→int8_t 映射

**常见陷阱**：

1. **`make` 默认走 bootstrap_shuxc target**：不带 `-B` 时，make 可能从 `seeds/bootstrap_shuxc.linux.x86_64`（6月30日旧版本）复制，不含最新源码修改。**必须用 `make -C compiler shux-c SHUX_LEGACY_C_FRONTEND=1 -B`**。
2. **shux-c vs shux_asm 链接路径差异**：
   - `shux_invoke_cc`（runtime_link_abi.c）是 **C 前端链接路径**
   - `shux_asm_ld_append_std_objs` 是 **ASM 后端链接路径**
   - `hello.x` 走 shux_invoke_cc 路径，不经过 shux_asm_ld_append_std_objs
3. **run-all.sh 的 SHUX 选择逻辑**：不设 SHUX 环境变量时，子脚本优先选 `./compiler/shux`（seed，.x pipeline），其次 `./compiler/shux-c`（C 前端，.x pipeline）。设 `SHUX=./compiler/shux-c` 会破坏 58+ 测试（seed 是正确测试工具）。
4. **每次测试调试前必须清理所有 .o 文件**：避免旧 .o 文件导致测试结果不准确。
5. **SHUX_COMPILE_STD_USE_C=1 环境变量**：必须设置，强制使用 shux-c 编译 `std/*.o`，确保与测试程序同源。
