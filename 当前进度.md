## §0.1.1 第 3 步 `cc -c` 类型前缀冲突：根因分析与修复

### 根因（上帝视角）

`codegen_gen.linux.x86_64.c`（pinned seed）与 `codegen.x`（.x 源码）严重失同步。seed 缺少以下 .x 源码已有的关键逻辑：

1. **emit_type 优先级反转**：seed 中 `struct_prefix`（当前 dep 模块前缀）优先于 `dep_prefix`（定义模块前缀），导致 backend 模块的 `*Module` 被 emit 为 `struct backend_Module` 而非 `struct ast_Module`
2. **codegen_type_dep_struct_prefix_into 缺裸名提取**：seed 用 `name_len` 而非 `bare_len` 比对 struct_layout 名称，限定名（如 `ast.Module`）匹配失败
3. **缺少 codegen_emit_dep_struct_forward_declarations 调用**：seed 中 `i==0` 块未调用此函数，导致 C struct tag 局部作用域 → conflicting types
4. **缺少 dep_enum_prefix 逻辑**：dep 模块 enum 类型（如 `ExprKind`）被 emit 为 `struct backend_ExprKind` 而非 `enum ast_ExprKind`
5. **缺少 u16/i16/i8 映射**：窄整型 TYPE_NAMED 未映射到 `uint16_t` / `int16_t` / `int8_t`
6. **缺少限定名兜底**：dep_prefix 查找失败时，未从限定名前缀推导 C 前缀

### 已同步的修复（codegen_gen.linux.x86_64.c）

| # | 修复项 | 对应 .x 源码位置 |
|---|--------|-----------------|
| 1 | emit_type 优先级：dep_prefix > struct_prefix > current_module > ast_ | codegen.x L2325-2348 |
| 2 | codegen_type_dep_struct_prefix_into 裸名提取（bare_off/bare_len） | codegen.x L2398-2410 |
| 3 | emit_type 限定名兜底（dep_prefix_len==0 且 name 含 `.`） | codegen.x L2286-2310 |
| 4 | emit_type 裸名发射（取最后一个 `.` 之后的部分） | codegen.x L2349-2358 |
| 5 | codegen_emit_dep_struct_forward_declarations 函数 + 调用 | codegen.x L3146-3172, L7450 |
| 6 | codegen_emit_module_struct_forward_declarations 函数 | codegen.x L2933-2968 |
| 7 | dep_enum_prefix 逻辑 + codegen_type_dep_enum_prefix_into 函数 | codegen.x L2249-2276, L2750-2812 |
| 8 | u16/i16/i8 TYPE_NAMED 映射 | codegen.x L2228-2245 |

### macOS 验证结果

- **原始 20 个 `conflicting types` error → 0** ✅（优先级反转 + forward declarations 生效）
- 新出现 20 个 error，均为 macOS pipeline 行为差异：
  - dep 模块 `num_funcs > 0`（macOS）vs `num_funcs == 0`（Ubuntu -E 单文件模式）
  - 导致 dep 模块函数在 enum 定义之前 emit → `incomplete type`
  - `backend_enc_dispatch` 模块未在 ctx dep 列表中 → extern 缺失
- **这些 error 在 Ubuntu 上不会出现**（因 dep 模块 `num_funcs == 0`，仅 entry 模块 codegen_x_ast 执行）

## 关键约束

- macOS 开发 → git push → `ssh ubuntu-server` → Linux 真机验证
- Linux x86_64 为金标准；macOS arm64 ASM 后端 stub，无法端到端验证 `invoke_cc`
- `shux -E` 只能对 **0 imports** 的 .x 文件运行（`asm_seed_full.x` 等通过 import 合并 dep 模块）
- `codegen_gen.c` / `parser_gen.c` / `pipeline_gen.c` 等生成文件受 pinned 机制保护，修改 `.x` 后需手工同步对应 `_gen.c`（`-E-extern` 模式 G-02a 后不可用）
- 修改 `pipeline_glue.c` 或 `ast_pool.c` 后需在 Ubuntu 重建 `pipeline_x.o`
- 每次测试调试前必须清理所有 `.o` 文件并重新编译
- 修复必须遵循**根源治理**：Bug 回溯到 IR 发射器 / 前端 typeck / 内存分配器，禁止热路径打补丁
