## 已修复问题（本会话）

### 1. import binding call typeck XT001 ✅

**commit**: `179071ac` + `0e26f19e`

**根因**: seed `typeck_gen.linux.x86_64.c` 的 `typeck_check_expr_method_call` 是旧的内联实现，只检查参数不解析 callee。`heap_ops.heap_mem_set_c(...)` 被解析为 EXPR_METHOD_CALL (kind=49)，seed 不委托 C glue。

**修复**: patch 脚本增加 `METHOD_CALL_BODY` 模板，将 seed 的 `typeck_check_expr_method_call` 替换为委托 `pipeline_typeck_check_expr_method_call_c`。

### 2. init_globals 重复定义 ✅

**commit**: `c6002ada` + `58d93a69`（seed 同步）

**根因**: `codegen_x_ast` 为每个有非 const 顶层 let 的模块 emit `init_globals`。dep 模块各自 emit 导致同一 C 文件中多个 `init_globals` 定义。

**修复**: dep 模块（dep_index >= 0）跳过 `init_globals` emission。入口模块（dep_index < 0）的 `init_globals` 合并 dep 模块的非 const 顶层 let 初始化。入口模块自身 0 个顶层 let 但 dep 有非 const let 时，在 `if (num_top_level_lets > 0)` 块外新增 init_globals emission。

### 3. `String` 裸类型名 ✅

**commit**: `0bf8d886` + `099fe14c`（seed 同步）

**根因**: codegen 对返回 String 的 call 表达式做 let 声明时，emit 裸 `String` 而非 `struct String`，导致 C 编译报 `unknown type name 'String'`。

**修复**: 将 `str_ty` 字节数组从 `"String"` (6 字节) 改为 `"struct String"` (13 字节)。

### 4. PREFER_X_O Makefile 规则无 seed 回退 ✅

**commit**: `83f0b33a`

**根因**: `if [ -x ./shux-c ]; then ... elif [ -f seed ]; then` 结构中，shux-c 存在但 -E 失败时不回退到 seed。

**修复**: 改为 `if [ -x ./shux-c ] && { ... }; then :; elif [ -f seed ]; then`，任何 -E 失败自动走 seed 回退。

### 5. `shux_compile_std_x.sh` 使用 ASM backend 失败 ✅

**commit**: `d2f13781`

**根因**: `shux-c -o` 使用 ASM backend，对含指针运算/数组的 .x 文件失败（`asm_codegen_elf_o failed`）。

**修复**: `shux-c` 分支改为 `-E + cc -c`（C backend），与 shux_asm 回退路径一致。

### 6. `std/string/string.o` ld -r 重复定义 ✅

**commit**: `0bf8d886`

**根因**: `ld -r` 合并 `.core` + `.fast` 时两者都定义 `shux_string_*_c` 函数。

**修复**: 添加 `$(LD_R_MULTIDEF_FLAGS)` 到 `ld -r` 命令。

### 7. `std/net/workers.x` 参数名冲突 ✅

**commit**: `8865e2b8`

**根因**: `net_worker_arg_slot_ptr_c(args: *NetWorkerArg, ...)` 的参数名 `args` 与 `net_run_accept_workers_c_real_c` 中的局部变量 `let args: NetWorkerArg[64]` 同名，typeck 将参数解析为数组类型 `[64]NetWorkerArg` 而非指针类型 `*NetWorkerArg`。

**修复**: 重命名参数 `args` → `base`。

### 8. parser `unsafe { ... }` 块后跟赋值导致函数体截断 + typeck weak seed 缺少 unsafe 检查 ✅

**commit**: `e1c473f9`

**根因（parser 侧）**: `parse_block_into` 和 `parse_one_function_impl` 的 unsafe handler 在 `parse_block_into` 返回后多调了一次 `lexer_next_into` 并设 `stmt_tok_ready=true`，而 `while`/`loop`/`for`/`if` handler 都不这么做。这导致 unsafe 块后跟赋值语句时函数体被截断，后续 let 被误识别为 top-level let，codegen emit 为 `static` 全局变量 + `init_globals`。

**根因（typeck 侧）**: 产品构建链接 `build_asm/pipeline_glue_strict_minimal.o`（从 seed 编译），其 weak 版 `pipeline_typeck_check_block_one_region_c` 不检查 `pipeline_block_region_is_unsafe`、不 push unsafe depth。新 parser 正确产生 unsafe region 后，weak seed 跳过 depth push 导致 `extern call requires unsafe block` 回归。

**修复**:
- `parser.x`: 两处 unsafe handler 移除 `lexer_next_into`，改 `stmt_tok_ready=false`（与 while/loop/for 对齐）
- `parser_gen.c`: 同步 seed C；补全 `parse_one_function_impl` unsafe handler（seed 之前缺失，走 expr_stmt GCC statement-expr 回退）
- `pipeline_glue_strict_minimal.from_x.c`: weak `pipeline_typeck_check_block_one_region_c` 添加 `pipeline_block_region_is_unsafe` 检查 + push/pop unsafe depth，与 `pipeline_glue.c` 强版本对齐

**验证**: macOS + Ubuntu 均通过 `test_w22.x`、`extern_putchar.x`、`std/net/workers.x`；Ubuntu `run-all-bstrict.sh` W3 best-effort `ok=21 skip=102 fail=0`。
