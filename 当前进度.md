# Shux 当前进度（2026-07-11）

## ✅ P0 已修：`shux -E` 在不完整代码上内存爆炸/Hang

### 问题现象

对包含 `if` 语句且**缺少闭合 `}`** 的 .x 文件执行 `shux -E`，进程会无限循环解析导致内存爆炸（macOS 下被 `alarm` 杀掉，RC=142）。期望行为：**报语法错误并退出（RC=1）**，而非 hang。

### 根因分析

两层缺陷叠加导致 hang：

**Fix 1（block parser loop 加 TOKEN_EOF 检查）**：`parse_block_into` 的 `while (1 == 1)` 循环只检查 `TOKEN_RBRACE` 跳出，未检查 `TOKEN_EOF`。

**Fix 2（核心根因：删除 stale realign + 加 TOKEN_EOF break 检查）**：`parser_gen.c` 中 `parse_one_function_impl` 的 if 分支有一个 `parser_realign_lex_after_if_stmt_onefunc_glue(&lex, source)` 调用，该调用在 `.x` 源码中**已被删除**（parser.x line 3833 注释："parse_if_stmt_into() 已返回下一 stmt 首 token；此处勿再二次 realign"），但 `parser_gen.c` 中仍残留。

`parser_asm_realign_lex_after_if_arm_c` 会从当前位置向后扫描 128 字节寻找 `if` 关键字。当 if 块 `}` 后跟 `;`、`// comment`、`let` 等非语句关键字时，回扫命中**当前 if 本身**，将 `lex` 重置回 if 位置 → 循环重复解析同一 if → 无限循环。

调试确认：循环卡在 `r.tok.kind=4 (TOKEN_IF)`、`lex.pos=22`（if 关键字位置），lex 不前进。

### 关键约束（来自 project_memory）

- `shux -E` 只能对 **0 imports** 的 .x 文件运行，否则会因递归依赖图加载导致内存爆炸
- `shux -E` 会丢弃含 `&&`/`||` 操作符的函数体；控制流需扁平化为 helper 函数
- `shux -E` 在 32-bit 平台会跳过含 `u64` 类型的函数；Ubuntu x86_64 上应使用 `usize`
- 修复必须遵循**根源治理**原则：禁止 if-else 兜底/硬编码/包裹临时逻辑的"狗屎补丁"
