## 一、语言特性感知缺口（Shux 差异化）

目前能力偏「通用系统语言编辑器」，对 **region 借用域**、**arena 分配域** 几乎没有视觉与导航支持。语言测试样例已存在，例如：

- `tests/typeck/slice_lifetime/region_block_escape.x` — `region ra { }`、`i32[]<rb>`
- `tests/mem/scope_allocator.x` — `with_arena(4096) { }`

### 1.1 region / lifetime / arena 语法高亮 — **已完成**

**痛点**：域标签、region 块与普通过程变量同色，借用冲突难以肉眼排查。

**已完成**：`grammars/x.tmLanguage.json` 已全量重写，实现以下分色：

| 语法 | TextMate scope | 实现状态 |
|------|----------------|----------|
| `region` | `keyword.control.region` | ✅ |
| `with_arena` | `keyword.control.arena` | ✅ |
| `i32[]<rb>` 中 `<rb>` | `storage.modifier.lifetime` | ✅（新增 `#lifetimes` 规则） |
| `export` / `packed` / `soa` / `align` | `storage.modifier` | ✅ |
| `type` / `run` / `spawn` | `storage.type` / `keyword.control` | ✅ |
| `extern "C"` / `extern "X"` | `keyword.other.abi` | ✅（新增 `#declarations` 规则） |
| `#[no_mangle]` / `#[cfg(...)]` / `#[repr(C)]` / `#[soa]` / `#[alloc]` | `meta.attribute.shux` | ✅（新增 `#attributes` 规则） |
| `allow(padding)` | `meta.attribute.shux` | ✅ |
| `i8` / `i16` / `u16` | `support.type.primitive` | ✅（`#types` 正则已补） |

**同步修复**：

1. ✅ `#comments` 中 `#.*$` → `#(?!\[).*$`（负 lookahead，排除 `#[` 开头的属性）
2. ✅ `language-configuration.json` 的 `unIndentedLinePattern` 中 `^\\s*#` → `^\\s*#(?!\\[)`
3. ✅ `language-configuration.json` 的 `folding.markers.start` 补充 `unsafe` 块

**验收**：打开 `region_block_escape.x`，`region` / `<rb>` 与 `let` 变量明显可分色；`#[no_mangle]` 不再被当注释灰化；`extern "C"` 中的 `"C"` 有独立着色。

### 1.2 大纲 / 面包屑中的 region·arena 边界 — **部分完成（P2）**

**痛点**：函数级大纲无法看到「这段代码在哪个借用域里」。

**现状**：

- ✅ **短期（扩展）**：`folding.ts` 已将 `region {` / `with_arena(...) {` / `unsafe {` 作为可折叠区域（标 `FoldingRangeKind.Region`）
- ❌ **中期（LSP）**：`documentSymbol` 仍只枚举顶层 func/struct/enum，**无 region/with_arena 子符号**，且 range 为占位 `(0,0)` — **阻塞于 LSP 服务端 `lsp_build_document_symbol_response` 改造，后期实现**
- ❌ **长期（LSP + typeck）**：悬停 region 块显示域标签、逃逸诊断关联符号 — **阻塞于 typeck region_label 元数据进 AST，后期实现**

**验收**：大纲树中出现 `region ra`、`with_arena` 节点，点击跳转到块首行（非 `(0,0)`）。

### 1.3 Semantic Tokens 扩展 region — **未做（P2，阻塞于 typeck）**

**现状**：`lsp_diag.x` → `lsp_collect_semantic_tokens` 覆盖 EXPR_VAR / CALL / STRUCT 等，未标记域标签；`initialize` legend `tokenModifiers` 仅含标准 10 项（declaration/definition/readonly/static/deprecated/abstract/async/modification/documentation/defaultLibrary），无 region/lifetime。

## 二、Inlay Hints — **完全缺失（P3，依赖 typeck）**

**痛点**：省略类型注解的极简写法（`let x = foo()`）长期阅读困难；函数调用实参缺少参数名提示。

**现状**：

- LSP `initialize` **无** `inlayHintProvider`
- 扩展 **无** `shux.features.inlayHints` 配置项

**目标体验**：

```x
// 用户编写
let config = init_config();
foo(a, b);

// Inlay Hints 开启后（灰色内联，非磁盘内容）
let config: Config = init_config();
foo(a: alpha, b: beta);
```

**实现路径**：

1. **LSP**：`textDocument/inlayHint` — typeck 推断类型、调用实参 ↔ 形参名映射
2. **扩展**：`shux.features.inlayHints` + 继承 VS Code `editor.inlayHints.enabled`
3. **README**：与 TypeScript / Rust Analyzer 同级说明

**阻塞**：typeck 局部推断 API 稳定性、`bootstrap_shuxc -E` 自举链路 — **后期实现，不纳入当前 Phase 0-1**。
