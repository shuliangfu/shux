## 0. 现状审计（2025-06 对照代码）

### 0.1 架构：扩展内置 vs LSP

| 能力 | 提供方 | 说明 |
|------|--------|------|
| 语法高亮、片段、括号配对、注释 | **扩展** TextMate + `language-configuration.json` | 不依赖 `shux` |
| 实时缩进（OnType）、语义折叠、CodeLens、状态栏、import 链接、构建任务 | **扩展** TypeScript Provider | 不依赖 LSP |
| 诊断、定义、引用、悬停、补全、签名、格式化、大纲、重命名、Semantic Tokens | **LSP** `shux --lsp` | 需 Open Folder + 可执行 `compiler/shux` |

`extension.ts` 注释已明确：**诊断 / 跳转 / 补全 / 悬停 / 格式化 / 大纲仅走 LSP**，不注册本地 LSP 回退 Provider。

### 0.2 已具备且基本可用

- `Ctrl+Shift+B` → `shux build`（`tasks.ts`，**未**硬编码 `make`）
- Pull 诊断（`textDocument/diagnostic`）
- 保存 / 粘贴时格式化（`[sx]` 默认 `formatOnSave` / `formatOnPaste`）
- 状态栏 `fn · st · en`（`statusbar.ts` 本地正则，**不依赖自举**）
- LSP 已 advertise：`renameProvider`、`semanticTokensProvider`（服务端 `lsp_diag.c` / `lsp_diag.sx`）

### 0.3 文档过度承诺（README 需同步修正）

| README 描述 | 实际 |
|-------------|------|
| 大纲含 trait、import | LSP `documentSymbol` 仅 **function / struct / enum**；本地 `symbols.ts` 虽有 trait/import 逻辑但 **未注册** |

## 一、语言特性感知缺口（Shux 差异化）

目前能力偏「通用系统语言编辑器」，对 **region 借用域**、**arena 分配域** 几乎没有视觉与导航支持。语言测试样例已存在，例如：

- `tests/typeck/slice_lifetime/region_block_escape.sx` — `region ra { }`、`i32[]<rb>`
- `tests/mem/scope_allocator.sx` — `with_arena(4096) { }`

### 1.1 region / lifetime / arena 语法高亮 — **缺失（P1）**

**痛点**：域标签、region 块与普通过程变量同色，借用冲突难以肉眼排查。

**现状**：`grammars/sx.tmLanguage.json` 的 `#keywords` 无 `region`、`with_arena`；无 `T[]<label>`、无 `region name` 标签分色。

**目标**（纯扩展侧，不阻塞自举）：

| 语法 | 建议 TextMate scope | 备注 |
|------|---------------------|------|
| `region` | `keyword.control.region` | 控制流级关键字 |
| `with_arena` | `keyword.control.arena` | 与 `defer` 同级感知 |
| `region ra` 中的 `ra` | `entity.name.tag.region` | 块顶域名 |
| `i32[]<rb>` 中 `<rb>` | `storage.modifier.lifetime` | 对齐 Rust lifetime 主题色 |
| `linear`（若源码出现） | `storage.modifier.linear` | 与 region 并列的安全修饰 |

**验收**：打开 `region_block_escape.sx`，`region` / `ra` / `<rb>` 与 `let` 变量明显可分色。

### 1.2 大纲 / 面包屑中的 region·arena 边界 — **缺失（P2）**

**痛点**：函数级大纲无法看到「这段代码在哪个借用域里」。

**现状**：

- 配置项 `shux.features.symbols` 存在，但仅影响（未接线的）本地 Provider；实际大纲来自 LSP
- LSP `lsp_build_document_symbol_response` 只枚举顶层 func/struct/enum，**无 region 块**

**目标**：

1. **短期（扩展）**：在 `folding.ts` 将 `region {` / `with_arena(...) {` 作为可折叠区域（与高亮同步）
2. **中期（LSP）**：`documentSymbol` 增加 `region` / `with_arena` 子符号，填充真实行列
3. **长期（LSP + typeck）**：悬停 region 块显示域标签、逃逸诊断关联符号

**验收**：大纲树中出现 `region ra`、`with_arena` 节点，点击跳转到块首行。

### 1.3 Semantic Tokens 扩展 region — **部分具备（P2）**

**现状**：`lsp_diag.sx` → `lsp_collect_semantic_tokens` 覆盖 EXPR_VAR / CALL / STRUCT 等，未标记域标签。

**目标**：在 typeck 已有 `region_label` 的 AST 节点上增量产出 token type `modifier` 或自定义 legend 项。

**扩展侧**：README 补充 Semantic Highlighting 说明；确认 `editor.semanticHighlighting.enabled` 对 `[sx]` 默认开启。

### 1.4 `.ai` 文件 — **未纳入（P3，待产品定稿）**

分析稿曾写 `.ai`；扩展 `contributes.languages` 仅 `.sx`。若 AI 子集与 `.sx` 语法一致，可复用同一 grammar；否则单独立项。

## 二、Inlay Hints — **完全缺失（P3，依赖 typeck）**

**痛点**：省略类型注解的极简写法（`let x = foo()`）长期阅读困难；函数调用实参缺少参数名提示。

**现状**：

- LSP `initialize` **无** `inlayHintProvider`
- 扩展 **无** `shux.features.inlayHints`

**目标体验**：

```sx
// 用户编写
let config = init_config();
foo(a, b);

// Inlay Hints 开启后（灰色内联，非磁盘内容）
let config: Config = init_config();
foo(a: alpha, b: beta);
```

**实现路径**：

1. **LSP**：`textDocument/inlayHint` — typeck 推断类型、调用实参 ↔ 形参名映射
2. **扩展**：`shux.features.inlayHints` + 继承 VS Code `editor.inlayHints.enabled`
3. **README**：与 TypeScript / Rust Analyzer 同级说明

**阻塞**：typeck 局部推断稳定性、`bootstrap_shuxc -E` 自举链路；**不纳入 P0/P1**。

## 三、工程与体验优化

### 3.1 README「快速开始」与 Makefile — **半对半错**

| 说法 | 核实 |
|------|------|
| 构建任务硬编码 `make` | **不成立** — `tasks.ts` 已是 `shux build` / `run` / `check` |
| README 引导 `make -C compiler bootstrap-driver-seed` | **成立** — 目的是**首次获得支持 `--lsp` 的 `shux` 二进制**（鸡生蛋），不是跑构建任务 |

**文档优化方向**（改 README，不改任务逻辑）：

```text
若已有 compiler/shux 且支持 --lsp → 跳过本步
否则任选其一：
  make -C compiler bootstrap-driver-seed   # 当前主流
  ./build.sh                               # 与仓库 build.sx 路线对齐（G-05）
```

`tests/lib/progress-run.sh` 为测试辅助脚本，**不宜**作为扩展默认任务。

### 3.2 删除 `formatOnBlur` — **建议采纳（P0）**

**现状**：`shux.features.formatOnBlur` 默认 `false`；`extension.ts` 在 `onDidChangeActiveTextEditor` 上对离开的 `.sx` 调 LSP 格式化。

**风险**：多标签切换、Cursor 多文件并发写入时，blur 格式化与缓冲区/磁盘写入可能竞态，存在覆盖风险。

**建议**：

- 删除 `formatOnBlur` 配置与 `extension.ts` 逻辑
- README 删除相关段落；统一依赖 `editor.formatOnSave` / `formatOnPaste`（`[sx]` 已默认开启）
- `shux.features.formatOnType` **保留** — 仅缩进对齐，非完整 LSP format，风险低

### 3.3 配置项精简与接线 — **建议采纳（P1）**

当前 `package.json` 约 **27** 项。问题不仅是「偏多」，而是部分**名不副实**：

| 配置 | 问题 | 建议 |
|------|------|------|
| `shux.format.insertFinalNewline` | 未读取 | **删除**；改 `[sx]` 默认或文档指向 `files.insertFinalNewline` |
| `shux.format.trimTrailingWhitespace` | 未读取 | **删除** → `files.trimTrailingWhitespace` |
| `shux.format.trimFinalNewlines` | 未读取 | **删除** → `files.trimFinalNewlines` |
| `shux.features.completion` | 未接 LSP middleware | **删除** 或实现 `middleware.provideCompletionItems` 短路 |
| `shux.features.hover` | 同上 | 同上 |
| `shux.features.signatureHelp` | 同上 | 同上 |
| `shux.features.symbols` | 同上 | 同上 |
| `shux.format.maxLineLength` | 需核实 LSP formatter 是否读取 | 若无实现则删除或接到 `shux --lsp` 参数 |

精简后目标：**约 18–20 项**，每项必有代码路径或明确文档指向 VS Code 全局设置。

### 3.4 死代码处理 — **P1**

二选一（推荐 A）：

- **A. 删除** `completion.ts` / `hover.ts` / `signature.ts` / `definitions.ts` / `references.ts` / `symbols.ts`，减轻维护与文档误导
- **B. 保留为 LSP 离线回退**：`extension.ts` 在 LSP 未启动时注册本地 Provider，并让 `shux.features.*` 真正生效

## 四、与编译器主线的关系

| 工作项 | 是否阻塞自举 | 说明 |
|--------|--------------|------|
| region 语法高亮 | 否 | 仅 TextMate |
| 删 formatOnBlur / 清死配置 | 否 | 扩展侧 |
| LSP 大纲真实 range | 部分 | 可先用 C parser 行列，不依赖 `.sx` typeck |
| Inlay Hints | **是** | 依赖 typeck 推断 API 稳定 |
| region semantic token | **是** | 依赖 typeck region 元数据进 AST |
| Code Action（region 逃逸一键修） | **是** | 依赖诊断 + 修复策略 |

**纠正旧说法**：状态栏 `fn · st · en` **现在即可用**，与 `bootstrap_shuxc` 无关；等自举的是 **LSP 诊断、补全、跳转、悬停、LSP 格式化**。
