# SHUX 编译器架构分析文档

## 一、核心二进制文件职责

### 1.1 shux-c — 冷启动种子编译器

**构建方式**：
- 默认：`bootstrap_shuxc`（预编译跨平台种子）
- `SHUX_LEGACY_C_FRONTEND=1`：完整 C 前端编译

**职责**：
- 生成 `pipeline_gen.c`、`driver_gen.c` 等中间 C 代码
- 供 `-E` 模式生成 C 代码，不支持 `-backend` 参数
- 冷启动时唯一可用的编译器

**关键约束**：
- 默认无完整 C 前端（lexer/parser/typeck/codegen 为 stubs）
- `SHUX_LEGACY_C_FRONTEND=1` 时才链接完整前端

### 1.2 shux — 主编译器入口

**构建方式**：`OBJS_CORE` 链接（无完整 C 前端）

**职责**：
- 主编译器入口，依赖 .x pipeline
- 支持 `-backend asm` 参数

**关键约束**：
- 依赖 shux-x 的 .x pipeline 生成代码

### 1.3 shux-x — 二次构建编译器

**构建方式**：带 .x pipeline

**职责**：
- 用 .x codegen 生成更完善的 driver_gen
- 供二次构建使用

### 1.4 shux_asm — ASM 后端编译器

**构建方式**：`build_shux_asm.sh`

**职责**：
- 编译 .x → .o（生成汇编）
- 支持 `-backend asm`

**关键约束**：
- 在 Darwin 上 `asm_backend_partial.o` 为 fallback 弱桩，真实指令发射器未被编译入
- 导致 `-backend asm` 产出 code_len=0

## 二、种子文件职责

### 2.1 bootstrap_shuxc.<os>.<arch>

**职责**：预编译的跨平台种子，供冷启动使用

**选择逻辑**：`select_bootstrap_shuxc.sh` 按 OS/架构自动选择

### 2.2 asm_backend_partial.<os>.<arch>.o

**职责**：ASM 后端部分实现

**关键符号**：
- 强符号：`backend_asm_codegen_ast_seed_mega`（seed_mega）
- 弱符号：`backend_emit_block_body`、`backend_emit_expr` 等（返回 0）

**问题**：Darwin 上仅 14 个弱符号，真实 ARM64 指令发射器缺失

### 2.3 *_gen.linux.x86_64.c

**职责**：Linux x86_64 平台预生成的 C 代码（pipeline/driver/lsp/lexer 等）

## 三、脚本职责

### 3.1 select_bootstrap_shuxc.sh

**职责**：按 OS/架构选择合适的 bootstrap_shuxc 种子

**调用时机**：`make bootstrap_shuxc`、冷启动

**选择顺序**：
1. 当前 ./bootstrap_shuxc 若可执行
2. ./seeds/bootstrap_shuxc.<os>.<arch>
3. 可执行的 ./shux-c / ./shux / ./shux_asm

### 3.2 build_shux_asm.sh

**职责**：构建 `shux_asm`（带 ASM 后端的编译器）

**调用时机**：手动调用、自举流程

### 3.3 build_seed_asm_host.sh

**职责**：构建 ASM 后端种子（`asm_backend_partial.o`）

**调用时机**：自举阶段、seed 刷新

**关键逻辑**：
- Darwin 默认跳过 `asm_seed_full.x -E`（需 `SHUX_DARWIN_ALLOW_ASM_SEED_FULL_E=1`）
- 从 `seeds/asm_backend_partial.${os}.${arch}.o` 复制

### 3.4 shux_compile_std_x.sh ⚠️（问题核心）

**职责**：编译 `std/*.x` 为 `.o`

**调用时机**：`make std-objs`、各 std 模块规则

**参数**：`shux_compile_std_x.sh <shux-bin|auto> <x-path> <out.o>`

**auto 选择逻辑**：
```
shux_asm → shux → shux-c
```

**当前问题**：两个分支都被改为 `-E + cc -c`，导致符号不兼容

**原始设计**：
- `shux-c)`：`exec "$shux_bin" -L .. "$x_path" -o "$out_o"`（直接编译）
- `*)`：`exec env SHUX_ASM_WPO_DCE=0 "$shux_bin" -backend asm -L .. "$x_path" -o "$out_o"`

**当前破坏**：
- `shux-c)`：改成 `-E + cc -c`，生成的 .o 符号不兼容
- `*)`：改成 `-backend asm` 失败时回退到 `-E + cc -c`

## 四、当前问题根因分析

### 4.1 问题链

```
shux_compile_std_x.sh 修改
       ↓
shux-c 分支从 -o 改成 -E + cc -c
       ↓
std/*.o 符号不兼容（mangling、include 路径差异）
       ↓
大量 std 模块测试失败
       ↓
失败数从 29 → 80 → 27（手动构建 runtime_*.o 后）
```

### 4.2 runtime_*.o 缺失问题

**设计预期**：
```
shux_asm_ld_append_std_objs
       ↓
link_abi_asm_ld_push_glue_after_std
       ↓
shux_ensure_runtime_*_o（自动 cc -c 生成）
```

**实际问题**：
- `make all` 不构建 `std-objs`（含 runtime_*.o）
- 自动构建函数可能因 argv0 解析失败而未触发

**修复状态**：手动构建所有 runtime_*.o 后，失败数从 80 降至 27

### 4.3 27 个剩余失败分类

| 类别 | 失败脚本 | 原因推测 |
|------|----------|----------|
| typeck | run-typeck.sh, run-check.sh, run-types-gate.sh | `?` propagation 等远期特性未实现 |
| stdlib-import | run-stdlib-import.sh | `core_types_placeholder` 重复符号 |
| 链接错误 | run-channel.sh, run-math.sh, run-encoding.sh, run-crypto.sh, run-stdtest.sh, run-backtrace.sh, run-hash.sh, run-ffi.sh, run-json.sh, run-csv.sh, run-unicode.sh, run-compress.sh, run-thread.sh, run-random.sh, run-http.sh, run-tar.sh, run-time.sh, run-io.sh, run-slice.sh, run-vector.sh, run-fmt-std.sh, run-net.sh, run-io-driver.sh | std .o 符号不兼容（shux-c -E + cc -c 导致） |
| 解析错误 | run-net.sh | `parse produced no functions` |

## 五、修复策略

### 5.1 策略A：恢复 shux_compile_std_x.sh（推荐）

**目标**：将 `shux-c)` 分支恢复为原始的直接 `-o` 编译

**步骤**：
1. 恢复 `shux-c)` 分支为 `exec "$shux_bin" -L .. "$x_path" -o "$out_o"`
2. 保留 `*)` 分支的 `-backend asm` → `shux-c` 回退（仅在 `-backend asm` 失败时触发）
3. 删除所有 std/*.o 和 runtime_*.o
4. 重新构建：`make -C compiler all` + 手动构建 runtime_*.o
5. `CI=1 ./tests/run-all.sh` 重测

**预期**：失败数应回到之前的 29 基线

### 5.2 策略B：修复 shux-c -o 在 macOS 上的冲突

如果 `shux-c -o` 在 macOS 上确实有 ntohs/htonl 宏冲突，需要在 shux-c 的 codegen 层面修复，而非在脚本层绕过。

**步骤**：
1. 定位 codegen 中生成 ntohs/htonl extern 声明的位置
2. 在 codegen 层面添加 `#undef` 处理
3. 测试 `shux-c -o` 是否正常工作

### 5.3 策略C：确保 make all 构建 runtime_*.o

修改 Makefile 的 `all` 目标，确保包含所有 runtime_*.o。

**风险**：可能增加构建时间，且 runtime_*.o 应按需构建（on-demand）

### 5.4 策略D：修复自动构建函数

排查 `shux_ensure_runtime_dynlib_os_o` 等自动构建函数为何未触发。

**步骤**：
1. 添加 debug 日志到自动构建函数
2. 测试 hello.x，查看自动构建是否触发
3. 修复 argv0 解析问题
