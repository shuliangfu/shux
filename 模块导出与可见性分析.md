## 0. 一句话结论

| 决策 | 内容 |
|------|------|
| **用什么** | 关键字 **`export`** 标记「本模块对外 API」，**不用** `public` / `private` 成对关键字 |
| **默认** | 目标态：**不写 `export` = 模块私有**（不可被其他 `.x` 的 `import` 点到） |
| **能导出什么** | 函数、方法、类型（struct/enum/…）、顶层常量；规划扩展 trait / 泛型 / re-export |
| **怎么上线** | **两步走**：① 语法与 AST 先认 `export`，默认仍全开兼容；② 再切「仅 export 可跨模块」 |
| **为啥必须做** | 没有「导出表」，就无法定义私有符号 → `shux check` 无法可靠报 unused 函数 → 全效化 roots 只能猜 |
| **和全效化关系** | 全效化 v0.3 反对 must_use/pure 等装饰属性；**`export` 是模块系统缺环**，属正当的一个关键字，不是标注爆炸 |

## 1. 问题背景

### 1.1 现状

| 项 | 现状 |
|----|------|
| 模块模型 | 每个 `.x` 文件 = 一个模块 |
| 跨模块使用 | `const m = import("path");` → `m.symbol()` / `m.Type` |
| 可见性关键字 | **无** `export` / `public` / `private`（见 `docs/01-关键字.md`） |
| 默认可见性 | 顶层 `function` / 类型 / 常量等 **实质均可被 import 使用**（无导出表） |
| 已有「export」字样 | `#[export_name("…")]`：改 **链接符号名**（C/目标文件），**不是**模块 API 可见性 |
| 防 DCE 属性 | `#[used]`、`#[no_mangle]`：链接/内核场景，与模块可见性正交 |

文档（`docs/05-函数与模块.md`）描述 import 可访问「函数、常量、跨模块 struct/enum」，但未定义「哪些名字属于模块对外表面」。

### 1.2 由此卡住的能力

| 能力 | 为何卡住 |
|------|----------|
| **unused 私有函数** | 无法区分「库 API」与「实现细节」→ 不敢默认报 unused function |
| **`shux check` 全效化** | 绑定级 unused（L6）可做；符号级 unused 依赖「非导出 + 不可达」 |
| **库模式 DCE roots** | 现在要么几乎全保留，要么从 main 乱裁；需要 **导出表 = roots** |
| **API 边界文档化** | 阅读源码无法一眼看出「对外承诺」与「内部 helper」 |

### 1.3 设计目标

1. **一个关键字**表达模块导出，与 `import` 语义对称。  
2. **默认可推断私有**：不写即内部，减少无意泄漏 API。  
3. **覆盖 import 真实用到的名字**（函数 / 方法 / 类型 / 常量），不限于「方法」。  
4. **可迁移**：不一夜改爆 `std` / `compiler/src`。  
5. **与链接层概念正交**：不混淆 `#[export_name]` / `#[no_mangle]`。  
6. **不阻塞自举 E-track**：实现与收紧可并行、可 env 回退。

### 1.4 非目标

- 不引入 `public` / `private` / `protected` 访问级别体系。  
- 不引入 `#[must_use]` / `pure` / `#[allow(dead_code)]` 等全效化装饰语法（见全效化 v0.3）。  
- 不在本文落地 re-export 全部语法糖（可规划，实现后置）。  
- 不把「模块 export」自动等同于「动态库 `.so` 导出符号表」（后者可用 no_mangle / 链接脚本另管）。

## 2. 为什么选 `export` 而不是 `public`

| 维度 | `public function` | **`export function`（推荐）** |
|------|-------------------|------------------------------|
| 语义重心 | 类型/成员访问级别（OOP） | **模块边界：放进导出表给 import** |
| 与现有词 | 语言中无 public | 与 **`import` 成对**，心智统一 |
| 覆盖面 | 易被理解成「只修饰方法」 | 自然覆盖函数 / 类型 / 常量 / … |
| 默认叙述 | public vs private 成对 | **export vs 不写** 即可，无需 `private` 关键字 |
| 学习成本 | 多一套 OOP 词汇 | 模块语言常见模型（类 JS/TS export、部分系统语言 pub export） |

**决议（规划）**：只采用 **`export`**；不并行保留 `public`。

## 3. 核心语义模型

### 3.1 导出表（Export Set）

每个模块 \(M\) 有导出表 \(E(M)\)：

```text
E(M) = { 本模块中所有带 export 修饰、且完成声明的名字 }

跨模块名解析（strict）：
  给定 const m = import("M");
  m.name 合法  ⇔  name ∈ E(M)（并满足类型/重载解析规则）

本模块内：
  任意本模块声明的名字均可使用（无论是否 export）
```

### 3.2 目标态默认规则（第 2 步之后）

| 声明 | 本模块内 | 其他模块 `m.name` | 是否进入库 DCE 的 export-root 候选 |
|------|----------|-------------------|-------------------------------------|
| `function f` | ✅ | ❌ | 否（仅按可达性） |
| `export function f` | ✅ | ✅ | 是 |
| `export struct S` 等 | ✅ | ✅ | 类型可达 root 相关 |
| `main` | 入口 | 一般不经 import | 可执行 root（特殊） |
| 未 export 且不可达 | — | — | 可 unused 诊断 + 不 emit |

### 3.3 与「链接导出」正交

```text
┌─────────────────────────────────────────────────────────────┐
│  语言层：export / 导出表 E(M)                                 │
│  → 其他 .x 能否通过 import 使用该名字                          │
└─────────────────────────────────────────────────────────────┘
                              ≠
┌─────────────────────────────────────────────────────────────┐
│  链接层：#[no_mangle] / #[export_name] / 符号全局与否          │
│  → 目标文件里符号如何命名、C/asm/dlsym 能否找到                 │
└─────────────────────────────────────────────────────────────┘
```

| 组合（示意） | 含义 |
|--------------|------|
| `export function f` | .x API；符号名可按现有 mangling |
| `#[no_mangle] function f` | C 可见符号；是否同时允许 .x 跨模块用 → 见 §3.4 |
| `export` + `#[export_name("c_name")]` | .x 名 `f`，链接名 `c_name` |
| 仅私有 `function helper` | 仅本模块；codegen 可内部链接 / 可 DCE |

### 3.4 `#[no_mangle]` 与 export 的推荐策略

| 策略 | 内容 | 推荐 |
|------|------|------|
| **S1** | no_mangle **不**自动进入 \(E(M)\)；.x 跨模块仍要 `export` | **推荐**：两概念彻底分离 |
| **S2** | no_mangle 自动视为导出 | 易把 C 桩泄漏成 .x API，不推荐 |

内核/中断等仅被 asm 引用：继续 **`#[used]`**，不强迫写 export（除非也要给 .x 用）。

### 3.5 `main` 与 `extern`

| 声明 | 规则 |
|------|------|
| `function main` | **入口特殊**；不要求 `export main`；不进入「库 import API」叙事 |
| `extern function …` | **从外部引进声明**，不是本模块生产的实现；若要把 C 能力以 .x API 形式提供，写 `export function` 包装或在模块内 export 再声明策略（实现期定一种） |

## 4. 可以 `export` 什么

按「`import` 绑定后能否作为 `m.Name` / 方法调用出现」划分。

### 4.1 v1 必须支持（与现网 import 对齐）

| 种类 | 语法示意 | 外部用法 |
|------|----------|----------|
| **函数** | `export function print_str(p: *u8, n: i32): void { … }` | `io.print_str(...)` |
| **方法** | `impl T { export function len(self): i32 { … } }` | `x.len()`（在 `T` 可见前提下） |
| **结构体** | `export struct Buffer { … }` | `m.Buffer`、`m.Buffer { … }` |
| **枚举** | `export enum Color { Red, Green }` | `m.Color`；变体策略见 §4.4 |
| **顶层常量** | `export const MAX: i32 = 10;` | `m.MAX` |

### 4.2 v1 建议写入语义、实现可紧随

| 种类 | 说明 |
|------|------|
| **`struct packed`** | 与 struct 同一可见性模型 |
| **泛型函数 / 泛型类型** | `export function id<T>(…)` / `export struct Box<T>` |
| **函数重载组** | **同一名字的所有 overload 共享可见性**（全 export 或全私有），禁止「只导出其中一个 overload」造成解析歧义 |
| **`trait`** | `export trait Reader { … }`；未 export 的 trait 仅模块内可 impl/约束 |

### 4.3 后置（v1.1+，规划占位）

| 种类 | 说明 |
|------|------|
| **字段级 export** | `export struct S { export x: i32; y: i32 }` — 类型导出、字段封装 |
| **`export { … }` 块** | 连续导出糖 |
| **re-export** | 门面模块把子模块 API 抬到本模块表面（见 §4.5） |
| **类型别名** | 若语言增加 `type A = B`，则 `export type` |
| **整块 `export impl`** | 块内方法默认 export 的糖 |

### 4.4 枚举变体与结构体字段（v1 简单规则）

**推荐 v1 采用简单规则，避免字段级语法：**

| 对象 | v1 规则（推荐） |
|------|-----------------|
| `export enum E` | 类型名 ∈ \(E(M)\)；**变体名对外可见**（构造/match 可用） |
| 未 export 的 enum | 仅本模块 |
| `export struct S` | 类型名 ∈ \(E(M)\)；**字段对外部按现有规则可访问**（与今天行为接近） |
| 未 export 的 struct | 外部不能写 `m.S`；本模块内随意 |

更强封装（字段默认私有、不透明类型）列入后置，不挡 v1。

### 4.5 方法与类型的约束

| 情况 | 规则 |
|------|------|
| 类型未 export，方法 export | **无意义 / 应诊断**：外部拿不到类型则无法调用 |
| 类型 export，方法未 export | 外部不能调用该方法；模块内可以 |
| 仅模块内 helper 方法 | 不写 `export` |

### 4.6 re-export（规划，非 v1 必做）

门面模块常见需求：

```text
// 示意（语法未定稿）
export function foo(...): T { return sub.foo(...); }   // 薄包装，v1 即可用
// 未来糖：export { foo } from "pkg.sub"; 或 export const sub = import(...);
```

v1 可用 **薄包装函数/类型别名（若有）** 完成 re-export；专用语法后置。

### 4.7 明确不纳入 `export` 的

| 项 | 原因 |
|----|------|
| 块内 `let` / 局部 `const` | 非模块级名字 |
| `import` 绑定名本身 | 本地绑定；再导出需显式 re-export 机制 |
| 链接符号改名 | 用 `#[export_name]` |
| 预处理宏 / 条件编译符号 | 另一套机制 |

### 4.8 导出能力总表

| 可导出对象 | v1 | 后置 | 备注 |
|------------|----|------|------|
| 顶层 function | ✅ | | 核心 |
| impl 方法 | ✅ | | 核心 |
| struct / packed struct | ✅ | | 核心 |
| enum | ✅ | | 变体随类型 |
| 顶层 const | ✅ | | 核心 |
| trait | 语义 ✅ / 实现可跟 | | |
| 泛型声明 | ✅ 随种类 | | |
| 重载组 | ✅ 整组 | | |
| 字段 | | ✅ | |
| export 块糖 | | ✅ | |
| re-export 糖 | | ✅ | 薄包装 v1 可用 |
| type alias | | ✅ | 依赖语言是否有 type |
| main | 特殊，不要求 export | | |
| extern 声明 | 不作为「模块生产 export」默认路径 | | |

## 5. 语法形态（规划稿）

### 5.1 基本形式

```text
export function name(params): Ret { ... }
export const NAME: Type = init;
export struct Name { field: Type, ... }
export enum Name { A, B }
export trait Name { ... }

impl Type {
  export function method(self, ...): Ret { ... }
  function private_helper(self): void { ... }
}
```

### 5.2 与属性顺序（建议）

```text
#[no_mangle]
export function foo(): void { ... }

#[export_name("c_foo")]
export function foo(): void { ... }
```

属性在前、`export` 紧贴声明关键字；实现期与现有属性解析对齐即可。

### 5.3 关键字

- `export` 进入**保留关键字**表（更新 `docs/01-关键字.md`）。  
- 全仓扫描标识符冲突（预期极少）。

### 5.4 不采用的语法

| 不采用 | 原因 |
|--------|------|
| `public` / `pub` 并行 | 双权威、双文档 |
| 必须写 `private` | 默认即私有即可 |
| `export default` | 与绑定 import 模型不符 |
| 用属性 `#[export]` 代替关键字 | 与「模块表面」第一公民地位不符；且与 `export_name` 更易混 |

## 6. 两步迁移策略（必须）

仓库体量（`std` / `core` / `compiler/src` / `examples`）大，**不能**第 1 天默认私有。

### 6.1 第 1 步：语法落地，行为兼容

| 项 | 行为 |
|----|------|
| 解析 | 识别 `export` 修饰多种声明 |
| AST | `is_export` / `vis = Export | Default` |
| 名解析 | **compat：未 export 仍允许跨模块访问**（与今天一致） |
| 导出表 | 已可计算 \(E(M)\)，供统计与 warn |
| 测试 | parse / AST 往返；不破坏现有 import 用例 |
| 文档 | 标明「迁移期：建议对外 API 写 export」 |

第 1 步价值：

- 工具链与语法就绪；  
- 可跑「被跨模块引用但未标 export」清单；  
- 人工/脚本给真正 API 补 `export`，不改运行语义。

### 6.2 过渡档（环境变量，非语法）

| 模式 | 建议 env（名称实现期可统一） | 行为 |
|------|------------------------------|------|
| **compat** | `SHUX_VISIBILITY=compat`（第 1 步默认） | 未 export 仍可跨模块 |
| **warn** | `SHUX_VISIBILITY=warn` | 跨模块使用未 export 名字 → **warning** |
| **strict** | `SHUX_VISIBILITY=strict`（第 2 步默认） | 跨模块使用未 export → **error** |

### 6.3 第 2 步：默认私有

| 项 | 行为 |
|----|------|
| 名解析 | 仅 \(E(M)\) 可被 `m.name` 使用 |
| 诊断 | `name is private / not exported`（文案实现期定） |
| DCE / check | private + 不可达 → unused；库 roots ⊆ 导出表 ∪ 特殊 root |
| 文档 | 删除「默认全导出」；更新 05/01 与 cookbook |
| CI | 产品路径 `strict`；冷启动 seed 可仍 compat |

### 6.4 迁移工作流（工程）

```text
1. 落地 export 语法（compat）
2. 开 warn + 脚本：列出「跨模块边 → 未 export 被引用」
3. 按包补 export：先 std/core 对外 API，再 compiler 模块边界
4. 单包切 strict 试点 → 全仓 strict
5. 打开 shux check 的 unused private（依赖导出表）
```

### 6.5 与自举

| 点 | 策略 |
|----|------|
| 编译器 .x 互 import 极多 | 第 2 步前 compiler 必须补标或分模块试点 |
| seed / PREFER_X_O | 可见性是 .x 语义；seed C 不走 export 表；双权威审计仍独立（全效化文档） |
| 不阻塞 E-track | 第 1 步可并行；strict 可等主线窗口 |

## 7. 与 `shux check`、全效化、WPO/DCE

### 7.1 check 应检测什么（规划）

| 检测项 | 依赖 export？ | 说明 |
|--------|---------------|------|
| 未使用局部绑定 / import | 否 | L6；应默认 warn（全效化 P1） |
| 跨模块访问未 export | 是（strict） | 可见性错误 |
| 未使用 **私有** 函数/方法 | 是（需能识别私有） | L7：非 export 且不可达；**仅 warning**；LSP 同步波浪线 |
| 未使用 export 函数 | 否（库）/ 视入口（exe） | 库：export 是 API root，不因「本仓无调用」删；exe：可按 reach 裁但不一定报 unused |

### 7.2 roots 统一口径

```text
roots =
    { main / entry }
  ∪ E(M)                          // 模块导出表（库与跨模块）
  ∪ { #[used] 符号 }
  ∪ { #[no_mangle] 等链接保留 }   // 按 §3.4，链接保留 ≠ 自动进 E(M)
  ∪ 测试入口（test 模式）

reachable = WPO/BFS(roots)        // 复用 codegen_wpo_reach_* / compute_used
dead_private = 非 root 私有声明 − reachable
```

**禁止**第二套 call graph；export 只多一个 **root 标记来源**。

### 7.3 与全效化 v0.3 的衔接

| 全效化主张 | 本文 |
|------------|------|
| 不增 must_use/pure 等 | 遵守；只增 **export** 一处语言表面 |
| 默认诊断 + 默认 DCE | check unused 在 export 语义清晰后加强 |
| 验收看 unused_private | **private ≔ 未 export（strict 后）** |
| seed 白名单 | 独立 gate，不与 export 表混写 |

### 7.4 库模块 `-E` 禁用「整库当 dead」

现有约束：无 main 时不能按 exe 乱 DCE。  
引入 export 后：

```text
库模块 roots 至少包含 E(M)
→ 所有 export 函数/类型相关实现按引用保留
→ 未 export 的 helper 仍可裁、可 unused 报
```

这是相对现状的明确改进。

## 8. 实现影响面（规划，非排期承诺）

| 层 | 工作 |
|----|------|
| **lexer** | `export` 关键字 token |
| **parser** | 声明前可选 `export`；impl 内方法；多种声明形 |
| **AST** | 函数/类型/const/trait/方法 的 `is_export`（或统一 Vis 枚举） |
| **typeck** | 跨模块解析查导出表；compat/warn/strict；方法与类型约束 |
| **codegen** | 不强制改符号命名；可选 private → 内部联动；DCE roots 接导出表 |
| **WPO** | `codegen_wpo_reach` root 集合并入 export |
| **driver / check** | 诊断出口；env `SHUX_VISIBILITY` |
| **文档** | `01` 关键字、`05` 模块、cookbook、语法对照表 |
| **测试** | parse；compat 回归；warn/strict 用例；std 试点包 |
| **迁移脚本** | 基于 import 边建议补 export（可先 Python/shell） |

C 与 .x 双 pipeline 若仍并存：须 **同步**（项目一贯要求），避免仅一边认 export。

## 9. 风险、代价与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| 全仓补 `export` 工作量大 | 第 2 步延期 | warn + 脚本 + 分包 strict |
| 与 `#[export_name]` 概念混淆 | 误用、误文档 | 文档强制正交表；错误信息区分「模块未导出」vs「链接名」 |
| 只 export 函数、类型仍全开 | 边界仍糊 | v1 同步支持类型/const |
| 重载半导出 | 解析灾难 | 整组同可见性 |
| 误报 unused export | 库 API 被骂 unused | export ∈ roots，库模式不报「未被本仓调用」 |
| 自举互依赖 | compiler 切 strict 痛 | compiler 最后切或按目录试点 |
| 实现拖累 E-track | 主线阻塞 | 仅规划/compat 语法可并行；strict 可选窗口 |

---

## 10. 与其它方案对比（简表）

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| 继续默认全导出 | 零迁移 | 无 private、unused/DCE 漂 | 不可长期 |
| 仅文档约定「别用内部函数」 | 零实现 | 无强制、必漂移 | 否 |
| `public` 关键字 | 常见 | 偏 OOP；与 import 不对称 | 不采用 |
| **`export` 关键字** | 与 import 对称；可扩类型/const | 一词与 export_name 需区分 | **采用** |
| 仅属性 `#[export]` | 少关键字 | 第二公民；易混 export_name | 不采用 |
| 文件级 `export *` | 迁移快 | 失去细粒度；等于没 private | 不作默认；最多迁移糖 |

## 11. 分阶段路线图（规划）

| 阶段 | 交付 | 行为变化 | 与主线 |
|------|------|----------|--------|
| **P0 本文** | 根目录分析定稿 | 无 | — |
| **P1 语法 compat** | lexer/parser/AST + 测试；文档「建议标注」 | 无语义破坏 | 可并行自举 |
| **P2 导出表 + warn** | typeck 可算 \(E(M)\)；`SHUX_VISIBILITY=warn` | 仅警告 | 可并行 |
| **P3 迁移** | 脚本 + std/core/compiler 补 export | 仍 compat 可过 CI | 工程波次 |
| **P4 strict** | 默认仅 export 可跨模块 | **破坏性**，需 major 说明 | 选窗口 |
| **P5 check/DCE 接轨** | unused private；WPO roots=导出表 | 全效化加强 | 接 `代码全效化` P1–P2 |
| **P6 后置** | 字段 export、export 块、re-export 糖 | 体验 | 不挡 P4 |

**当前执行建议**：先接受本文决议；**不急着写代码**；实现窗口优先 P1 compat（低风险），strict 与自举里程碑错开。

## 17. 摘要（给评审 30 秒）

1. **加 `export`，不加 `public`。**  
2. **目标：不写就不导出**；import 只能用导出表。  
3. **v1 导出：函数、方法、类型、顶层常量**；字段/re-export 糖后置。  
4. **两步：先 compat 语法，再 strict。**  
5. **链接层属性保持正交。**  
6. **据此才能让 `shux check` 报 unused 私有符号、让 DCE roots 干净。**  
7. **现在：规划落地本文即可，实现可排期。**
