## 0. 开波前 60 秒决策

每次用户说「继续下一步」或你准备开新波时，先回答三个问题：

| # | 问题 | 若答是 → 走哪条轨 |
|---|------|-------------------|
| 1 | 多个 rest 被**同一能力缺口**卡住？ | **Track Cap**（先解锁，再批量 R） |
| 2 | 某 rest **能力已够**，产品 hybrid 仍有业务 C 体？ | **Track R / R2 真迁**（当前主线默认） |
| 3 | mega 仍 pin，想试替换 `*_x.o`？ | **Track M**（与 R **并行**，不互相阻塞） |

**禁止**作为「下一步」：

- 仅为 N+1 写/扩 `*_surface.from_x.c`（thin surface prove / **thin N++**）
- 产品已 PREFER_X_O、rest 未动，却宣称「又自举了一步」
- 无 Ubuntu 金标就改进度 KPI 为「完成」

**默认下一波选择（后 T，2026-07 中旬）**：

1. 优先 **R2**（runtime / labi 上仍 H>0 的 rest）  
2. 或 **Cap-global-bss** 等阻塞 ≥2 模块真迁的能力  
3. 并行可选 **typeck M2 smoke**  
4. **L** 仅：未产品化叶 / R2 后回归锁

## 2. 通用闭环（所有轨共用）

无论 Cap / R2 / L / M，**一波的外形**都是：

```
[0] 选题（见 §0）
[1] 读代码：rest seed + .x + g05 接线 + prove 行
[2] 实现（根源修；禁 symptom 补丁）
[3] 本机验证（mac：-E / prove / hybrid nm / 可选 smoke）
[4] git commit（feat）→ push
[5] Ubuntu：pull → 同门禁复跑 → 记录 SHA
[6] 更新 analysis/自举进度.md → commit（docs）→ push
[7] 更新前排；开列「下一波候选」
```

| 步骤 | 未完成则… |
|------|-----------|
| 无 [3] | 禁止 push 宣称绿 |
| 无 [5] | 禁止改 KPI「完成」、禁止 docs 写 Ubuntu 金标 |
| 无 [6] | 下一会话会丢仪表盘，视为波次未收口 |

**两条 commit 惯例**（与近几波 R2 一致）：

| 顺序 | 类型 | 内容 |
|------|------|------|
| 1 | `feat(selfhost): …` | 代码 + prove/g05/seed/surface |
| 2 | `docs(selfhost): Ubuntu gold …` | 仅进度文（及必要时方法/步骤文） |

禁止把「未金标的 KPI」和 feat 糊在同一 docs 提交里当完成。

## 3. Track R2 真迁 — 逐步清单（当前默认主线）

> **目标**：产品路径上该 TU **业务符号** 来自 `.x→.o`；`FROM_X` rest **业务 T=0**（通常只留 `labi_*_slice_marker`）。  
> **不算完成**：仅 thin 门闩 + rest 仍全 C（那是 R0/R1，N 可能不动或假涨）。

### 3.1 选题

1. 打开 `analysis/自举进度.md` §3 前排、§4.2 仍标 **thin + hybrid rest C** 的 runtime/labi 行。  
2. 读 `compiler/seeds/<mod>.from_x.c`：业务函数体 vs marker。  
3. 评估能力缺口：

| rest 特征 | 处置 |
|-----------|------|
| 纯逻辑 / 栈缓冲 / 简单 libc extern / 已有 `_buf` 非 slice API | **直接 R2** |
| 需全局 BSS / 超长字面量 / va_list 深路径 / 函数指针+pthread 复杂 | 先 **Cap** 或换更小 rest |
| 仅 thin 已 PREFER、改 surface 刷 nm | **不做** |

**近期已完成 R2 参考**（勿重复）：`rt_util` / `rt_lib_root` / `rt_emit_flags` / `rt_content` / `rt_fs_open` / `rt_parse_diag`。

### 3.2 代码改哪些文件

| 文件 | 动作 | 说明 |
|------|------|------|
| `compiler/src/runtime/<mod>.x`（或 labi 路径） | **吃满业务** | `#[no_mangle]` 公共符号；删 `_impl` 薄门闩依赖（若 R2 后不再需要） |
| `compiler/seeds/<mod>.from_x.c` | 拆 `#ifndef SHUX_*_FROM_X` | **无 FROM_X**：保留冷启动全 C 体；**有 FROM_X**：仅前向声明 + `labi_*_slice_marker` |
| `compiler/seeds/<mod>_surface.from_x.c` | **与 full .x 同构** | 供 prove：由 `shux -E` 抛光生成，勿手写第二套语义 |
| `tests/prove_module_selfhost.sh` | 注释改 R2 full | 模块行路径通常不变；注释写清「full / H=0 / rest 仅 marker」 |
| `compiler/scripts/g05_ensure_relink_prereqs.sh` | 通常**只改 echo/注释** | 若已 PREFER thin+.x + `-DSHUX_*_FROM_X`，拓扑可不动；确认 rest 编译仍带 FROM_X 宏 |
| 其它 | 仅当拓扑未接线 | 新模块才需加 g05 片段、Makefile、宏名 |

**`.x` 实现纪律（R2 常见坑）**：

| 坑 | 做法 |
|----|------|
| C 里 `struct slice` + 指针 | 优先 `*_buf(data, len)` 类 API |
| `diag_reportf` / va_list | 改 `diag_report_with_code` 固定 msg |
| 长字符串 `*u8`（历史 slen 限制） | 栈 `u8[N]` 字节赋值（见 `rt_parse_diag`） |
| 平台 flag 差异 | `#[cfg(target_os = "linux")]` / macos（见 `rt_fs_open`） |
| 空串 / basename | 已由 Cap-empty-str 等解锁时再迁 |
| 全局大数组 BSS | **勿硬迁** → Cap-global-bss |

### 3.3 本机验证（mac，commit 前）

在仓库根或 `compiler/` 下按顺序：

#### A. `-E` 能出 C

```bash
cd compiler
# 选存在的二进制
SHUX_BIN=./shux; [ -x "$SHUX_BIN" ] || SHUX_BIN=./shux-c
perl -e 'alarm 45; exec @ARGV' "$SHUX_BIN" -E \
  -L .. -L src/lexer -L src/ast -L src/parser -L src/typeck \
  -L src/codegen -L src/lsp -L src/driver -L src/preprocess \
  src/runtime/<mod>.x 2>/tmp/mod_e.err | grep -v '^DBG-' > /tmp/mod_e.c
# 期望：exit 0、/tmp/mod_e.c 非空、含业务函数定义
```

失败：先 typeck/codegen 根因，**不要**先改 seed 糊过去。

#### B. 生成 / 对齐 surface

```bash
# 用 /tmp/mod_e.c 抛光写入 seeds/<mod>_surface.from_x.c
# 头部：POSIX prologue（stddef/stdint/…），去掉 -E 自带 #include
# 与 prove 的 gen_x_o prologue 对齐（见 tests/prove_module_selfhost.sh）
```

#### C. FROM_X rest 业务 H=0

```bash
cd compiler
cc -Wall -Wextra -I. -Iinclude -Isrc -DSHUX_<MOD>_FROM_X \
  -c -o /tmp/rest.o seeds/<mod>.from_x.c
nm /tmp/rest.o
# 期望：仅 labi_<mod>_slice_marker（及局部符号）
# 不得再有业务 T 符号（如 driver_fs_open_* / runtime_report_*）
```

冷启动路径（**无** FROM_X）仍应定义业务符号：

```bash
cc -Wall -Wextra -I. -Iinclude -Isrc -c -o /tmp/cold.o seeds/<mod>.from_x.c
nm /tmp/cold.o   # 应有业务 T + marker
```

#### D. hybrid merge smoke

```bash
# thin = -E 产物 cc -c；rest = FROM_X；ld -r 合并
cc -r -nostdlib -o /tmp/merged.o /tmp/thin.o /tmp/rest.o
nm /tmp/merged.o
# 期望：业务 T 来自 thin；marker 来自 rest；无 duplicate symbol
echo SMOKE_OK
```

#### E. prove（单模块 + 全量）

```bash
cd /path/to/shux   # 仓库根
bash tests/prove_module_selfhost.sh <mod>     # 若脚本支持过滤；否则看全量表格行
bash tests/prove_module_selfhost.sh 2>&1 | tail -30
# 期望：该模块 IDENTICAL；全量 DIFF=0；N 附注（R2 同模块 thin→full 时 N 往往不变）
```

**主判据**：`IDENTICAL` = 已定义**全局业务符号集合**一致（非 C 文本 diff；非 rodata 形名）。

#### F. 可选产品链

```bash
# 若本机 g05 / shux-build 可用：
# 确认 g05 log 出现 “R2 full H=0” 或 prefer .x 合并成功
# SHUX=./compiler/shux bash tests/run-g-ffi-5-release-ci-gate.sh   # 大改后建议
```

## 4. Track Cap — 能力解锁逐步清单

> **目标**：修一类阻塞 **真迁** 的编译器/语言能力；结束时列出 **可立即 R2 的模块清单** 并尽量同波或紧随批量 R。

### 4.1 选题

1. 从失败 log / 未完成 rest 聚类（空串、串池、BSS、va、unsafe 表达式…）。  
2. 确认 **≥2 模块** 或 **≥1 大 rest** 被同一缺口卡住。  
3. **一条 Cap 未做完，不开同层冲突 Cap**（落点文件不冲突的不同域可并行）。

### 4.2 代码落点（根源）

| 症状在 | 必须修 |
|--------|--------|
| typeck 报错 | `typeck.x` **且** 实际链接的 `typeck_gen` patch / pin 路径 |
| codegen 坏 C | `codegen.x` + pinned gen 同源 |
| glue 逃逸/分派 | `pipeline_glue_strict_minimal` **不是**只改未链接的全量 glue |
| 仅 seed | **不够** — 语义权威在 .x |

最小复现：≤20 行 `.x` 测试或 tests 下专用 case。

### 4.3 验证

```bash
# 1) 最小复现：修前红 → 修后绿
# 2) 地图 T 不回退
timeout 180 bash tests/diff_single_link.sh 2>&1 | tail -40
# 3) 相关 prove / 产品 smoke
bash tests/prove_module_selfhost.sh 2>&1 | tail -20
# 4) Ubuntu 同命令 + 冷启动（若动 pin/glue）
```

改 `pipeline_glue.c` / `ast_pool.c`：**Ubuntu 重建**相关 `pipeline_x.o` / 冷启动（见 AGENTS.md）。

### 4.4 文档

| 文档 | 更新 |
|------|------|
| `analysis/自举进度.md` §4.1 Track W/Cap 表 | 新波次一行：症状 / 状态 commit / **解锁的 R 候选** |
| §3 前排 | 改为「批量 R2：…」 |
| §8 | 一行 Cap 摘要 + Ubuntu 金标 |
| `自举方法.md` | 仅当新增 Cap 类别或纪律时 |

**禁止**：Cap 合并后只迁 1 个 rest 就回去刷 thin N。

### 4.5 Cap 完成判据

- [ ] 最小复现绿（mac + Ubuntu）  
- [ ] 热路径与 .x 同源  
- [ ] T 未回退（或有意范围已说明）  
- [ ] 文档写明 **解锁 R 清单**  
- [ ] 下一提交进入 R2，不进入 thin surface 冲刺

## 5. Track L — 叶子 prove（辅轨，ROI 明确才做）

### 5.1 仅在这些情况做

- 产品 **尚未** PREFER_X_O，prove 推动首次切 hybrid/纯 .x  
- 模块本就是 **纯叶 / 无 rest**，可去构建硬依赖  
- **R2 之后**的回归锁（防 rest 回潮）  

### 5.2 步骤摘要

1. `.x` `-E` 绿 → surface 对齐  
2. `prove_module_selfhost.sh` IDENTICAL  
3. g05 / Makefile **产品路径**切 PREFER（若尚未）  
4. Ubuntu prove  
5. 进度文：N 附注可 +1；**主叙事仍写是否产品 C 下降**；§4.2 表更新  

### 5.3 明确禁止

- 产品已 PREFER、rest 仍全 C，只扩 surface 让 N+1  
- 用 L 步骤冒充 R2

- **正式构建与金标：git pull**，禁止 scp 源树代替同步  
- 工作分支：通常 `self-hosting`  
- 金标输出贴进 §8 时带 **短 SHA**

### 8.3 KPI 填写约定

| KPI | 何时 +1 / 改写 |
|-----|----------------|
| **R2** | FROM_X rest 业务符号 T=0 **且** 产品 PREFER 吃 .x 体 |
| **H** | 该 hybrid 切片业务 rest 清空 → 计数下降（文案写「上述 k 切片 H=0」） |

| **T** | 仅 typeck 地图变化时 |
| **S** | 产品 pin/PREFER 面变化时 |
| **G** | 冷启动 / FFI / Stage2 状态变化时 |

```markdown

## 10. Commit / Push / 分支

| 项 | 约定 |
|----|------|
| 分支 | 日常自举：`self-hosting`（以团队当前主分支为准） |
| feat | 代码可编译证明；message 含模块名 + R2/Cap/M 层级 |

| push | feat 后立刻 push，便于 Ubuntu pull 同 SHA |

git status

git push origin self-hosting
# Ubuntu 验证 …
git add analysis/自举进度.md
git commit -m "docs(selfhost): Ubuntu gold …"
git push origin self-hosting
```

## 11. 反模式速查（做了等于没做）

| 反模式 | 正确做法 |
|--------|----------|
| thin N++ | R2 吃 rest 或 Cap |
| 只改 seed C | `.x` 语义 + 链接路径 |
| 只改 .x 不改 FROM_X 拆分 | rest 仍提供同名 C → H 不降 |
| 无 hybrid nm 就说 H=0 | `nm` FROM_X rest |
| 无 Ubuntu 写金标 | 先 pull prove 再 docs |
| C 文本 diff=0 卡退役 | nm / 行为门禁 |
| 双套 ABI 表 | 单一权威 |
| Cap 解锁后只迁 1 个就刷 L | 批量 R2 |
| 等 hybrid 全 thin 再 M | M 与 R 并行 |
| scp 当正式同步 | git |
