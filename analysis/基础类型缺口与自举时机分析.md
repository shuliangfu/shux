## 0. 一句话结论

| 问题 | 结论 |
|------|------|
| **现在要不要加语言级 `string` / `str`？** | **不要。** 等 **自举完成（Cap residual / host residual 主闸门关闭）后再做。** |
| **自举阶段还要不要动「基础类型」？** | **只做不搅所有权/分配的薄补全**：文档正式化 `i8`/`i16`/`u16`（实现已基本具备）；**禁止**引入拥有型 `string`、隐式分配、析构/移动语义、插值糖。 |
| **当前最刺眼的应用层缺口** | 语言级字符串体验差 → 但 **库层已有** `std.string.String` / `StrView` + `core.str.BytesView` + 字面量 `*u8`/`u8[]` 路径，**自举不堵死**。 |
| **优先级一句话** | **自举轨优先于应用糖**；糖必须等编译器产品面稳定，否则爆炸半径 × 双权威 × residual 会拖死自举。 |

## 1. 决策框架：用什么尺子量「现在 vs 自举后」

自举阶段改语言面，只允许同时满足：

1. **不增加新的所有权 / 隐式堆分配 / 析构路径**（这些会碰到 heap、link on-demand、lifetime、codegen drop）。  
2. **不强迫 seed / glue / typeck / parser 再开第三条权威**（G.7）。  
3. **爆炸半径可 L2 探针 + 产品矩阵收住**；不必为「好看 API」触发 tip L4 全量 bstrict 大波。  
4. **对编译器自吃 dogfood 非硬依赖**（编译器内部已用 `*u8` + len / cstr helpers 跑通）。  
5. **不与当前 Cap residual leave-off 抢主航道**（name cap 63、host `_impl`、freestanding 等）。

| 尺子 | 语言级 `string` | 正式化 `i16`/`u16`/`i8` | `char`/`never`/`f16` |
|------|-----------------|-------------------------|----------------------|
| 分配/所有权 | 高风险 | 无 | 低～中 |
| typeck/codegen 面 | 极大 | 小（多已存在） | 中 |
| 与自举相关性 | 低（应用糖） | 中（已在 residual 修 widen） | 低 |
| 现在做？ | **否** | **文档 + 可选 residual 补洞** | **否 / 极后** |

### 2.2 实现已有、主表未正式列出

| 类型 | 证据 | 缺口性质 |
|------|------|----------|
| **`i8` / `i16` / `u16`** | typeck `TYPE_NAMED` 族 + wave312/313 integer widen；`core.types` `size_of_i16`/`u16`；atomic/vec/u16 gate；wave315 探针 i8=42 | **文档落后于实现**；个别 emit/cast residual 可能仍有，属 Cap 叶，不是「从零引入」 |
| **字面量字符串语义** | typeck：默认 `*u8`（C interop），并可为 `u8[]` 打高分匹配 print 等 | **不是**语言类型 `string`，是 lit 类型规则 |
| **库级字符串** | `std.string.String` / `StrView`；`core.str.BytesView` | 库类型，非 keyword 内建 |

### 2.3 语言级真正「没有」的

| 缺失 | 用户期望示例 | 今天怎么写 |
|------|--------------|------------|
| 内建 `string`（拥有） | `let s: string = "Hello"` | `std.string` API 或 `*u8` + len / 栈 `u8[N]` |
| 内建 `str`（视图） | `let v: str = s` / 参数 `s: str` | `StrView` / `BytesView` / `u8[]` |
| 字符串插值 | `"x={n}"` | `fmt` / 手动缓冲 |
| `*u8 + *u8` 运算符 | 运行时拼接 | leave-off 已登记；应用层用 concat API |
| 语言级 `char` / `rune` | `'中'` | `u8` 字节或 `u32` 码点 + unicode 库 |
| `never` / `!`、unit `()` | 发散 / 空元组 | `void` 返回 + 不写表达式类型 |
| `f16` / `f128`、更多 SIMD 车道 | 数值/向量 | 按需后期 |

## 3. 核心问题：`string` 自举前还是自举后？

### 3.1 用户想要的形态

```x
let str: string = "Hello World";
```

这不是「再注册一个 NAMED 别名」那么简单，至少要定义整条产品语义链：

```text
源语法 "…" / string 注解
  → parser（token / 类型名 / 可选插值）
  → AST 类型表示（内建 kind？还是 std 注入？）
  → typeck（lit 凝聚、赋值、重载、泛型）
  → codegen / asm emit（布局、拷贝、移动、常量池）
  → 运行时（alloc / free / SSO？）
  → 链接门控（heap/string .o 是否 on-demand）
  → 与现有 *u8 / u8[] / StrView / String 的兼容与迁移
```

### 3.2 两种实现档位（爆炸半径差一个数量级）

| 档位 | 语义 | 工作量级 | 自举期是否可做 |
|------|------|----------|----------------|
| **A. 薄别名 / 糖** | `string` ≡ `u8[]` 或 `*u8`+len 的语法糖；**无**隐式 alloc | 中：parser+typeck+文档+大量兼容测试 | **勉强可做但价值低**（库已有 String/StrView；还会制造「假 string」） |
| **B. 真拥有类型** | 堆缓冲 + 析构/移动/拷贝策略 + 字面量→owned 提升 | 大：接近一门子语言的值语义 | **自举期禁止** |
| **C. 仅文档/库推** | 继续 `std.string`；示例教 `String`/`StrView` | 小 | **推荐现状** |

用户示例 `let str: string = "Hello World"` 在体验上指向 **B**（或至少「看起来像 B」的 A+隐式提升）。  
**一旦字面量自动变成拥有串，就必然碰到堆与生命周期**——这与当前「Cap residual 纯叶、钉盘不乱升、禁 host 硬开」策略冲突。

### 3.3 自举前做 string 的真实代价

| 成本项 | 说明 |
|--------|------|
| **与 residual 抢航道** | 当前 leave-off 是 name cap 63、host `_impl`、freestanding 等；string 会淹没归因 |
| **双权威风险** | 已有 `String`/`StrView`/`BytesView` + lit `*u8`；再加 keyword `string`/`str` 若语义不完全同一路径 → 三套文本类型 |
| **G.7 / seed 同步** | 类型名解析、overload 评分、asm promote、C 后端类型串 —— 处处要同 commit 镜像 |
| **std/编译器 dogfood** | 若半推半就，编译器内部不会立刻受益，却要承担全矩阵回归 |
| **验收膨胀** | 字符串 = Unicode 边界 × 空串 × SSO × move × print 重载 × link 入链；一次 tip L4 不够扫完 |
| **假绿窗口** | mac dead_strip / 部分路径 const-fold 易藏 emit 错（wave315 已有浮点先例） |

### 3.4 自举后做 string 的收益

| 收益 | 说明 |
|------|------|
| **产品面稳定** | parser/typeck/codegen/link 主链不再每周 residual 改热路径 |
| **可一次做对** | 拥有 vs 视图、`str` vs `string`、与 `u8[]` 关系、是否 UTF-8 校验默认开 |
| **插值可并列设计** | `"a{x}"` 依赖 fmt + 临时缓冲策略，适合自举后整包 RFC |
| **不挡自举 KPI** | 自举完成定义不依赖「用户写 string 关键字」 |

### 3.5 判定

```text
应用层刚需强度：高
自举阻塞强度：低（库路径可绕）
实现复杂度：高（若做真 string）
与当前 wave 策略契合度：低

→ 决策：自举完成后再做语言级 string/str（优先 B 档 RFC，禁止自举期半糖 A 冒充 B）
```

**对用户问题的直接回答**：  
- **不是「现在就加 string 类型」**。  
- **也不是「永远用 *u8」**——自举成功后 **最高优先级应用体验** 就是 string/str（+ 插值路线）。  
- **自举前**：继续 `*u8` / `u8[]` / `std.string` / `core.str`；文档写清迁移预期。

## 4. 其它基础类型：缺什么、何时补

### 4.1 总表（优先级 × 时机）

| 优先级 | 类型 | 现状（2026-07-24） | 建议时机 | 理由 |
|--------|------|-------------------|----------|------|
| **P0 应用** | `string` / `str` | 仅库 + lit 规则 | **自举后** | 见 §3 |
| **P1 文档/薄补** | `i8`、`i16`、`u16` | 实现近完整；`02` 主表未列 | **现在：文档正式化**；路径洞走 Cap residual 小叶 | 已 widen/size_of；正式化成本低、不搅所有权 |
| **P2** | `char` / `rune` | 无 | **自举后，与 string 同包或紧随** | 宜 `u32` 语义别名 + unicode；单独上价值有限 |
| **P2** | 运行时 `*u8 + *u8` | leave-off | **自举后或 Cap 很晚的纯叶（慎）** | 运算符糖；有 concat API 可先活；真做要定 alloc 归属 |
| **P3** | `void` 一等类型 | 返回 void 已有 | **可选 / 后期** | 多数场景不需要类型层 void |
| **P3** | `never` / `!` | 无 | **后期** | panic/无限循环标记；提升精确度非自举必需 |
| **P3** | `()` unit | 无 | **可选** | 可用 void / 不写返回 |
| **P4** | `f16` / `f128` | 无 | **很后期** | f32/f64 足够 |
| **P4** | 更多 SIMD（`i8x16`、`f32x4`…） | 部分 i32/u32 向量 | **按性能热点** | 非自举主线 |
| **P4** | `u128` / 大整数 | 无 | **按需** | 密码/宽算术另案 |

### 4.2 `i8` / `i16` / `u16`：为什么「现在可以动文档」

- **CORE-013** 已定 `size_of`/`align_of` = 2（i16/u16）。  
- **wave312–313** 已把 NAMED `i8`/`i16`/`u16` 纳入 integer widen 权威路径。  
- **文档债**：`docs/02-类型定义.md` 主表仍不列它们 → 用户误以为「没有」。  

**自举期允许的动作**（小、可回滚、不升钉盘）：

1. 更新 `02-类型定义.md`：补 `i8`/`i16`/`u16` 与宽度表。  
2. 若探针仍发现 emit/cast 洞 → **Cap residual 纯叶**（与 wave302/313 同模式），**禁止**借机上 string。  
3. **不要**为了「类型整齐」去改 std 批量 API 签名。

### 4.3 `char`：不要单独提前

- 无 string 时，`char` 几乎只是 `u32`/`u8` 别名。  
- 有 string 后，`char` 才进入「码点迭代 / 字面量 `'x'`」设计。  
- **结论**：绑在 string RFC 里，不单独插队自举。

### 4.4 `void` / `never` / `()`

- 函数返回 `void` 已够编译器与 std 使用。  
- `never` 提升类型精密度，**不缩短自举**。  
- 自举期 **零优先级**。

## 5. 字符串能力分层（避免「没有 string 就不能写程序」的误解）

```text
L0  字面量 "…"          → 类型 *u8（默认可匹配 u8[]）
L1  切片 / 缓冲         → u8[] 、u8[N]、*u8 + 显式 len
L2  零拷贝视图          → core.str.BytesView 、std.string.StrView
L3  库拥有串            → std.string.String（堆 + API）
L4  语言级 string/str   → keyword + 字面量凝聚 +（可选）插值   ← 自举后
L5  插值 / 格式化糖     → "a{x}" 等                             ← L4 后或并行 RFC
```

| 层级 | 自举期 | 说明 |
|------|--------|------|
| L0–L3 | **必须保持绿** | 产品 hello、fmt、path、json 等依赖 |
| L4–L5 | **禁止开工实现** | 仅可写 RFC / 本分析级文档 |

**体验差是真的；阻塞自举是假的。**  
编译器自举 dogfood 走的是 L0–L2（cstr、固定缓冲、BytesView 风格），不是 L4。

## 6. 与当前自举 leave-off 的关系

摘自 `analysis/当前进度.md`（wave315 后）与近期 residual：

| leave-off | 与「基础类型」关系 | 自举期态度 |
|-----------|-------------------|------------|
| AST `name`/`var_name` cap **63** | 标识符/路径长度，不是类型族 | 可继续 Cap 叶；**不等于**上 string |
| `*u8 + *u8` 拼接运算符 | 字符串体验相关 | **勿与 keyword string 捆成一波**；有 API 可先活 |
| host residual `_impl` / freestanding | 与类型无关 | 主航道；勿被应用糖打断 |
| 整数/浮点 widen、cast residual | 标量补全 | **允许**纯叶（已在做） |

**纪律**：一条债一层一个 commit；**禁止**「顺便把 string 类型也加上」。

## 7. 推荐路线图（阶段闸门）

### 阶段 S0 — 现在（自举进行中）

| 做 | 不做 |
|----|------|
| Cap residual / host residual 按 pin 推进 | 语言级 `string`/`str` keyword |
| `02` 文档补 `i8`/`i16`/`u16` | 隐式堆字符串字面量 |
| 保持 L0–L3 字符串路径绿 | 插值语法 |
| 可选：string **RFC 草稿**（仅设计，不写产品代码） | 半残 keyword 别名制造双权威 |

### 阶段 S1 — 自举主闸门关闭后（定义见 skill：双端 L4 + 全量 bstrict 等产品叙事）

1. **RFC：语言级文本类型**  
   - `str` = 视图（布局是否 ≡ `StrView` / `u8[]`）  
   - `string` = 拥有（布局是否 ≡ `String`，析构/移动规则）  
   - 字面量默认类型：`str` vs `*u8` 兼容策略（C interop 不能断）  
   - 与现有 std API 迁移表（deprecate 周期）  
2. **实现波次（建议）**  
   - Wave α：`str` 视图 + lit 凝聚（尽量无 alloc）  
   - Wave β：`string` 拥有 + 从 lit/从 str 构造  
   - Wave γ：插值 / 运算符 `+`（若仍要）  
3. **并行可插**：`char`（`u32` 语义）、`i8` 若仍有文档/边角。

### 阶段 S2 — 应用层打磨

- cookbook 全面改 `string`/`str`  
- LSP / 语法高亮 keyword  
- 性能：SSO、堆策略与 `std.heap` 契约

## 8. 风险与反模式

| 反模式 | 为何坏 |
|--------|--------|
| 「先做一个 string = u8[] 别名让示例好看」 | 用户以为有拥有语义；日后真 string 破坏兼容 |
| 自举期上插值 | 强依赖临时分配与 fmt；爆炸半径不可控 |
| 只改文档宣称有 string | 假绿叙事；与「文档/实现双权威」同罪 |
| 自举期为 string 改 AST name cap / host 一锅炖 | 归因死亡 |
| 废弃 `*u8` 字面量过早 | 打断 C interop 与现有 std |

| 点 | 你的判断 | 校准 |

## 10. 最终决策摘要（可贴进度/RFC 文首）

```text
【基础类型 · 自举时机】

1. 语言级 string / str / 插值 / 拥有语义
   → 自举完成后做（最高优先级应用体验）；自举期禁止实现。

2. 自举期字符串策略
   → *u8 + len / u8[] / core.str.BytesView / std.string.{String,StrView}
   → 保持 L0–L3 绿即可。

3. i8 / i16 / u16
   → 实现基本具备；现在补 docs/02 正式地位；路径洞走 Cap residual 纯叶。

4. char / never / unit / f16 / 更多 SIMD
   → 自举后或按需；不插队。

5. 纪律
   → 不抢 Cap residual 主航道；不制造 string 与 String 双权威；
   → 一条债一层；SHARED 双端；谈自举仍 tip L4 金标。
```
