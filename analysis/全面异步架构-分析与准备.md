## 0. 结论（先读）

| 问题 | 结论 |
|------|------|
| **自举后能否做到 Zig 0.16 式「全面、无函数染色」异步？** | **能。** Xlang 已有 CPS 语言面、协作调度、io_uring/async IO、Future/Poll 雏形、TaskGroup、Context；再加可注入执行模型（Io/Executor）与库 API 去染色，是可行长期方向。 |
| **现在要不要暂停自举去大改？** | **不要。** 自举（G 清 seed、零 libc、产品 L4 金标）是地基；此时大改执行模型会反复砸 bootstrap gate，风险/收益比差。 |
| **自举期间要不要为全面异步做准备？** | **要，但只做「低成本、不破闸门」的并行准备**：扩展点、契约文档、单权威、平台矩阵、no-libc 下 IO 原语可用性、**禁止**第二套 async 栈。 |
| **最佳推进窗口** | **G 阶段末期～自举金标（Stage2 纯 .x + 零 seed / L4–L5）达成后**，再开「执行模型抽象层」主工程；自举收尾期只做可用级强化（现有 async/await + 调度 + io_uring）。 |
| **时间预期（量级）** | G 若在 2026 年底前收口，**2027 年初**具备做全面重构的工程条件（非承诺日历）。 |

**一句话**：自举不是障碍，而是前提；反馈里的判断整体正确。本文在反馈之上补齐 **现状差距地图、染色根因、目标架构分层、分阶段实现路径、自举期硬清单/禁清单**。

## 1. 用户反馈的评估（同意 / 细化 / 校正）

### 1.1 同意的核心判断

1. **Zig 也是自举相对稳定后才推 std.Io + 无染色** —— 时机叙事正确：先有可自举编译器与稳定 runtime，再动执行模型。  
2. **纯 .x 环境后迭代 async 抽象更安全** —— 与 Xlang G 目标一致：std/core/runtime 去 seed 后，改调度与 Io 接口不必同时打穿 C seed + `.x` 双权威。  
3. **已有基础（async/await、scheduler、io、thread/channel、liveness）** —— 不是从零做 async 语言。  
4. **显式内存模型（region/arena/noalias）利于高性能 async** —— 无 GC 暂停、便于 zero-copy 与帧布局，是相对 Rust/Go 的潜在优势。  
5. **自举期：预留扩展点 + 打磨 liveness/cancel/structured concurrency；不做完整无染色重构** —— 正确。

### 1.2 需要细化 / 校正的点

| 反馈表述 | 细化 |
|----------|------|
| 「完全有潜力 / 甚至超越 Zig」 | **潜力成立**；「超越」取决于：Io 接口设计质量、与 region/borrow 的融合、生态 API 收敛速度，**不是**现状已具备。 |
| 「async_liveness 已在推进」 | 自举语境下 **async_liveness / async_cps_codegen 主要是编译器 TU 自举（R2 pure / Cap residual）**，不是「全面异步产品重构」本身。两者共用同一套 CPS 语义，但 **工程目标不同**。 |
| 「std.Io 风格」 | Zig 的 Io 是 **可注入的 I/O + 调度能力面**；Xlang 今天是 **语言 `async` 染色 + 固定 scheduler C ABI + std.io 提交槽**。目标应对齐「**单一能力面、可换后端**」，不必照抄 Zig API 名字。 |
| 「no-libc 下 io_uring 也能工作」 | 方向对；当前 **产品轨零 libc** 与 **async 全链路 freestanding** 尚未同一验收叙事。准备期必须把「async 原语是否依赖 libc/pthread 哪些符号」画进矩阵（见 §6）。 |
| 「2027 年初全面重构」 | 可作 **规划窗口**；触发条件应是 **金标准清单**（§5.4），而不是日历。 |

### 1.3 反馈未写透、但决定成败的项

1. **函数染色在 Xlang 里具体发生在哪一层**（语言关键字 / codegen / 库 API / 运行时）。  
2. **单一权威**：禁止 std.async v2 旁路另起一套 scheduler。  
3. **与自举 G 闸门的交叉**：scheduler_glue、io stubs、link_abi on-demand 都是 SHARED 产品面，乱动会假绿。  
4. **平台三端**：Linux io_uring 金标 vs macOS kqueue vs Windows IOCP，全面架构必须从第一天写 PLATFORM 矩阵。  
5. **泛型 Future / trait 对象**：无染色库 API 往往依赖更好的泛型与 trait；语言能力债要进路径，不能只谈 runtime。

## 2. Zig 级「全面异步」到底指什么（目标定义）

### 2.1 问题：函数染色（function coloring）

经典分裂：

- 同步函数只能调同步；异步函数只能 `await` 异步。  
- 库被迫提供 `read` / `read_async` 两套 API，或整库染成 async。  
- 生态分裂（JS/C# 历史；Rust async trait 长期痛苦）。

Zig 0.16 方向（抽象描述，非抄接口）：

- 业务代码写 **一套** I/O 逻辑。  
- 通过注入 **`Io` / 执行环境**（线程池阻塞、事件循环、测试假 Io、自定义）决定「这次调用如何完成」。  
- **不靠** `async fn` 把调用图染成两套类型世界。

### 2.2 Xlang 目标定义（本文采用的「全面异步架构」）

**目标态 T\*（全面）** 同时满足：

| # | 能力 | 说明 |
|---|------|------|
| T1 | **单一库 API** | `std.io` / `std.net` / `std.fs` 热路径对用户呈现 **一套** 读写/连接 API；不强制 `*_async` 第二命名空间（内部可有 backend）。 |
| T2 | **可注入执行模型** | 显式 `Io` / `Executor`（名字可再议）参数或线程/任务局部绑定；同一函数体可在 Threaded / Evented / Test / Custom 下跑。 |
| T3 | **无强制染色** | 用户库函数 **默认** 不因「内部会挂起」而必须标 `async`；若保留 `async`/`await` 语法糖，应是 **可选优化/显式控制流**，不是生态通行证。 |
| T4 | **结构化并发** | TaskGroup / cancel 传播 / deadline 为默认纪律，而不是事后补丁。 |
| T5 | **高性能后端可选** | Linux io_uring（及 multishot 等）、macOS kqueue、Windows IOCP；freestanding/no-libc 路径可退化但语义清晰。 |
| T6 | **与内存模型融合** | 帧/缓冲生命周期可用 region/arena/noalias 表达；zero-copy 读视图有合法寿命。 |
| T7 | **单一权威实现** | 调度、挂起、完成队列、链接门控 **一条路径**（G.7）；seed 仅过渡。 |

**非目标（本阶段明确不做）**：

- 把整个编译器本体改成「必须 async 才能编译」（dogfood 可选，不阻塞）。  
- 在自举未金标前替换全部 `xlang_async_*` C ABI。  
- 兼容「无限栈协程 + 任意抢占」而不谈帧布局与 unsafe 边界。

### 2.3 与「当前可用异步」的分层

| 层级 | 名称 | 用户能做什么 | 相对 T\* |
|------|------|--------------|----------|
| L-now | **实用异步（现状）** | `async`/`await`/`run`/`spawn` + io 槽 + drain；1M coop 门禁 | 有染色、固定 scheduler、Future 弱 |
| L-usable | **可用高性能** | 现模型打磨：liveness、多 worker、io_uring 边角、Context 全路径 | 仍染色，但生产可用 |
| L-inject | **可注入执行模型** | 出现 `Io`/`Executor` 参数与 2～3 个后端 | 开始去染色 |
| L-full | **全面（T\*）** | 库 API 单一 + 无强制染色 + 结构并发默认 + 三端后端 | 目标 |

## 3. 现状地图（上帝视角）

### 3.1 分层数据流（当前）

```text
.x 源
  async function / await / run / spawn
        │
        ▼
  parser / typeck          （async 语法与 is_async 等）
        │
        ▼
  async_liveness           跨 await 活性 · 帧布局 · needs_cps_frame
  async_cps_codegen        switch(__phase) · suspend[_io] · sched wrapper
  async_asm_pool           帧池 / layout 辅助
        │
        ▼
  生成 C / .o
  符号：xlang_async_cps_suspend[_io]、xlang_async_run_i32、
        xlang_async_task_submit、xlang_io_submit_*_async …
        │
        ▼
  runtime scheduler glue   协作帧 · MPSC · workers · io wait/wake
  std.io backend           io_uring / kqueue / IOCP · async 槽
        │
        ▼
  std.async 门面           wait / future_* / drain / re-export
  std.task / std.context   TaskGroup · cancel · deadline
        │
        ▼
  link_abi                 用户 .o 有 UNDEF 时按需链 scheduler.o
```

### 3.2 模块与权威路径（禁止再开第二套）

| 能力 | 当前权威 | 备注 |
|------|----------|------|
| 语言 await 边界 emit | `compiler/src/async/async_cps_codegen.*` + seed | R2 pure 大量已迁；`driver_preamble_fputs` Cap residual |
| 帧 liveness / layout | `async_liveness.*` + seed | 与 CPS 共用 AST 布局契约 |
| 帧池 | `async_asm_pool.*` | 已 PREFER unbundle |
| 调度运行时 | `runtime_scheduler_glue`（seed + thin `.x`） | **仍重度 C**；全面架构中期要收敛到纯 .x 或单一 .x 权威 |
| IO submit/complete | `std/io/*` + platform backend + runtime io stubs | 名称门闩表在 cps（`read`/`write`/`xlang_io_*`） |
| 用户门面 | `std/async/mod.x` | 薄封装；Future 在 `future.x` |
| 结构化并发 | `std/task/task.x` | 仍绑 `xlang_async_spawn_i32` 等 |
| 取消 | `std/context/*` | 已与 async bind_context 对接 |
| 按需链接 | `runtime_link_abi` `needs_async_scheduler` | 符号表扩展时 **只改权威表** |

### 3.3 已有能力（资产清单）

| 资产 | 状态 | 对 T\* 的价值 |
|------|------|----------------|
| 语言 `async`/`await`/`run`/`spawn` | 已对接 STD-041 | L-now 核心；T3 时可能降级为糖 |
| CPS 帧 + phase switch | 生产可用 | 事件式后端的编译器基础 |
| coop ping-pong ~ns 级 | 门禁 + bench | 调度热路径可信 |
| io_uring 路径 + async slots | Linux 主路径 | T5 后端 |
| Future i32 槽（max 64） | 雏形 | **不够** 做通用无染色库 |
| TaskGroup / JoinSet | v1 可用 | T4 起点 |
| Context cancel/deadline | 可用 | T4 必备 |
| Reader/Writer trait（std.io） | 已有 | T1 扩展点 |
| 平台 backend 分流 | linux/macos/windows | T5 骨架 |
| freestanding / 零 libc 战略 | 产品决议 | 异步后端必须进同一战略 |

### 3.4 差距（相对 T\*）

| ID | 差距 | 层 | 严重度 |
|----|------|-----|--------|
| G-color | **`async function` 染色**：含 await 的函数必须是 async；调用侧 `run`/`spawn` 特殊 | 语言 + codegen | 高（生态） |
| G-dual-api | **双 API**：sync `read`/`write` vs async submit/complete；await 靠 **名字门闩** 识别 IO | 语言 + std | 高 |
| G-io-inject | **无可注入 Io/Executor**；执行模型绑死 scheduler_glue | std + runtime | 高 |
| G-future | Future 仅 i32 槽、固定容量、非泛型 | std.async | 中高 |
| G-fp | 函数指针 / 类型统一未完成（STD-041 已记 extern 冲突） | 语言 | 中（阻塞优雅 spawn） |
| G-sched-c | 调度内核仍大量 C seed | runtime | 中（自举与迭代成本） |
| G-trait | trait/interface 表达力决定「一套 API 多后端」成本 | 语言 | 中 |
| G-name-gate | `async_cps_callee_is_io` 等 **字符串名表** 识别 await 目标 | codegen | 中（扩展脆弱） |
| G-platform | 非 Linux 为 ⚠️ 回退；全面矩阵未当 T\* 硬门 | 平台 | 中 |
| G-freestanding | async 全链 freestanding 验收未与零 libc 金标对齐 | 产品 | 中 |
| G-structured | TaskGroup 未与语言 `spawn` 语法糖默认合一 | 语言 + std | 低中 |
| G-doc | 活文档缺「全面异步」权威 RFC（本文补） | 文档 | — |

### 3.5 染色根因（为什么现状必然有 coloring）

1. **语法层**：`async function` 与普通 `function` 分流；`is_async` 进 AST。  
2. **编译层**：`needs_cps_frame` / await 点数驱动 **帧结构 + switch 状态机**；同步函数不生成该形态。  
3. **运行时层**：挂起点依赖 `xlang_async_cps_suspend[_io]` 与固定就绪环。  
4. **库层**：高性能路径显式 `*_async`；语言 await 只识别 **固定名字集合** 的 callee。  

因此：**不是「再写几个 async 函数」就能无染色**；无染色要求 **调用约定与完成机制从「函数类型世界」挪到「注入的 Io 能力」**（或等价：编译器按调用图自动变换但对外类型不分裂——实现更重）。

## 4. 目标架构（建议蓝图）

> 名称暂定；实现时以 **单一权威 RFC 定名**，禁止并行 `std.async2` / `std.runtime_v2`。

### 4.1 四层模型

```text
┌─────────────────────────────────────────────────────────┐
│  L4  用户业务 / 库 API                                     │
│      一套 read/write/connect；可选显式 await 控制流        │
├─────────────────────────────────────────────────────────┤
│  L3  能力面 Io / Executor（可注入）                         │
│      sleep · read · write · spawn · cancel · poll        │
│      后端：Threaded | Evented(io_uring/kqueue/IOCP) | Test │
├─────────────────────────────────────────────────────────┤
│  L2  运行时内核                                             │
│      就绪队列 · worker · 完成队列 · Context 绑定            │
│      （演化自今日 scheduler_glue + io backend）            │
├─────────────────────────────────────────────────────────┤
│  L1  编译器支持（可选加速）                                 │
│      CPS 变换 / 栈切 / 帧布局 / liveness                    │
│      今日 async_* TU；T* 中变为「优化路径」而非染色门槛     │
└─────────────────────────────────────────────────────────┘
```

### 4.2 两种实现策略（择一为主，另一可兼容）

| 策略 | 描述 | 优点 | 风险 |
|------|------|------|------|
| **A. 参数注入（Zig 近亲）** | 热路径函数接受 `io: Io`（或等价），内部 `io.read(...)` | 去染色清晰；可测试；与 trait 契合 | API 改面大；需 trait/泛型成熟 |
| **B. 隐式上下文** | 任务局部 / 线程局部当前 Io；语法糖不写参数 | 调用点干净 | 隐式状态难推理；与 region/unsafe 纪律冲突风险 |

**建议主路径：A（显式 Io 参数）**，与 Xlang「显式内存/安全」气质一致；B 仅作薄糖（宏/包装）且必须可关闭。

### 4.3 与现有 CPS 的关系

| 阶段 | CPS 角色 |
|------|----------|
| 现在 | **主路径**：语言 async 的唯一实现 |
| L-inject | Evented 后端 **内部** 仍可用 CPS 帧；Threaded 后端可直接阻塞 syscall |
| L-full | 用户 **不必** 写 `async fn` 才能用 Evented；编译器可对「在 Evented 下调用的 Io 方法」做自动变换或库内状态机 |

**关键纪律**：Io 后端实现可以内部染色；**用户可见类型世界不染色**。

### 4.4 与内存 / 安全模型

- **帧**：async 帧 = 跨挂起点的活局部；应对齐 `async_liveness` 与 region（挂起期间借用规则）。  
- **zero-copy**：`u8[]<io_read_ptr>` 类视图寿命必须挂在 **Io 操作 / arena** 上，禁止无标注逃逸。  
- **cancel**：Context 取消必须能 **中止 in-flight Io**（今天部分路径有 bind；T\* 要求 Io 层统一契约）。  
- **noalias / 线性**：通道与缓冲所有权转移比「共享 Future 回调」更贴 Xlang。

### 4.5 平台矩阵（T5 强制）

| 后端能力 | Linux | macOS | Windows | freestanding Linux |
|----------|-------|-------|---------|---------------------|
| 阻塞 Threaded Io | ✅ | ✅ | ✅ | ⚠️ 需 syscall 面 |
| Evented 主路径 | io_uring | kqueue | IOCP | io_uring 子集 / 轮询 |
| multishot / batch | 部分已有 | 回退 | 回退 | 按需 |
| 验收金标 | **Ubuntu** | 产品 SHARED | 探针 / 后 G | NL 金标对齐 |

所有新 Io 代码：**PLATFORM 注释 + G.8 双端**（SHARED 不得只 mac 绿）。

## 5. 分阶段实现路径

### 5.1 阶段 0 — 自举并行准备（**现在～G 收尾**）

**目标**：不破坏 L4/bstrict；为 T\* 铺「不会被迫推倒」的地基。

| 动作 | 优先级 | 说明 |
|------|--------|------|
| 本文档 + README 挂链 | P0 | 单一叙事入口 |
| **禁止第二套** async/Io 栈 | P0 | G.7；扩展只补权威 |
| 扩展点 **文档化**（见 §6） | P0 | 代码可只加注释/薄 trait 坑，不大改 |
| 现有门禁保持绿 | P0 | `run-async` / language / io-cps / 1m / task / context |
| Context 在 await 路径全覆盖 | P1 | cancel/deadline 不丢 |
| TaskGroup 与 spawn 语义文档对齐 | P1 | 为结构并发默认化 |
| async  freestanding 符号依赖清单 | P1 | 对照 `零libc产品策略.md` |
| Future 泛型化 **设计**（不急实现） | P2 | 依赖语言泛型债 |
| Io trait 草案（仅 analysis + 空 trait？） | P2 | 若加代码须极薄且不改链接默认 |

**禁止**：替换 scheduler ABI、删 `async` 关键字、大规模改 std.io 用户面、为「无染色」改 typeck 主路径。

### 5.2 阶段 1 — 自举收尾期「可用高性能」（L-usable）

**触发**：产品 tip 稳定、G 主链 residual 可控。

- 打磨：多 await 槽、io_uring 边角、worker 亲和、trace（已有 obs 方向）。  
- 编译器：name-gate → **typeck 标记 / resolved 属性**（仍可保留名字回退），减少字符串表脆弱性。  
- 调度 thin 继续 `.x` 化（跟自举 R2 节奏，**不单开异步重构项目**）。  
- 文档：cookbook 结构并发 + await IO 最佳实践。

### 5.3 阶段 2 — 引入可注入执行模型（L-inject）

**触发**：§5.4 金标准满足。

1. 定稿 **Io / Executor 接口**（方法集最小：read/write/sleep/spawn/cancel 查询）。  
2. 实现 **ThreadedIo**（先）：直接同步 syscall；证明「一套业务代码」。  
3. 实现 **EventedIo**：内部接今日 scheduler + async 槽 + io_uring。  
4. **TestIo**：录制/回放；门禁友好。  
5. 选 1～2 个 std 模块（建议 `std.io` 核心 + `std.net` 一点）做 **双路径**：旧 API 转调 `default_io()`，新 API 显式 `io`。  
6. 迁移纪律：**一条债一个 commit**；Ubuntu L4 + async 门禁 + 产品矩阵。

### 5.4 阶段 3 — 去染色与生态默认（L-full / T\*）

- 新库默认 **只暴露** Io 参数 API；旧 `async fn` 语法保留为可选糖或逐步软弃。  
- codegen：Evented 下自动变换或库内状态机成熟后，弱化「必须 async 关键字」。  
- TaskGroup 成为 spawn 默认；泄漏检测进 CI。  
- 三端 Evented 矩阵进 gate；freestanding Evented 子集有硬标准。  
- 删除/收敛 `*_async` 用户面双名（内部符号可留）。

### 5.5 金标准：何时允许开「全面异步主工程」

同时满足（可裁剪，但不可空降日历）：

- [ ] 自举叙事：Stage2 / 零 seed 主链或项目定义的 **G 金标** 已宣称且 L4 双端绿  
- [ ] 产品：`run-all-bstrict` 金标持续绿；async 相关 gate 全绿  
- [ ] 调度与 IO：**权威符号表** 文档化；无双实现  
- [ ] 零 libc：async 相关符号依赖清单闭合或有明确 residual 列表  
- [ ] 语言：函数指针 / trait 足以表达 Io 方法表（或临时 C ABI vtable 可接受）  
- [ ] 有 **迁移 RFC**（本文下一代）+ 试点模块清单

## 6. 自举期间必须准备的基础（清单）

### 6.1 必须做（低成本 · 高回报）

1. **单一权威纪律（强制）**  
   - 调度 / await emit / IO 完成 / link 门控：只扩现有符号与文件。  
   - 发现 `*_v2` / 平行 Future 实现 → 合并或开明确废止票，禁止静默双栈。

2. **扩展点预留（文档 + 接口形状，少动代码）**  
   - `std.io`：Reader/Writer 已存在 → 文档写明未来 `Io` 为 **后端能力**，Reader 为 **字节流适配**。  
   - `std.async`：门面保持薄；新能力优先挂 **scheduler 权威** 再 re-export。  
   - Context：所有新异步 API **必须** 有 cancel/deadline 挂钩位（参数或 bind）。  
   - link_abi：新增挂起/完成符号时 **只改** `needs_async_scheduler` 权威表。

3. **平台与 freestanding 诚实矩阵**  
   - 每个 async 原语注释 `PLATFORM:`。  
   - 维护「依赖 libc/pthread/liburing 哪些」表（可附本文附录，随改更新）。  
   - 零 libc 产品路径改动时，**顺带**跑 async smoke（不必全量 1M）。

4. **编译器侧：契约稳定优先于功能膨胀**  
   - `async_liveness` / `async_cps_codegen` 的 AST 布局注释与 host LE 契约是 **自举 + 未来 Io** 共用资产。  
   - Cap residual 只收敛 fputs 类 opaque，不借机改 CPS 语义。  
   - name-gate 表扩展时：优先考虑 **typeck 标志位** 设计备注（实现可后置）。

5. **结构化并发与取消**  
   - 保持 `std.task` / `std.context` gate 绿。  
   - 新示例默认展示 TaskGroup，而不是裸 spawn 泄漏。

6. **测试资产不腐烂**  
   - `tests/run-async.sh`、language / io-cps / 1m / task / context 作为 **回归锚**。  
   - 全面架构试点必须挂在这些锚上，禁止「新目录绿、旧门禁红」。

### 6.2 应该做（中成本 · 自举不冲突）

| 项 | 说明 |
|----|------|
| Future 能力债列表 | 泛型、多类型 payload、取消、组合子；进语言债 backlog |
| trait 对象 / 静态分发 Io | 与 `lang-trait` 路线对齐 |
| await 目标从名字表 → 语义属性 | 减少 G-name-gate |
| scheduler 继续 thin `.x` | 跟 G-02f/R2，不另起炉灶 |
| 观测：async runtime trace | 已有 obs 方向；重构时要能对比延迟 |

### 6.3 不要做（自举期禁令）

| 禁令 | 原因 |
|------|------|
| 暂停自举专攻无染色 | 地基未稳，返工成本高 |
| 大规模改 `async` 语法语义 | 砸语言门禁与 CPS 测试 |
| 新建 `std.async2` / 第二 scheduler | 双权威 |
| 在 seed 与 `.x` 只改一侧调度语义 | 漂移 |
| 用 soft-skip / BEST_EFFORT 顶 async 红 | 假绿 |
| 只在 mac 验 Evented 当 Linux 金标 | G.8 |
| 把「async_liveness R2 绿」写成「全面异步完成」 | 目标混淆 |

### 6.4 代码触碰时的「未来友好」微规则

编写/修改 async 相关代码时：

1. **完成机制与业务逻辑分离**：业务函数尽量不直接依赖全局 `g_xlang_futures` 形态。  
2. **挂起点显式**：suspend 只经权威 `xlang_async_cps_suspend*` 或未来 `Io` 方法。  
3. **缓冲所有权写清**：谁分配、谁在 complete 后释放、取消时谁释放。  
4. **错误码统一叙事**：对齐 `std.error` / IO_ASYNC_NOT_READY 等已有常量，不发明第三套。  
5. **英文 docblock（G.9）** + **PLATFORM（G.8）**。

## 7. 风险与假绿

| 风险 | 表现 | 缓解 |
|------|------|------|
| 双执行栈 | 旧 async 与新 Io 行为不一致 | 单一默认后端；对照测试同输入同输出 |
| 链接假绿 | mac dead_strip 藏 UNDEF | Ubuntu 全链 + needs_* 表 |
| 自举假绿 | 只 prove async TU 当产品 async 绿 | 产品矩阵 + run-async 门禁 |
| 染色「假去除」 | 语法去掉 async 但类型仍分裂 | 以 T1/T3 验收，不以关键字有无 |
| cancel 半吊子 | 任务停了 IO 还在跑 | Io 层统一 cancel 契约 + 测试 |
| freestanding 谎言 | 文档写无 libc，async 仍链 pthread 未声明 | 符号依赖表 + ldd/nm 门禁 |
| 范围爆炸 | 同时改语言+std+net+http | 试点模块制；一波一模块 |

## 8. 与自举路线的关系（优先级）

```text
当前主线（不可让）
  G 清 seed / 零 libc / 产品 L4 / bstrict / 双端
        │ 并行（低成本）
        ├─ 本文纪律 + 扩展点文档
        ├─ async 门禁不绿不扩 feature
        └─ Context/Task/liveness 小步
        │
        ▼
G 金标 / Stage2 窗口
        │
        ▼
全面异步主工程（阶段 2～3）
  Io 接口 → Threaded → Evented → 去染色迁移
```

**资源分配建议**：

- 自举冲刺期：全面异步 **≤10%** 工程注意力（文档 + 纪律 + 门禁）。  
- 金标后第一季：可升到 **主项目之一**，仍建议与 Windows/MS ABI 等大项 **错峰**。
