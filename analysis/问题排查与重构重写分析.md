# 问题排查与重构/重写分析

> 本文档记录近期全面检查与修复结论，以及「若问题难以根治时」的重构/重写选项。  
> 与 `完全脱离C与Makefile路线图.md` 配套使用。

---

## 一、本次已修复问题

### 1. 入口模块 num_funcs=0 导致 run-return-value / run-multi-file 失败

**现象**：`[pipeline] before_codegen num_funcs=0`，pipeline 报错或产出空。

**原因**：slice 版 `parser_parse_into` 的 **-E 生成 C** 存在两种形态：

- 形态 A：生成 `int32_t so = 0; int32_t ci = 0; }` 后直接到 `parser_collect_imports`，漏掉「写回 block + module.funcs + num_funcs++ + lex」整块（仅 1.3 补丁覆盖）。
- 形态 B：只生成 `int32_t so = 0; }`（无 `ci = 0`），同样漏掉写回块，且原 1.3 补丁的匹配串不匹配形态 B，导致**入口模块**解析后 num_funcs 仍为 0。

**修复**：在 `scripts/fix_pipeline_gen.py` 中新增 **1.3b 补丁**：匹配形态 B（`"    int32_t so = 0;\n  }"` + `parser_collect_imports(lex, source, module, ...)`），插入与 1.3 相同的写回块（`num_stmt_order`、`ast_arena_block_set`、填充 `f`、`module.funcs[num_funcs]=f`、`num_funcs++`、`parser_lex_from_onefunc_next_into`）。

**验收**：`run-return-value.sh`、`run-multi-file.sh`、`run-su-multi-file.sh` 通过；`before_codegen num_funcs=1`（或 2 等）正确。

### 2. Parser 在 EOF 处死循环（-backend asm 编仅 enum 模块卡住）

**现象**：`shuc -backend asm -o out.o token.sx` 或仅含 `enum E { A }` 的文件长时间无输出并卡住。

**原因**：`parse_into` / `parse_into_buf` 中，当当前 token 不是 `FUNCTION` 且 `try_skip_allow` 未跳过时会 `break`，随后执行「解析一个函数」分支；若此时 token 为 **EOF**，会反复在 EOF 上尝试解析函数并 `continue`，形成死循环。

**修复**：在 **parser.sx** 的 slice 与 buf 两路中，在 `break` 之后、「解析函数」分支之前，增加：若 `r.tok.kind == TokenKind.TOKEN_EOF` 则直接返回成功（`num_funcs==0` 时 `main_idx=-1`，否则返回当前 `main_idx`）。

**验收**：逻辑上 -backend asm 编仅 enum 模块应能结束；当前 -backend asm 仍会卡住（见下节），EOF 修复为必要不充分条件。

## 二、尚未解决 / 已知问题

### 1. -backend asm 路径仍卡住

**现象**：即使用最小文件（如 `enum E { A }` 或空文件），`./shuxc -backend asm -o out.o file.sx` 仍会长时间无输出并超时。

**已排除**：

- 非 asm 路径下同一文件可完成解析（num_funcs=0 或 1），说明 parser 在非 asm 路径能正常结束。
- `typeck_su_ast_library` 在 `num_funcs==0` 时直接返回；`backend_asm_codegen_ast` 在 `num_funcs==0` 时 while 不进入，直接返回。
- Parser 已做 EOF 早退，理论上仅 enum 模块应能结束。

**可能位置**：卡住更可能发生在**进入 pipeline 之前**（如 driver 的读文件/预处理），或 **pipeline 内 asm 分支的早期**（在打印 `[pipeline] before_codegen` 之前）。精确定位需在 driver 入口、`pipeline_run_sx_pipeline_impl` 入口等处加调试输出或 gdb 断点。

**影响**：`build_shuc_asm.sh` 无法完整跑通，asm 自举目前不可用；日常构建与 run-all 不依赖 -backend asm，仍可正常使用。

### 2. run-hello 输出为空

**现象**：`run-hello.sh` 报 "expected 'Hello World' in output, got: "。

**可能原因**：stdout 未正确捕获、或 print/puts 类能力在 -sx 路径下未实现/未链接。与 num_funcs 写回无关，属独立问题。

### 3. 其它 run-* 脚本

未做全量 run-all 完整扫一遍；建议在 1.3b 与 EOF 修复合入后，定期跑 `./tests/run-all.sh` 或 CI 做回归。

## 三、何时考虑重构/重写

### 1. 不建议立刻大面积重写

- 当前**业务逻辑已在 .sx**（parser/typeck/codegen/pipeline/driver/build），路线图 6.0～6.4 已达成；问题主要集中在：
  - **-E 生成 C 的形态**与 **fix_pipeline_gen.py 的补丁覆盖**不完全一致；
  - **-backend asm** 一条路径上的卡住（可能为 driver/preprocess/parser 中某一处）。
- 这类问题更适合**定点修复 + 补丁/生成逻辑收敛**，而不是推倒重写。

### 2. 适合做「局部重构」的方向

| 方向 | 内容 | 目的 |
|------|------|------|
| **生成形态收敛** | 让 -E 对 parser_parse_into 的 slice 路径只生成一种形态（例如统一有 `so = 0; ci = 0;` 及写回块），或把「写回 module.funcs」作为必须生成的语义在 codegen 层保证，减少对 fix_pipeline_gen.py 的形态匹配依赖。 | 避免「新 -E 输出换一种格式就漏补丁」。 |
| **driver/pipeline 入口可观测** | 在 main_driver_run_sx_emit_su、pipeline_run_sx_pipeline_impl 等处加可选 debug 打印（如环境变量控制），便于下次快速定位 -backend asm 卡住点。 | 不改变行为，只提升可排查性。 |
| **asm 路径最小复现** | 用最小 main（如仅 return 0）+ 无 import 的单一文件，在 -backend asm 下做单步/断点，确认卡在 preprocess / parser 首 token / typeck / asm 哪一阶段。 | 为 -backend asm 修 bug 或文档化「当前 asm 仅支持 xxx」提供依据。 |

### 3. 若未来真的「重写」编译器前端

仅在以下情况考虑**局部或模块级重写**（不是全盘重写）：

- **parser -E 生成**长期不稳定，且修 codegen 成本高于用更简单、生成形态单一的 parser 子路径（例如「仅支持无歧义语法」的瘦 parser）专门用于 -E；
- **-backend asm** 经排查确认为设计或状态机缺陷，且修不如重写 asm 入口与 pipeline 的对接层。

重写时应保持：

- 与现有 AST/typeck/codegen 的接口一致（Module、ASTArena、PipelineDepCtx 等）；
- 测试套件（run-return-value、run-multi-file、run-su-multi-file、run-all）作为回归基准。

## 四、策略建议：先 C 实现完整功能，再自举

**结论：是的，先不急着自举、先用 C 把功能实现完整，是更稳妥的路线。**

### 4.1 为何容易「陷入误区」

- 当前是 **C 与 .sx 混用**：parser/typeck/codegen 既有 C 也有 .sx，pipeline/main 用 .sx 写再 **-E 展开成 C**（pipeline_gen.c、driver_gen.c），依赖 **fix_pipeline_gen.py** 等补丁去修生成结果的多种形态。
- 问题变成双重：既要保证「**语义对不对**」（该写回 module.funcs、该在 EOF 返回），又要对付「**生成出来的 C 长什么样**」（形态 A/B、漏块、补丁没匹配到）。相当于**同时在做「实现功能」和「自举管线」**，容易互相拖累。

### 4.2 更稳妥的顺序

| 阶段 | 做法 | 目的 |
|------|------|------|
| **一、初级版本：C 实现** | 用 **纯 C** 实现完整编译器：parser、typeck、codegen、driver、pipeline、必要时的 build 脚本。不依赖 .sx → -E 生成，不依赖 fix_pipeline_gen.py。 | **功能跑通、行为稳定**；所有 run-* 测试通过；C 版本成为**唯一事实标准**。 |
| **二、再来自举** | 在「C 版行为已固定」的前提下，再按模块用 .sx 重写（或逐段替换），对同一输入与 C 版对比输出，一致即通过。 | 自举变成「**移植已知正确的实现**」，而不是「一边改语义一边修生成/补丁」。 |

这样做的效果：

- **先 C**：只关心「逻辑对不对」，不关心「-E 生成了啥、要补几处」；bug 范围小、回归简单。
- **后自举**：.sx 的职责是「和 C 版等价」，生成形态可以慢慢收敛，甚至先不追求 -E 一条龙，用 C 链 .sx 的中间形态也可以接受。

### 4.3 和当前路线图的关系

- 路线图里的「完全脱离 C」「自举」等目标**仍然成立**，只是**建议把顺序调成**：
  1. 先把「**功能完整、测试全过**」的 **C 版编译器** 做扎实；
  2. 再在不动行为的前提下，做「C → .sx 迁移」和「自举/asm 后端」。
- 若当前 .sx 已较多、全部回退到 C 成本高，也可以采用**折中**：以 **C 版为参考实现**，.sx 只做「与 C 版对齐」的修补和回归，不再为自举提前铺 -E 形态、asm 等，等 C 版稳定后再系统性做自举。

### 4.4 自举相关代码：保留，不删除

**结论：自举相关代码建议全部保留，不删除。**

| 保留内容 | 说明 |
|----------|------|
| **.sx 源码**（parser.sx、pipeline.sx、main.sx、typeck.sx、codegen.sx、build.sx 等） | 日后做「C 行为稳定后再自举」时直接复用；删掉后将来要重写一遍。 |
| **-E 生成与 fix_pipeline_gen.py**（pipeline_gen.c、driver_gen.c 及补丁脚本） | 当前构建若仍依赖 seed（driver_su.o + pipeline_sx.o），删掉会断链；保留则现有 `make bootstrap-driver-seed` 仍可用，只是不作为「主开发路径」。 |
| **build_shuc_asm.sh、-backend asm 路径** | 虽当前 asm 自举未跑通，代码占位不多，保留不影响「先用 C 完善功能」；以后 C 稳定再修 asm 或重接自举即可。 |
| **build_tool（build.sx）** | 可选构建方式，保留不碍事；主路线用 Makefile + C 时可以不调用它。 |

**当前主路线建议**：

- **主路径**：以 **C 实现** 为准，完善 parser/typeck/codegen/driver/pipeline 的 C 版逻辑，跑通 run-all，把 C 版当作唯一事实标准。
- **自举相关**：**保留不删**，也不主动改；需要跑测试时仍可用 `make bootstrap-driver-seed` 或现有脚本，但新功能、修 bug 以 **C 为准**，.sx/-E/asm 仅「跟着 C 行为对齐」或暂不投入。
- 等 C 版功能完整、测试全过之后，再以现有 .sx 与生成管线为基础做自举，无需从零恢复代码。

## 五、建议的下一步

1. **合入 1.3b 与 EOF 修复**：确保 `make bootstrap-driver-seed` 后 run-return-value、run-multi-file、run-su-multi-file 稳定通过。
2. **策略选择**：若采纳「先 C 再自举」——优先用 C 把 parser/typeck/codegen/driver/pipeline 行为做到 run-all 全过，再谈 -E 生成与 asm 自举；若继续当前混用——则收敛 -E 生成形态、减少对 fix_pipeline_gen.py 的依赖。
3. **可选**：为 -backend asm 增加最小复现与一次 gdb/printf 定位，将卡住点记入本文档或 issue；若短期不修 asm，在 README 或 asm 文档中注明「asm 自举当前未验证，建议使用 seed + C 生成」。
4. **定期**：跑 `./tests/run-all.sh`，把新增失败用例记入「尚未解决」并优先修回归，再考虑局部重构或自举节奏。
