## 八、std.net 实现状态（功能完整）

已在 **std/net/mod.sx** 与 **std/net/net.c** 落地，功能与 Zig/Rust 常用 API 对等：

- **TCP**：Ipv4Addr、TcpStream、TcpListener；connect/listen/accept(..., timeout_ms)、close_stream/close_listener；accept_many/connect_many（当前循环实现）。
- **UDP**：UdpSocket；udp_bind(addr, port)、udp_send_to(sock, addr, port, buf, len)、udp_recv_from(sock, buf, len, timeout_ms, out_addr, out_port)、close_udp。
- **IPv6**：Ipv6Addr（16 字节）；connect_ipv6(addr, port, timeout_ms)、listen_ipv6(addr, port)；accept 复用（listener 可为 IPv6）。
- **地址与端口**：SocketAddrV4(addr, port)；local_addr(stream)、peer_addr(stream)、listener_local_addr(listener)。
- **DNS**：resolve(hostname: *u8) -> Ipv4Addr（阻塞，hostname 须 NUL 结尾；失败返回 0.0.0.0）。
- C 层：net_tcp_*、net_udp_*、net_tcp_local_addr_c/net_tcp_peer_addr_c（返回 i64 打包）、net_tcp_*_ipv6_c、net_resolve_ipv4_c；Unix 用 poll+netdb.h，Windows 用 select+ws2tcpip；链接 net.o，Windows 需 ws2_32。

数据读写：TcpStream.fd 与 io.handle_from_fd 统一，用 **io.read_fd/write_fd** 或 **read/write(handle, ptr, len, timeout_ms)** 走 io_uring/kqueue/IOCP。

## 八点一、与 Zig / Rust 对标与缺口

| 能力 | Zig std.net | Rust std::net | 当前 std.net | 说明 |
|------|-------------|---------------|--------------|------|
| TCP connect/listen/accept | ✅ connectToHost、listen | ✅ TcpStream::connect、TcpListener::bind/accept | ✅ connect/listen/accept | 已实现，且带 timeout_ms |
| 连接/接受超时 | 可配合 async/超时 | connect_timeout、set_read/write_timeout | ✅ 统一 timeout_ms（connect/accept） | 我们读写超时在 std.io 每次 read/write |
| 默认非阻塞 | 可配置 | 默认阻塞，需 set_nonblocking | ✅ 默认非阻塞 | 与 driver 统一调度，利于超越 |
| 关闭连接/监听 | drop / close | drop / shutdown | ✅ close_stream/close_listener | 已实现 |
| IPv4 地址 | ✅ Ipv4Address | SocketAddrV4 | ✅ Ipv4Addr(a,b,c,d) | 已实现 |
| IPv6 / 双栈 | ✅ | SocketAddrV6、SocketAddr | ✅ Ipv6Addr、connect_ipv6、listen_ipv6 | 已实现 |
| 主机名解析（DNS） | ✅ connectToHost 等 | 需 std::net::ToSocketAddrs | ✅ resolve(hostname)->Ipv4Addr | 阻塞解析，失败返回 0.0.0.0 |
| UDP | ✅ | UdpSocket | ✅ UdpSocket、udp_bind/send_to/recv_from、close_udp | 已实现 |
| local_addr / peer_addr | 有 | 有 | ✅ local_addr、peer_addr、listener_local_addr（返回 SocketAddrV4） | 已实现 |
| 批量 accept/connect | 可循环 | 可循环 | ✅ accept_many/connect_many（当前循环实现） | 接口已有；真批量需与 driver Completion 对接 |
| 读写抽象 | Reader/Writer | Read/Write trait | 走 std.io（同一 handle） | 统一 I/O 环，利于批量和零拷贝 |

**结论（功能完整性）**：**std.net 功能已完整**：TCP、UDP、IPv4、IPv6、DNS、local/peer_addr 均已实现；与 Zig/Rust 常用网络 API 对等。后续可专注性能优化（批量 accept/connect、[]Buffer、与 driver Completion 对接等）。

**结论（是否具备超越能力）**：有。依据在《高并发IO与多平台》§六、§七：  
- Socket **默认非阻塞** + 与 **std.io.driver 同一套 Buffer/submit/timeout**，便于做批量和多段 I/O（[]Buffer）、热路径零分配。  
- 接口**不绑定** io_uring/kqueue/IOCP 形状，自举后可做自研驱动或内核/ bypass，在「实际表现超越典型用法」或「换赛道（如 bypass）」上具备条件。  
- 当前 accept_many/connect_many 为循环版；**真正超越**需：driver 层 []Buffer 批量 submit、accept/connect 与 Completion 对接、以及（可选）io_uring 批量 accept/connect 或自研路径。

## 九、性能压榨与超越 Zig

详见 **《[std.net 性能压榨与超越 Zig](std.net性能压榨与超越Zig.md)》**，要点：

- **Zig 瓶颈**：典型用法单次 accept/connect、单次 read/write，多路复用未必用 io_uring，批量化与预注册不暴露。
- **我们路线**：阶段 1～2 用 io_uring 做 accept/connect 及真批量 accept_many/connect_many；阶段 3 将 []Buffer 与 net 打通；阶段 4～5 fixed fd/buffer 与 UDP recvmmsg/sendmmsg；阶段 6 可选 bypass。
- **多段缓冲区**：scatter-gather、[]Buffer 批量 submit 的 API 设计与实现见该文档阶段 3。

## 十、网络协议检查项

与《自举前-目标与缺口分析》汇总表「网络协议检查项」对应；自举后实现须遵守以下三项。

| 检查项 | 约定 | 自举前状态 |
|--------|------|------------|
| **Socket 默认非阻塞** | 通过 std.net 创建的套接字默认非阻塞，由 std.io.driver 统一调度 | 已在本文档与 std.net 占位注释中写明；自举后实现时遵守 |
| **submit 预留 []Buffer** | submit_read / submit_write 在签名或文档中**预留** buf 接受 **[]Buffer**（多段缓冲区）的能力，自举后实现 | driver 当前签名为单 Buffer；文档与注释已预留 []Buffer 扩展 |
| **超时统一 u32 ms** | 超时粒度统一为 **u32 毫秒**，与 driver 的 timeout_ms 一致 | 已满足：std.io.driver 的 submit_read/submit_write 均为 timeout_ms: u32 |

以上三项在自举前已文档化并与现有占位一致；自举后实现网络栈与多段 submit 时须符合本第十节。
