# 自举实现 — 逐步计划（参考 C 一一实现 .sx）

> 目标：从头到尾、参考 C 实现，把编译器全链路在 .sx 中实现并完成自举。本文档列出各环节状态与下一步。

## 一、流水线总览（C 当前 vs .sx 目标）

| 环节 | C 实现 | .sx 实现 | 状态 |
|------|--------|----------|------|
| 词法 | lexer.c (lexer_new, lexer_next) | lexer.sx (lexer_init, lexer_next) | ✅ 已有 |
| 语法 | parser.c (parse → ASTModule) | parser.sx (parse, parse_into → Module+Arena) | ✅ 最小集已有 |
| AST | ast.h / ast.c | ast.sx (Type, Expr, Block, Func, Module, ASTArena) | ✅ 已有 |

## 二、已完成的 .sx 模块

- **lexer.sx**：Lexer、lexer_init、lexer_next(source: []u8)；关键字、标识符、整数/浮点、符号；与 token.sx 一致。
- **token.sx**：TokenKind、Token（kind, line, col, int_val, float_val, ident, ident_len）。
- **ast.sx**：TypeKind/ExprKind、Type/Expr/Block/ConstDecl/LetDecl/WhileLoop/ForLoop/Func/Module、ASTArena、ref_is_null、ast_arena_*。
- **parser.sx**：parse(source) → ParseResult；parse_into(arena, module, source) 解析最小程序 `function main(): i32 { return <int>; }` 并填 Arena+Module。
- **typeck.sx**：typeck_su_ast(module, arena)；仅读 .sx AST，无 extern 业务逻辑。
- **codegen.sx**：codegen_sx_ast(module, arena, out)；唯一 extern 为 codegen_sx_emit_byte；类型/表达式/块/函数/模块全在 .sx。
- **pipeline.sx**：run_sx_pipeline(arena, module, source, out) = parse_into → typeck_su_ast → codegen_sx_ast。

## 三、C 与 .sx 的衔接

- **调用 .sx 流水线**：C 在 run_compiler_c 中（如带 `-su` 或 `SHUC_USE_SU_PIPELINE`）读入并预处理后，分配 arena/module，调用 `pipeline_run_sx_pipeline(arena, module, src, len, out)`。
- **分配**：arena/module 由 C 分配；需知 sizeof(ASTArena)、sizeof(Module)，由生成 pipeline_gen.c 时在同一编译单元内提供 `pipeline_sizeof_arena()`、`pipeline_sizeof_module()`。
- **写输出**：out 为 FILE* 传予 codegen_sx_emit_byte；由 codegen.c 实现。

## 四、建议实施顺序

1. **打通单文件最小程序**  
   - 使 C driver 在某种条件下（如 `-su`）调用 run_sx_pipeline。  
   - 验收：`shuc -sx -E tests/return-value/main.sx`（或等价最小 main）：解析、typeck、codegen 全走 .sx，输出与 C 版一致或可编译的 C。

2. **扩展 parser.sx**  
   - 参考 parser.c，逐步支持：多函数、参数、let/const、if/while/for、二元/一元运算、return 表达式、块末表达式等。  
   - 保持 parse_into 产出 .sx AST（Arena+Module），与 typeck.sx/codegen.sx 一致。

3. **扩展 codegen.sx**  
   - 对照 codegen.c，补全尚未支持的 ExprKind/块内语句（defer、labeled、stmt_order 等）、多文件/import 占位或简化。

4. **多文件与 import**  
   - 先由 C 做 resolve_imports 与 preprocess，再对每个模块调用 .sx 的 parse_into；或后续在 .sx 内实现简化版 import 解析。

5. **driver.sx 作为唯一入口（不要 main.c）**  
   - 将 main.c 中的命令行解析、读文件、调 pipeline、调 cc 全部迁入 driver.sx（如 `function main(): i32`）；生成的 C 自带 `main()`，链接最小 C 运行时（提供 extern）即可，不再需要 main.c。

6. **build.sx（不要 Makefile）**  
   - 用 .sx 写构建逻辑：列出 compiler 下需编译的 .sx、调用 shuc 生成 C、调用 cc 链接、产出 shuc；用现有 shuc 编译并运行 build.sx 即可完成一次构建，替代 Makefile。

7. **自举验证**  
   - 用宿主编译器编出 shuc₁；用 shuc₁ 编 compiler 下 .sx 得 shuc₂；两代对同一测试套件行为一致。

## 五、参考的 C 文件

- 流水线：`compiler/src/main.c`（read_file → preprocess → parse → typeck → codegen → invoke_cc）。
- 词法：`compiler/src/lexer/lexer.c`。
- 语法：`compiler/src/parser/parser.c`。
- 类型检查：`compiler/src/typeck/typeck.c`。
- 代码生成：`compiler/src/codegen/codegen.c`。
- AST：`compiler/include/ast.h`、`compiler/src/ast/ast.c`。

## 六、构建说明（pipeline）

- 将 pipeline.sx 编为 C：`shuc -E -L .. src/pipeline/pipeline.sx > pipeline_gen.c`（需能解析 parser/typeck/codegen/ast/lexer/token 及 std.fs）。
- 在 pipeline_gen.c 末尾提供：`pipeline_sizeof_arena()`、`pipeline_sizeof_module()`（返回对应 struct 的 sizeof），以便 main.c 用 malloc 分配后调用 `pipeline_run_sx_pipeline`。

## 七、目标：build.sx 与不要 Makefile / main.c

- **build.sx**：用 .sx 实现的构建程序，替代 Makefile；负责「用 shuc 编译 compiler 下哪些 .sx、调 cc 链接、产出 shuc」。
- **不要 main.c**：编译器入口与驱动逻辑放在 driver.sx（`function main(): i32`）；生成的 C 含 `main()`，仅需链接最小 C 运行时（实现 extern），不再依赖 main.c。  
详见 **自举-全面实现与C扩展约定.md** §七。
