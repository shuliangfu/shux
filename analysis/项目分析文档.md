# Shux 项目全面分析

> 本文档基于对仓库实际代码、构建配置、测试体系与自举进度的逐项审查，系统梳理 Shux 的定位、架构、实现现状、工程治理与风险点。
> 数据采集时间：2026-07-02。版本号：`0.2.0`。当前分支：`dev`。

---

## 一、项目概要

**Shux** 是一门**系统级编程语言**，源文件后缀 `.x`，编译器二进制名 `shux`，构建配置文件 `build.x`（对标 Zig 的 `build.zig`）。许可证 AGPL-3.0-or-later，单一作者 Shuliang Fu。

| 维度 | 取值 |
|------|------|
| 语言名 / 后缀 | Shux / `.x` |
| 版本 | 0.2.0 |
| 许可证 | AGPL-3.0-or-later |
| 仓库提交数 | ~4460（自 `init` 起至 2026-07-01） |
| 主分支 | `dev`（当前）、`main`；远端 `origin/dev`、`origin/main` |
| 当前主线 | 阶段 G 终局清场 |

### 核心定位

用现代化语言写高性能、强安全代码——**参考 C 的类型与执行模型，语法与语义尽量简单、易读、易维护**；不靠复杂类型论，而靠「显式 + 少未定义 + 默认安全」同时达成性能与内存安全。对标关系：

- 相对 C：开发体验更简单，错误带位置与建议，工具链一体化。
- 相对 Rust：类型系统刻意克制，在可空、边界、所有权/借用、别名处**足够显式**，让编译器静态证明安全并放心优化。

### 核心目标

| 目标 | 含义 |
|------|------|
| 极致性能 | 无 GC、零成本抽象、ASM 直出 + 可选 C 后端；别名与 noalias 驱动自动向量化 |
| 内存安全 | 安全子集内无泄漏、无悬垂、无越界、无数据竞争；`unsafe` 边界清晰可审计 |
| 轻量级 | 依赖少、可执行文件小、支持 freestanding 与嵌入式；标准库按需链接 |
| 标准库丰富 | `core/` + `std/` 全量模块，一套 API 全平台 |
| 易于上手 | 比 C 简单太多，学习曲线平缓，工具链顺手 |
| 编译自举 | 编译器与标准库最终 100% `.x`；宿主 C / seed 仅冷启动与清场过渡期 |

## 二、设计哲学

### 2.1 零成本高性能

- **无 GC**：内存由栈 + 堆 + Arena 管理，编译期 region / linear / borrow 拒错。
- **单态化**：泛型函数/结构体在编译期实例化，消除运行时泛型代价。
- **别名分析**：`noalias`、scope borrow gate，为 autovec 与 DCE 提供依据（`src/codegen/autovec.c`）。
- **编译期自动内存管理**：`defer`、`owned`、作用域 `Allocator`（`with_arena`）、SROA、BCE。

### 2.2 显式安全

刻意简化 Rust 式的「所有权 + 借用」体系，采用更直白的模型：

- **Region / Borrowing**：在 typeck 层管理内存生命周期，拒绝区域逃逸；切片 `T[]<label>` 携带域标注。
- **Arena / 作用域分配器**：`with_arena` 等基于作用域的内存管理。
- **分级安全**：默认 Safe；裸指针、底层 syscall 必须在 `unsafe { ... }` 块内，边界清晰可审计。

### 2.3 简洁易用

- 避开复杂类型理论，学习曲线平缓。
- 错误信息带位置与建议。
- 语法比 C 更清晰，避免特性膨胀。

详细设计见 `analysis/语法与类型设计-高性能与内存安全.md`、`analysis/需求分析.md`。

## 三、语言特性概览

### 类型与语义

- 基础类型：`i8`/`i16`/`i32`/`i64`、`u8`/`u16`/`u32`/`u64`、`f32`/`f64`、`bool`、`usize`/`isize`。
- 结构体与泛型：泛型函数/结构体 + 单态化；trait / impl 接口。
- 可空与错误：`Option<T>`、`Result<T, E>`，显式处理。
- 切片：`T[]` 携带长度；域标注 `T[]<label>` 与 region 借用域（typeck 拒逃逸）。
- 模块：`import("std.io")` / `import("core.mem")`；目录即模块，入口 `mod.x`。

### 内存与安全

- 无 GC：栈 + 堆 + Arena；编译期 region / linear / borrow 拒错。
- 编译期自动内存管理：`defer`、`owned`、作用域 `Allocator`、SROA、BCE。
- 分级安全：默认 Safe，仅 `unsafe { ... }` 内允许裸指针与底层 syscall。
- 别名分析：`noalias`、scope borrow gate。

### 条件编译与平台

- 条件编译（`#if` / target 分支）支持多目标。
- 全平台一套 API：平台差异封装在 `std/sys` 内部。
- 支持 freestanding（`-freestanding`、nostdlib 静态链）。
- 多目标：`x86_64-linux`、`arm64-macos`、`riscv64` 等（`-target`）。

Hello World 示例（`examples/hello.x`）：

```sx
const fmt = import("std.fmt");

function main(): i32 {
  let n: i32 = fmt.println("Hello World");
  return if (n >= 0) { 0 } else { 1 };
}
```

语法速查见 `docs/README.md`（关键字、类型、控制流、模块等共 9 篇）。

## 四、编译器架构

### 4.1 编译流水线

```
.x 源码 → 预处理 → 词法 → 语法 → AST → 类型检查 → 代码生成 → 后端 → .o / 可执行文件
```

| 阶段 | 目录 | 核心文件 | 规模（行） |
|------|------|----------|-----------|
| 预处理 | `src/preprocess/` | `preprocess.x` | — |
| 词法分析 | `src/lexer/` | `lexer.x`、`token.x`、`cfg_eval.x` | lexer.x 2,181 |
| 语法分析 | `src/parser/` | `parser.x` | 7,924 |
| AST | `src/ast/` | `ast.x` | 1,101 |
| 类型检查 | `src/typeck/` | `typeck.x` | 6,061 |
| 代码生成 | `src/codegen/` | `codegen.x`、`codegen_mini.x`、`autovec.c` | codegen.x 6,853 |
| 汇编后端 | `src/asm/` | `asm.x`、`backend.x`、`arch/*_enc.x` | x86_64_enc 918 / arm64_enc 845 / riscv64_enc 611 |
| 中间表示 | `src/ir/` | （仅 README，IR 尚未落地） | — |
| 驱动编排 | `src/main.x`、`src/driver/` | build/check/compile/emit/fmt/run/test 子命令 | — |
| 异步 | `src/async/` | `async_liveness.c`、`async_cps_codegen.c` | — |
| 语言服务 | `src/lsp/` | `lsp.x`、`lsp_diag.x`、`lsp_io.x` | — |
| 流水线编排 | `src/pipeline/pipeline.x` | 解析→依赖加载→typeck→codegen 串联 | — |

### 4.2 两条产物路径

由 `build.x` 与 `build_tool` 决定：

- **默认路径（legacy）**：shux 把各 `.x` 模块编译为 C（`-E -E-extern`），再用宿主 `cc` 编成 `.o` 并链接 → `compiler/shux`。依赖少、调试友好，中间产物 `*_gen.c`。
- **asm 路径**：`./build_tool <shux> asm` → 用同一套 shux 以 `-backend asm` 把 `.x` 直接编成目标文件 `.o`（构建顺序见 `src/asm/asm_build_list.x`），再以 crt0 + 极少 C 运行时节链接出 `compiler/shux_asm`。这是「尽量走机器码、少依赖 C 翻译」的路径；链接阶段仍需宿主链接器。

### 4.3 自举分层

- **shux-c**：C 前端版本，仅用于**首次冷启动**（`bootstrap.sh` / `make build-tool`）或非 x86_64 链接回退，**不是日常发布二进制**。
- **shux（B-strict）**：日常开发入口，由 `make -C compiler bootstrap-driver-bstrict` 产出。
- **shux_asm**：asm 路径产物，`build_use_asm_only()` 决定是否默认走 asm 脚本。

### 4.4 多后端与多架构

- **C 后端**：`.x` → C 文本 → 宿主 `cc`，调试友好。
- **ASM 直出后端**：`.x` → 目标文件 `.o`，三套架构编码器：
  - `src/asm/arch/x86_64_enc.x`（918 行）
  - `src/asm/arch/arm64_enc.x`（845 行）
  - `src/asm/arch/riscv64_enc.x`（611 行）
- 平台目标文件格式：`src/asm/platform/elf.x`、`macho.x`、`coff.x`。
- 运行时 crt0：`crt0_x86_64.s`、`crt0_arm64.s`、`crt0_darwin_x86_64.s`、`crt0_mingw.c`、`freestanding_io_x86_64.s`。

### 4.5 代码规模现状

| 区域 | `.x` 文件 | `.x` 行数 | `.c` 文件 | 备注 |
|------|-----------|-----------|-----------|------|
| `compiler/src` | 43 | 38,608 | 186（149,562 行） | 主体已 `.x` 化，C 仍大量保留（阶段 G 待清） |
| `core` | 12 | 1,915 | 0 | 纯 `.x`，无 OS 依赖 |
| `std` | 164 | 52,758 | 16 | 16 个 `.c` 均为 `*_import_alias.c` 绑定桩，非业务逻辑（F-ZC ✅） |

> 注：README 声称 `std/` 0 `.c` + 0 `.h`（F-ZC ✅）指**业务逻辑无 C**；实际残留的 16 个 `*_import_alias.c` 是 `import("std.xxx")` 在 asm `-o` 链接时生成的 `std_xxx_*` 符号桥桩，与 `mod.x` 语义对齐，属链接胶水而非功能实现。

## 五、标准库

### 5.1 分层

- **`core/`（无 OS 依赖）**：可在裸机、内核、嵌入式使用。12 个模块：`types`、`mem`、`option`、`result`、`slice`、`fmt`、`builtin`、`assert`、`cmp`、`iterator`、`str`，`debug` 为兼容 alias。
- **`std/`（依赖 OS）**：在 core 之上提供统一 API，用户只写一套 API，平台差异封装在 `std/sys` 内部，按需链接未用模块不进二进制。

### 5.2 std 模块清单（50+）

已完善可用的模块包括：`runtime`、`io`、`mem`、`fs`、`process`、`path`、`heap`、`string`、`vec`、`map`、`error`、`net`、`thread`、`time`、`random`、`env`、`fmt`、`debug`、`sync`、`encoding`、`base64`、`crypto`。

其余模块覆盖：`async`、`atomic`、`backtrace`、`bytes`、`cache`、`channel`、`cli`、`codec`、`compress`、`config`、`context`、`csv`、`db`（SQLite）、`dynlib`、`elf`、`ffi`、`hash`、`http`（含 HTTP/2）、`json`、`log`、`math`、`metrics`、`queue`、`regex`、`schema`、`security`、`set`、`simd`、`socketio`、`sort`、`tar`、`task`、`test`、`trace`、`unicode`、`url`、`uuid`、`websocket`。

### 5.3 平台策略

- **Linux**：`std/sys/linux.x` — 内联汇编 `svc/syscall`，可 freestanding，拒绝 libc 脐带。
- **macOS**：`std/sys/macos.x` — `extern "C"` 绑 `libSystem.dylib`（稳定，规避 syscall 号漂移）。
- **Windows**：`std/sys/win32.x` — `extern "stdcall"` 绑 `kernel32.dll` / `ws2_32.dll` + IOCP（禁止硬编码 Nt* syscall 号）。


## 七、构建与测试体系

### 7.1 构建

- **顶层 Makefile**：委托 `compiler/Makefile`，统一入口 `all/clean/test/test_c/test_x/bootstrap-lexer/bootstrap-token`。
- **`compiler/Makefile`**：核心构建规则，定义 `OBJS_CORE`、`SHUX_C`/`SHUX_X`、各 bootstrap-target（bstrict / asm / crt0 / seed-user-asm 等）、`-E -E-extern` 生成 `*_gen.c` 的规则。
- **`build.x`**：最终目标是以本文件为唯一「构建策略」来源，去掉 Makefile；当前 `build_tool`（C 版 + 未来 .x 版）按 `build_use_asm_only()` 决定走 asm 或 legacy。
- **`bootstrap.sh`**：仅依赖 `cc`、不依赖 Makefile 的冷启动入口，产出 shux、shux-c、build_tool。

日常流程：

```bash
make -C compiler build-tool              # 首次：从零产出 shux-c、build_tool，链出 shux
make -C compiler bootstrap-driver-bstrict  # 日常：B-strict 产出 shux_asm 覆盖 compiler/shux
SHUX=./compiler/shux ./tests/run-hello.sh
make -C compiler bootstrap-verify        # 语义自举烟测（两代 shux 测试 outcome 一致）
```

### 7.2 测试规模

`tests/` 目录约 **3,517 个文件**，其中：

- `*.sh`：1,170（含 `run-all.sh`、`run-all-x.sh`、`run-all-c.sh`、`run-all-bstrict.sh`、各 `run-*-gate.sh` 门禁）

- `tests/run-all-bstrict.sh`、`tests/run-all-seed.sh`：分别验证 B-strict 与 seed 路径。

## 九、风险与改进建议

### 9.1 当前风险点

1. **仓库卫生**：工作树有 **145 个未提交改动**，根目录散落大量 `debug-*.md`（hello-stage1-segv、memory-spike、regex-asm-crash、check-segv 等）及临时二进制（`debug_join`、`tmp_string_user.o`、多个 `shux_asm*` 变体如 `.broken_strict`、`.pre_net_fix`、`.orch`）。建议归档 debug 笔记到 `analysis/`、清理临时产物、纳入 `.gitignore`。
2. **编译器 C 代码体量**：`compiler/src` 仍有 186 个 `.c`（约 14.9 万行），是阶段 G 物理清场的主要工作量；同时 `compiler/` 目录散落大量 `shux*`、`shux_asm*`、`*_gen.c`、`*_link_alias.c` 副本与备份，应建立产物隔离目录。
3. **测试体量与时长**：1,170 个 `run-*.sh` 全量回归极慢（`run-all-x.sh` 注释提到 asm 编译 typeck.x 致 25min 超时，CI 内需 skip）；需分层快测与夜间全测。
4. **IR 缺位**：`src/ir/` 仅有 README，IR 优化 Pass 尚未落地，长期可能制约 WPO 与跨过程优化上限。
5. **平台覆盖不均**：Windows 仅 B-hybrid（关卡一🟡、二⬜、四⬜），riscv64 依赖 QEMU 烟测；macOS 与 Linux 的自举成熟度差异需持续维护。
6. **A-10 悬置**：std path/net hang 仍未解决，阻塞构建链关卡 A 的完全闭合。

### 9.2 改进建议

- **建立产物分区**：`compiler/build/` 集中 `*_gen.c`、`*_x.o`、`shux*` 副本，源码区只留 `src/`。
- **分层 CI**：fast（核心语义 ~5min）/ full（nightly）/ gold（selfhost stage2），避免单次 PR 跑 25min+。
- **debug 笔记归档**：根目录 `debug-*.md` 统一移入 `analysis/debug/` 或 `.dbg/`（已存在该目录）。
- **IR 规划**：在阶段 G 收尾后启动 IR 中间表示与优化 Pass 设计，为后续 WPO 默认化铺路。
- **Windows 推进**：将 B-hybrid 推向 B-strict，补齐关卡二/四的自举闭环。

---

## 十、总结

Shux 是一个雄心明确、执行密度极高的系统级语言自举项目：以「C 的性能 + Rust 的安全 + 更简单」为定位，已经走通了前端 `.x` 化、黄金自举（Stage2 哈希一致）、std 无 C 三大里程碑，标准库覆盖 50+ 模块且全平台一套 API，编译器同时具备 C 后端与三架构（x86_64/arm64/riscv64）ASM 直出后端。

当前处于**阶段 G 终局清场**：核心矛盾从「能不能自举」转向「能不能把宿主 C 依赖与仓库手写 C 物理清零、实现 Linux 零 libc bootstrap、用 `build.x` 彻底替代 Makefile」。工程层面的主要改进空间在仓库卫生、产物隔离、分层 CI 与 IR 规划。语言与标准库的能力面已相当完整，下一步更多是工程收口与长期优化（WPO 默认化、IR、Windows 自举）。
