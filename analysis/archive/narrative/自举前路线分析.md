## 三、自举前「语言能力」缺口（写编译器所需）

用 .x 写编译器时，至少需要：

1. **多函数定义与函数调用** ✅  
   - ~~当前：AST 仅有 `main_func`，Parser 只接受 `function main() -> i32 { ... }`，无「其他函数」与「调用表达式」。~~  
   - 已实现：支持多个 `function name(params) -> ret { body }`，以及表达式中的 `name(args)` 调用；typeck 校验参数/返回值；codegen 生成多函数 C 与 call。

2. **泛型（7.1）** ✅  
   - 已实现：泛型函数 `function name<T>(...)`、调用 `name<Type>(args)`；typeck 单态化与实例登记、泛型体代入检查；codegen 按 mono_instances 生成 C 函数（mangle 名）。Option\<T\>、Result\<T,E\>、Vec\<T\> 等标准库依赖泛型；编译器数据结构也可用泛型容器。

3. **Trait（7.2）** ✅  
   - 已实现：trait 定义、impl for Type、方法调用 base.method(args)、typeck 校验与静态分派；标准库接口（如 Read/Write、迭代）可在此基础上扩展。

4. **模块/可见性（7.3）** ✅ 多文件已支持  
   - 多文件：import 可从入口目录解析（同目录 import foo）；库模块生成带前缀 C 并参与链接。后续可补跨模块调用、export/可见性。

因此：**在 7.1 泛型之前，应先补齐「多函数 + 函数调用」**，否则无法用 .x 写任何多函数程序（包括未来的编译器前端）。**（多函数 + 函数调用已按 5.1 完成 ✅）**

## 五、建议实施顺序（自举前）

按依赖关系与「先能写多函数 .x 再写泛型库」的原则，建议顺序如下。

### 5.1 第一步：多函数 + 函数调用（7 的前置，可记为 6.5 或 7.0）✅ 已完成

- **AST**：模块内支持多个函数（如 `ASTFunc* funcs; int num_funcs`，或保留 `main_func` 并增加 `other_funcs`）；函数有 name、params、return_type、body。 ✅
- **Parser**：解析 `function <ident>(...) -> type { block }`，不再强制仅 main；顶层可有多条 import、多函数、struct/enum 等。 ✅
- **Typeck**：维护符号表（含函数名与类型）；调用表达式 `callee(args)` 时检查 callee 存在、参数个数与类型、返回值类型。 ✅
- **Codegen**：为每个函数生成 C 函数定义；对调用表达式生成 C 的 `callee(...)`。 ✅
- **测试**：多函数、互相调用、main 调辅助函数并返回结果。 ✅

完成标志：能编译并运行「含至少两个函数且 main 调用另一函数」的 .x。 ✅

### 5.2 第二步：泛型（7.1）✅ 已完成

- 泛型函数与泛型结构体语法（Parser：`function name<T>(...)`、`struct Name<T> { ... }`、`name<Type>(args)`）；typeck 单态化与实例登记、代入后形参/返回值检查、泛型体按实例再检查；codegen 按 mono_instances 生成 C 函数（mangle 名如 id_i32）、调用处生成对应实例名。 ✅
- 完成标志：泛型 .x（如 `id<T>(x: T) -> T { x }` 与 `main() -> i32 { id<i32>(42) }`）能编译并运行。 ✅

### 5.3 第三步：Trait（7.2）✅ 已完成

- trait 定义与 impl；typeck 解析 impl、约束检查；codegen 单态化 + 静态分派（impl 方法 mangle 名、base.method(args) 生成对应 C 调用）；词法修正：数字后 `.` 仅当后接数字时解析为浮点，避免 `21.double()` 被吞成 FLOAT。
- 完成标志：带 trait 的 .x 能通过 typeck 并编译、运行（如 tests/trait/main.x、run-trait.sh）。✅

### 5.4 第四步：模块系统完善（7.3）✅ 阶段性完成

- 多文件编译：import 路径先试 lib_root，单段路径再试入口文件所在目录（entry_dir），实现同目录 import foo；库模块可不含 main（parser/typeck 放行）；codegen_library_module_to_c 生成带前缀的 C（类型/函数名如 foo_bar），与入口模块符号不冲突。
- 完成标志：多文件项目能正确编译与链接（tests/multi-file、run-multi-file.sh）。✅  
- 跨模块调用：入口可调用 import 的普通函数与泛型函数（typeck 在依赖中解析并在 dep 模块登记单态化实例，codegen 生成前缀调用与 extern 声明）。✅  
- 后续可扩展：export/可见性规则。

### 5.5 第五步：标准库扩展（7.4）✅ 阶段性完成

- 按 core/README 与 std/README 的「自举前必须」清单，实现各模块类型与最小实现；Option/Result/Vec 等依赖 7.1、7.2。
- 完成标志：自举前必须的 core/std 模块均可 import 并有最小可用实现。✅
- 当前：已新增 core.mem、core.option、core.result、core.slice、core.fmt、core.builtin、core.debug；core.types 与所有 std 模块均含至少一个占位函数或泛型占位，可 import 并参与编译/链接；tests/stdlib-import 与 run-stdlib-import.sh 验证多模块 import 与跨模块泛型调用。
- 后续：各模块内补齐真实类型与 API（如 Option\<T\>、Result\<T,E\>、Vec\<T\>、std.process.exit 等）。

5. **测试**：tests/generic/main.x（id\<T\>(x: T) -> T { x }、main 返回 id\<i32\>(42)）、run-generic.sh；Makefile test 纳入 run-generic.sh。 ✅
