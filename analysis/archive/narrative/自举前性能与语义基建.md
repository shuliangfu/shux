# 自举前性能与语义基建分析

> 本文档说明：在**尝试用 .x 自举编译器之前**，宿主编译器（用 C 实现的 shux 编译器）必须完成的**语义传递、零开销抽象与最小标准库**，以便自举后生成的代码能**超越 C**，而不是「另一个 C」。  
> 关联文档：《[需求分析](需求分析.md)》《[语法与类型设计-高性能与内存安全](语法与类型设计-高性能与内存安全.md)》《[控制流与语法构造清单](控制流与语法构造清单.md)》。

---

## 一、文档目的与范围

### 1.1 为何单独成文

- **自举是阶段闸门**：只有在「宿主编译器能正确编译用 .x 写的完整编译器、且该编译器能再次编译自身」时，才进入 M5 及之后的 .x 内迭代。在此之前，所有「超越 C」的能力都必须由**宿主编译器**在 typeck/codegen 中实现并传递给后端。
- **避免「只是另一个 C」**：若仅做源码到 IR/机器码的直译，没有别名信息、没有零开销 defer、没有边界检查消除与平台特化，生成代码难以在优化上超越手写 C。本文档把**自举前必须完成的性能与语义基建**收拢在一处，便于排期与评审。
- **与现有文档的分工**：
  - 《需求分析》：做什么产品、为什么、标准库全量清单、自举路线（§7）与里程碑（§9）。
  - 《语法与类型设计-高性能与内存安全》：语言层面的显式语义、别名/不可变/切片/错误处理等设计（§7、§8、§10）。
  - 《控制流与语法构造清单》：defer 静态内联算法、边界检查消除等控制流相关优化（§8）。
  - **本文档**：自举前的**阶段闸门清单**——上述设计中，哪些必须在自举前落地、落地到什么程度、与「超越 C」的对应关系。

### 1.2 范围与约束

- **读者**：编译器与语言设计参与者；评审「是否可启动自举」时对照本文档。
- **不重复设计细节**：各节仅概括「要做什么、做到什么程度、为何自举需要」，具体算法与类型规则见上述关联文档的对应章节。
- **项目目标与宗旨**：本文档与根目录 `.cursor/rules/00-项目目标与宗旨.mdc` 一致，不引入违背「性能比肩 C 甚至超越 C、内存安全、轻量级、全平台一套 API、编译自举」的假设。

## 二、核心语义：别名分析与不可变默认化

要在生成代码上超越 C，必须在 **typeck（类型检查）** 阶段就为后端提供比 C 更明确的**别名与可变性**信息，使 LLVM（或自研后端）能放心做自动向量化与指令重排。

### 2.1 唯一所有权 / 借用与 noalias

- **目标**：在自举代码中，编译器能识别哪些指针是**唯一**的（unique/owned 或单写借用），在生成 LLVM IR 时为这些指针打上 **noalias** 属性。
- **超越点**：C 中指针可任意重叠，优化器不敢做激进假设；.x 通过类型或注解标明「此处无其他指针修改该内存」，可直接开启 C 难以企及的**自动向量化**与**指令重排**（详见《语法与类型设计-高性能与内存安全》§7）。
- **自举前最低要求**：typeck 能区分「唯一/单写」指针与普通指针，并将该信息传递到 codegen；codegen 生成 LLVM IR 时对相应参数/返回值附加 `noalias`（或等价元数据）。不要求完整的生命周期标注，但**至少**支持《语法与类型设计》§8.3 所述的 **Scope-based Borrowing**（引用不超出定义它的花括号），以便在常见场景下推断无别名。

### 2.2 不可变性默认化（let / mut）

- **目标**：自举前必须支持 **let（不可变）** 与 **mut（可变）**，且默认不可变。
- **压榨点**：不可变数据可安全放入只读段、参与更激进的常量折叠与优化（《语法与类型设计-高性能与内存安全》§10 设计清单）。
- **自举前最低要求**：typeck 正确区分 let/mut，并在 IR 或后端可见的元数据中保留「只读」信息，便于后端做只读优化与常量传播。

### 2.3 交叉引用

- 别名与 noalias 的语义设计：《语法与类型设计-高性能与内存安全》§7（显式语义→极致性能）、§8.3（Scope-based Borrowing）。
- 类型层面清单：《语法与类型设计-高性能与内存安全》§10（压榨性能与安全的设计清单）。

## 三、零开销控制流：Defer 静态内联

C 中资源释放往往依赖手写 `goto cleanup`，易错且逻辑分散。.x 的 **defer** 必须在编译期**完全静态内联**，零运行时栈，同时保证所有出口（return、break、continue、块自然结束）都按逆序执行 defer。

### 3.1 算法要求

- **收集阶段**：Parser 进入块时为该作用域维护 **defer_stack**；遇到 `defer { ... }` 仅将代码块压栈，不生成任何运行时调用。
- **出口点注入**：
  - **return**：在求值返回表达式之后、真正返回之前，按**逆序**执行当前块及外层块的 defer。
  - **break / continue**：若跨越带 defer 的作用域，在跳转前逆序执行该作用域内 defer。
  - **块自然结束**：在块右大括号前逆序插入当前 defer_stack 中的代码。
- **代码膨胀控制**：当出口多、defer 多时，采用 **goto cleanup** 模式：统一跳到 `cleanup:` 标签执行所有 defer 再 return，避免在每个 return 处重复展开（详见《控制流与语法构造清单》§8.3）。
- **寄存器与 spill**：defer 注入时需考虑寄存器压力；若 defer 块体积或涉及变量超过阈值，应强制走 goto cleanup，避免多出口展开导致 spill（《控制流与语法构造清单》§8.7）。

### 3.2 自举前最低要求

- 宿主编译器在 **AST 变换阶段** 完成上述 defer 静态内联，**禁止**生成 `runtime_push_defer` 等运行时调用。
- 所有 return、break、continue、块出口均正确注入；嵌套作用域与「break/continue 跨越带 defer 的块」行为与文档一致。

### 3.3 交叉引用

- 完整算法与陷阱说明：《控制流与语法构造清单》§8.3（Defer 编译期静态内联算法）、§8.7（寄存器保存策略）。

## 四、边界检查消除（BCE）与值域分析

为兼顾**安全**（默认有边界检查）与**性能**（循环内不越界时达到 C 裸指针级别），自举前需实现初级的**值域分析（Range Analysis）**，在可静态证明不越界时**物理删除**边界检查指令。

### 4.1 目标

- 当 for 循环的索引范围与切片/数组长度关系可被**静态证明**（例如 `for i in 0..len` 且数组长度为 `len`）时，在 codegen 中**不生成**边界检查指令。
- **超越点**：安全子集内仍有默认保护，但在热点循环中消除分支与检查，为自动向量化（§8.2）与指令流水线扫清障碍（《控制流与语法构造清单》§8.8）。

### 4.2 自举前最低要求

- 在 **typeck** 或独立分析 pass 中，对简单循环（如 C 风格 for 或 `0..len` 范围）做值域分析，记录「索引变量在 0..len 内」的约束。
- **codegen**：当能证明某次切片/数组访问的索引恒在合法范围内时，不生成边界检查分支；否则保留检查（或按配置在 Release 下移除并由程序员负责）。

### 4.3 交叉引用

- 设计说明与「可证明的安全消除」：《控制流与语法构造清单》§8.8（静态可证明的边界检查消除）；《语法与类型设计-高性能与内存安全》§8.1（切片与值域分析）。

## 五、跨平台一套 API 与内建特化

为实现「全平台一套 API」且不损失性能，自举前需要**条件编译**与**内建特化**能力，使标准库与运行时内部能按目标平台选择实现路径，而不把 `#ifdef` 暴露给用户。

### 5.1 条件编译与平台选择

- **目标**：宿主编译器支持根据**目标三元组**（如 `os == "linux"`、`arch == "x86_64"`）在编译期选择不同代码路径。
- **语法或工具**：可采用 `@if(os == "linux") { ... } else @if(os == "windows") { ... }` 或等价的条件编译构造（《需求分析》§4.2；《控制流与语法构造清单》中条件编译为「未实现」项，自举前需达到「标准库内部可用」）。
- **压榨点**：平台相关 syscall 或 I/O 包装通过内联展开，消除跨平台抽象带来的间接调用开销。

### 5.2 内建函数（Intrinsics）与 SIMD 映射

- **目标**：为不同 CPU（x86_64、arm64 等）预留或实现**内建函数**映射，使标准库（如 core.simd）能生成目标平台最优的 SIMD 指令。
- **自举前最低要求**：至少具备「内建函数名 → 后端 intrinsic」的映射框架，使自举期使用的核心运算（如 memcpy、简单向量化）不依赖未实现的 intrinsic；完整 SIMD 库可在自举后补齐。

### 5.3 交叉引用

- 全平台一套 API 与条件编译：《需求分析》§4（全平台一套 API）、§4.2（条件编译）。
- 标准库 core 中的 builtin/simd：《需求分析》§5.1（core.builtin、core.simd）。

## 六、自举最小标准库（M5 准备）

在尝试自举前，宿主编译器必须能正确编译以下 **.x 模块**，且这些模块的实现满足「写编译器」所需功能。下表对应《需求分析》§5.4「自举前必须写的库」，此处按「自举前必须完成」做最小聚焦。

| 模块 | 自举前必须完成的功能 | 自举中的用途 |
|------|----------------------|--------------|
| **core.mem** | 指针运算、字节拷贝、对齐（size_of、align_of、offset_of）、安全封装与 volatile 读写 | 后端分配与布局、二进制与 IR 处理 |
| **std.vec** | 动态扩容数组（push/pop、索引、迭代） | 存储 AST 节点、Token 列表、符号表条目 |
| **std.map** | 高性能哈希表（get/insert/remove、迭代） | 变量名/函数名/类型定义等符号表查询 |
| **std.fs** | 文件打开、读、写、关闭与路径解析 | 读取 .x 源码、写出目标文件或 IR |
| **std.string** | 可变字符串与基本格式化 | 词法/语法分析中的缓冲、编译错误信息拼接与输出 |

- **说明**：core 中 **types、option、result、slice、fmt、hash、alloc** 等为类型与算法基础，自举前同样必须可用（详见《需求分析》§5.4 表格）；上表突出「与超越 C 及自举直接相关」的五个模块。
- **可自举后补齐**：网络、加密、完整并发、高级算法与集合等见《需求分析》§5.4「可自举后写的库」。

## 七、自举前性能生死线（三项）

若以下三项在自举前未满足，建议**推迟自举**，否则自举后的编译器在「性能比肩 C 甚至超越 C」上会留下结构性短板。

### 7.1 ABI 与 C 互操作零成本

- **要求**：.x 调用 C 函数（以及 C 调用 .x 生成的函数）时，**调用约定与寄存器/栈使用与 C 一致**，无额外装箱、封送或内存拷贝。
- **意义**：标准库与运行时大量依赖 C 库（如 libc、LLVM C API）；若 FFI 有额外成本，整体性能与「比肩 C」目标不符。

### 7.2 错误处理的寄存器化

- **要求**：多返回值（如 `(T, Error)` 或 Result 类型）在 codegen 中优化为**寄存器返回**（或等同的高效 ABI），而不是在内存中分配临时结构再拷贝。
- **意义**：错误处理为「显式、零成本」的一部分（《语法与类型设计-高性能与内存安全》§8.2）；若错误路径导致每次调用都写内存，会拖累整体性能与自举后编译器的表现。

### 7.3 编译期常量求值（CTFE）最小集

- **要求**：编译器能在**编译期**求值简单常量表达式（如字面量运算、常量折叠、简单类型与长度计算），不把可在编译期算完的逻辑留到运行时。
- **意义**：自举后的编译器自身会大量使用常量（如类型大小、对齐、表大小）；若 CTFE 缺失，编译器运行时会多做无谓计算，且不利于后续「更多逻辑编译期化」的扩展。

## 八、总结与自举前检查表

### 8.1 总结

- 自举前完成的**语义与优化**（别名/noalias、let/mut、defer 静态内联、边界检查消除、条件编译与内建特化）决定了自举后生成的代码能否在**自动向量化、指令重排、零开销控制流**上超越 C。
- 自举前完成的**最小标准库**（core.mem、std.vec、std.map、std.fs、std.string 及 §6 与《需求分析》§5.4 所列 core/std 子集）决定了能否用 .x 写出可编译自身的编译器。
- **三项性能生死线**（ABI 零成本、错误处理寄存器化、CTFE 最小集）未满足时，建议先补齐再启动自举，避免自举后陷入性能与架构债。

### 8.2 自举前检查表（勾选用）

实现或评审时，可在自举前逐项确认：

| 类别 | 项 | 状态 |
|------|----|------|
| **语义** | typeck 区分唯一/单写指针并传递至 codegen，生成 noalias（或等价） | ⬜ |
| **语义** | let/mut 正确实现，不可变信息可供后端优化 | ⬜ |
| **控制流** | defer 全量静态内联，无运行时 defer 栈；return/break/continue/块出口均正确注入 | ⬜ |
