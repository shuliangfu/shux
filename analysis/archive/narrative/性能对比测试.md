# SHUX × C × Zig 横向性能基准测试方案 (v1)

> **目标**：贯彻"性能优化高于一切"底层意志，直击编译器优化能力、内存吞吐、流水线效率与 CPU 缓存命中率，验证 SHUX 在系统级底层能否对标 / 超越 C 与 Zig。
> **状态**：方案定稿，待 Step 1 落地。
> **关联**：[`性能压榨.md`](性能压榨.md) §7（六层模型）、[`自举进度.md`](自举进度.md) §9（PERF 门禁）、`tests/bench/`（现有基准模式）。

---

## 0. 背景与定位

SHUX 定位为 C++/Zig 级系统语言，性能是生死线。本方案设计与 C / Zig 的横向性能基准，**直击编译器后端优化能力**：循环优化、自动向量化、内存布局、分支预测、缓存友好性。

**不做什么**：
- 不用 Web / 高层业务基准（不能反映编译器底层能力）；
- 不与 Mojo / MLIR 路线混淆（见 [`性能压榨.md`](性能压榨.md) §4，产品形态不同）；
- 不在自举未稳定时改 Lexer 热路径（L5 推迟至 M2 后）。

**与现有基建的关系**：本方案**扩展 `tests/bench/`**（已有 `loop_i32` / `mem_copy` / `simd_dot` 三组 `.x` + `.c` 对照模式），新增语言级负载，并对齐 PERF-001 门禁（SHUX asm median ≤ C -O2，再追 ≤ Zig -O2）。

## 1. 测试负载设计 (Benchmark Substrates)

负载覆盖系统语言的五个核心维度，每个信息密度极高。

### 负载 1：整数循环累加（已有 `loop_i32`）

| 项 | 内容 |
| --- | --- |
| **维度** | 基本算术 + 循环优化 |
| **规模** | 100,000,000 次循环累加 |
| **考核** | 循环展开、归纳变量分析、常量折叠防御 |
| **现状** | `tests/bench/loop_i32.{x,c}` 已存在 |

```x
function main(): i32 {
  let n: i32 = 100000000;
  let i: i32 = 0;
  let sum: i32 = 0;
  while (i < n) { sum = sum + i; i = i + 1; }
  return sum;
}
```

`.c` 版用 `__asm__ volatile("" : "+r"(sum) : : "memory")` 阻止 `-O3` 常量折叠。

### 负载 2：内存拷贝吞吐（已有 `mem_copy`）

| 项 | 内容 |
| --- | --- |
| **维度** | 内存带宽 + 连续布局 |
| **规模** | 4096 字节缓冲区，1,000,000 次拷贝 |
| **考核** | 连续内存访问、`memcpy` 识别与向量化、边界检查开销 |
| **现状** | `tests/bench/mem_copy.{x,c}` 已存在 |

### 负载 3：SIMD 点积（已有 `simd_dot`）

| 项 | 内容 |
| --- | --- |
| **维度** | 自动向量化 + 循环尾部剥离 |
| **规模** | **1,000,007** 个 Vec4f 点积累加（非 4/8 整除，强制考核 Loop Remainder） |
| **考核** | SIMD 指令编码（SSE/AVX2/Neon）、`@shuffle`/`@select` codegen 质量、尾部标量收尾代码生成 |
| **现状** | `tests/bench/simd_dot.{x,c}` 已存在（规模须由 1M 调整为 1,000,007） |

### 负载 4：结构体数组遍历（新增 `struct_arena`）

| 项 | 内容 |
| --- | --- |
| **维度** | 缓存命中 + 布局 + 内存对齐 |
| **规模** | 100,000 个结构体，遍历累加某字段 |
| **考核** | AoS 布局下 L1/L2 cache miss、region/arena 显式内存控制、**非对齐结构体的 Padding 策略** |
| **对比要点** | 三者强制等价连续布局；SHUX region 优势单独列项说明 |

**必须测试两组结构体**：
- **A 组（完美对齐）**：`Point { x,y,z,w: i32 }`（16 字节，天然对齐）—— 考核峰值吞吐；
- **B 组（非幂等大小）**：如 `Node { tag: i8; val: i32; flag: i8; ptr: i32; extra: i8 }`（裸 11 字节，Padding 后 12 字节）—— 考核编译器 Padding 策略与非对齐访问惩罚。

`.x` 骨架（A 组）：

```x
// struct_arena.x — 负载4：结构体数组遍历，考察缓存命中
struct Point { x: i32; y: i32; z: i32; w: i32; }

function main(): i32 {
  let N: i32 = 100000;
  let pts: Point[N] = [];
  let i: i32 = 0;
  while (i < N) { pts[i].x = i; pts[i].y = i; pts[i].z = i; pts[i].w = i; i = i + 1; }
  let sum: i32 = 0;
  i = 0;
  while (i < N) { sum = sum + pts[i].x; i = i + 1; }
  return sum;
}
```

**B 组对齐锁死**：三者须通过显式对齐属性（C `__attribute__((aligned(X)))`、Zig `extern struct` + `align`、SHUX 对应语法）保证**绝对等价的物理 Padding 步长**，防御因 Padding 策略差异导致的非对齐访问惩罚失真。

### 负载 5：分支密集（新增 `branch_sort`）

| 项 | 内容 |
| --- | --- |
| **维度** | 分支预测 |
| **规模** | 1,000,000 个元素，对随机 / 已排序两组数据做条件分支遍历 |
| **考核** | 分支预测器友好性、条件移动（cmov）生成、`-O3` 下条件化重写 |
| **对比要点** | 随机数据 branch-miss 高、已排序低；三者差异反映 codegen 分支化简能力 |

**RNG 同源约束**（关键）：严禁调用各语言标准库 RNG（C `rand()` / Zig `std.rand.DefaultPrng` / SHUX std）——分布熵不同会导致 TAGE 预测器命中率差异，使 branch-miss 对比失真。三者必须**内联实现完全相同的 LCG**（同算法、同常数、同 seed），运行时现场生成 1,000,000 个元素：

```c
// 三者等价实现的 LCG（以 C 示意，.x / .zig 须逐行对译）
uint64_t rng_state = 0x9E3779B97F4A7C15ULL;  // 固定 seed
static inline uint64_t lcg_next() {
  rng_state = rng_state * 6364136223846793005ULL + 1442695040888963407ULL;
  return rng_state;
}
```

**进程级隔离约束**（关键）：随机 / 已排序两组测试**严禁在同一进程内顺序执行**——TAGE 分支预测器的 BHT 残余状态会跨循环污染，劣化第二轮预测。必须设计为 argv 切换的两条独立 CLI 路径（`--mode=random` / `--mode=sorted`），Runner 以**独立进程**分别拉起，物理清空硬件预测器状态。

## 2. 测试环境与约束 (Strict Constraints)

为防止编译器"作弊"或配置不对称导致数据注水，锁死以下物理边界：

### 2.1 硬件平台

- 统一在 **M5 Max（ARM64）** 或指定 **x86_64** 物理服务器上跑；
- **锁单核**：`taskset -c 1`（Linux）或 `pthread_setaffinity_np`，macOS 用 `thread_policy_set` 设置 affinity，排除多核调度抖动；
- **Pin CPU 频率**（若可解锁）：避免 DVFS 引入噪声；不可解锁则记录室温/热状态。

### 2.2 编译器激进等级

| 语言 | 选项 | 备注 |
| --- | --- | --- |
| **C** | `clang -O3 -march=native` | 整数负载不加 `-ffast-math`；浮点负载（如 `simd_dot`）单独标注 `-ffast-math` 分支 |
| **Zig** | `zig build-exe -O ReleaseFast` + `@setRuntimeSafety(false)` | 启用 LLVM 全优化；SIMD/循环核心路径须显式关闭边界检查并改用裸指针步进，否则 Zig 在 ARM64 上多生成 `B.HS` 边界分支，阻断 NEON FMLA 流水线，使对比变成"安全开销"而非"codegen 质量"。次轮可保留 safety=on 报告真实世界开销 |
| **SHUX** | 自研后端最高等级（循环展开 + DCE + autovec + simd_loop peel） | 记录所用 codegen 路径（asm 直出 / cc 回退） |

### 2.3 内存测量口径

- **严禁统计平均值**；
- 统一用系统调用精确捕捉 **Peak RSS**：Linux `getrusage(RUSAGE_SELF)`，macOS `task_info(MACH_TASK_BASIC_INFO)`；
- 采样多次取中位数 + P99，报告原始分布而非单点。

### 2.4 测量噪声控制

- **Warmup**：前 3 次运行丢弃，避免冷 cache / 页表未预热；
- **重复**：每负载 ≥ 20 次，报告 median / P99；
- **后台静默**：关闭无关进程，记录运行期间系统负载基线。

## 3. 监控指标与工具对齐

| 维度 | 核心监测项 | 采集工具 (Linux / macOS) | 胜负判定 |
| --- | --- | --- | --- |
| **时间开销** | Wall Time / CPU Time | `clock_gettime(CLOCK_MONOTONIC)` / `mach_absolute_time` | 越低越好（微秒级） |
| **缓存效率** | L1-dcache-loads / misses、LLC misses | Linux `perf stat -e`；macOS **`kpc` / `m1counters` 直读硬件寄存器**（非 `xctrace` 采样） | Miss 率越低，证明物理布局与重排能力越强 |
| **流水线** | Branches / Branch-misses | `perf stat -e branches,branch-misses` | Miss 率低，证明条件分支对硬件预测器友好 |
| **内存** | Peak RSS | `getrusage` / `task_info` | 越低越好 |

> **公平性前提**：cache miss 与 branch miss 的横向对比**只在三者使用等价连续布局时成立**。SHUX 的优势应体现在 region/arena 自动保证布局，而 C/Zig 需手动确保等价布局——这一差异本身是 SHUX 的卖点，但须显式记录而非混入算法差异。
>
> **macOS 计数器精度**：`xcrun xctrace` 是采样工具，对毫秒级 bench 无法给出精确绝对值，且解析 `.trace` 文件耗时数秒，破坏 CI 敏捷性。M5 Max 须用 `kpc`（kernel performance counter）系统调用或 `m1counters` 开源包装，直读 `L1_D_CACHE_MISS` 寄存器。注意：`kpc` 需 root / 专属 entitlement，且默认系统级计数——须配合 §2.1 锁核隔离单进程噪声。

## 4. 测试自动化流水线 (Execution Pipeline)

不引入第三方测试框架，写极简 Shell / Python Runner，与现有 `tests/run-perf-baseline.sh` 风格一致。

```
[Test Runner]
    ├─── 执行 SHUX 二进制 ───> 采样 ───> JSON 日志
    ├─── 执行 C 二进制    ───> 采样 ───> JSON 日志
    └─── 执行 Zig 二进制   ───> 采样 ───> JSON 日志
                            ↓
                    汇总 → tests/baseline/lang-perf.tsv
                            ↓
                    生成雷达图 / 柱状图
```

**JSON 日志 schema**（每负载每语言每轮一条）：

```json
{
  "load": "loop_i32" | "mem_copy" | "simd_dot" | "struct_arena" | "branch_sort",
  "lang": "shux" | "c" | "zig",
  "run": 7,
  "wall_us": 1234567,
  "cpu_us": 1234000,
  "peak_rss_kb": 24576,
  "l1_dcache_miss_rate": 0.032,
  "llc_miss_rate": 0.011,
  "branch_miss_rate": 0.004,
  "codegen": "asm" | "cc",
  "host": "m5max-arm64",
  "ts": "2026-07-04T10:00:00+08:00"
}
```

Runner 输出汇总 TSV 至 `tests/baseline/lang-perf.tsv`，对齐 PERF-001 现有 `tests/baseline/zig-perf.tsv` 模式。

## 5. 落地执行三步走

### Step 1：补齐 C 和 Zig 对照基准

- 现有 `loop_i32` / `mem_copy` / `simd_dot` 已有 `.c` 版，补齐对应 `.zig` 版；
- 新增 `struct_arena.c` / `struct_arena.zig`、`branch_sort.c` / `branch_sort.zig`；
- 沿用现有 `loop_i32.c` 的 `__asm__ volatile` 防 const-folding 模式；
- 要求三者使用**等价连续布局**，保证 cache/branch 指标可比。

### Step 2：用 SHUX 编写相同逻辑

- `loop_i32.x` / `mem_copy.x` / `simd_dot.x` 已存在，新增 `struct_arena.x` / `branch_sort.x`，逻辑与 C/Zig 一一对应；
- 这是 SHUX 编译器的极限大压测：若编译报错或机器码有冗余，**根据测试结果反向修编译器后端优化器**（autovec、simd_loop、peephole）；
- 记录每次 compiler 修复对应的 bench 变化，写入 commit message。

### Step 3：横向对比图表 + 门禁接入

- Runner 跑完后吐出 `tests/baseline/lang-perf.tsv`；
- 生成雷达图（4 维：Wall / Cache Miss / Branch Miss / RSS）与柱状图；
- **接入 PERF-001 门禁**：SHUX asm median ≤ C -O2（硬门禁），再追 ≤ Zig -O2（软目标）；
- 若 SHUX 在 Cache Miss 上击败 C/Zig → 证明 region/arena 物理布局在底层完胜；
- 若 SHUX 在 Branch Miss 上完胜 → 证明条件化简成功引导 CPU 流水线优化。

## 6. 验收与门禁

| 门禁 | 标准 | 失败动作 |
| --- | --- | --- |
| **PERF-001 硬门禁** | SHUX asm median ≤ C -O2（每负载） | CI 红，阻塞合入 |
| **PERF-001 软目标** | SHUX asm median ≤ Zig -O2 | 记录 gap，不阻塞 |
| **回归门禁** | bench 较 `tests/baseline/lang-perf.tsv` 退化 > 5% | CI 红，须 `SHUX_PERF_UPDATE_*` 刷新基线 |

**禁止**：无 baseline 的"感觉更快"合入；**禁止**为过 gate 改 bench 用例语义（与 [`性能压榨.md`](性能压榨.md) §7.2 L6 一致）。

## 7. 风险与回避

| 风险 | 回避措施 |
| --- | --- |
| 编译器常量折叠 / DCE 掉整个循环 | **三语言均须防御**：C/Zig 用 `__asm__ volatile`（现有 `loop_i32.c` 已验证）；**SHUX 侧**须用运行时输入（argv / 环境变量）或编译期屏障 builtin 阻止常量折叠，且 **Runner 必须外部校验 main 返回值**与预期一致——既验证正确性，又迫使编译器真实生成循环机器码，杜绝 Wall Time = 0 的虚假神话 |
| 测量噪声（调度 / DVFS / 温度） | 锁核 + warmup + ≥20 轮 + 报告 P99 + 记录环境基线 |
| 布局不对等导致 cache 对比失真 | 三者强制等价连续布局；SHUX region 优势单独列项说明 |
| 平台差异（ARM vs x86） | 每平台独立 baseline，不跨平台比；记录 `host` 字段 |
| SHUX 自举未完成导致 bench 不可跑 | 优先用 cc 回退后端跑出 C-cc 对照，asm 路径稳定后再补；记录 `codegen` 字段 |
| `-ffast-math` 误用 | 整数负载禁用；浮点负载单独分支标注 |
| 分支预测器跨模式污染（负载 5） | 随机 / 已排序测试须独立进程拉起（argv 切换），禁止同进程顺序执行，物理清空 BHT 残余 |
| Zig safety 阻断 NEON FMLA（负载 3） | 核心循环显式 `@setRuntimeSafety(false)` + 裸指针步进；次轮保留 safety=on 对照真实世界开销 |
| macOS 硬件计数器精度不足 | 弃用 `xctrace` 采样；改用 `kpc`/`m1counters` 直读寄存器，须 root + 锁核隔离 |
