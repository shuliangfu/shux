## 0. 核心命题（读这一节就够）

Shux 的设计方向可以概括为一句话：

```text
把能在编译期证明的事，全部压进 typeck；
把必须在运行期做的事，压到最少、最可预测的 syscall / 寄存器路径；
证不出 → 保守回退，禁止 silent wrong code。
```

这与「运行时堆 GC / 隐式分配 / 全指针 restrict / 编译器盲目 autovec」等路径刻意相反。目标不是「业务妥协型通用语言」，而是 **系统级、零 GC、直出 ASM、可 freestanding** 的语言栈；但 **不等于** 所有程序在任何路径上都能做到「0-malloc、0-syscall、100% SIMD」——那是 **特定子集 + 特定 gate 绿** 下的可验收结果，不是默认保证。

**进度口径（2026-06-23）**：

| 里程碑 | 含义 | 状态 |
|--------|------|------|
| D+E+F v1 | 黄金自举 + E-soft 编译器 + std 零 C 源码 | ✅ |
| 终局 G | 物理删尽 `.c/.h`、`-nostdlib` 硬绿、`build.x` 替代 Makefile | 🟡 ~20% |
| F-no-libc | Linux bootstrap 不链 libc | 🟡 NL-07 推进中 |

详见 [`自举进度.md`](自举进度.md) §2。

## 1. 设计哲学：编译期证明 ↔ 运行期解放

### 1.1 三层分工

| 层 | 职责 | 典型机制 | 运行期成本 |
|----|------|----------|------------|
| **Typeck** | 拒错、域、move、noalias 证明 | region、Linear(T)、AL-06、scope borrow | **0** |
| **Codegen** | 在证明成立后删检查、发 SIMD、SROA | BCE v1、autovec、栈晋升 | **0**（相对未优化基线） |
| **Runtime / std** | 不可避免的 OS 交互 | syscall、io_uring / kqueue / IOCP | **有界、可审计** |

**关键边界**：安全子集内的「自动」= **IR 可见的静态展开**（defer、cleanup 链、Arena deinit），不是 GC/RC 追踪。块外 API、FFI 边界仍 **显式** `Allocator`（与 Zig 同级抽象，块内 Shux 更省事）。

### 1.2 与常见表述的纠偏

| 宣传式说法 | 务实表述 |
|------------|----------|
| 「malloc = 0 的奇迹」 | **`with_arena` 热路径** gate 已验 0 malloc；默认堆、`owned` 长生命周期仍走显式分配 |
| 「0-syscall 极致并发」 | io_uring **批量化、减少 syscall**；connect/accept/read 仍经内核，不是零 syscall |
| 「100% 裸奔 SIMD」 | loop autovec **v1/v2 有 gate**；SoA 列向量化为 **v3 路线**，非全仓库默认 |
| 「F-09 白名单为空」 | **std ✅**；**compiler 仍 226 `.c`**（E-soft）；终局 G-04 未闭合 |
| 「只等 Stage 2 点火」 | **D v1 ✅**；当前阻塞在 **G-06 LSP seed、d03 哈希、G-02 物理删 C** |

## 2. 五支柱深度拆解

### 2.1 编译时自动内存管理（零 GC，零运行时追踪）

**设计**：安全与释放收敛为 **作用域效应（Scope Effects）** —— 借用域（region）、分配域（`with_arena` / `#[alloc]`）、释放域（defer / `owned T`）叠加，typeck 闭环，不进 ABI 元数据表。

| 机制 | 设计意图 | 验收 / 状态 |
|------|----------|-------------|
| **L1 `with_arena(cap)`** | 块内隐式 threading `Allocator`，单态化后等价手写传参 | ✅ `run-with-arena-vec-gate.sh`、`run-perf-alloc-hotspot.sh` |
| **`owned(T)` + cleanup** | early return / break → goto cleanup 链 | ✅ `run-owned-return-gate.sh` |
| **SROA / 小 struct 拍平** | ≤64B、不逃逸 → 寄存器/栈标量 | ✅ SROA v1；**ASP（Arena→栈晋升）** 为增强项 |
| **CTFE 折叠** | 编译期常量链 | ✅ 部分；大整数见 `run-i64-ctfe-gate.sh` |
| **L0 默认堆** | 公共 API 仍显式 | ✅ `std.heap` + `*_deinit` |

**相对 Zig**：边界同样显式；**块内**少写 allocator 参数；**安全**上多 region/Linear 拒错。性能 closure 目标 **≥ Zig**，非更慢。

**未闭合 / 增强**：SoA 布局自动切换（§2.3 v3）；跨模块逃逸的更强域分析；编译器自身 Lexer/typeck 仍大量 Arena（见 [`自举进度.md`](自举进度.md) §9）。

---

### 2.2 编译期安全（静态拒错，非运行时防御）

**设计**：不在 Release 热路径堆 `assert` 式防御；能证明的删，证不出的 **编译错误** 或 **保留检查**。

| 机制 | 管什么 | Gate | 状态 |
|------|--------|------|------|
| **region + slice 域** | 视图不逃出 `L` | `run-typeck-region.sh` | ✅ |
| **Linear(T)** | double-free、use-after-move | `run-typeck-linear.sh` | ✅ |
| **AL-06** | slice + Allocator 双重逃逸 | `run-al06-gate.sh` | ✅ |
| **scope borrow** | 悬垂借用 | `run-scope-borrow-gate.sh` | ✅ |
| **BCE v1** | `for i in 0..len` 内下标合法 | codegen release 删守卫 | ✅ |
| **精细 noalias** | 仅 proof bit 置位的 disjoint 块 | 替代全指针 `restrict` | ✅ 主路径；粗 `*T→restrict` 仍待收窄 |

**因果**：noalias + BCE 是 **autovec 与指令重排的许可证**（见 [`安全路线`](安全路线.md) §3）。证不出 → 标量 + 保留边界检查，**禁止** silent 错误优化。

---

### 2.3 别名证明驱动的自动向量化（Loop Autovec）

**设计**：SIMD 不是「猜循环能向量」，而是 **noalias ∧ BCE（+ 无 call 等）→ autovec_eligible → 平台 intrinsic**。

```text
typeck: disjoint + in-bounds
    → autovec_eligible = true
    → codegen: SSE2 / AVX2 / NEON（按 target）
    → 否则: 标量 + 可选诊断
```

| 版本 | 范围 | 状态 |
|------|------|------|
| **V1** | 双指针、无 call 的基础 `for` | ✅ `run-autovec-loop-gate.sh` |
| **V2** | `while i < n` 下 f32 reduction / 点积 | ✅ `run-std-simd-autovec-strategy-gate.sh` |
| **V3 SoA** | AoS → 字段列、stride/gather | 🟡 策略与 bench 在；**非默认全模块** |
| **显式 SIMD** | `Vec4f`、`@shuffle`、`@select` | ✅ STD-S3/S4 |

**不宜承诺**：「所有数值循环自动 SIMD」——需满足 gate 条件；SoA v3 面向数值/量化场景，与通用业务循环无关。

---

### 2.4 独立性：无 libc、斩断 C 脐带（Freestanding）

**设计**：默认不依赖臃肿 libc；`std/sys/*` + 条件编译直抵 OS；用户一套 API，平台差异收进 std 内部。

| 平台 | 路径 | 现状 |
|------|------|------|
| **Linux** | 内联 asm `syscall`；`std.io` io_uring 批 accept/connect/read | ✅ 主链；multishot / provided buffer 部分 ✅（见 [`std.net性能压榨与超越Zig.md`](std.net性能压榨与超越Zig.md)） |
| **macOS** | 薄 libSystem + kqueue / mmap | ✅ 开发机常用；B-strict 与 Linux 金标准有 parity 口径 |
| **Windows** | IOCP 完成端口 | 🟡 门面 / hybrid；非 Linux 同级硬绿 |

**阶段对照**：

| 阶段 | 含义 | 2026-06-23 |
|------|------|------------|
| **F（F-ZC）** | `std/` **0 手写 `.c/.h`** | ✅ |
| **E-soft** | 编译器 **运行时**不链前端 C | ✅ |
| **F-no-libc** | bootstrap **`-nostdlib`**、`ldd` 无 libc | 🟡 G-03 |
| **G 终局** | compiler **226 `.c` 物理删除** + 零 seed 冷启动 | ⬜ / 🟡 |

**纠偏**：「不调用 C 的 malloc/fopen」指 **Shux 程序默认路径**；仓库 **冷启动 seed、Makefile cc** 在 G 完成前仍存在 C 工具链——与「用户程序 freestanding」分层理解。

---

### 2.5 零运行时抽象成本 + 自举终局

#### 2.5.1 零成本单态化与 `#[cfg]`

跨平台 async/IO 在 **编译期** 被 `#[cfg(target_os)]` + monomorphization 拍平为单平台实现——Linux 产物是 io_uring 路径，macOS 是 kqueue/mmap 路径，**无** libuv 式运行时 vtable 中间层（在 std 设计内；用户若自己引入 C 库则另论）。

#### 2.5.2 自举：D+E+F 与 G 的区别

| 层次 | 判据 | 状态 |
|------|------|------|
| **语义自举** | `make bootstrap-verify` | ✅ |
| **黄金自举 D** | Stage1 SHA256 == Stage2 | ✅（Linux x86_64 金标准） |
| **编译器无 C E** | 默认 bootstrap 不 cc 前端 C | ✅ E-soft |
| **std 无 C F** | `std/` 0 `.c` | ✅ F-ZC |
| **完全自举 D+E+F** | 三者齐备 | ✅ v1 |
| **终局 G** | 全仓库物理零 C、`-nostdlib` 硬绿、F-09 STRICT 空 | 🟡 ~20% |

**A-09 Stage2 哈希** 是 **D 的硬门禁**，不是「尚未点火」——当前工作重心在 **G-06 seed（尤其 LSP）、G-02 删 C、G-03 no-libc**。

## 3. 安全 ↔ 性能统一因果链

```text
region / Linear / AL-06 / scope borrow
        ↓
   精细 noalias + BCE
        ↓
   autovec_eligible / 指令调度 / SROA
        ↓
   with_arena → 热路径 0 malloc（可 bench）
        ↓
   Release 用户程序：更少分支、更少分配、更多向量指令
```

**铁律**（与 [`安全路线`](安全路线.md) 一致）：证不出 → **不** autovec、**不** 删边界检查；`unsafe {}` 内行为文档化，等同 C 责任。

## 5. 一句话画像（修订版）

**Shux** 是以 **编译期 typeck 证明** 换取 **运行期零 GC、可证热路径零 malloc、可证循环 SIMD 化** 的系统语言；通过 **region/Linear/noalias/BCE/arena** 把安全与优化绑在同一条因果链上。它不是「什么都自动极致」的魔法，而是 **gate 可复验** 的工程栈——**D+E+F v1 已闭合**，**终局 G 与 F-no-libc** 仍在推进；在此之前，表述应区分 **「设计目标 / 已验收子集 / 路线中」** 三档。
