# 补丁与胶水清单 — 待从源头去除

本文档列出**曾存在的**生成后修补与胶水追加，及「从源头解决问题」后的**当前状态**。  
（std.io 的 io_externs 已去除：改为 std/io/io_abi.h + #include。）

**当前状态小结**：清单内列出的补丁/胶水均已去除或收缩——preprocess/driver/parser_export 已去除；pipeline 仅保留「可选插入 parser_parse_into extern + 追加一行 #include "pipeline_glue.c"」；runtime.c 不再追写整份 pipeline_glue.c，改为输出该 #include。无剩余「待去除」项；下文为历史与实现说明。

**澄清（是否「全部去掉」）**：  
- **生成后文本替换类补丁**（sed、整块 replace、追写大段 C）：已全部去掉。  
- **仍保留的**：① pipeline 的「一行 extern 插入 + 一行 #include 追加」；② driver 的「读回写回」（内容不改）；③ **胶水代码** `pipeline_glue.c` 整文件仍在，通过 `#include` 参与编译（包装、sizeof、debug 等）。若目标为「零补丁、零胶水」，需将 pipeline_glue.c 内逻辑迁入 .x 或独立 C 模块+头文件，并去掉上述两处最小补丁。

**runtime.c 中的 C 桩常量（std_fs_stub_c / std_process_stub_c）**：  
曾在 `SHUX_USE_SX_DRIVER && SHUX_USE_SX_PIPELINE` 下定义，为未使用的死代码，**已删除**。std/process/process.c 中对应说明已同步去掉。

## 三、其他提及但非「文本补丁」的用法

- **asm 中的 patch**：`elf_add_patch`、`elf_resolve_patches` 等为**指令级重定位/回填**（jz/jmp/call 的偏移），属于正常代码生成流程，不是「对生成 C 的文本补丁」，无需去除。
- **typeck/codegen 注释中的「从源头消除 sed」**：已从类型推断/生成侧消除对 sed 的依赖，无额外补丁列表。

## 四、建议去除顺序（按依赖与风险）

1. **preprocess codegen**：对 slice 指针生成 `->`，去掉 `build_patch_preprocess_gen_c` 中的 `.`→`->` 替换（以及 build_runtime_sx.x 中对应逻辑）。
2. **driver_gen.c**：main.x/codegen 生成时即用 `->` 和正确的 `preprocess_preprocess_sx_buf` extern；并让 `run_compiler_c` 由生成或 ABI 提供，去掉 `build_patch_driver_gen_c` 的替换与追写。
3. **pipeline_gen.c**（已做）：codegen/pipeline 已从源头产出 _impl、get_ndep()、slice→指针；build 仅「可选插入 parser_parse_into extern + 追加 #include "pipeline_glue.c"」。
4. **pipeline_glue.c**（已做）：改为由 pipeline_gen.c 末尾 `#include "pipeline_glue.c"` 引入，不再追写整份内容。
5. **parser_export**：确认 -E-extern 路径全覆盖后，删除 `build_patch_parser_export`，或改为仅文档说明「非 -E-extern 时由 parser 侧产出 ABI」。
