## 二、分析目录文档综述（与自举相关的结论）

以下为从 analysis 目录各文档中提炼的、与自举直接相关的结论，便于在实现时对照。

### 2.1 需求与路线（需求分析.md）

- **自举路线**：先用宿主语言（C）实现编译器 → 用 .sx 重写前端 → 用 .sx 重写中端与后端 → 自举验证；自举对应里程碑 **M5**。
- **自举前必须的标准库**：core（types, mem, alloc, option, result, cmp, slice, array, fmt, iterator, str 基本, hash, builtin, debug）；std（runtime, heap, io, fs, path, string, vec, map, process 的 exit+args, error 基础）。缺一不可。
- **可自举后写的库**：网络、加密、完整并发、高级算法与集合、完整 std 扩展等，见需求分析 §5.4。

### 2.2 构架与目录（构架分析.md）

- **编译器流水线**：源码 → 词法 → 语法 → AST → 类型检查 + 泛型单态化 → IR 生成 → 优化 → 代码生成 → 目标码；前端（lexer/parser/ast/typeck）、中端（IR、优化）、后端（codegen）分阶段，便于自举时分步用 .sx 替换。
- **自举前后差异**：自举前 compiler/ 为 C 源码，由系统 C 编译器构建；自举后 compiler/ 全部为 .sx，由上一代 shux 编译；构建命令从 `make` 编译 C 变为「shux 编译 compiler/*.sx + 指定 core/std 根路径」。
- **标准库**：core/ 与 std/ 扁平化置于仓库顶层，以 .sx 源码形式存在，按需编译与链接；编译器内置标准库根路径，不把标准库打进编译器二进制。

### 2.3 自举前必须清单（自举前必须清单.md）

- **核心路径（三步顺序）**：  
  1）**锁定内存契约**：与 C 互操作、struct 默认对齐、Packed 显式、Slice 固定 (ptr, len)，文档化并可验证。  
  2）**定义 I/O 接口协议**：std.io 的 Reader/Writer、ReadOnlySlice/WriteOnlySlice、超时预留；std.mem Buffer(ptr, len, handle?)；std.io.driver 占位。  
  3）**完善编译器诊断**：带「面包屑」的错误（文件:行:列、期望类型 vs 实际类型、出错子表达式）。
- **与阶段 9 的对应**：9.1 用 .sx 写前端依赖 L1–L5、C1–C2、S1–S4；9.2 用 .sx 写中端与后端依赖 ABI/Result、I/O 协议与诊断；9.3 自举验证依赖全部检查项通过。

### 2.4 自举语言子集（自举语言子集.md）

- **约定**：自举期间**少动语法与类型**；仅允许小步扩展（修复遗漏、与文档一致的小补丁）。**新语法、新类型、新关键字**在自举验证通过后再加。
- **当前子集**：关键字（function, return, let, const, if, else, while, for, loop, break, continue, match, struct, enum, trait, impl, import, extern 等）；类型（内建整数/浮点/bool、指针、数组、切片、struct/enum）；表达式与语句（字面量、算术、比较、条件、match、调用、块与循环）；顶层定义（import、struct、enum、trait、impl、function、extern）；多文件与泛型（import 解析、泛型函数/结构体、单态化）。具体以当前 shux 实现与测试为准。

## 四、自举实现路线（9.1 / 9.2 / 9.3）

与 README 阶段 9 一致，自举分三步实施；每步的产出与验收标准如下。

### 4.1 阶段 9.1：用 .sx 写编译器前端

**目标**：将 lexer、parser、ast、typeck、以及（若存在）从 AST 到 IR 的生成，改为 .sx 实现；宿主编译器（C 版 shux）编译该 .sx 前端，与**现有 C 后端**或占位后端联调，形成「.sx 前端 + C 后端」的混合编译器。

**范围**：

- **lexer**：.sx 源码 → Token 流（与现有 token 类型、关键字、字面量一致）。
- **parser**：Token 流 → AST（与现有 AST 结构一致或可映射，便于后端消费）。
- **ast**：AST 数据结构用 .sx 定义（struct/enum），与 C 后端约定好内存布局或通过 C 导出接口传递。
- **typeck**：基于 AST 做类型检查、泛型单态化、符号表；输出带类型 AST 或等价 IR 输入。
- **IR 生成（若当前有独立步骤）**：由带类型 AST 生成供后端使用的 IR 或直接生成当前 codegen 可接受的中间表示。

**联调方式**（二选一或组合）：

- **方式 A**：.sx 前端产出与现有 C 后端一致的中间表示（如生成的 C 或内部数据结构），通过 FFI 或约定好的 C 接口把「前端结果」交给现有 C 的 codegen/driver 使用。
- **方式 B**：.sx 前端直接生成当前 codegen 的输入格式（如某种序列化或内存布局），由 C 后端读入并继续 codegen + 调用 cc。

**验收标准**：

- 宿主编译器能编译「.sx 前端」源码（含其依赖的 core/std 自举子集），得到可执行文件或库。
- 该可执行文件（或与 C 后端链接后的整机）能对若干示例 .sx（含多文件、泛型、trait）完成「词法→语法→类型检查→（至 IR 或至 C 后端入口）」并得到与当前 C 版 shux 一致或可对比的结果（如同样能生成可编译的 C 或同样报错）。

**依赖**：阶段 7 的 .sx 子集 + core/std 自举子集；自举前必须清单中的语言、标准库、诊断项。

---

### 4.2 阶段 9.2：用 .sx 写中端与后端

**目标**：将优化、codegen、driver 改为 .sx 实现；整机「.sx 编译器」由宿主编译器（或上一代 .sx 编译器）编译，能编译任意 .sx（含自身源码）。

**范围**：

- **中端**：泛型单态化（若未完全在前端）、内联、DCE、常量传播等；输入为前端产出的带类型 AST 或 IR，输出为优化后 IR 或可直接生成目标码的表示。
- **后端（codegen）**：将 IR 或等价表示生成 C 代码（或未来 LLVM IR/机器码）；调用系统 cc 生成目标文件/可执行文件。
- **driver**：命令行解析、文件列表、依赖 core/std 路径、调用中端与后端、调用链接器（cc）；错误汇总与退出码。

**数据流**：.sx 源码 → [.sx 前端] → 带类型 AST/IR → [.sx 中端] → 优化后 IR → [.sx 后端] → C 或目标码 → cc/链接 → 可执行文件。

**验收标准**：

- 用宿主编译器（或上一代 shux）编译「完整 .sx 编译器」源码，得到新 shux。
- 新 shux 能编译任意 .sx 程序（含多文件、import core/std、泛型、trait），行为与当前 C 版 shux 一致（同一输入得到可运行程序或相同错误）。
- **关键**：新 shux 能编译**自身源码**（compiler 目录下全部 .sx），产出下一代 shux 可执行文件。

**依赖**：9.1 完成；自举前必须清单中的 ABI、Result 寄存器化、I/O 协议（若编译器内部用 std.io/std.fs 等）。

---

### 4.3 阶段 9.3：自举验证

**目标**：确认「.sx 编译器编译自身」的闭环成立，且两代编译器行为一致。

**动作**：

1. 用宿主编译器（C 版 shux）编译「.sx 编译器」源码，得到 **shux₁**。
2. 用 **shux₁** 编译「.sx 编译器」源码，得到 **shux₂**。
3. 用 **shux₂** 编译「.sx 编译器」源码，得到 **shux₃**（可选，用于进一步确认稳定性）。
4. 用 shux₁ 与 shux₂（及可选 shux₃）分别编译同一批测试用例（run-all.sh 或 make test），对比输出与退出码，**两代行为一致**。

**验收标准**：

- shux₁、shux₂ 均能成功编译自身源码，无未解决错误。
- 对全量测试套件，shux₁ 与 shux₂ 的通过/失败一致，无回归。
- 文档或 CI 中记录「自举验证通过」及所用命令与哈希，便于日后回归。

**依赖**：9.2 完成；测试套件完整且稳定。

### 4.4 C 源码的保留与删除时机

自举过程中 **C 文件会一直保留**，直到你确认 .sx 版完全可替代后再决定是否删除；**不建议在 9.3 通过前删除任何 C 源码**。

| 阶段 | C 文件状态 | 说明 |
|------|------------|------|
| **9.1** | **全部保留** | 在现有目录（如 src/lexer/、src/parser/ 等）下新增 .sx 与 .c 并存，与 C 后端联调；构建仍是 `make` 编 C + 可选链接 .sx。C 文件不删除，全部写好后统一删 C 留 .sx，保持同一布局。 |
| **9.2** | **全部保留** | 用 .sx 实现中端与后端后，可用「宿主编译器编译全 .sx 编译器」产出新 shux；此时主构建可切换为「只编 .sx」，但 C 源码**建议仍保留**在目录中，便于对比、回退和排查。 |
| **9.3** | **保留** | 两代 shux 行为一致验证通过；C 仍**不删除**，作为参考与安全网。 |
| **9.3 通过之后** | **可选删除** | 当团队确认 .sx 版为唯一主路径、且 CI/文档均已切换为「用 shux 编译 compiler/*.sx」后，可**选择**将 compiler 下 C 源码删除或迁出（例如移到 `compiler/legacy-c/` 或单独分支）。删除前建议打 tag 或备份，便于追溯。 |

**结论**：写自举时 C 文件**不会**被自动删除；**只有在 9.3 通过后、你主动决定删除时**才删。在此之前 C 一直保留，用于联调、对照和回退。

## 五、技术要点：语言子集、标准库、构建、测试

### 5.1 语言子集冻结

- **依据**：《自举语言子集》与「自举前-全面检查」子集稳定约定。
- **实践**：自举期间（从 9.1 开始到 9.3 通过）仅允许小步扩展（bug 修复、与现有文档一致的小补丁）；不引入新关键字、新类型、破坏性语法变更。新特性在自举验证通过后再加入，并同步更新《自举语言子集》。

### 5.2 标准库使用边界

- **编译器 .sx 代码**：仅使用「自举前必须」的 core/std 模块（见需求分析 §5.4、自举前路线分析 §4）；避免依赖「可自举后写的库」。
- **占位与补齐**：若 std.vec、std.map、std.fs、std.string 等当前为占位，建议在 9.1 前或 9.1 中补齐到「最小可用」（如 vec 的 push/len、map 的 get/insert、fs 的 open/read/close、string 的拼接与长度），否则用 .sx 写编译器时会大量自造数据结构，拖慢进度。

### 5.3 构建系统

- **自举前**：`make -C compiler` 编译 C 源码，产出 shux。
- **9.1**：新增目标或脚本，例如 `make bootstrap-frontend`：用当前 shux 编译 compiler/frontend/*.sx（或约定目录），并链接 C 后端，产出混合编译器。
- **9.2**：`make bootstrap` 或等价：用当前 shux 编译 compiler/*.sx（全部 .sx），指定 core/std 根路径，产出新 shux；文档中明确「上一代 shux」路径或环境变量。
- **9.3**：脚本或 make 目标：执行「shux → 编译自身 → 新 shux → 再编译自身 → 对比测试」，并输出成功/失败。

### 5.4 测试与回归

- **全量测试**：run-all.sh 或 make test 覆盖泛型、trait、多文件、import、core/std 模块、边界与负向用例；自举前必须全部通过。
- **自举验证**：9.3 中用 shux₁ 与 shux₂ 分别跑同一套测试，结果一致；任何不一致即回归，需排查并修复后再标「自举成功」。

## 六、风险与应对

| 风险 | 说明 | 应对 |
|------|------|------|
| **前端与后端接口不稳定** | .sx 前端产出的 AST/IR 与 C 后端期望格式不一致，联调成本高 | 9.1 先严格复用现有 AST/IR 的内存布局或序列化格式，或先做「.sx 前端 → 生成与现有 codegen 一致输入」的薄适配层 |
| **标准库占位不足** | std.vec/map/fs/string 仅占位，写编译器时缺 API | 在 9.1 前或 9.1 初期补齐最小可用实现，优先 vec、map、fs、string |
| **两代编译器行为不一致** | shuxc₁ 与 shux₂ 对同一输入结果不同 | 严格用同一测试套件对比；差异定位到具体用例与模块，修复后再验证 |
| **性能与语义基建未完全满足** | 如 noalias、defer 内联、BCE、CTFE 未做或未达标 | 不阻塞启动 9.1；在自举验证通过后在 .sx 内迭代补齐，或在文档中明确「自举后优先补齐项」 |
| **子集漂移** | 自举过程中无意引入新语法或破坏性变更 | 严格遵守《自举语言子集》与「仅小步扩展」约定；变更前对照子集文档与测试 |

**何时实现「.sx 写自己的方法、不再调用 .c」？（完全自举路线）**

当前状态是：**.sx 做入口与分发，具体逻辑仍多由 C 实现**（如 `typeck_sx_fill_resolved_types` 填类型、`codegen_sx_emit_struct_lit_full` 输出结构体字面量等）。要实现「.sx 写自己的方法、不再调用 .c 的逻辑」，即**编译器业务逻辑全部在 .sx、C 只做最小运行时**，可按以下阶段推进：

| 阶段 | 目标 | 做法 |
|------|------|------|
| **现在** | 自举验证通过 | .sx 入口 + 能迁的表达式/检查在 .sx，其余逻辑仍调 C（`*_full`、`fill_resolved_types` 等）。 |
| **阶段 A：逻辑逐步迁到 .sx** | 每个「仍调 C」的节点改为 .sx 实现 | 对每一块（如 struct_lit、array_lit、if_expr、typeck 填类型）：C 只提供**只读访问器**（如库前缀、依赖模块、类型名、字段列表、子表达式等），**.sx 里用这些访问器自己实现**整段逻辑（拼接字符串/逐字节写、类型推导与填类型等），**不再调用** C 的 `*_full` 或 `typeck_module` 的填类型循环。 |
| **阶段 B：C 只做「壳」** | 编译器逻辑 100% 在 .sx | AST 仍可由 C 分配并提供访问器（或将来改为 .sx 侧数据结构）；类型检查、代码生成、语句/顶层输出的**全部算法**在 .sx 中实现。C 仅保留：进程/IO（main、FILE*、读文件）、可选的一层「按访问器写字节」、以及构建时用系统 cc 链接。 |
| **阶段 C（可选）：去掉 C 源码** | 编译器源码 100% .sx | 若运行时也由 .sx + core/std 实现（或通过 FFI 只调系统 API），则 compiler 目录可不再包含 .c 文件；构建变为「上一代 shux + 系统 cc」只编 .sx 并链接少量运行时。 |

**关键点**：  
- 「不再调用 .c」指的是**不调用 C 里的业务逻辑**（类型推导、代码生成规则等）。  
- **C 仍可保留**：AST 内存布局与访问器、`emit_byte`/FILE*、或运行时；这些视为「运行时/接口层」，不是「编译器逻辑」。  
- **何时能实现**：**每迁完一块逻辑**（例如 struct_lit 在 .sx 里用访问器自己拼 C 代码、typeck 在 .sx 里自己遍历并填类型），就少一块对 C 的依赖；当所有这类块都迁完，即达到「.sx 写自己的方法、不再调用 .c 的编译器逻辑」的完全自举。

**我们现在不能把逻辑全部写到 .sx 吗？—— 能。**

没有「语言或工具链不允许」的硬性限制，逻辑**可以**全部写到 .sx。当前 .sx 已具备：extern 调 C、循环/条件/函数、访问器返回 ptr+len 或标量、逐字节写输出等，足以用「.sx 写算法 + C 只提供最小原语」的方式实现 typeck 与 codegen 的整块逻辑。未全部写在 .sx 的原因主要是**历史与优先级**（先打通自举验证，再逐步迁），而不是「写不了」。

- **可以这样做**：  
  - **逐步迁**：按「当前仍由 C 实现、需迁到 .sx」的清单，一块一块把 C 里的逻辑改成「C 只提供访问器/写字节」，.sx 里实现分支、拼接、遍历与输出。每迁完一块，.sx 就多一块自己的方法，C 就少一块业务逻辑。  
  - **集中迁**：若优先「尽快逻辑全在 .sx」，可定一个里程碑（例如 codegen 的 type→字符串、初值生成、块体顺序全部在 .sx 内用访问器实现），按清单集中把对应 C 函数拆成小原语并在 .sx 里重写调用方。  

- **需要 C 保留的（若暂不重写 parser）**：AST 的分配与内存布局（malloc/struct 定义）仍可在 C；.sx 只通过访问器（只读）操作 AST。这样「逻辑」= 所有基于 AST 的决策与输出，都在 .sx；C 只做「分配 + 访问器 + FILE* / emit_byte」。  

- **结论**：**现在就可以把逻辑全部写到 .sx**，没有不可逾越的障碍；区别只是**做多少、按什么顺序做**。选好一块（例如 codegen 的 const/let 前缀与初值、或 typeck 的填类型循环），按「C 只提供 is_used/emit_prefix/emit_init 等原语，.sx 里自己写流程」的方式实现，即可逐步逼近「逻辑全部在 .sx」。

---

**自举目标：全部用 .sx 写，和 C 无关**

自举 = 编译器**源码 100% .sx**。compiler 目录下不再有 .c/.h；构建 = 上一代 shux 编译 compiler/*.sx（及 core/std），产出新 shux。**与 C 无关系**：不依赖 C 实现任何编译器逻辑，也不依赖 C 的 AST/访问器；所有东西（词法、语法、AST、类型检查、代码生成、驱动）全部写在 .sx 里。

**为达「全部 .sx」必须完成的迁移**：

| 层级 | 当前状态 | 目标（全部 .sx） |
|------|----------|------------------|
| **AST** | C 的 struct/typedef，.sx 通过 extern 访问器读 | AST 在 .sx 里用 struct/enum 定义；parser 在 .sx 里分配、填充 AST |
| **词法/语法** | parser.c、lexer.c 实现 | lexer.sx、parser.sx 已存在；需与 .sx AST 对接，不再调 C 的 AST 分配 |
| **类型检查** | typeck.sx 调 C 的 typeck_sx_fill_resolved_types 等 | 填类型、类型相等、泛型单态化等**全部**在 typeck.sx 里实现，不调 C 业务逻辑 |
| **代码生成** | codegen.sx 调 C 的 emit_byte、访问器、c_type_str、codegen_init 等 | 类型→字符串、初值生成、块体顺序、函数签名等**全部**在 codegen.sx 里实现；输出为 .sx 管理的缓冲区或通过 core/std 写文件，不依赖 C 的 FILE* |
| **驱动** | driver.sx 调 C 的 run_compiler_c（读文件、调 cc） | 读文件、调 typeck/codegen、调 cc 等**全部**在 driver.sx 里用 core/std（如 std.fs、std.process）实现 |
| **构建** | make 编 C + 编 .sx 生成的 .c → 链接 | `shux -L core -L std compiler/*.sx -o shux`，无 .c 参与 |

完成后：**没有任何编译器逻辑在 C 里**；C 仅可在可选的极小运行时（如 crt0）中存在，且可由 core/std + FFI 替代。

---

**什么时候才能删除 .c/.h，实现真正的自举？**

当前做法（.sx 做入口与编排、具体逻辑仍调 C）**不能**自动导致「可删 C」：只要业务逻辑还在 C 里，.c/.h 就删不掉。要**真正删掉 .c/.h**，必须满足下面两条之一：

| 目标 | 含义 | 前提与步骤 |
|------|------|------------|
| **删除「编译器逻辑」所在的 C** | compiler 目录下不再有实现 lexer/parser/typeck/codegen **算法**的 .c 文件 | **阶段 A→B 做完**：所有「仍调 C」的节点/块在 .sx 里**用 C 提供的只读访问器自己实现**（类型→字符串、初值生成、AST 遍历、语句顺序等），C 只保留：AST 分配与访问器、FILE* / emit_byte、可选的最小运行时。此时可把 codegen.c / typeck.c / parser.c 等里的**业务逻辑全部迁到 .sx**，.c 只留「壳」或删除。 |
| **删除所有 C 源码** | 仓库中不再包含任何 .c/.h，构建 = 上一代 shux 只编 .sx + 链接运行时 | 在上一行基础上，**阶段 C**：把「壳」和运行时也换成 .sx + core/std（或 FFI 调系统 API）；构建命令变为「shux 编译 compiler/*.sx 与 runtime」，不再用系统 cc 编任何 .c。 |

**可删除 C 的充要条件（检查清单）**：

1. **编译器逻辑 100% 在 .sx**  
   以下不再由 C 实现，而由 .sx（或 .sx 调用的、仅做「写字节/读数据」的 C 小接口）实现：  
   - 词法/语法分析、AST 构建与遍历；  
   - 类型检查、泛型单态化、类型→C 字符串；  
   - 代码生成：表达式/语句/声明/顶层输出的**全部**生成规则（const/let 前缀与初值、函数签名、defer、cleanup 等）。  

2. **C 仅剩「可替代层」**  
   若仍保留 C，则只允许：进程/IO（main、读文件、写 FILE*）、AST 内存分配与访问器、emit_byte 等「无业务含义」的接口。这些层必须能在阶段 C 用 .sx + 标准库或 FFI 重写，否则无法彻底删 .c。  

3. **构建与自举流程**  
   - 删除前：`make` = cc 编 .c + 编 .sx 生成的 .c → 链接 → shux。  
   - 删除后：`shux -L core -L std compiler/*.sx -o shux`（或等价命令），无 .c 参与。  

**结论**：  
- **现在**：不能删 .c；当前是「.sx 入口 + 部分逻辑迁入 .sx」，大量逻辑仍在 C。  
- **能删「逻辑 C」**：当阶段 A→B 完成，.sx 用访问器自己实现所有类型检查与代码生成逻辑，C 只做壳/IO/访问器时，可删除 parser.c、typeck.c、codegen.c 里的业务实现（或整文件删除，若访问器与壳也迁到 .sx 或极小 C）。  
- **能删「全部 C」**：再完成阶段 C，用 .sx + core/std 替代剩余 C 壳与运行时，即可删除全部 .c/.h，实现**源码级**真正自举。

**当前仍由 C 实现、需迁到 .sx 才能删 C 的主要逻辑（非穷举）**：  
- **parser**：整词法/语法、AST 构建、stmt_order、const/let/loop/labeled 解析。  
- **typeck**：填类型（typeck_sx_fill_resolved_types 调用的 C 实现）、类型相等/兼容判断、泛型单态化。  
- **codegen**：类型→C 字符串（c_type_str、codegen_emit_type）、ensure_slice_struct/ensure_vector_typedef、codegen_init（初值生成）、整块 body 顺序与 const/let/expr/loop 分支（含 stmt_order）、函数签名与顶层声明、run_defers 以外的 defer 相关逻辑。  
- **driver/main**：读文件、调 typeck/codegen、调 cc、错误输出。  

每把上述一块在 .sx 里**用访问器与最小 C 接口自己实现**，就离「可删 .c」近一步；全部迁完且 C 仅剩壳/IO 时，即可删除 .c/.h 实现真正自举。

**全部 .sx 的推荐执行顺序**（与 C 脱钩）：  
1. **AST 迁到 .sx**：在 .sx 中定义与现有 C AST 同构的 struct/enum（或简化版），提供分配/访问接口；parser.sx 改为构建该 .sx AST，不再调 C 的 AST。  
   - **已做**：`compiler/src/ast/ast.sx` 已新增，纯 .sx 定义 TypeKind、Type、ExprKind、Expr、ConstDecl、LetDecl、WhileLoop、ForLoop、StmtOrderItem、LabeledStmt、Block、Param、Func、Module；无任何 C 或 extern。解析顺序已满足（enum* 再 struct*），StmtOrderItem/Expr/ASTArena 已加 allow(padding)。typeck 已支持 NAMED 内建类型名（i32 等）的 size/align，以及「指针到结构体」的字段访问（arena.num_types 等）；codegen 已对指针 base 输出 `->`。**AST 池化**：已实现 ASTArena（types[512]、exprs[4096]、blocks[512] + num_*）、ast_arena_init、ast_arena_type/expr/block_alloc、ast_arena_*_get/ast_arena_*_set；无 C malloc，ref 为 1-based，0 表示空。  
2. **Typeck 全 .sx**：typeck.sx 只操作 .sx AST；填类型、类型相等、泛型单态化均在 .sx 内实现，不再调用 typeck_sx_fill_resolved_types 等 C 实现。  
3. **Codegen 全 .sx**：codegen.sx 只操作 .sx AST；类型→字符串、初值、块体、顶层输出均在 .sx 内实现；输出写入 .sx 缓冲区或通过 std.io/std.fs 写文件，不依赖 C 的 FILE* 与 c_type_str/codegen_init。  
4. **Driver 全 .sx**：driver.sx 用 std.fs 读文件、std.process 调 cc，不再调 run_compiler_c。  
5. **删除 compiler 下全部 .c/.h**，构建仅保留「shux 编 compiler/*.sx」。

**自举前未完成项（建议全部完成再自举）**：  
- **CTFE 最小集**：✅ 已完成。  
- **noalias/restrict 传递**：✅ 已完成（指针形参标 restrict，codegen 生成 C restrict）。  
- **defer 多出口 goto cleanup**：✅ 已完成（多 return 时统一 shux_cleanup + 一次 defer）。  
- **边界检查消除（BCE）**：✅ 最小版已完成（arr[0] 且数组长度≥1 时省略边界检查；AST 已留 index_proven_in_bounds 便于扩展）。  
- **Result 寄存器化**：✅ 已完成（Result_i32 双槽布局，x86-64 两寄存器返回，见 Result寄存器化.md）。

### 7.3 为何生成 .c、多模块卡点与解决（bootstrap-parser 相关）

**为何「写纯 .sx 自举」还要生成 .c？**  
自举指**编译器源码**用 .sx 写；当前 shux 的**后端**是「.sx → C → 调系统 cc」，所以编译任意 .sx（含自举用的 lexer/parser）时，**输出**仍是 C。这是后端设计选择（也可选 LLVM IR/机器码），与「源码是否纯 .sx」无关。生成 .c 是编译器产出的中间结果，再由 cc 生成最终可执行文件。

**多模块时的两个 C 问题：**

1. **多文件 → incomplete type**  
   若每个 .sx 模块生成一个 .c（token.c、lexer.c、parser.c），则 lexer.c 里会出现 `struct lexer_LexerResult { ...; struct token_Token tok; }`。C 规定：**按值嵌入的结构体**必须在同一翻译单元内有完整定义。`token_Token` 定义在 token.c，编译 lexer.c 时看不到，故报错「incomplete type」。

2. **合并为单文件 → 可能重定义**  
   若把依赖与入口写进**同一个 .c**（先 token → lexer → parser），类型定义顺序正确，可消除 incomplete type。但若某模块的 codegen 或 AST 里把依赖模块的类型也输出一遍，同一 .c 里会重复出现同一 enum/struct 定义，C 报「redefinition」。

**解决方式（已实现）：**  
- **有依赖时改为单文件输出**：driver 先按拓扑序把全部依赖模块写进**同一**临时 .c，再写入口模块，最后只调用一次 cc 编译该单文件。  
- **类型去重**：codegen 维护「本文件已输出的类型名」集合；输出 enum/struct 前若该 C 名已在集合中则跳过，否则输出并加入集合，避免重定义。  
这样既消除 incomplete type（依赖类型先于使用处定义），又避免重定义，bootstrap-parser 可跑通。

### 7.4 9.1 联调进展（能处理示例 .sx）

- **多 -L**：driver 支持多个 `-L`，依次在各自根下解析 import（如 `-L .. -L src/lexer` 可同时找到 std.fs 与 lexer）。
- **parser.sx 读文件**：若存在 `/tmp/shux_parse_test.sx`，parser.sx 的 main 从该路径读入最多 64 字节并解析，否则用固定源码；lexer 遇 NUL 视为 EOF。
- **类型与 codegen**：typeck 允许数组实参传给 *T 形参（数组到指针衰减）、非负整数字面量传给 usize；codegen 对「切片从外层块数组变量初始化」用 init->resolved_type 生成 `.data`/`.length`。
- **验收**：`make bootstrap-parse-file` 写示例到 `/tmp/shux_parse_test.sx`，跑 .sx 解析器得「parse OK」，再跑 C shux 同文件得「parse OK」，二者一致即联调初步通过；后续可扩展为 .sx 前端产出与 C 兼容的 AST 并由 C 后端消费。
- **6.1 typeck 入口**：`typeck.sx` 提供 `typeck_entry`，extern 声明 C 的 `typeck_module`，转调 C 实现；无 main 时 `-E` 按库模块生成（`typeck_typeck_entry`）；`make bootstrap-typeck` 将 typeck.sx 编进 shux，main 在 `SHUX_USE_SX_TYPECK` 下调用 `typeck_typeck_entry(mod, deps, ndep)`，形成 .sx 入口与 C AST 对接。
- **6.2 codegen 入口**：`codegen.sx` 提供 `codegen_entry_module_to_c` 与 `codegen_entry_library_module_to_c`，转调 C 的 `codegen_module_to_c` / `codegen_library_module_to_c`；无 main 时 `-E` 按库模块生成（`codegen_codegen_entry_*`）；`make bootstrap-codegen` 将 typeck.sx + codegen.sx 编进 shux，main 在 `SHUX_USE_SX_CODEGEN` 下调用两处 .sx 入口，产出与当前 C 一致的 .c。首块迁入：LIT/CTFE、BOOL_LIT、VAR、FLOAT_LIT、ENUM_VARIANT、二元运算（ADD/SUB/MUL、==..||）、一元运算（NEG/BITNOT/LOGNOT）均由 .sx 或经 .sx 输出；C 提供 emit_byte、char_at、emit_float、emit_expr、needs_*_parens、emit_binop/emit_unary 由 .sx 实现。
- **6.3 driver 入口**：`driver.sx` 提供 `driver_main(argc, argv)`，转调 C 的 `run_compiler_c`；C main 将原逻辑抽成 `run_compiler_c`，在 `SHUX_USE_SX_DRIVER` 下 main 调用 `driver_driver_main`；`make bootstrap-driver` 将 typeck + codegen + driver.sx 编进 shux，形成「命令行→读文件→typeck→codegen→调 cc」的 .sx driver 入口编排。
- **6.4 验收**：`make check-6.4` = 宿主编译器（C-only shux）执行 `make bootstrap-driver` 产出新 shux，并用新 shux 编译运行 tests/return-value/main.sx，通过即 6.4 达成。
- **6.5 自举**：`make bootstrap-self` = 用 bootstrap-driver 产出的 shux₁ 再编 typeck/codegen/driver.sx 得到 shux₂（输出为 shux_stage2），并用 shux₂ 编译运行同一测试；通过即「新 shux 能编译自身源码产出下一代」达成。
- **7.1～7.3 两代一致与验证**：7.1 由 `bootstrap-self` 完成（shux_stage1=shux₁，shux_stage2=shux₂）。7.2：`tests/run-all.sh` 支持环境变量 `SHUX`，指定时用该二进制代替 `compiler/shuxc`；`make check-7.2` 用 shux₁ 与 shux₂ 各跑一遍全量测试，均通过则 OK。7.3：`make bootstrap-verify` 依赖 check-7.2，通过即视为自举验证通过；可选在 CI 或文档中记录命令（`make -C compiler bootstrap-verify`）及此时 `shux_stage1`/`shux_stage2` 的哈希。
