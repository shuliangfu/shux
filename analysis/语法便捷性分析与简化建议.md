## 二、繁复语法清单（问题 → 原因 → 建议）

下列按**开发者体感繁琐程度**与**bootstrap / 标准库修错频率**排序。

### 2.0 标准库 `as` 分布（粗扫，2026-06）

`std/**/*.sx` 中 cast 高度集中；**多数可用「期望类型传播」在编译期消掉**，与手写 C cast 生成相同 IR，零运行时成本。

| 模式 | 典型写法 | 集中模块 | 能否 contextual 化 |
|------|----------|----------|-------------------|
| 下标拓宽 | `buf[i as usize]` | url、kv、schema、tar、socketio | ✅ §2.13 |
| 空指针 | `p == 0 as *u8`、`return 0 as *T` | fs_posix、regex、base64 | ✅ 部分 §2.14 |
| 整型字面量 | `0 as u32`、`1 as i16` | io_sync、config | ✅ §2.1、§2.11 |
| 变量拓宽 | `nr as usize`、`st.st_size as usize` | fs、heap | ✅ assignable 链 |
| 指针/FFI | `&buf[0] as *u8`、`p as i64` | io、mmap | ⚠️ 部分保留 §2.15 |
| 位运算/协议 | `(c >> 4) as u8` | crypto、hash | ❌ 应保留显式 cast |

**原则**：变量**已有确定类型**时不应对变量写 `as`；cast 只留给类型真对不上、且 join/期望类型都解决不了之处，以及指针/unsafe 边界。

### 2.1 整型字面量与上下文类型（高频）

**现状**

- 整数字面量默认 **`i32`**。
- 下列场景常被迫写 `as`：
  - `let x: i64 = 0;` → 需 `0 as i64`（或依赖 typeck 拓宽，实现不一致）
  - `return n >= 0 ? (n as i64) : (-1 as i64);` → 旧规则要求两分支显式同类型；`n` 已是 `i64` 时仍被迫写 `as`
  - `let e: i32 = __errno_location()[0];` → 部分路径下 call 后缀 `[]` 与 let 初值解析仍弱
  - `io_fixed_nr = 0`（`u32` 变量）→ 需 `0 as u32`
  - `pfd.events = 1`（`i16` 字段）→ 需 `1 as i16`

**原因**

- typeck 对**块内 let** 有 `typeck_coerce_let_int_lit` / 整型拓宽；**顶层 let/const、三元、部分 return** 长期更严。
- 三元运算符原要求 `type_equal`，不自动 `i32 → i64`。

**建议（零运行时成本）**

| 方案 | 说明 | 状态 |
|------|------|------|
| **A. 期望类型传播（contextual typing）** | 在 `let/const` 初值、**赋值**、**return**、三元、实参、**struct 字段**位置，用已知类型收窄/拓宽 | ✅ 五入口已落地（`typeck_expr_matches_expected`） |
| **B. 三元整型汇合（join）** | `i64` 分支 + `i32` 字面量 `-1` → 结果类型 `i64` | ✅ 已实现（`typeck_join_ternary_types`） |
| **C. 顶层 let/const 与块内对齐** | 顶层也走 `coerce_let_int_lit` + `type_assignable_to` | ✅ 已实现 |
| **D. 拓宽 isize** | `i32 ↔ isize ↔ i64` 在赋值/三元/return 一致 | ✅ 已加 isize 拓宽 |

**简化后写法示例**

```su
// 目标：上下文已知时少写 as；n 已有确定类型（如 i64）时不必对 n 写 cast
function f(n: i64): i64 {
  return n >= 0 ? n : -1;   // then 为 i64，else 字面量 -1 经 join 拓宽为 i64
}
let total: isize = 0;
let nr: u32 = 0;
let r: i64 = n >= 0 ? n : -1;  // n: i64 时同上；n: i32/isize 且目标 i64 依赖期望类型传播（P0）
```

**性能**：纯 typeck/codegen 阶段；生成 C 与手写 cast 相同，**无额外运行时开销**。

### 2.2 结构体字面量必须写类型名（高频）

**现状**

```su
let pfd: PollFd = PollFd { fd: out_fd, events: POLLOUT, revents: 0 };
// 期望：
let pfd: PollFd = { fd: out_fd, events: POLLOUT, revents: 0 };
```

**原因**

- Parser 仅在「大写标识符 + `{`」时解析为 `AST_EXPR_STRUCT_LIT`；裸 `{` 在语句位置是块，在表达式位置历史上未接类型上下文。

**建议**

| 场景 | 写法 | 状态 |
|------|------|------|
| `let/const name: StructType = { ... }` | 类型注解提供 struct 名 | ✅ 已实现（`parse_init_with_type_context`） |
| `return expr` 且返回类型为 struct | `return { a: 1, b: 2 };` | ✅ parser `ctx_return_type` |
| 字段类型已知 | `{ events: 1 }` 中 `1` 自动匹配 `i16` | ✅ struct 字段期望类型 |
| 跨模块 | 仍须 `mod.Type { }` 或注解 `: mod.Type` | 保持（消除歧义） |

**性能**：无影响。

### 2.3 struct 定义用分号、字面量用逗号（中频，认知负担）

**现状**

```su
struct Point { x: i32; y: i32; }      // 字段间 ;
let p: Point = Point { x: 1, y: 2 }; // 字面量字段间 ,
```

**原因**

- 解析器历史选择：定义区按「声明」用 `;`，字面量按「表达式列表」用 `,`（接近 C designated initializer / Rust struct expr）。

**建议**

| 方案 | 评价 |
|------|------|
| **保持双规则，文档强调** | 成本低；在 [02-类型定义](../docs/02-类型定义.md) 与 [08-语法规范](../docs/08-语法规范.md) 用对照表固定 |
| **字面量也允许 `;`** | 解析器接受 `{ x: 1; y: 2 }` | ✅ P1（`contextual_typing_p1`） |
| **定义改逗号** | 与 C/Java 习惯冲突，**不建议** |

### 2.4 let 必须写类型且无推断（中高频）

**现状**

```su
let x: i32 = 1;           // 必须写 : i32
let x = 1;                // ❌ 不允许
```

**原因**

- 项目早期约定：**显式类型、少隐式**，利于 C 后端与可读性（见类型系统设计 §一）。

**建议（分档）**

| 档位 | 行为 | 性能 | 建议 |
|------|------|------|------|
| **保守** | 仅 `let x = StructLit` 且右侧无歧义时推断 | 零成本 | 可选，阶段 6+ |
| **中等** | 从字面量推断标量：`let x = 1` → `i32` | 零成本 | 与「显式」宗旨略冲突 |
| **激进** | 全推断 | 零成本 | **不建议**作默认 |

**推荐**：保持「**必须写类型**」为默认；仅对已实现的 **`let x: T = { ... }` 简写**继续扩展，不全面引入推断。

### 2.5 `as Type` 过多（中高频）

**现状**

- FFI / 多宽度整数 / 跨模块 struct 混用导致标准库大量 `as usize`、`as i32`、`as *u8`（分布见 §2.0）。

**建议：用「期望类型」替代显式 cast 的场景**

| 场景 | 现写法 | 目标写法 | 状态 |
|------|--------|----------|------|
| 函数实参 | `poll(&pfd, 1 as u64, -1)` | `poll(&pfd, 1, -1)` | ✅ 字面量 + assignable |
| 泛型调用实参 | 仅 `type_equal_subst` | 走 `type_assignable_to` | ✅ |
| struct 字段初值 | `revents: 0 as i16` | `revents: 0` | ✅ §2.2 |
| 赋值 | `io_fixed_nr = 0 as u32` | `io_fixed_nr = 0` | ✅ §2.11 |
| return / 三元 | 见 §2.1、§2.12 | 见 §2.1、§2.12 | ✅ |
| 下标 | `buf[i as usize]` | `buf[i]`（`i: i32`） | ✅ §2.13 |
| 指针算术 | `out + n as usize` | `out + n`（`*T + i32`） | ✅ §2.13.1 |
| 空指针 | `path == 0 as *u8` | `path == 0` | ✅ §2.14 |
| 指针桥接 | `bufs as *u8` | **仍建议显式** | 保持 |

**不应取消 as 的场景**

- 指针 ↔ 整数、不同 struct 名义类型、unsafe 边界、位运算语义 cast：必须显式。

### 2.6 分号规则（中频，新手易错）

**现状**（[08-语法规范](../docs/08-语法规范.md)）

- 一般语句要 `;`；**表达式以 `}` 结尾则禁止 `;`**（`return Foo { }` 不能写 `};`）。

**繁复点**

- 与 JavaScript/TypeScript「块尾可省略分号」类似，但规则更严；`return expr;` 与 `return expr`（块尾）两套。

**建议**

| 方案 | 说明 |
|------|------|
| **保持** | 与当前 parser 一致；靠 formatter / LSP 提示 |
| **可选放宽** | `};` 在 struct 字面量后允许但 warning | 低优先级 |
| **统一「块尾 return 可无分号」** | 文档已写；main 仍强制 `return expr;` | 保持 |

### 2.7 条件编译两套语法（中频）

**现状**

| 机制 | 形式 | 作用域 |
|------|------|--------|
| 行预处理 | `#if target_os = "linux"` … `#endif` | 整行；改行号 |
| 属性 | `#[cfg(target_os = "linux")]` | 顶层项；函数体内 `{ ... }` / 单语句 |

**繁复点**

- 标准库 `fs_posix.sx` 等大量函数内 `#[cfg] { ... }`，C 前端 parser 曾不支持块内 cfg（已在 parser 增强）。
- 用户需同时理解 `#if` 与 `#[cfg]`。

**建议**

| 方案 | 说明 |
|------|------|
| **文档统一** | 顶层用 `#[cfg]` 剪枝；长段平台代码用 `#if` 包文件级块 |
| **函数内优先 `#[cfg]`** | 单平台语句：`#[cfg(linux)] if (e == 11) { ... }` |
| **远期** | 考虑 `cfg_if!` 式宏（自举后） | 非当前必须 |

### 2.8 import：仅绑定形式（已定稿，不扩展）

**结论（项目决策）**

- **只保留** `const m = import("path");` + `m.symbol()`；**够用，不再加**其它 import 语法糖。
- 下列形式**明确不做**：
  - 解构 import：`const { read } = import("std.io")`
  - 关键字 import：`import "path" as io`
  - 链式：`import("std.io").read(...)`
  - 通配：`use std.io.*`

**标准写法**

```su
const io = import("std.io");
const driver = import("std.io.driver");

function main(): i32 {
  io.print_str(...);
  let buf: driver.Buffer = { ptr: p, len: l, handle: h };  // 跨模块类型仍用绑定名限定
  return 0;
}
```

**原因（保持简单）**

- 单一符号表、无 re-export / 通配泛滥；与 Zig `@import` 绑定风格一致。
- 绑定名即命名空间，冲突时换绑定名（`const io2 = import("std.io")`）即可。

**可优化方向（不增 import 语法）**

| 方向 | 说明 | 章节 |
|------|------|------|
| 跨模块 struct 简写 | `let b: driver.Buffer = { ... }` | §2.2、§2.9 |
| 模块 API 稳定 | 平台差异收进 `std.*`，用户只 bind 一个模块 | 库设计 |
| 文档示例统一 | [05-函数与模块](../docs/05-函数与模块.md) 只展示绑定形式 | §2.25 |

**状态**：✅ 语法定稿；后续优化走 typeck/struct 简写，**不改 import 本身**。

### 2.9 跨模块类型与字面量（中频）

**现状**

```su
const driver = import("std.io.driver");
let buf: driver.Buffer = driver.Buffer { ptr: p, len: l, handle: h };
```

**繁复点**

- `*Buffer` vs `*IoBatchBuf` 同名布局不同 type，module 边界需 `*u8` 桥接或 core 层包装（bootstrap 已遇）。

**建议**

| 方案 | 说明 |
|------|------|
| **布局相同 struct 互认** | typeck 对「同 ABI 字段序」指针互 assign | ✅ MOD-02 `#[repr(compatible)]` opt-in |
| **模块 API 用 `*u8` + 文档** | 当前 io_backend 做法 | 已实现 |
| **简写 + 注解** | `let b: driver.Buffer = { ptr: p, ... }` | 依赖 §2.2 |

### 2.11 赋值（`=`）缺少期望类型传播（高频）

**现状**

- `AST_EXPR_ASSIGN` 除 `u8` 小字面量、`ptr = 0` 外要求 `type_equal`，**未**走 `type_assignable_to`。
- 标准库典型：

```su
io_fixed_nr = 0 as u32;
pfd.events = 1 as i16;
i = 1 as u32;
```

**原因**

- 历史实现只特判了 C 式 `NULL` 赋指针与 `u8` 字节写入；整型变量/字段赋值未接期望类型。

**建议**

| 方案 | 说明 | 状态 |
|------|------|------|
| 左值类型作期望类型 | `lhs: u32` + 右端字面量 `0` → coerce | ✅ |
| 复用 `type_assignable_to` | `i32` 赋给 `isize/usize` 等拓宽链 | ✅ |
| 字段赋值 | `obj.field = 1` 以字段类型为期望类型 | ✅ |

**简化后**

```su
io_fixed_nr = 0;
pfd[i].events = POLLOUT;
```

**性能**：纯 typeck；codegen 与显式 cast 相同。

### 2.12 return 整型拓宽（§2.1 延伸）

**现状**

- `typeck_return_value_matches` 以 `type_equal` 为主；`function f(): i64 { return -1; }` 及「三元汇合为 `i32/isize`、return 要 `i64`」仍可能报错。
- `n: i64` 时 `return n >= 0 ? n : -1` **已可**（三元 join）；`n: i32/isize` 尚需 return 上下文再拓宽。

**建议**

- return 操作数检查：`type_equal` **或** `type_assignable_to(got, expect, op)`。✅ 已实现。
- 与 §2.1 三元 join 叠加：先 join 分支，再按函数返回类型拓宽。✅

**简化后**

```su
function g(n: i32): i64 {
  return n >= 0 ? n : -1;
}
function h(): i64 {
  return -1;
}
```

### 2.13 下标 `arr[i as usize]`（cast 量最大）

**现状**

- 循环计数常用 `i: i32`，数组/切片下标期望 `usize`，标准库大量 `arr[i as usize]`（url、kv、tar 等模块尤甚）。

**原因**

- `INDEX` 不将 `i32/i64` 下标自动拓宽为 `usize`；开发者被迫在每次下标处 cast。

**建议**

| 方案 | 说明 | 状态 |
|------|------|------|
| 下标期望 `usize` | `i32`/`isize` 在拓宽链内可隐式用于 `[ ]` | ✅ |
| 安全约束 | 不引入 C 式有符号/无符号混用；仅允许已定义的 `typeck_integer_widen_ok` 链 | ✅ |

**优先级**：P0.5 ✅ 已落地（`contextual_typing_p1` / `IDX-01`）。

#### 2.13.1 指针算术 `ptr + n as usize`（§2.13 延伸）

**现状**：`memcpy(out + n as usize, ...)` 中 `n: i32` 时 cast 冗余。

**规则**：`*T + i32/isize/usize` → `*T`（与下标拓宽同链）。✅ 已实现（`ptr_arith_i32.sx`）。

**书写原则**：类型能对上（同类型或 assignable 链）**不写 `as`**；对不上则 **编译期 typeck error**，不用运行期异常；跨边界（指针↔整数、位语义）**必须显式 `as`**。

**显式禁止隐式（2026-06 定稿）**

| 转换 | 规则 | 示例 |
|------|------|------|
| **`u64` → `usize`** | **禁止隐式**，必须 `as usize` | `hdr.write_pos as usize` |
| **`i32` 下标 / `*T + i32`** | 允许隐式（P0.5） | `buf[i]`、`out + i` |
| **整数字面量 + 已知类型** | 允许隐式 | `let x: u8 = 48`、`buf[i] = 48` |
| **三元 + 期望类型** | 分支按左值/字段类型收窄（CTX-08） | `flag: u8 = cond ? 53 : 48` |

**简化后**

```su
let i: i32 = 0;
while (i < n) {
  let c: u8 = buf[i];
  memcpy(out + i, src, len);  // *u8 + i32，无需 as usize
  i = i + 1;
}
```

### 2.14 空指针：`0 as *T`（高频）

**现状**

```su
if (path == 0 as *u8) { ... }
return 0 as *u8;
let p: *u8 = 0 as *u8;   // let 初值、ptr=0 赋值已可省略 as
```

**原因**

- `typeck_coerce_let_int_lit` 与赋值 `ptr = 0` 已支持；**比较、return、部分实参**尚未把 `0` 视为 typed null。

**建议**

- 在期望类型为 `*T` 的 `==`/`!=`、return、实参位置，允许整数字面量 `0` 匹配空指针。✅
- **不**推广到整数语境（避免 `0 as i64` 与 null 混淆）。✅

**简化后**

```su
if (path == 0) { return 0; }
```

### 2.15 数组首元素取址与 decay（中频）

**现状**

```su
memcpy(dst, &buf[0] as *u8, n);
extern_fn(&line[0] as *u8);
```

**原因**

- 部分路径已有 `T[N]` → `*T` 实参衰减；`&arr[0]` 到 `*u8` extern 仍常需 cast。

**建议**

| 场景 | 目标 | 状态 |
|------|------|------|
| `extern fn f(p: *u8)` + `f(&buf[0])` | 数组 decay + elem `u8` 自动匹配 | ✅ |
| 不同 elem 或跨模块 slice | **保持显式 `as`** | 保持 |

### 2.16 循环与计数器冗长（中高频）

**现状**

- 仅有 C 风格 `for (init; cond; step)` 与 `while`；标准库大量：

```su
let i: i32 = 0;
while (i < n) {
  // ...
  i = i + 1;
}
```

- 复合赋值 `i += 1` **已实现**（`tests/compound-assign/main.sx`），但循环头仍冗长。

**原因**

- 无「范围循环」或「整数区间」语法；`for` 的 init 须写完整 `let i: i32 = 0`。

**建议**

| 方案 | 示例 | 优先级 | 性能 |
|------|------|--------|------|
| **保持 C for，推广 `i += 1`** | 文档与 std 示例统一 | 低成本 | 零 |
| **范围 for（远期）** | `for (i : 0 .. n) { }` 半开区间 | P2+ | ✅ LOOP-02 |
| **while + 隐式下标** | 依赖 §2.13 `buf[i]` | P0.5 | 零 |

**不推荐**：`i++` / `++i`（与显式、少未定义宗旨不一致）。

### 2.17 struct 字段 init 简写（中频）

**现状**

```su
let pfd: PollFd = PollFd { fd: out_fd, events: POLLOUT, revents: 0 };
// Rust 可写：PollFd { fd: out_fd, ... } 或字段同名简写 fd
```

**建议**

| 方案 | 示例 | 状态 |
|------|------|------|
| 类型上下文 + 裸 `{ }` | `let p: Point = { x: 1, y: 2 }` | ✅ let 路径 |
| **字段同名简写** | `{ fd, events: POLLOUT }`（`fd` 即 `fd: fd`） | ✅ P2 STR-02 |
| return struct 简写 | `return { x: 1, y: 2 }` | ✅ CTX-07 |

### 2.18 match 能力偏弱（中频）

**现状**

- 支持：整数字面量、枚举变体、`_` 通配（见 [03-控制流](../docs/03-控制流.md)）。
- **不支持**：多模式 `1 \| 2 =>`、范围 `1..10 =>`、guard `n if n > 0 =>`、struct 解构。

**繁复点**

- 相近整型值须多行重复；错误码分支冗长。

**建议**

| 阶段 | 能力 | 优先级 |
|------|------|--------|
| P1 | 多字面量合并：`1 \| 2 \| 3 => expr` | 解析 + typeck 小改 |
| P2 | guard：`pat if cond =>` | ✅ MCH-02 guard |
| P2 | struct 解构：`Point { x, y } =>` | ✅ MCH-02 解构 |
| P2+ | enum 载荷 match | 大；与类型系统联动 |

**性能**：编译期 desugar 为 if-else 链或跳转表，零运行时抽象成本。

### 2.19 错误处理：Result 无 `?` / try（中高频，远期）

**现状**

```su
// 须手动判 err、层层传递
let r: Result_i32 = foo();
if (result.is_err_i32(r)) { return result.unwrap_err_i32(r); }
let v: i32 = result.unwrap_i32(r);
```

- 控制流清单：**ERR-01 `?` ✅**、**ERR-02 try/catch ✅**（见 [控制流与语法构造清单.md](./控制流与语法构造清单.md) §六）。

**建议**

| 方案 | 说明 | 优先级 |
|------|------|--------|
| **保持显式判错** | 与 C 风格、零成本一致 | 当前默认 |
| **`?` 传播** | `let v = foo()?` desugar 为 early return | ✅ ERR-01 |
| **try 块** | `try { ... } catch (r) { ... }`；try 内 ? goto catch | ✅ ERR-02 |

**边界**：不引入异常栈/GC；须 desugar 为显式 `return err`，保持零成本。

## 四、建议优先级路线图

同一套 **`type_assignable_to` + 字面量 coerce** 引擎，按入口分批落地（全部编译期，零运行时成本）。

### P0（立即价值）— 期望类型五入口

## 五、不应为「便捷」而弱化的边界

以下规则**故意比 C 严格**，不建议用隐式转换简化：

| 规则 | 原因 |
|------|------|
| 条件必须 `bool`，禁止 `if (1)` | 消除 C 式隐式 truthiness |
| 指针不默认可互转 | 内存安全、显式边界 |
| struct 隐式 padding 报错（除非 `allow(padding)`） | 性能与布局可预期 |
| `main` 显式 `return` | 入口与退出状态清晰 |
| unsafe / extern 区域显式标注 | 与安全子集分界 |
| import 仅绑定形式 | 不引入解构/通配/关键字 import，避免模块语法膨胀 |
