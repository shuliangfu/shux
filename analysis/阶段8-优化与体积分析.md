# 阶段 8：优化与体积/性能分析

> 本文档在阶段 7 完成后，针对「优化、体积/性能」给出分析、优先级与实施建议，与《自举前路线分析》及项目目标（性能比肩 C、轻量级）一致。

---

## 一、当前流水线（与优化切入点）

当前链路：

```
.sx 源码 → lexer → parser → typeck → codegen → 若干 .c 文件 → 系统 cc → 可执行文件
```

- **shuc** 自身：C 编译时使用 `CFLAGS ?= -Wall -Wextra -I...`，**无 -O**，即默认多为 -O0。
- **用户产物**：`invoke_cc()` 只传 `cc -o out c1.c c2.c ...`（及可选 `-target`），**未传 -O2/-Os**，故生成 C 被 cc 以默认优化级别编译。

因此优化可从三处着手：

1. **后端（C 编译器）**：给 cc 传优化/体积相关参数（见效快、零成本）。
2. **生成 C 的质量**：让生成的 C 更利于 C 编译器优化（不丢失可优化信息、少制造障碍）。
3. **编译器自身**：shuc 的编译选项与（可选）编译速度/内存占用。

---

## 二、与项目目标的对应

| 目标 | 阶段 8 对应动作 |
|------|-----------------|
| **性能比肩 C 甚至超越 C** | 后端用 -O2/-O3；codegen 不引入多余间接、不隐藏常量/只读信息；必要时用 `restrict`/`const` 等助 C 编译器优化。 |
| **轻量级（一切都要小）** | 可选 -Os、strip、减少生成 C 中的冗余（如未用到的库模块不生成 .c）。 |
| **编译器前端生成高质量 IR，不丢失可优化信息** | 当前“IR”即生成的 C；保证 C 表达直接、类型与语义清晰，便于 cc 做常量传播、内联、循环优化等。 |

---

## 三、优化项拆解与优先级

### 3.1 后端（调用 cc 时的参数）—— 优先做

| 项 | 说明 | 建议 |
|----|------|------|
| **默认优化级别** | 当前未传 -O，多为 -O0，性能与体积都未发挥。 | 在 `invoke_cc` 中默认增加 **-O2**（或通过 `-O`/环境变量/新命令行参数让用户选 -O0/-O2/-Os）。 |
| **发布/体积敏感** | 需要小二进制时。 | 支持 **-Os**（如 `shuc -O size` 或 `SHUX_OPT=size` 时传 -Os）。 |
| **strip** | 去掉符号表减小体积。 | 可选：在 `-o` 且优化非 debug 时，cc 成功后对 out_path 执行 `strip`（或交给用户自行 strip）。 |
| **-DNDEBUG** | 若生成 C 里用了 assert，发布构建可关掉。 | 可选：在“release”模式传 -DNDEBUG。 |

**实施顺序建议**：先实现「默认 -O2」与「可选 -Os」（例如 `-O0`/`-O2`/`-Os` 或环境变量），再考虑 strip/NDEBUG。

### 3.2 生成 C 的质量（codegen）—— 次优先

目标：**不拖累 C 编译器优化，并尽量提供可优化信息**。

| 项 | 说明 | 建议 |
|----|------|------|
| **只读/常量** | 已用 `const` 的 const/let，生成 C 时保持 `const`，便于常量传播与只读假设。 | 检查 codegen 中 const/let 的 C 声明是否带 `const`，缺失则补全。 |
| **指针别名** | 指针参数若语义上不重叠，可用 `restrict` 助优化（需谨慎、可先选关键路径）。 | 可选：对已知“唯一写”的指针参数在生成 C 时加 `restrict`（先小范围试点）。 |
| **冷路径** | 已有 panic 分支 `__attribute__((cold))`，利于分支预测与代码布局。 | 保持；若有其他“极少走”的分支可同样标记。 |
| **内联与可见性** | 库函数若只在单 TU 使用，可 `static` 或 `inline` 减少调用开销、便于内联。 | 可选：对单模块内使用的辅助函数生成 `static inline` 或 `static`（需先区分“导出/仅内部”）。 |
| **避免多余临时** | 简单表达式不必都落成临时变量，可直接嵌入到使用处，便于常量折叠与死代码消除。 | 在不大幅增加 codegen 复杂度的前提下，对明显可内联的表达式直接生成到父表达式。 |
| **循环与数组** | 循环边界与数组下标若为常量或简单表达式，C 编译器更容易做向量化/展开。 | 保持当前对常量与局部变量的直接生成；避免对循环边界做不必要的间接或函数调用。 |

**实施顺序建议**：先做「const 一致性检查与补全」，再视需要做「restrict/static 等」试点。

### 3.3 编译器自身（shuc）—— 可选

| 项 | 说明 | 建议 |
|----|------|------|
| **编译 shuc 时的优化** | 当前 Makefile 未加 -O，shuc 自身可能偏慢。 | 在 compiler/Makefile 中增加 **CFLAGS += -O2**（或通过 make 变量覆盖），开发时可保留 -O0 以方便调试。 |
| **编译速度/内存** | 大项目时 lexer/parser/typeck 的分配与重复解析。 | 阶段 8 可只做简单观测（如 time/内存）；大规模优化留待后续。 |

### 3.4 体积与性能的可测性—— 建议有

| 项 | 说明 | 建议 |
|----|------|------|
| **体积基线** | 对固定用例（如 hello、小标准库程序）记录「可执行文件大小」。 | 增加脚本或 make 目标：编译若干用例，输出 size，便于 -O2/-Os 前后对比。 |
| **运行性能基线** | 对简单热点用例（如循环、整数运算）测运行时间。 | 可选：用 time 或简单 benchmark 脚本记录，避免性能回退。 |

## 四、实施顺序小结

1. **第一步（立即见效）** ✅ 已做  
   - 在 **invoke_cc** 中默认传 **-O2**。  
   - 支持用户选择优化级别：**-O 0|2|s**（调试/默认/体积），环境变量 **SHUX_OPT** 可覆盖默认 "2"。

2. **第二步（巩固质量）** ✅ 已做  
   - codegen 中 const/let 已正确：const 声明生成 C `const`，let 不生成 const。  
   - Makefile 支持 **make OPT=1** 时为 shuc 启用 -O2 构建。

3. **第三步（可测、可迭代）** ✅ 已做  
   - **体积基线**：**tests/run-size-baseline.sh** 与 **make -C compiler size-baseline**，编译 hello.sx 与 option/main.sx 并输出可执行文件大小。  
   - 可选：简单**性能基线**（如某循环/算术用例的 time）。

4. **后续（按需）** ✅ strip 已做  
   - **strip 集成**：在 invoke_cc 成功且非 -O0 时，对产出可执行文件执行 `strip`，减小体积（不传 -s 给 cc，避免 ld 的 obsolete 警告）。  
   - 其余：-DNDEBUG、codegen 的 restrict/static/内联等，按实际收益逐步加。
