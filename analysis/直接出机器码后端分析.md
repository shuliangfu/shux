# 直接出机器码后端 — 完整分析文档

> 本文档描述如何实现「AST → 机器码 → 可执行文件」的后端，不经过 C 或 LLVM，产出 ELF/Mach-O 等目标文件并经系统链接器生成可执行文件。适用于对零 C/零 LLVM 依赖、体积与可控性有要求的场景。

---

## 一、概述与目标

### 1.1 目标

- **输入**：经 typeck 的 AST（Module + Arena），与当前 codegen.x 的输入一致。
- **输出**：可执行文件（或 .o 目标文件 + 调用系统 ld 链接）。
- **约束**：不依赖 C 编译器、不依赖 LLVM；可依赖系统 **汇编器/链接器**（如 `as`、`ld`）以降低首版工作量，或完全自研。

### 1.2 在流水线中的位置

```
当前：  AST --[codegen.x]--> C 源码 --[cc]--> .o --[ld]--> 可执行文件
目标：  AST --[native codegen]--> 机器码/汇编 --[as 或自研]--> .o --[ld]--> 可执行文件
```

若采用「先出汇编再调 as」的折中方案，则：

```
AST --[native codegen]--> 汇编文本 (.s) --[as]--> .o --[ld]--> 可执行文件
```

这样可暂不实现二进制目标文件格式与编码，先打通端到端，再替换为直接写 .o。

---

## 二、总体架构

直接出机器码后端可拆成以下阶段（顺序可略有重叠）：

| 阶段 | 输入 | 输出 | 说明 |
|------|------|------|------|
| **1.  lowering** | AST（函数/表达式/语句） | 线性 IR 或带基本块的 IR | 将 AST 转为更接近机器指令的表示（虚拟寄存器、线性指令、基本块+分支）。 |
| **2. 指令选择** | IR / 抽象指令 | 目标架构指令（如 x86-64 的 add/sub/mov） | 每条「抽象操作」映射到一至多条机器指令。 |
| **3. 寄存器分配** | 带虚拟寄存器的指令序列 | 物理寄存器或栈槽 | 把虚拟寄存器映射到物理寄存器（或溢出到栈）。 |
| **4. 指令编码** | 指令 + 操作数 | 字节流（机器码） | 按 CPU 手册把指令编码为字节（含 REX/ModR/M/SIB 等）。 |
| **5. 布局与重定位** | 字节流 + 符号/标签 | 带重定位信息的段 | 确定 .text/.data 等布局，记录哪些位置需要链接时填写的地址。 |
| **6. 目标文件生成** | 段 + 符号表 + 重定位表 | .o 文件（ELF 或 Mach-O） | 按目标文件格式写出头部、节表、节内容、符号表、重定位表。 |
| **7. 链接** | 一个或多个 .o + 运行时 | 可执行文件 | 调用系统 `ld` 或自研简单链接器（解析符号、合并段、填重定位）。 |

**折中路径**：先做 **AST → 汇编文本 (.s)**，用系统 `as` 生成 .o，再用 `ld` 链接。这样 4、5、6 由 as 完成，只做 1～3 再加「生成汇编字符串」即可，工作量显著减小。

## 三、是否引入 IR

### 3.1 选项 A：AST 直接到机器码/汇编

- **做法**：为每种 AST 结点（如 ADD、CALL、RETURN、IF）写一个「生成若干条机器指令或汇编行」的函数，递归遍历 AST，边遍历边往缓冲区写指令或汇编文本。
- **优点**：实现路径短，无需定义 IR 数据结构；与现有 codegen.x「AST → 线性输出」的思路一致，只是输出从 C 文本变成汇编/机器码。
- **缺点**：优化空间小（如公共子表达式、寄存器复用难做）；控制流（if/loop）要手写标签与跳转，易出错；后续若做优化或换架构，改动面大。

### 3.2 选项 B：引入简单线性 IR 再生成机器码

- **做法**：先做 **AST → 线性 IR**。IR 由「抽象指令」组成，例如：`mov v1, imm`、`add v2, v1, v3`、`br cond, label_else`、`ret v2`。操作数为虚拟寄存器（v0, v1, …）或立即数或标签。再写 **IR → 指令选择 → 寄存器分配 → 编码**。
- **优点**：指令选择、寄存器分配与「AST 递归」解耦，便于优化（如将来做常量传播、死代码消除）；多架构时只需换「IR → 机器码」这一层。
- **缺点**：要多一套 IR 的数据结构与 lowering 代码，首版工作量更大。

### 3.3 建议

- **首版 / 验证路径**：采用 **选项 A + 出汇编文本**。即 AST 直接生成汇编（.s），不定义 IR；每个 ExprKind/语句对应一段「生成汇编字符串」的逻辑。用系统 `as` 得到 .o，用 `ld` 链接。先跑通最小子集（如 main 里 return 42、简单加减、一次函数调用）。
- **后续**：若要做优化或多架构，再引入线性 IR（选项 B），把现有「AST → 汇编」改成「AST → IR → 汇编/机器码」。

## 四、各阶段详解

### 4.1 Lowering（AST → 可生成指令的表示）

**若采用 AST 直接出汇编**，则「lowering」就是遍历 AST 并针对每种结点生成汇编：

- **字面量**（LIT、BOOL_LIT）：生成 `mov reg, imm` 或对应架构的立即数加载。
- **变量**（VAR）：若在栈上，生成 `mov reg, [rbp - offset]`；若在寄存器，直接引用该寄存器。
- **二元运算**（ADD/SUB/MUL…）：先递归生成左、右操作数到寄存器（或栈），再生成一条运算指令（如 `add rax, rbx`）。
- **函数调用**（CALL）：按 ABI 把参数放入约定寄存器/栈，生成 `call 符号名`，返回值在约定寄存器（如 rax）。
- **return**：把返回值放入约定寄存器，生成 `ret`。
- **if/block**：生成标签（如 `.L_then`、`.L_else`、`.L_end`），条件为假时 `jmp .L_else`，then 块末 `jmp .L_end`，再生成 else 与 end 的代码。
- **循环**（for/while）：同理，用标签 + `jmp`/`jcc` 实现。

**局部变量与栈帧**：为每个函数维护「局部变量 → 栈偏移」的映射；函数入口生成 `push rbp; mov rbp, rsp; sub rsp, N`，出口生成 `mov rsp, rbp; pop rbp; ret`。

**与现有 codegen 的对应**：当前 codegen.x 的 `emit_*` 是「写 C 字符串」；直接出汇编时改为「写汇编行」即可（如 `emit_asm(out, "mov eax, %d", lit_val)`），数据结构可复用「输出缓冲区」思路。

---

### 4.2 指令选择（若用 IR）

若采用线性 IR，则「指令选择」把每条 IR 指令映射到目标架构的一至多条机器指令，例如（x86-64）：

- `add v1, v2, v3` → `mov rax, [v2]; add rax, [v3]; mov [v1], rax`（若 v 都在栈上）或直接用寄存器指令。
- `br cond, label` → 比较 + `jcc label`（如 `jne`、`je`）。
- `ret v` → `mov rax, [v]; ret`。

若 **AST 直接出汇编**，则没有单独的「IR 指令选择」阶段，而是在 lowering 时直接选好目标指令（如 ADD 结点直接生成 `add` 汇编）。

---

### 4.3 寄存器分配

**目标**：把「虚拟寄存器」或「临时值」映射到物理寄存器（如 rax、rbx、rcx…）或栈槽（当寄存器不够时「溢出」到栈）。

**简单做法（首版）**：

- **不显式做寄存器分配**：所有局部变量和临时值都放在栈上（如 `[rbp-8]`、`[rbp-16]`…），每次运算从栈 load 到固定寄存器（如 rax）算完再 store 回栈。这样无需分配算法，代码生成简单，但性能与代码体积较差。
- **简单分配**：为每个函数维护「当前可用的物理寄存器」集合；生成一条运算时，优先给结果分配一个空闲物理寄存器，用完后释放；若没有空闲则选一个溢出到栈（或复用已溢出槽）。可先做「线性扫描」分配：按指令顺序扫描，为每个虚拟寄存器分配一个物理寄存器或栈槽，在生命周期结束处释放。

**ABI**：需遵守目标平台的调用约定（如 x86-64 System V：参数用 rdi, rsi, rdx, rcx, r8, r9；返回值 rax；被调用者保存 rbx, rbp, r12–r15）。在生成 `call` 前后按约定使用/保存寄存器。

---

### 4.4 指令编码（x86-64 示例）

若 **直接出二进制 .o**（不经过 as），则需要把每条指令编码成字节。x86-64 指令格式大致为：

- **前缀**：REX（0x40–0x4f）用于 64 位与扩展寄存器；必要时 VEX/EVEX（SIMD）。
- **opcode**：1–3 字节，如 ADD 的 0x01、0x03 等。
- **ModR/M**：指定操作数（寄存器/内存、寄存器编号）。
- **SIB**：索引寻址时使用（base + index*scale + disp）。
- **立即数/偏移**：1/2/4/8 字节。

**编码表**：需要按 Intel/AMD 手册或现有表格，为每条用到的指令维护「opcode + ModR/M 规则」。例如：

- `mov rax, imm64`：REX.W + B8 + rd + imm64。
- `add r64, r/m64`：REX.W + 01 /r 或 03 /r。
- `ret`：C3。
- `call rel32`：E8 + rel32（相对当前 PC 的偏移）。

**实现方式**：为每种要支持的指令写一个 `encode_xxx(buf, &len, operands)`，把编码结果追加到 `buf`，并记录该指令占用的长度（用于后续重定位时计算偏移）。重定位处先填 0，在「布局与重定位」阶段再填实际值或交给链接器。

**首版建议**：若走「先出汇编再 as」的路径，可**不做指令编码**，只生成 .s 文本，由 as 完成编码。

---

### 4.5 节、标签与重定位

- **节（section）**：至少需要 `.text`（代码）、`.data`（已初始化数据）、`.bss`（未初始化数据，可选）。每个函数对应 .text 里一段连续字节；全局变量对应 .data/.bss 中的标签。
- **标签**：每个函数入口、每个全局变量、每个内部分支目标（如 if/loop 的 label）对应一个符号或局部标签。生成指令时，跳转目标用标签名表示；若出汇编，则写 `jmp .L_foo`；若出二进制，则记「此处需要重定位」，重定位类型为 PC 相对或绝对。
- **重定位**：凡「需要链接时才能确定的值」都记一条重定位项，例如：`call foo` 的偏移、`mov rax, [rip + 全局变量]` 的偏移、.data 中对其他符号的引用。格式依目标文件（ELF/Mach-O）而定。

**若出汇编**：标签和重定位由汇编器处理，我们只负责生成正确语法的 `.globl main`、`.text`、`label:`、`call foo` 等。

---

### 4.6 目标文件格式（ELF 为例）

若要 **直接写 .o**（不调 as），需要生成 ELF（Linux）或 Mach-O（macOS）。以 ELF 为例，结构大致为：

1. **ELF header**：魔数、类型（RELOCATABLE）、机器类型（x86-64）、入口、program header 数量（.o 可为 0）、section header 偏移等。
2. **Section headers**：每个节一条：名字、类型、标志、虚拟地址（可为 0）、文件偏移、大小、对齐、链接信息等。
3. **.text**：代码字节流。
4. **.data**：已初始化数据。
5. **.symtab**：符号表（函数名、变量名、节索引、值、大小、绑定/类型）。
6. **.strtab**：符号名字符串表。
7. **.rela.text**（或 .rel.text）：.text 的重定位项（偏移、类型、符号、加数）。
8. **.shstrtab**：节名字符串表。

**实现要点**：先确定各节的大小与内容，再计算每个节在文件中的偏移，最后写 header 和 section headers。重定位类型需与 ABI 一致（如 x86-64 的 R_X86_64_PC32、R_X86_64_64 等）。

**参考**：可对照 `man elf`、binutils 源码、或现有小编译器（如 8cc、chibicc）的 .o 输出逻辑。

---

### 4.7 链接

- **调用系统 ld**：首版推荐。生成一个或多个 .o 后，执行 `ld -o a.out a.o -lc`（或等价），让系统链接器解析符号、合并段、填重定位、链 C 运行时。我们只需正确生成 .o。
- **自研简单链接器**：若要求「零外部工具」，则需自研：读入多个 .o，解析 ELF 头与节、符号表、重定位表；按符号解析结果把段合并到同一地址空间，根据重定位类型计算并回填地址；写出可执行 ELF（类型为 EXEC，需 program header、加载地址等）。工作量较大，可作为后续阶段。

## 五、数据结构建议（.x 或 C）

以下用「模块 + 核心结构」方式列出，便于在 .x 或 C 中实现。

### 5.1 若采用「AST 直接出汇编」

- **AsmBuf**：类似现有 `CodegenOutBuf`，用于累积汇编文本；`append_str`、`append_int`、`label(name)`、`insn(op, operands)` 等。
- **FunctionContext**：当前函数内「局部变量/临时 → 栈偏移」、栈帧大小、下一个可用偏移；在进入/退出函数时重置。
- **符号表**：函数名、全局变量名 → 是否已定义、所在节、偏移（若单模块可简化）。

### 5.2 若引入线性 IR

- **Instr**：一条指令，如 `{ op: ADD, dst: v1, src1: v2, src2: v3 }` 或 `{ op: BR, cond: v, target: label }`。
- **BasicBlock**：指令列表 + 入口/出口（fallthrough / 跳转目标）。
- **FunctionIR**：基本块列表、标签到块索引的映射、虚拟寄存器数量。
- **LiveRange**：每个虚拟寄存器的定义与使用位置，供寄存器分配使用。

### 5.3 若直接出二进制 .o

- **Section**：名字、类型、字节内容（可动态扩展）、重定位项列表（偏移、类型、符号索引、加数）。
- **SymbolTable**：符号名、节索引、在节内偏移、大小、绑定（LOCAL/GLOBAL）、类型（FUNC/OBJECT）。
- **ObjectFile**：ELF 头 + Section 数组 + 符号表 + 字符串表；最后 `write_elf(f, obj)` 写出文件。

## 六、与现有 codegen 的衔接

- **入口**：与 codegen.x 一致，由 pipeline 在 typeck 之后调用「某 codegen 入口」，传入 Module + Arena + 输出缓冲。可增加「后端选择」：`-backend c`（现有）、`-backend asm`（出 .s）、`-backend obj`（出 .o）。
- **复用**：类型到「寄存器/栈槽大小」的映射（i32→4、i64→8、指针→8）、ABI 参数顺序、函数名/全局变量名，可与现有逻辑共享或从 AST/typeck 结果读取。
- **替换点**：当前 `emit_*` 写 C 字符串；新后端改为写汇编行或二进制段。可抽一层「CodegenBackend」接口：`emit_lit(out, val)`、`emit_binop(out, op, left, right)`、`emit_call(out, name, args)` 等，C 后端与 asm/obj 后端分别实现。

### 6.1 代码放在哪：目录布局建议

**是的，直接写在 compiler 目录里**，和现有 lexer/parser/typeck/codegen 同层，不要单独再起一个“编译器二”的仓库。

两种常见放法都合理：

**方案 A：`src/asm/`（与 lexer/parser/typeck/codegen 平级）**

```
compiler/src/
├── lexer/
├── parser/
├── typeck/
├── codegen/            # 现有：AST → C 源码
├── asm/                # 直接出机器码（汇编）后端，与 codegen 平级
│   ├── backend.x     # AST → 汇编入口、缓冲区、分派
│   ├── x86_64.x      # x86-64 指令/ABI/汇编
│   └── (可选) arm64.x
├── pipeline/
└── ...
```

- **优点**：asm 和 codegen 一样是「一种输出路径」，平级更直观；目录名短、路径简单；以后 asm 下可再分子目录（如 asm/elf、asm/x86_64）而不挤在 codegen 里。
- **适用**：把「直接出机器码」当成和「出 C」并列的一等公民时用。

**方案 B：`src/codegen/asm/`（作为 codegen 的子后端）**

```
compiler/src/codegen/
├── codegen.x         # 现有 C 后端
├── asm/               # 子目录：asm 后端
│   ├── backend_asm.x
│   └── x86_64.x
└── ...
```

- **优点**：所有「从 AST 到目标码」的逻辑都收在 codegen 下，概念上统一；和「codegen/ — 代码生成」的 README 一致。
- **适用**：强调 asm 只是 codegen 的多种实现之一（C / asm / 将来 LLVM）时用。

**建议**：若你更在意「asm 和 C 后端平起平坐、路径简单」，用 **`src/asm/`** 即可；若更在意「所有目标码生成都在 codegen 名下」，用 **`src/codegen/asm/`**。两种都合理，选一种统一即可。

- **pipeline**：无论哪种，都在 `src/pipeline/pipeline.x` 里根据 `-backend asm` 等选择调用 asm 后端或现有 codegen；多架构仍按架构分文件（如 x86_64.x、arm64.x）。

## 七、工作量与阶段拆分（估算）

| 阶段 | 内容 | 工作量（人月，量级） |
|------|------|----------------------|
| **7.1** | AST → 汇编文本 (.s)，仅支持 main + return 42 + 简单加减 + 一个 call | 0.5～1 |
| **7.2** | 扩展：局部变量、if/else、循环、多函数；调 as + ld 产出可执行文件 | 1～2 |
| **7.3** | 简单寄存器分配（线性扫描），减少栈溢出，代码质量可接受 | 0.5～1 |
| **7.4** | 直接出 ELF .o（不调 as）：指令编码 + 节 + 符号表 + 重定位 | 2～3 |
| **7.5** | 多架构（如再加 ARM64）：指令编码与 ABI 另一套 | 1～2 每架构 |
| **7.6** | 自研链接器（可选）：读 .o、合并段、重定位、写可执行文件 | 2～3 |

**首版最小闭环**：7.1 + 7.2（约 1.5～3 人月）可得到「shux -backend asm file.x -o a.out」且不依赖 C 编译器，仅依赖 as/ld。再按需做 7.3～7.6。

## 八、参考资料与可借鉴实现

- **指令编码**：Intel 64 and IA-32 Architectures Software Developer’s Manual, Vol. 2（Instruction Set Reference）；AMD64 ABI（调用约定、ELF 重定位类型）。
- **ELF**：Tool Interface Standard (TIS) ELF Specification 1.2；`readelf -a xxx.o` 对照现有 .o。
- **小编译器**：chibicc（C 编译器，有从 AST 生成 .s 的代码）；8cc（C 子集，有简单 codegen）；tcc（可直接生成 x86 机器码，可参考其编码与重定位）。
- **寄存器分配**：Linear Scan 算法（如 Poletto & Sarkar）；若只做栈分配可先不做。

## 九、小结

| 决策点 | 建议 |
|--------|------|
| 首版输出形式 | 先 **汇编文本 (.s)**，用系统 as 生成 .o，再 ld 链接。 |
| 是否要 IR | 首版 **不要**，AST 直接生成汇编；后续若做优化再引入线性 IR。 |
| 寄存器分配 | 首版 **全栈** 或极简线性扫描；后续再加强。 |
| 直接写 .o | 作为 **第二阶段**，在「出 .s + as」稳定后再做。 |
| 链接 | 首版 **调系统 ld**；自研链接器可选、后期再做。 |

按上述分阶段推进，可在约 2～3 人月内实现「直接出机器码（经汇编器）」的可用后端，再视需求做二进制 .o 与自研链接，逐步逼近「零 C、零 LLVM、体积与行为完全可控」的目标。
