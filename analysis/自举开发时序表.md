## 从部分实现到完全自举：何时、如何达成

阶段 2 表中 **2.8～2.10** 已与 **`parse_expr_into`**、显式 **`return`**、块末 **`EXPR_RETURN`** 包装对齐；宿主 **C typeck** 与 **`-su` typeck** 在个别规则上仍可收敛（见 NEXT §四、tests/typeck）。

### 第一步：解阻（改 C，使 .sx 能继续扩展）

| 阻塞点 | 在 C 中的处理 | 目的 |
|--------|----------------|------|
| **单函数 let 上限** | C/`parser.c` 与 `parser.sx`/`OneFuncResult`/`ast.sx::Block` 统一到 **256**（`MAX_CONST_DECLS` / `MAX_LET_DECLS`） | 避免因 SU/C 一端截断另一端「too many let」；更复杂解析若仍不足可提高上限或可配置 |
| **bootstrap-parser 失败** | 修复 codegen / 结构体与 dummy 数组尺寸（如 `OneFuncResult` forward、数组字面量、[256] 槽位对齐） | `make bootstrap-parser`、`make bootstrap-parse-file`（`compiler` 目录）通过 |
| **pipeline.sx 无法由 C shuc 编出 pipeline_gen.c** | 1) typeck 入口时传入**全部传递依赖**（all_dep_mods）供布局阶段解析跨模块类型（如 parser 的 OneFuncResult.next_lex: Lexer）；2) codegen 库模块路径设置 **codegen_current_module** 以便结构体字面量含数组字段时走块形式；3) -E 单文件输出增加 `#include <string.h>` | 使 `make shuc-sx-pipeline` 成功，得到 shuc_su |
| **其他 .sx 语法/类型缺失** | 在 parser.c、lexer.c、ast.h、codegen.c 等中**扩展语法/AST/输出**（仅作能力扩展，业务逻辑仍在 .sx） | 使 compiler 下所有 .sx 所需语法与类型均被当前 shuc 支持 |

完成「解阻」后：**重新编译 shuc**，得到能编译更复杂 parser.sx 的宿主编译器。

### 第二步：在 .sx 中补全阶段 2～4

- **阶段 2**：新语法按需扩 `parser.sx`；**`parse_expr_into` 与各语句路径** 已接通 return、循环条件、`let` 初值。
- **阶段 3**：在 typeck.sx 中持续与 C **typeck_module** 对齐（含 **RETURN** 等与 **`-su`** 共用规则）。
- **阶段 4**：在 codegen.sx 中补全 ExprKind / 收口 **extern glue**（如 import 路径常量判断迁入 .sx），使 **`-su`** 链路可稳定编编译器主体。

### 第三步：阶段 5～8（多文件、driver、build、自举验证）

- **阶段 5**：多文件与 import（C 辅助或 .sx 实现）。
- **阶段 6**：driver.sx 为唯一入口，不再依赖 main.c 的驱动逻辑。
- **阶段 7**：build.sx 替代 Makefile，用 .sx 描述构建。
- **阶段 8**：用宿主编出 shuc₁，用 shuc₁ 编出 shuc₂，两代对同一测试套件一致，即**完全自举达成**。

### 时间线（概念）

- **短期**：**let 上限**、`bootstrap-parser` / **`bootstrap-parse-file`**（含括号 return）、**`-su` return 类型负例** 已就位；接续 **收窄 codegen/typeck extern**、**宿主与 -sx typeck** 对齐。
- **中期**：阶段 3/4 在 `.sx` 侧与宿主深化一致，使 **`-su`** 流水线编编译器更稳。
- **长期**：阶段 5～8 完成，`driver.sx` + `build.sx` + 自举验证，实现**完全自举**。

结论：**先在 .sx 内落地再在宿主扩孔**；遇阻时可改 **C** 再接续 **.sx**。

## 迁移到 .sx 进度（逻辑迁出 C、少调 C 方法）

- **已就绪**：`make migrate-su-objs` 可生成 parser_sx.o、typeck_su.o、codegen_su.o（由 parser.sx / typeck.sx / codegen.sx 经 shuc -E 产出 C 再编成 .o）。
- **当前构建**：bootstrap-driver 仍用「pipeline.sx -E 单文件内联 parser/typeck/codegen」的 pipeline_gen.c，故 pipeline_sx.o 已含 .sx 逻辑；宿主编译 pipeline_gen.c 时仍走 C 前端。
- **下一步**：若要让「编出 shuc 的整条链路」也走 .sx（即链入 parser_sx.o / typeck_su.o / codegen_su.o 而不再链 parser.o / typeck.o / codegen.o），需 shuc 支持「-E 仅展开入口、import 用 extern」或等价机制，避免 pipeline_gen.c 内联依赖导致重复符号。
- **完全脱离 C / 不再需要 Makefile**：分步表格与验收见 **《[完全脱离C与Makefile路线图](完全脱离C与Makefile路线图.md)》**（阶段 1～5，逐项实现并打勾）。

### 自举验证（不依赖 make 的流程）

若已有 `build_tool` 与任一可用 `shuc`：在仓库根执行 `cd compiler && ./build_tool ./shuxc` 产出新 shuc，再执行 `./tests/run-all.sh`；全部通过即等价于自举验证通过。从零构建时需先 `make -C compiler` 得到初始 shuc，再 `make -C compiler build-tool` 得到 build_tool，之后即可仅用 build_tool 构建。可选：`make -C compiler bootstrap-verify` 仍可用作兜底验收（内部会 make）。
