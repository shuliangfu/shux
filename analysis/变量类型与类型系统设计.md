## 变量类型与类型系统设计（草案）

> 本文档专门回答：「**Shux 要支持哪些变量类型？**」「**类型系统大致长什么样？**」  
> 目标是**列全 + 分阶段**，为后续 parser/typeck/codegen 与标准库设计提供硬约束。

---

## 一、设计目标与约束

- **参考 C，但比 C 简单、语义更清晰**（对应《语法与类型设计-高性能与内存安全.md》的定位）：
  - 有整型、指针、结构体、数组/切片、函数类型等；
  - 但用更显式的可空/边界/所有权语义，减少未定义行为。
- **分阶段演进**：
  - 目前实现：仅 `i32`（字面量与变量类型）、`function main() -> i32`；
  - 后续在不推翻现有代码的前提下，逐步引入更多类型。
- **实现难度可控**：
  - 编译器前期仍是「简单 AST + 直接生成 C」；
  - 复杂能力（所有权、Option/Result、trait）更多放在阶段 7 之后。

---

## 二、类型家族总览（按优先级）

### 2.1 按「必须 / 推荐 / 可选」划分

| 类别 | 类型示例 | 用途 | 优先级 |
|------|----------|------|--------|
| **整数标量** | `i32`、`i64`、`u32`、`u64`、`isize/usize` | 算术、索引、指针大小 | **必须（最先）** |
| **布尔** | `bool` | 条件、逻辑表达式 | 必须 |
| **字符/字节** | `u8`、`char`（Unicode 标量） | 字符串、字节序列 | 推荐（较早） |
| **指针/引用** | 裸指针 `*T`、借用/引用 `&T` | 与 C 互操作，手动内存管理 | 推荐（随所有权设计） |
| **数组/切片** | 固定长 `T[N]`、切片 `T[]` 或 `slice<T>` | 容器、字符串、缓冲区 | 推荐（配合安全边界） |
| **结构体** | `struct Point { x: i32, y: i32 }` | 自定义复合类型 | 必须 |
| **枚举/代数数据类型** | `enum Option<T>`、`Result<T, E>` | 可空、错误处理 | 推荐（阶段 7） |
| **函数类型** | `fn(i32) -> i32` 等 | 回调、高阶函数 | 可选（视需求） |
| **浮点** | `f32`、`f64` | 计算、图形等 | 可选（后期） |

### 2.2 与变量的关系

- 变量始终是「**持有某个具体类型的值**」：
  - `let x: i32 = 1;`
  - `let p: *i32 = ...;`
  - `let buf: u8[16] = ...;`
- 类型的复杂度**集中在类型本身**，不是语法糖：
  - `Option<T>` 真的就是一个枚举，不是编译器偷改；
  - `T[]` 若采用切片语法，本质等同于 `{ ptr: *T, len: usize }` 的结构体。

## 三、阶段划分：什么时候支持哪些类型？

> 这里只给出「建议时序」，方便你把实现任务分批放到不同阶段。

| 阶段 | 主要目标 | 必须类型 | 建议顺带上的类型 |
|------|----------|----------|------------------|
| 当前 ~ 阶段 3 | 表达式/控制流打底 | `i32` | （已完成） |
| 阶段 4–5 | 多文件、基础 std | `i32`、`bool`、`u8`、`struct` | `i64`、`u64`、`usize/isize` |
| 阶段 6 | 标准库初版 | 上述 + 指针 `*T`、固定长数组 `T[N]` | `T[]` 切片（值语义） |
| 阶段 7 | 泛型 + trait | `Option<T>`、`Result<T, E>`、更丰富 struct/enum | 结合 trait 的接口类型 |
| 阶段 8+ | 性能与安全增强 | 所有权/借用标注、unsafe 区域 | 浮点 `f32/f64` 等 |

**结论**：我们现在写的分析文档，重点是**阶段 4–7** 的类型设计；实现可以按阶段渐进落地。

### 3.1 极限压榨性能：阶段与特性映射

在「极致压榨性能」目标下，类型系统可从**内存布局、零开销抽象、编译器 Hint** 三个维度做深度优化。下表将相关特性按阶段落地，与上表配合使用：

| 阶段 | 极限压榨相关特性 | 目标场景 |
|------|------------------|----------|
| **阶段 6** | `align(N)` 属性、`packed`、结构体 padding 检查与 **Zero-Padding 强制校验**；`*unaligned T` 与对齐指针区分 | 喂满 NVMe 随机读、Direct I/O、Cache Line 友好 |
| **阶段 8** | 饱和算术操作符（如 `+!`）、回绕算术（如 `+%`）；Option 空指针优化（零成本布局）；**`*restrict T` 别名优化** | 消除多余跳转与内存浪费，激进向量化、音频/图像饱和运算 |
| **阶段 10** | **专用 Mask 类型 `m32x4`**（映射 AVX-512/AMX 掩码寄存器）；多维/跨步切片视图 `{ ptr, len, stride }`；**Const-Size Slice `slice<T, const N>`**；`'static` 与只读段 | 压榨 GPU/AVX-512 吞吐、张量零拷贝视图、边界检查消除与循环展开、权重预载 |

**压榨性能硬指标（实现进度细项）**：以下细项作为 typeck/codegen 不掉速的硬指标，实现后可将状态改为 ✅：

| 特性 | 压榨点 | 状态 |
|------|--------|------|
| 显式 Padding 校验（Zero-Padding + `allow(padding)`） | 消除无效内存读取与缓存污染 | ✅ 已实现（§11.1） |
| `*restrict T` 指针 | 激活编译器激进向量化优化 | ⬜ 待加入 Stage 8 |
| SIMD Mask `m32x4` | 映射至 AVX-512/AMX 掩码寄存器 | ⬜ 待加入 Stage 10 |

## 四、基础标量类型设计

### 4.1 整数族

目标：**和 C 接近，但语义更清晰**。

- 有符号：`i8`、`i16`、`i32`、`i64`、`isize`
- 无符号：`u8`、`u16`、`u32`、`u64`、`usize`

建议语义（后续在类型文档里详细写）：

- 溢出行为：**调试构建下检查并 panic；发布构建下默认为 wrapping**，或明确选择其一；
- **极限压榨**：阶段 8 支持显式饱和算术（如 `+!`）、回绕算术（如 `+%`），便于 SIMD 饱和指令与 C 互操作，见 §11.2。
- `isize/usize`：大小与目标指针宽度一致，用于索引、指针差值；
- 与 C 互操作时：
  - `i32` ↔ `int` 或 `int32_t`；
  - `u8` ↔ `uint8_t`/`unsigned char`；
  - `usize` ↔ `size_t`。

### 4.2 布尔与条件

- 类型：`bool`；
- 字面量：`true` / `false`；
- 仅允许：
  - `if (bool_expr) { ... }`；
  - `while (bool_expr) { ... }`；
  - `&&` / `||` / `!` 作用于 `bool`；
- **不允许**隐式从整数转 bool（避免 C 那种隐式规则），需要显式比较或转换：
  - 写成 `x != 0`，而不是 `if (x) {...}`。

### 4.3 字节与字符

- `u8`：**字节**，用于缓冲区、二进制数据；
- `char`（可选，后期）：**Unicode 标量值**或 UTF-8 codepoint，语义要在文档中说清：
  - 若早期不想踩 Unicode 坑，可以只用 `u8` + `u8[]` 表示字符串。

## 五、指针与引用（值 vs 地址）

> 这是变量类型设计里最敏感的一块，要兼顾「底层可控」与「默认安全」。

### 5.1 裸指针 `*T`

- 语法形态：`*T`；
- 语义：接近 C 的指针，**默认可为 null**，可算术、可别名；
- 用途：
  - 与 C 函数/库互操作；
  - 在 `unsafe` 区域手写高性能代码。
- 安全性：
  - 在「安全子集」之外使用，建议需要 `unsafe` 标注（阶段 8 再精化）；  
  - typeck 层不保证越界/空指针安全，交给用户与运行时检查。
- **极限压榨**：阶段 6 起区分对齐指针 `*T` 与非对齐 `*unaligned T`；阶段 8 起支持 `*restrict T` 以激活激进向量化，见 §11.3。

### 5.2 引用/借用 `&T`（后期）

- 语法形态：`&T`（只读）、`&mut T`（可变）——具体语法可后定；
- 语义：**受限指针**，编译器能在类型上保证：
  - 生命周期内不悬空；
  - 不会与另一个可变引用形成数据竞争。
- 用途：
  - 绝大多数「传地址」场景都用 `&T`，而不是 `*T`；
  - 在类型系统里表达借用关系，为后续所有权规则做准备。

### 5.3 指针与切片的关系

- 裸指针：`*T`；
- 切片：`T[]` / `slice<T>`，本质是 `{ data: *T, len: usize }`；
- 字符串可以是：
  - `u8[]`（UTF-8），或
  - `slice<u8>`，在 `core` 里 typedef/alias。

## 六、复合类型：结构体 / 数组 / 切片

### 6.1 结构体 `struct`

- 语法示例：

```su
struct Point {
  x: i32
  y: i32
}

let p: Point = Point { x: 1, y: 2 };
```

- 语义：
  - 与 C 的 `struct` 类似，字段顺序决定内存布局；
  - 默认按目标 ABI 对齐；
  - 后续可引入 `packed`、`repr(C)` 等属性。
- **极限压榨**：支持 `align(N)`（如 Cache Line 64）、`packed`，Typeck 对 padding 发出警告或建议，见 §11.1。

### 6.2 固定长数组 `T[N]`

- 语法示例：`let a: i32[4] = [1, 2, 3, 4];`、`a[0]`、`a[i]`
- 语义：
  - 编译期常量 `N`，值语义，存储在栈或静态区；
  - 边界检查策略：  
    - 编译期可证明安全 → 消除检查；  
    - 否则调试构建下检查，出错 panic。
- 与指针：
  - `T[N]` 可以显式转换为 `*T` 或 `T[]`（切片），不做隐式退化。

**当前实现要点（§三 进度表 6.2）**：类型 `T[N]` 在 parser 中解析为 `AST_TYPE_ARRAY`（elem_type + array_size）；const/let/结构体字段支持数组类型，codegen 生成 `elem_type name[N]`；**数组字面量** `[e1, e2, ...]` 解析为 `AST_EXPR_ARRAY_LIT`，typeck 要求与声明类型长度一致，codegen 输出 `{ e1, e2, ... }`；**下标访问** `base[index]` 解析为 `AST_EXPR_INDEX`，typeck 要求 base 为数组类型（符号表通过 type_ptrs 解析 elem_type），codegen 输出 `(base)[index]`。零初始化 `= 0` 生成 C 的 `{0}`。

### 6.3 切片 `T[]` 或 `slice<T>`

两种可选设计：

- **方案 A：语法级 `T[]`**（更易用）
  - 语法：`let s: u8[] = buf[0..n];`
  - IR/实现：等价于 `struct { data: *u8, len: usize }`。
- **方案 B：库类型 `slice<T>`**（实现简单）
  - 语法：`let s: slice<u8> = make_slice(&buf, n);`
  - 对编译器而言只是一种普通泛型结构体。

建议：**语义按方案 A 设计，第一版实现可以用方案 B 落地**，等语言成熟再开放 `T[]` 语法糖。  
**极限压榨**：扩展为带步长的视图 `{ ptr, len, stride }`（多维/跨步切片），以及**已知长度切片** `slice<T, const N>`（消除边界检查、循环展开），见 §11.4。

**当前实现要点（§三 进度表 6.3）**：类型 `T[]` 解析为 `AST_TYPE_SLICE`（elem_type）；codegen 为每种元素类型生成 `struct shux_slice_<elem> { elem* data; size_t len; }`。仅支持从**数组变量**初始化切片：`let s: i32[] = a`（a 为 `i32[N]`），生成 `{ .data = a, .len = N }`。下标 `s[i]` 生成 `(s).data[i]`。typeck 允许下标基类型为数组或切片，并设置 `base_is_slice` 供 codegen 区分。

## 七、枚举 / Option / Result（为错误处理与 try 做准备）

> 这部分主要支撑 `Option`/`Result` 以及未来的 `try` / `?` 语法。

### 7.1 枚举 `enum`

- 语法示例：

```su
enum Color {
  Red,
  Green,
  Blue,
}
```

- 语义：
  - 类似 C 枚举 + Rust 枚举：可以是纯标签，也可以带数据；
  - 带数据形式示例：

```su
enum Option<T> {
  None,
  Some(T),
}
```

### 7.2 Option 与 Result

- `Option<T>`：表示「可能没有值」，取代「可为 null 的指针」：
  - `None`：无值；
  - `Some(T)`：有值。
- `Result<T, E>`：表示「结果或错误」：

```su
enum Result<T, E> {
  Ok(T),
  Err(E),
}
```

- 变量可以写成：
  - `let x: Option<i32>;`
  - `let r: Result<i32, MyError>;`
- 将来 `try` / `?` 语法就以这些类型为基础：
  - `try { expr } catch { handler }` 可以被翻译为操作 `Result`；
  - `expr?` 可以被翻译为：若 `Err` 则向上传播，否则解包 `Ok`。
- **极限压榨**：对 `Option<&T>` 等做空指针优化，使 `Option` 与 `T` 尺寸一致、与 C 可空指针兼容，见 §11.5。

### 7.3 与泛型/trait 的关系

- `Option`、`Result` 均是泛型枚举，因此依赖阶段 7 的泛型实现；
- `Result` 的 `E` 类型可以实现某个 `Error` trait，以便统一处理。

### 7.4 当前实现规划（无负载枚举先行）

- **本阶段**：实现**无负载**枚举（仅标签，无泛型、无变体数据）：
  - 语法：`enum Name { A, B, C }`（顶层定义）；类型写 `Name`（与 struct 同为 NAMED）；构造值写 `Name::Variant`（如 `Color::Red`）。
  - 语义：枚举类型与 C 枚举一致，codegen 生成 C 的 `enum Name { A, B, C }`，值为 `int`。
  - **不实现**：带数据的变体（如 `Some(T)`）、`match` 对枚举分支、Option/Result 库类型；留待阶段 7 泛型一并实现。
- **标准库 API 草案（阶段 7 实现时再落地）**：
  - `Option<T>`：`Option::None`、`Option::Some(x)`；方法 `.is_some()`/`.is_none()`、`.unwrap()`、`.unwrap_or(default)`。
  - `Result<T,E>`：`Result::Ok(x)`、`Result::Err(e)`；方法 `.is_ok()`/`.is_err()`、`.unwrap()`、`?` 传播。

## 八、变量声明语法与类型位置

> 上面讨论的是「类型有哪些」，这里补充「变量上如何写」。

### 8.1 变量声明

- 统一形态：`let name: Type = expr;`
  - 允许省略类型：`let x = expr;`（后期加类型推断）；
  - 当前阶段继续要求显式类型（便于 typeck 简单正确）。

示例：

```su
let a: i32 = 1;
let b: bool = a > 0;
let p: *i32 = ...;
let buf: u8[16] = ...;
let slice: u8[] = buf[0..8];
let opt: Option<i32> = Option::Some(1);
```

### 8.2 函数参数与返回类型

- 参数：`function f(x: i32, y: *u8) -> bool { ... }`
- 返回：
  - 标量：`-> i32`、`-> bool` 等；
  - 复合：`-> Option<i32>`、`-> Result<i32, Error>`；
  - 无返回值：可以定义 `-> void` 或省略，后续再精化。

## 九、与当前实现的衔接建议

结合现在的代码现状（**已有** `i32`、`bool`、`u8`/`u32`/`u64`/`i64`/`usize`/`isize`、`let`/`const`、`function main() -> i32`，见 §三 当前实现进度），可以按以下顺序推进类型支持：

1. **在 typeck 与 codegen 中引入 `bool`**  
   - lexer/parser 已经支持比较与逻辑运算；只需在 typeck 中认定它们产出 `bool`，并允许 `if/while` 条件为 `bool`。
2. **扩展整数族：`u8`、`u32`、`i64`、`usize/isize`**  
   - AST 类型描述目前是字符串 `"i32"`，可以先按「字符串枚举」方式扩展，不立刻引入完整类型系统；
   - codegen 侧按不同类型映射到 C 的 `int32_t`/`uint8_t` 等。
3. **加入 `struct` 与基本字段访问**  
   - 为变量和函数返回值提供复合类型能力，为后续 `Option`/`Result` 铺路。
4. **在分析文档中先固定好 `Option`/`Result` 形态**  
   - 真正实现放在阶段 7，与泛型一并落地；
   - 但可以先在标准库文档中写出预期 API。

这样一来，「变量类型还没写好」的问题就有了一个**完整的设计蓝图**：  
前期你只需实现 `bool + 更多整数 + struct`，后期按文档补上指针、切片、枚举、Option/Result 与所有权/借用即可。

### 9.1 下一步行动建议（Stage 4-5 之后）

**已完成**：struct 定义与字段访问、结构体字面量；裸指针 `*T`；固定长数组 `T[N]`（类型、数组字面量、下标访问）；**结构体布局计算（§11.1）**（typeck 中计算每字段偏移与结构体总大小/对齐，为 codegen 预留 `size_of`/`offset_of`）。多文件 import 已支持。

## 十、向量（SIMD）类型设计

> 本节单独规划「**向量类型 / SIMD 类型**」，它们与前面几类（标量整数、指针/引用、struct/数组/切片、枚举）**一起**构成高性能代码的核心类型族。

### 10.1 设计目标

- 直接映射到底层 SIMD 寄存器（如 SSE/AVX、NEON 等），**不引入额外运行时开销**；
- 语法简单明确，可以在**类型层面表达“这是一组一起算的标量”**，方便：
  - 手写热点循环（图像处理、数值计算、音频/视频）；
  - 给编译器清晰 hint，便于自动/半自动 vectorize；
  - 与将来的 `unsafe`/intrinsics 一起用。

### 10.2 向量类型形态（候选方案）

我们需要兼顾「**易读**」与「**接近硬件**」，建议先支持**固定宽度整数/浮点向量**：

- 命名方案（建议）：
  - 整数：`i8x16`、`i16x8`、`i32x4`、`i64x2`；
  - 无符号：`u8x16`、`u16x8`、`u32x4`、`u64x2`；
  - 浮点（后期）：`f32x4`、`f64x2` 等。
- 这些类型在编译器内部有**专门的 kind**（不是普通 struct），以便：
  - 在 IR 里映射到目标后端的 vector type（LLVM 的 `<4 x i32>` 等）；
  - 做特定指令选择与对齐优化。

后续若要更抽象，可以在标准库里引入：

- 泛型封装：`Simd<T, N>` 或 `VecN<T>`，由编译器对其中常见实例（如 `Simd<i32, 4>`）做特殊处理；
- 但第一步建议**直接用语言级别的 `i32x4` 这类名字**，实现成本最低、最直观。

### 10.3 运算与操作

向量类型的运算分为几大类：

1. **逐元素算术/位运算**
   - `+ - * /`、`& | ^ ~`、移位等；
   - 对应到底层 SIMD 指令：如 `addps/addpd/addq` 等。
2. **比较与掩码**
   - `== != < <= > >=` 生成向量掩码（布尔向量或全 0/全 1 向量），配合 `select` 使用；
   - 极限压榨下引入**专用掩码类型**（如 `m32x4`），直接映射到 AVX-512 等掩码寄存器，见 §11.6。
3. **装载与存储**
   - 从切片/数组加载：`load(&buf[i]) -> i32x4`；
   - 回写：`store(v, &buf[i])`；
   - 要求地址与长度满足对齐/越界约束，必要时在调试下检查，release 下由调用者保证。
4. **与标量的互操作**
   - **广播**：`i32x4::splat(1)` 或 `splat(1_i32)`；
   - **提取/插入**：`extract(v, idx) -> i32`、`insert(v, idx, x) -> i32x4`。

这些操作在语言层面可以定义为：

- **内建函数 / 关键字**（如 `simd_add`, `simd_load`），由编译器识别；
- 或配合标准库提供 `core.simd` 模块，将一些操作以内联函数形式暴露。

### 10.4 与变量/函数的使用方式

变量上直接使用向量类型：

```su
let v: i32x4 = i32x4::splat(0);

function dot4(a: i32x4, b: i32x4) -> i32 {
  let mul: i32x4 = a * b;
  // 假设提供水平加法 horizontal_add
  return horizontal_add(mul);
}
```

与数组/切片结合：

```su
function add_inplace(dst: i32[], src: i32[]) -> void {
  // 假设长度已对齐为 4 的倍数，边界检查由调用方保证或在调试下校验
  let i: usize = 0;
  while (i < len(dst)) {
    let d: i32x4 = load_i32x4(&dst[i]);
    let s: i32x4 = load_i32x4(&src[i]);
    store_i32x4(d + s, &dst[i]);
    i = i + 4;
  }
}
```

### 10.5 与其它类型设计的关系

- **与标量整数族**：  
  -`i32x4` 基于 `i32`，`i64x2` 基于 `i64` 等；  
  - 标量语义（溢出、符号）要先定义好，向量运算直接复用这些语义。
- **与数组/切片**：  
  - 向量类型主要从 `T[N]` / `T[]` 上做批量读写；  
  - 若数组/切片边界语义清晰，向量 load/store 的安全策略也就清晰。
- **与 Option/Result**：  
  - 性能敏感场景下，一般在**外层控制流程**用 Option/Result，而向量运算内部保持无额外装箱；
  - 这要求编译器对 `Result<Simd, E>` 等组合类型有良好布局与优化。

### 10.6 实现时序建议

在整体类型系统路线中，向量类型的落地顺序建议是：

1. **定义语法与类型名**（如 `i32x4`），在 AST/typeck 中加入对应 kind；
2. **在 codegen 层先用「C + intrinsics」模拟**：
   - 对于 x86 平台，用 `__m128i/__m128` 等封装；
   - 非支持平台可退化为 struct `{ i32, i32, i32, i32 }` 的标量实现（功能正确但不一定加速）。
3. 等 IR 与后端完善后，再将这些类型映射到统一的中间表示（如 LLVM `<4 x i32>`），并做真正的目标特定优化。

这样可以保证：**从现在开始设计所有与“极致性能”相关的类型/接口时，就把向量类型一起考虑进去**，但在实现上仍然可以渐进式演进，不一次性把所有后端都做完。

## 十一、极限压榨性能：类型系统深度优化

> 在「极致压榨性能」目标下，类型系统可从**内存布局控制、零开销抽象、编译器 Hint** 三个维度做深度优化。本节对现有设计做补充，给出具体语法/语义建议与阶段归属。

### 11.1 内存布局：显式对齐与 Padding 控制

在追求极限 I/O 和大模型读写的场景下，**Cache Line（64B）对齐**是胜负手：结构体若跨越 Cache Line 会引发两次内存访问，跨越 Page 边界会引发两次 TLB 查找。

- **结构体属性**：在 `struct` 设计中加入 `align(N)` 和 `packed`。
  - `align(N)`：强制结构体整体对齐到 N 字节（如 64 对应 Cache Line）。
  - `packed`：取消字段间 padding，与 C 的 `__attribute__((packed))` 语义一致。
- **示例**：

```su
align(64)  // 强制对齐到 Cache Line
struct Weights {
  data: f32x16
  version: i32
}
```

- **显式 Padding 检查**：Typeck 阶段默认对结构体内部的空隙（padding）发出**警告**或提供填充建议，便于用户主动消除跨 Cache Line 布局。

- **Zero-Padding 强制校验**（深度优化）：隐式 padding 不仅浪费内存，还会破坏缓存对齐、在大数据扫描时读入无用字节。建议在 struct 语义中增加**显式校验**：
  - 若结构体存在编译器自动插入的 Padding，且**未**标记 `allow(padding)`，则**编译报错**。
  - 强制开发者用显式填充字段定义每一字节，例如 `_pad: u8[N]`，保证结构体每个字节都在开发者掌控之下。
  - 需要允许自动 padding 时，显式写 `allow(padding)`，便于与 C ABI 或遗留布局兼容。
- **示例**：

```su
align(64)
struct Weights {
  data: f32x16
  version: i32
  _pad: u8[4]   // 显式补齐，避免隐式 padding；否则未写 allow(padding) 时报错
}
```

**结构体布局算法：Padding 探测（离线计算）**  
在 typeck 中处理 struct 时，可按以下逻辑执行 Zero-Padding 强制校验，便于实现时对齐算法一致、可复现：

1. **初始化**：`current_offset = 0`。
2. **遍历字段** f_i（i = 0, 1, …）：
   - 取字段 f_i 的**对齐要求** A_i（如 `i32` 为 4 字节，`f32x16` 按目标 ABI）。
   - 计算**对齐间隙**：`Gap = (A_i - (current_offset mod A_i)) mod A_i`（即当前偏移按 A_i 对齐所需补足字节数）。
   - **校验**：若 `Gap > 0` 且该结构体**未**标记 `allow(padding)`（或 `@allow(padding)`）：  
     编译报错，提示：*「在字段 f_i 之前存在 Gap 字节的隐式 Padding，请显式定义填充字段或添加 allow(padding)」*。
   - **更新偏移**：`current_offset = current_offset + Gap + size_of(f_i)`。
3. **末尾对齐**：设结构体整体对齐为各字段对齐的最大值（或 `align(N)` 若存在）。若最终 `current_offset` 不是该对齐值的倍数，则存在**结尾空隙**；若该空隙 > 0 且未标记 `allow(padding)`，同样**编译报错**。

该模型为纯“离线”计算，不依赖运行时；typeck 阶段即可完成布局与校验，为后续 codegen 提供确定的 `size_of` 与字段偏移。

**当前实现要点**：布局算法已在 typeck 中实现（`type_align_of`/`type_size_of`/`compute_struct_layouts`），结果写入 `ASTStructDef` 的 `field_offsets`、`struct_size`、`struct_align`。Zero-Padding 强制校验已实现：若存在字段前 Gap 或末尾空隙且未在 struct 前写 `allow(padding)`，编译报错；parser 支持可选 `allow(padding)` 后接 `struct Name { ... }`。

**阶段**：与阶段 6 的 `struct`、指针一起落地。

### 11.2 整数族：溢出感知与饱和算术

高性能计算（音频、图像）中，溢出处理开销巨大。SIMD 指令集（SSE/AVX）有专门的饱和加减指令；若类型层面不支持，编译器难以优化。

- **饱和算术**：显式饱和运算，如 `a +! b`（饱和加）、`a -! b`（饱和减）。
- **回绕算术**：显式 wrapping，如 `a +% b`、`a -% b`，便于与 C 的 wrapping 语义及 SIMD 指令对应。
- **理由**：在类型/操作符层面区分后，codegen 可直接选择 `_mm_adds_*` / 普通 `add` 等，无需插入运行时分支。

**阶段**：阶段 8，与溢出策略、发布构建语义一起细化。

### 11.3 指针：区分「对齐」「非对齐」与「Restrict 别名」

在大数据读写中，非对齐访问是吞吐量杀手；io_uring、GDS 等 Direct I/O 常要求 512B 或 4KB 对齐。

- **类型层面区分**：
  - `*T`：默认表示**对齐**指针（按 `T` 的自然对齐），codegen 可生成最高效的对齐 load/store。
  - `*unaligned T`：显式非对齐指针，用于需要逐字节或非对齐访问的场景；必要时生成 unaligned 指令或多次小粒度访问。
- **理由**：在类型系统里区分后，codegen 不必插入运行时对齐检查，可直接生成对应指令序列。

- **Restrict 别名优化**（深度优化）：C 的 `restrict` 在类型系统里常被忽视，但对自动向量化至关重要。
  - **方案**：在指针类型中支持 `*restrict T`，表示「当前作用域内，该指针所指向的内存没有其他指针会访问」。
  - **极致压榨点**：编译器可据此做**激进循环向量化**，无需担心内存别名（Aliasing）导致的写覆盖，从而避免保守的标量展开。大模型权重读取后的 Tensor 运算中，这是性能翻倍级别的优化。
  - **与 C 互操作**：codegen 可直接映射为 C 的 `restrict` 限定符，便于与现有 C 库对接。

- **向量化利器：`*restrict` 的 Codegen 映射**（Stage 8 实现时）：后端逻辑保持简单，即可释放优化效果。
  - **C 后端**：直接映射为 `T *restrict`，由 C 编译器利用其别名分析做向量化。
  - **LLVM 后端**（未来）：为对应 load/store 指令附加 **noalias** 元数据，使中端/后端放心做循环向量化与重排。
  - **压榨效果**：消除“内存重叠”顾虑，使编译器将 `for` 循环替换为 `vmovups` 等宽指令，而非保守的逐元素标量代码。

**阶段**：对齐/非对齐在阶段 6；`*restrict T` 在阶段 8，与饱和算术、Option 零成本布局一起引入。

### 11.4 复合类型：切片的「视图」化与 Const-Size 提示

当前切片设计为 `{ ptr, len }`，len 为运行时变量。从两方面扩展可进一步压榨 codegen：

- **多维/跨步切片（Strided Slice）**：`{ ptr, len, stride }`，或类型层面支持 `T[]` 的 stride 属性。
  - **用途**：大模型张量运算中「跳跃式」读取内存；转置等操作变为零拷贝的视图切换，无需复制数据。
  - **与现有 `T[]` 的关系**：可视为 `T[]` 的扩展形态，stride 默认为 `size_of(T)` 时退化为连续切片。

- **Const-Size Slice（深度优化）**：支持**已知长度的切片视图**，如 `slice<T, const N>` 或等价的类型表示。
  - **极致压榨点**：在局部作用域内长度确定时（例如处理 4×4 矩阵块），编译器可**完全消除边界检查**并**直接展开循环**，利于指令流水线。
  - **与 `T[N]` 的关系**：比固定长数组 `T[N]` 更灵活（可指向内存任意位置），比普通切片更高效（已知 N，便于优化）。
  - **示例**：`let block: slice<f32, 16> = ...;` 表示长度为 16 的只读视图，循环可无边界检查、可完全 unroll。

**阶段**：阶段 10，与向量、多维数组一起落地。

### 11.5 枚举与 Option：零成本布局（空指针优化）

`Option<T>` 若用「tag +  payload」实现，会额外占用 4–8 字节，且与 C 指针接口不兼容。

- **空指针优化（Non-Null Optimization）**：当 `T` 为引用 `&T` 或非空指针类型时，保证 `Option<T>` 与 `T` **尺寸相同**，用 `0x0` 表示 `None`。
- **效果**：节省内存，且 `Option<&T>` 可直接与 C 的「可空指针」互操作，无需转换层。
- **实现约束**：编译器在布局阶段识别「可空优化友好」类型，对 `Option<&T>`、`Option<*T>`（若引入非空指针）等做特殊布局。

**阶段**：阶段 8，与泛型枚举、Option/Result 一起实现。

### 11.6 SIMD：专用掩码类型（Mask Types）

向量比较（如 `v1 > v2`）的结果不应只是「布尔向量」，而应映射到硬件掩码寄存器，避免浪费位宽。

- **掩码类型**：如 `m32x4`、`m64x2`，与 `i32x4` 等向量类型配套；比较运算结果类型为掩码类型。
- **示例**：`let mask: m32x4 = v1 > v2;`，codegen 映射到 AVX-512 的 k0–k7 等掩码寄存器。
- **理由**：用普通向量模拟掩码会占用完整向量位宽；专用掩码类型可直接对应指令集能力，提升吞吐。

**阶段**：阶段 10，与 10.3 比较与掩码、向量类型一起实现。

### 11.7 静态生命周期：`'static` 与常量预载

大模型权重等只读、长期存在的数据，若在类型上标记为「静态只读」，可放入 `.rodata`，利用操作系统与硬件的预载/映射优化。

- **`'static` 或等价标注**：表示该引用/指针所指向的内存生命周期为整个程序，且只读（或不可变）。
- **codegen**：将对应全局/常量放入 `.rodata`（或只读段），便于内核与 CPU 做预取、写保护等。
- **与 const/全局**：可与 `const`、全局变量语义结合，在类型层面区分「栈/堆」与「静态只读」。

**阶段**：阶段 10 或更晚，与所有权/生命周期、只读语义一起细化。

**小结**：上述内容与 §3.1 的「极限压榨特性」表及「压榨性能硬指标」细项一一对应。Zero-Padding 强制校验、`*restrict T`、Const-Size Slice 等深度优化确保 typeck/codegen 阶段不掉速；实现时按阶段 6 → 8 → 10 分批接入，在不推翻现有类型设计的前提下，持续向「性能比肩 C 甚至超越 C」靠拢。
