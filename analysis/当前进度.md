## 下一波前排（按优先级排序）

### 1. C5 EXPR_MATCH guard 扩展（immediate next step）

**做什么**：本波已 fold 不带 guard 的 const match；带 guard `_ if cond => ...` 因 `MatchArmEntry`（[compiler/ast_pool.c](../compiler/ast_pool.c):36-43）无 guard 字段暂未处理。当前 wildcard fallback 仅在 arm result 是 const 时 stamp，理论上不影响 runtime 正确性，但严格性需后续加 guard 字段收口。

**根修路径（产生点单点）**：
1. [compiler/ast_pool.c](../compiler/ast_pool.c) `MatchArmEntry` 加 `guard_ref` 字段（i32，存 guard 表达式 ref）
2. [compiler/ast_pool.c](../compiler/ast_pool.c) `pipeline_expr_append_match_arm` 加 guard_ref 参数 + 对应 accessor `pipeline_expr_match_arm_guard_ref`
3. [compiler/src/parser/parser.x](../compiler/src/parser/parser.x) match arm 解析处把 `_ if cond => ...` 的 cond 存入 guard_ref
4. [compiler/pipeline_glue.c](../compiler/pipeline_glue.c):14029-14104 EXPR_MATCH handler 加 guard 分支：若 arm 有 guard_ref，递归 fold guard → 若 guard fold 成功且 `const_folded_val != 0`，则视为匹配（与 lit/variant 同优先级 first-match wins）；若 guard fold 失败，跳过该 arm（保守，不 stamp）
5. 新 lang-const case `c_match_guard_const.x`（`match 2 { 1 if true => 10; 2 if X > 0 => 20; _ => 40 }` want=20）+ baseline tsv 注册

**平台**：PLATFORM: SHARED（parser + ast_pool + pipeline_glue 三处都是 SHARED）

**验收**：mac L2 + Ubuntu L2；c_match_guard_const exit=20；lang-const 14/14；bstrict 125 全绿

**升钉**：本子项也是 typeck CTFE 扩展非 ABI / 非 link 改动；不升钉
