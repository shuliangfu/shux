## 本波摘要（2026-07-20 · shux-x -E 爆内存方案 B · 1 commit · L2 验证未跑 L4）

| 项 | 结果 |
|----|------|
| **入口** | [shux-x-5GB-内存根因调查.md](shux-x-5GB-内存根因调查.md) §4 真凶：`g_arena_sc[512]` ArenaSidecar 池（18 GrowVec × 4.3 MB init cap）+ `g_module_sc[512]` ModuleSidecar 池（11 GrowVec）永不释放；`ast_pool_arena_reset` 只 reset len=0 不动 data/cap；dep 循环每个新 arena 指针触发 `arena_sidecar_get(create=1)` → 4.3 MB × N 累积 → GB 级 |
| **方案 B 实施（ast_pool.c）** | 新增 `arena_sidecar_free(ArenaSidecar *sc)` static：调用 18 个 `grow_vec_free`（types/exprs/blocks/funcs/consts/lets/ifs/regions/loops/for_loops/defer_block_refs/labeled_stmts/expr_stmt_refs/stmt_order/expr_call_arg_refs/expr_method_call_arg_refs/expr_match_arms/expr_struct_lit_fields/expr_array_lit_elem_refs/func_params）+ `memset(sc, 0, sizeof(*sc))` 标记 slot 可复用；新增 `module_sidecar_free(ModuleSidecar *sc)` static：调用 11 个 `grow_vec_free` + memset；新增 `ast_pool_arena_release(struct ast_ASTArena *a)` 公开 API：线性查找 `g_arena_sc[]` 匹配 arena 指针，找到则 free，找不到则 no-op；新增 `ast_pool_module_release(struct ast_Module *m)` 公开 API：同模式查找 `g_module_sc[]` |
| **方案 B 实施（rt_run_x_emit.from_x.c）** | 添加 `extern void ast_pool_arena_release(void *arena)` + `extern void ast_pool_module_release(void *module)` 声明（void* 参数避免拉 struct 定义，与 ast_pool.c 的 struct 指针兼容）；所有 `free(arena)` / `free(dep_arenas[i/di/j])` 路径（~20 处含错误 goto）前调用 `ast_pool_arena_release(...) + ast_pool_module_release(...)`；错误路径 `!arena || !module` / `src_len > INT32_MAX` / `pr.ok != 0` / `n_funcs <= 0` / `shux_load_direct_imports_for_asm_layout` 失败 / `shux_collect_dep_paths_transitive` 失败 / `shux_merge_direct_then_transitive_dep_paths` 失败 / `!out_buf || !pctx_e` / `!dep_arenas[i] || !dep_modules[i]` 全部成对调用 release + free |
| **安全性** | ① typeck 在主 emit 完成前需要访问 `typeck_dep_module_ptrs[]` / `typeck_dep_arena_ptrs[]`，release 只在 cleanup 路径（成功或错误 goto）调用，在 typeck + codegen 完成之后；② `ast_pool_arena_release` 幂等：slot 已释放或从未初始化时静默返回；③ slot 被 `memset(0)` 后可被未来 `arena_sidecar_get(create=1)` 复用 |
