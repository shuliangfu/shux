## 本波摘要（2026-07-19 · shux-x 5GB 内存根因方案 A · 日常 L2 · 双端 L2 全绿）

| 波 | 结果 |
|----|------|
| **入口** | 推进 [shux-x-5GB-内存根因调查.md](shux-x-5GB-内存根因调查.md) §7 推荐执行顺序「A → B，C/D 顺延自举后」首项。方案 A 为最小改动单 commit：拆分 init cap 与 grow step 两个常量，只减少 init 阶段的 zerofill 浪费，不改变 dep-arena-per-instance 架构，不触碰 free 路径（free API 留给方案 B） |
| **根因** | `compiler/ast_pool.c` L14-15 定义 `AST_POOL_GROW 4096`，被同时用于：① `grow_vec_init` 的 default fallback（L155 `if (initial_cap <= 0) initial_cap = AST_POOL_GROW`）；② 80 处 `grow_vec_init(..., AST_POOL_GROW)` 调用作为初始 cap；③ `grow_vec_ensure` 的 grow step（L173/175 `nc = v->cap > 0 ? v->cap : AST_POOL_GROW; nc += AST_POOL_GROW`）。结果：每个 ArenaSidecar init 时 18 个 GrowVec 各 calloc 4096 元素 × ~200B 平均 elem_sz ≈ 4.3MB zerofill；50 deps × 4.3MB = 215MB 仅 init cap 浪费（不含实际 AST 节点） |
| **根修（产生点权威）** | `compiler/ast_pool.c` 三处改动：① L18-40 新增 `AST_POOL_INIT_CAP 256` 常量 + 3-dim 英文 docblock（Why/Invariant/Asm-Perf/PLATFORM: SHARED）；② L140-145 改 `grow_vec_init` default fallback：`initial_cap = AST_POOL_GROW` → `initial_cap = AST_POOL_INIT_CAP`，加英文注释说明 init 用小 cap / grow 路径仍用大 step；③ 80 处 `grow_vec_init(..., AST_POOL_GROW)` → `grow_vec_init(..., AST_POOL_INIT_CAP)`（Edit replace_all=true）。**保留** `grow_vec_ensure` 内的 `AST_POOL_GROW`（L173/175，grow step 不变）—— 池超 INIT_CAP 时按 4096 元素线性扩容，保持 amortized O(1) |
| **不变量** | `0 < AST_POOL_INIT_CAP (256) <= AST_POOL_GROW (4096)`；init 路径走 `calloc`（zerofill 小），grow 路径走 `realloc` + `memset` tail（线性扩容保留 amortized cost） |
| **内存收益** | per-ArenaSidecar init：4.3MB → 270KB（减 ~94%）；50 deps init RSS：215MB → 14MB；OneFuncSidecar / ModuleSidecar / DepCtxSidecar 同步受益（init cap 全部从 4096 → 256）；不解决「ArenaSidecar 永不释放」根因（留给方案 B），只削减 init 浪费 |
