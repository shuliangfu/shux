# Shux 项目构建与架构分析

> 本文档分析 shux（.sx）项目**应如何构建**：仓库结构、模块划分、编译器架构、构建系统、标准库布局、嵌入式与多目标、自举前后组织方式。与根目录分析文档中的「目标与路线」相配合，本文档侧重**工程落地的构架**。

---

## 一、项目整体应如何构建

### 1.1 原则

- **轻量**：仓库内无巨型依赖；构建脚本/工具简单，不依赖复杂构建系统。
- **分层清晰**：编译器（宿主编译器 / 自举后 .sx 编译器）、运行时、标准库、工具（格式化、测试）分离，便于按需编译与裁剪。
- **单源多目标**：同一套源码支持「宿主编译器」与「.sx 自举编译器」两种产出；同一套标准库支持桌面、服务器、嵌入式等多种目标。
- **可嵌入、可自举**：从设计期就支持 freestanding 与最小运行时，并为自举保留明确的「自举子集」与目录边界。

### 1.2 顶层产出物

| 产出物 | 说明 | 阶段 |
|--------|------|------|
| **shuc**（或 su 等） | 编译器可执行文件：.sx → 目标码/IR | 全阶段 |
| **libshurt**（可选） | 最小运行时（启动、backtrace 等），仅在需要时链接 | 全阶段 |
| **core / std** | 标准库：以源码形式随编译器分发，按需编译进用户程序 | 全阶段 |
| **工具** | 格式化、测试运行器、包管理（若做）等 | M3 及以后 |

## 二、推荐仓库与目录结构

以下为**推荐**的仓库布局；标准库**扁平化**，不设 `stdlib/`，直接将 `core/` 与 `std/` 放在仓库顶层。实际可随里程碑裁剪（如先只有 `compiler/` 与 `core/`）。

```
shux/
├── README.md                 # 或链接到 analysis/ 下的分析文档
├── Makefile 或 build.sx       # 顶层构建入口，尽量简单
├── analysis/                  # 分析与设计文档
│   ├── README.md              # 语言实现分析（现有）
│   └── architecture.md        # 本文档：项目构建与架构
├── compiler/                  # 编译器实现（先 C，后 .sx）
│   ├── src/                   # 源码
│   │   ├── main.c             # 或 main.sx（自举后）
│   │   ├── lexer/
│   │   ├── parser/
│   │   ├── ast/
│   │   ├── typeck/
│   │   ├── ir/                # 或 codegen/（直接到 LLVM IR）
│   │   ├── codegen/           # LLVM 或自研后端
│   │   └── driver/            # 命令行、多文件、链接等
│   ├── include/               # C 头文件（若用 C 实现）
│   └── CMakeLists.txt 或 Makefile  # 编译器自身构建
├── core/                      # 标准库 core（无 OS 依赖），扁平化，与 std 并列
│   ├── types.sx
│   ├── mem.sx
│   ├── alloc.sx
│   └── ...
├── std/                       # 标准库 std（依赖 OS），扁平化，与 core 并列
│   ├── runtime.sx
│   ├── io.sx
│   ├── fs.sx
│   └── ...
├── runtime/                   # 最小运行时（可选，与 std 解耦）
│   └── ...                    # 启动、panic、backtrace 等
├── tests/                     # 测试用例与测试脚本
│   ├── run-tests.sh
│   └── ...
├── tools/                     # 可选：格式化、测试运行器等
│   └── ...
├── mcp/                       # MCP 服务（方便开发者用 MCP 开发）
│   └── ...                    # Shux MCP Server 实现（见第八章）
└── plugins/                   # IDE 插件（语法高亮、语法检查、语法提示等）
    └── ...                    # 各 IDE 插件实现（见第九章）
```

- **compiler/**：自举前为 C 源码，自举后可逐步替换为 .sx；目录结构（lexer/parser/ast/typeck/ir/codegen/driver）可保持一致，便于对照与迁移。
- **core/ 与 std/**：扁平化放在仓库顶层，与编译器解耦；由编译器在编译用户程序（或自身）时按模块引用；**不**作为巨型单库编译，而是按需链接，满足「轻量级、一切都要小」。
- **runtime/**：若与 std.runtime 或 panic 等强相关，也可并入 std/，仅保留极薄一层在 runtime/（如 crt0、栈检查）；视实现取舍。

### 2.1 目录结构补充：还缺什么？

在以上**最小必要**结构基础上，可按阶段补充下列项，使目录更完整、可维护性更好：

| 目录/文件 | 用途 | 建议 |
|-----------|------|------|
| **examples/** | 示例 .sx 程序（Hello World、使用 core/std、嵌入式最小示例等），便于用户上手与回归测试 | 建议有；M1/M2 即可加 |
| **scripts/** | 构建/CI 辅助脚本（如 `build.sh`、`ci.sh`、交叉编译脚本），与顶层 Makefile 配合 | 可选；构建复杂时再抽 |
| **LICENSE** | 开源协议文件 | 建议有 |
| **.gitignore** | Git 忽略规则（构建产物、analysis/ 等） | 建议有（已存在则可不再列） |
| **compiler/src/diagnostics/** | 错误/警告的格式化、多行提示、修复建议（与 typeck、parser 解耦） | 可选；错误多时可单列 |
| **compiler/src/target/** | 目标三元组、ABI、平台特定选项（与 codegen 配合） | 可选；多目标稳定后再抽 |
| **tests/unit/**、**tests/integration/** | 单元测试（单模块）与集成测试（完整编译+运行）；与 `tests/` 根下脚本区分 | 可选；测试多了再分子目录 |
| **benchmarks/** | 性能基准用例（配合 std.bench 或独立脚本），用于回归「性能比肩 C」 | 可选；M4 优化阶段加 |
| **config/** 或 **etc/** | 默认配置（如默认 target、默认库路径）；用户项目可覆盖 | 可选；有 `shuc build` 配置后再加 |
| **mcp/**（或 **tools/mcp/**） | Shux MCP Server：供 IDE/AI 通过 MCP 调用解析、诊断、符号等，方便开发者用 MCP 开发 | 建议有；后续实现，见第八章 |
| **plugins/** | IDE 插件：各编辑器/IDE 的 .sx 语言支持（语法高亮、语法检查、补全/提示等） | 建议有；后续实现，见第九章 |

- **core/、std/ 内部**：当前按「单文件一模块」（如 `core/types.sx`）即可；若某模块过大，可改为子目录（如 `core/alloc/` 下多文件），由编译器按模块名解析，保持对外仍为 `import("core.alloc")`。
- **结论**：上述树状图已覆盖**编译器、标准库、运行时、测试、工具、MCP、插件**等必要部分；**examples/** 与 **LICENSE** 建议尽早补上，**mcp/** 与 **plugins/** 后续实现，其余按需增量添加即可，不强制一次到位。

## 三、编译器架构（如何构建编译器本身）

### 3.1 流水线划分

编译器建议按**阶段**划分，便于分步实现与自举时分步用 .sx 重写：

```
.sx 源码
    → [词法] → Token 流
    → [语法] → AST
    → [类型检查 + 泛型单态化] → 带类型 AST / IR
    → [IR 生成] → 自定义 IR 或 LLVM IR
    → [优化] → 优化后 IR（DCE、内联、常量传播等）
    → [代码生成] → 目标码（.o / 可执行文件）
```

- **前端**：lexer、parser、ast、typeck；产出「正确且带类型信息的 AST 或高层 IR」。
- **中端**：泛型单态化、内联、常量传播、DCE；产出「优化后、可多后端复用的 IR」。
- **后端**：目标无关的 IR → 目标相关代码（先 LLVM，可选自研）；产出机器码或目标文件。

### 3.2 模块与依赖关系（编译器内部）

- **driver** 依赖 lexer、parser、ast、typeck、ir、codegen，负责命令行、文件列表、调用链接器。
- **lexer / parser / ast** 不依赖 typeck 与后端，便于先跑通「.sx → AST」。
- **typeck** 依赖 ast，产出带类型 AST 或直接产出 IR 输入。
- **codegen** 仅依赖 IR 与目标描述，不依赖具体 .sx 语法，便于换后端（LLVM / 自研）。

这样构建时可按阶段增量：先构建出「仅到 AST」的编译器，再逐步加上 typeck、IR、codegen。

### 3.3 宿主编译器（C）与自举后（.sx）的构建差异

| 方面 | 宿主编译器（C） | 自举后（.sx） |
|------|-----------------|----------------|
| **入口** | compiler/src/main.c | compiler/src/main.sx |
| **构建系统** | Makefile / CMake，调用系统 C 编译器 | 用上一代 shuc 编译 compiler/src/*.sx + 链接 |
| **标准库** | 仅「用户程序」用 core/std；编译器自身不依赖 .sx 标准库 | 编译器自身也由 .sx 写成，依赖 core/std 自举子集 |
| **产出** | 单二进制 shuc | 单二进制 shuc（新一代） |

自举后，**compiler/** 下全部为 .sx，由上一代 shuc 或宿主编译器一次性编译；构建脚本只需「列出所有 .sx + 指定 core/std 根路径」，无需再维护 C 与 .sx 的混合构建。

## 四、标准库在项目中的构建方式

### 4.1 不单独构建为可执行文件

- 标准库（core + std）以 **.sx 源码** 形式**扁平化**存在于仓库顶层目录 `core/` 与 `std/`，**不**预先编译成一个大 libshux.a。
- 用户执行 `shuc build` 或 `shuc app.sx` 时，编译器根据用户代码的 `import`，只编译并链接用到的 core/std 模块，满足「按需链接、可执行文件非常小」。

### 4.2 与编译器的耦合方式

- 编译器内置「标准库根路径」（如 `SHUX_STDLIB` 或相对可执行文件路径），解析 `import("core.xxx")` 时到 `core/xxx.sx` 查找，`import("std.yyy")` 时到 `std/yyy.sx` 查找（扁平化：core 与 std 为顶层目录，无 stdlib 包裹）。
- 构建编译器时**不**把 core/std 打包进编译器二进制；编译器只带「路径」或「源码内嵌可选」，避免编译器体积膨胀。

### 4.3 嵌入式与 freestanding

- 用户指定目标为 freestanding（如 `-target arm-none-eabi`）时，仅允许引用 `core` 及可选的最小 std 子集；链接时只拉入这些模块，不链接完整 std，满足嵌入式「一切都要小」。

## 五、构建系统与构建脚本

### 5.1 原则

- **简单**：优先 Makefile 或单脚本（shell / 最小 Python），避免引入 CMake/大型构建系统，除非跨平台需求明确。
- **依赖少**：构建编译器只需 C 编译器（及可选 LLVM 开发库）；构建 .sx 程序只需 shuc。
- **可重复**：指定相同源码与目标，多次构建结果一致（如固定随机种子、时间戳不写进二进制）。

### 5.2 建议的顶层命令

| 命令 | 含义 |
|------|------|
| `make` 或 `./build` | 构建宿主编译器 shuc（C）或当前编译器 |
| `make test` | 运行 tests/ 下用例，用当前 shuc 编译并执行、对比输出 |
| `make bootstrap` | 使用宿主编译器编译 compiler/*.sx，产出自举版 shuc（自举阶段） |
| `make stdlib-check` | 仅编译 core/std 中指定模块，确保无语法/类型错误 |

### 5.3 用户侧构建（将来）

- 用户项目内可提供 `build.sx` 或 `shux.toml` 等简单配置，由 `shuc build` 读取，指定入口、目标、依赖路径；不追求功能大而全，先满足「单命令出可执行文件」。

## 六、嵌入式与多目标在构建中的体现

### 6.1 目标三元组

- 构建时通过 **-target** 指定目标三元组（如 `x86_64-pc-windows-msvc`、`arm-none-eabi`、`riscv32imac-unknown-none-elf`），编译器据此选择：
  - 是否链接 libc、链接哪家 libc（glibc / musl / newlib-nano）；
  - 是否包含 std、允许哪些 std 子集；
  - 代码生成与 ABI（调用约定、栈对齐等）。

### 6.2 条件编译在标准库内

- std 内按目标或特性做条件编译（如 `@if(target.os == "none") { ... }`），**不**向用户暴露；用户始终写同一套 API，构建时由编译器与标准库内部决定落哪段实现。这样「全平台一套 API」与「嵌入式最小体积」在构架上统一。

### 6.3 运行时与启动

- 嵌入式目标可提供最小 **crt0** 或 **startup**（在 runtime/ 或 core/ 内），仅负责栈初始化、调用用户 main；不提供完整 libc，满足「可执行文件非常小」。

## 七、自举前后的项目结构对照

| 阶段 | compiler/ 内容 | 谁编译 compiler | 标准库使用者 |
|------|----------------|-----------------|--------------|
| **自举前** | C 源码（lexer/parser/...） | 系统 C 编译器（Make/CMake） | 仅用户程序；编译器自身不用 .sx |
| **自举中** | 部分 .sx（如前端）+ 部分仍 C（后端） | 宿主编译器 + C 编译器混合（或全由宿主编译器编译 .sx，C 仅链接） | 用户程序 + 用 .sx 写的那部分编译器 |
| **自举后** | 全部 .sx | 上一代 shuc | 用户程序 + 完整编译器 |

自举后，**analysis/**、**core/**、**std/**、**tests/**、**tools/** 可保持不变；仅 **compiler/** 从 C 全部变为 .sx，构建脚本从「编译 C」改为「调用 shuc 编译 .sx」。

## 八、MCP 与开发者工具

### 8.1 定位与目标

- **MCP（Model Context Protocol）**：为 Shux 提供**官方 MCP 服务**，使开发者能在 IDE（如 Cursor、支持 MCP 的编辑器）中通过 AI 助手完成：解析 .sx、查询符号、补全、诊断、重构建议等，**方便开发者使用 MCP 开发**。
- **目标**：后续实现一个 **Shux MCP Server**，作为项目的一部分维护；与编译器、标准库解耦，通过调用 shuc 或复用编译器前端（lexer/parser/ast）提供能力。

### 8.2 在构架中的位置

| 角色 | 说明 |
|------|------|
| **产出物** | 独立进程或库：`shux-mcp`（或集成在 tools/ 下），通过 stdio/HTTP 与 MCP 客户端通信。 |
| **依赖** | 依赖编译器能力：可执行 `shuc`（解析、typeck、编译）或直接链接 compiler 前端库（若拆出 libshuc_frontend）。 |
| **仓库位置** | 建议 **tools/mcp/** 或顶层 **mcp/**：实现 MCP 服务、工具注册、资源与 Prompt 等；与 tests/、examples/ 并列。 |

### 8.3 推荐目录与能力

- **目录**：在「二、推荐仓库与目录结构」中增加 **mcp/**（或 **tools/mcp/**），与 tools/、tests/ 同级，便于开发者克隆后即用 MCP 开发。
- **能力建议**（后续实现时按需取舍）：
  - **Tools**：如 `parse_file`（返回 AST/错误）、`compile`（调用 shuc 编译）、`symbol_at`（光标处符号信息）、`diagnostics`（当前文件诊断）。
  - **Resources**：如当前项目的语法/架构文档、内置函数清单、控制流清单等只读资源，供 AI 查阅。
  - **Prompts**（可选）：与 Shux 编码规范、项目约定相关的提示模板。

### 8.4 与构建系统的关系

- MCP 服务**不**参与「编译出 shuc」的主线；可作为可选组件：`make mcp` 或 `cd mcp && go build`（若用 Go 实现）等。
- 文档与清单（如 analysis/ 下 md）可被 MCP 以 Resource 形式暴露，方便「用 MCP 开发」时 AI 对齐项目规范与架构。

## 九、IDE 插件

### 9.1 定位与目标

- **插件目录（plugins/）**：用于存放**各 IDE/编辑器的 Shux 语言插件**，后续实现；支持**语法高亮、语法检查、语法提示**等一系列能力，提升在 IDE 内编写 .sx 的体验。
- **目标**：与编译器、MCP 解耦；插件通过调用 shuc（或 LSP/本地库）获取诊断与补全数据，不强制绑死某一 IDE。

### 9.2 在构架中的位置

| 角色 | 说明 |
|------|------|
| **产出物** | 各 IDE 的扩展/插件包（如 VS Code 扩展、JetBrains 插件、Vim/Neovim 插件等），按子目录或子仓库组织。 |
| **依赖** | 依赖编译器能力（如 `shuc` 可执行文件、或未来 LSP 服务）；可选依赖 MCP 提供更丰富的 AI 辅助。 |
| **仓库位置** | 顶层 **plugins/**：其下按 IDE 分子目录（如 `plugins/vscode/`、`plugins/idea/`），便于统一维护与发布。 |

### 9.3 推荐能力（按需实现）

- **语法高亮**：.sx 关键字、字面量、注释、字符串等的高亮规则（TextMate grammar 或各 IDE 等价机制）。
- **语法检查**：实时或保存时调用 shuc/前端，展示错误与警告（下划线、Problems 面板等）。
- **语法提示 / 补全**：基于 AST 或 LSP 的补全（关键字、符号、函数参数等）；悬停显示类型或文档。
- **跳转与大纲**：符号定义/引用跳转、文档符号树（Outline）；可选与 MCP 联动做「AI 补全/解释」。

### 9.4 与 MCP、构建的关系

- **与 MCP**：插件可与 MCP 并存——插件负责「编辑器内交互」（高亮、诊断、补全）；MCP 负责「AI 上下文与工具」。二者可共享对 shuc 或前端库的调用约定。
- **构建**：插件**不**参与「编译出 shuc」；各插件按各自 IDE 的构建/打包方式独立构建（如 VS Code 的 `npm run compile`、JetBrains 的 Gradle 等）。

## 十、总结：项目应如何构建

1. **仓库**：顶层区分 `compiler/`、`core/`、`std/`（标准库扁平化，无 stdlib 目录）、`runtime/`（可选）、`tests/`、`analysis/`、`tools/`、**`mcp/`（或 tools/mcp/）**、**`plugins/`**；构建入口简单（Makefile 或单脚本）。
2. **编译器**：按前端（lexer/parser/ast/typeck）→ 中端（IR、优化）→ 后端（LLVM 或自研）分模块；先 C 实现，再逐步用 .sx 重写并自举。
3. **标准库**：以 .sx 源码形式**扁平化**放在顶层 `core/` 与 `std/`，按需编译与链接，不整包打进编译器；嵌入式仅用 core + 最小 std。
4. **构建系统**：轻量、依赖少、可重复；支持 -target 多目标与 freestanding；自举后构建仅需「shuc + core/std 根路径」。
5. **嵌入式与全平台**：同一套 API、同一套 core/std 源码，通过目标与条件编译在构建时裁剪，实现「一切都要小」与「支持嵌入式」的构架目标。
6. **MCP**：提供官方 **Shux MCP Server**（mcp/ 或 tools/mcp/），方便开发者通过 MCP 使用 AI 助手开发 .sx；与编译器解耦，可选构建，后续实现。
7. **IDE 插件**：提供 **plugins/** 目录，后续实现各 IDE 插件，支持语法高亮、语法检查、语法提示等；与编译器、MCP 解耦，按 IDE 独立构建（见第九章）。

按上述方式构建，可与分析文档中的「轻量级、功能强、自举、全平台一套 API」等目标一致，并便于分阶段落地与长期维护。
