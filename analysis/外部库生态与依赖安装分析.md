## 2. 问题背景：`std.socketio` 样例

### 2.1 模块概况

`std/socketio` 是一套完整的 **Engine.IO / Socket.IO** 实现（客户端 + 简易服务端），并非通用 WebSocket 封装：

| 维度 | 情况 |
|------|------|
| 代码量 | `mod.sx` ~1300 行 + `socketio.sx` ~3500 行，约 **241 个 API** |
| 版本 | v0–v16 已收口（编解码、polling、WS 升级、namespace、room、cluster adapter 等） |
| 依赖 | `std.http`、`std.net`（`net_ws_*`）、TLS |
| 文档定位 | `analysis/std-socketio-v1.md`：**P3 可选，非 core 默认 std** |

底层实时通信能力已由 `std.http` + `std.net`（WebSocket）提供；Socket.IO 是在其之上的 **Node 生态专用协议层**。

### 2.2 实际使用情况

全仓库 `import("std.socketio")` **仅出现在 `tests/socketio/`**：

- 编译器 / 自举路径 **无依赖**
- 其他 std 模块 **无依赖**
- `需求分析.md`、`自举进度.md` **无规划条目**
- **不在** `run-lang-unsafe-gate`、`run-no-legacy-shux-gate` 等主语言门禁
- **在** `run-f-std-de-c-batch-gate.sh`（F 阶段 std 去 C 聚合）

结论：它是 F 阶段 std 去 C 的验收资产，不是语言核心或自举硬依赖。

### 2.3 与项目宗旨的冲突

对照 `.cursor/rules/00-项目目标与宗旨.mdc`：

| 宗旨 | 冲突点 |
|------|--------|
| **轻量级** | 3500+ 行、241 API，服务 niche 协议，与「一切都要小」冲突 |
| **标准库丰富但按需链接** | 可按需链接，但维护成本与测试体积（含 npm live）仍高 |
| **自举** | 与编译器自举无直接关系；`socketio.sx` 仍在 typeck 旧债列表 |
| **全平台一套 API** | Socket.IO 本质是 Node `socket.io` 互操作，非跨平台基础能力 |

### 2.4 结论

| 问题 | 答案 |
|------|------|
| 对 Shux 语言核心必要吗？ | **否** |
| 对自举必要吗？ | **否** |
| 对一般网络开发必要吗？ | **否**（用 `std.net` WebSocket 即可） |
| 对 Node Socket.IO 互操作有价值吗？ | **有**，但受众窄 |
| 应留在 `std/` 默认体系吗？ | **不合适** |
| 适合作为官方外部库吗？ | **适合** |

**推荐方向**：将 `socketio` 迁出 `std/`，作为 **官方维护的外部包**（首个样板：`shux.socketio`），需要的人按需安装，不需要的人零成本。

## 3. 现状能力审计（2026-06）

### 3.1 已有能力

| 能力 | 状态 | 说明 |
|------|------|------|
| `import` + `-L` 多库根 | ✅ | 编译器 `resolve_import_file_path_multi` |
| `shux.toml` 项目清单 | ⏳ 规划 | 定版格式 TOML；v1 原型为 `shux.pkg.tsv`（见 `tool-pkgmgr-v1.md`） |
| `shux-deps-resolve.sh` | ✅ 原型 | 校验 require；自举后由 `shux deps` + TOML 解析替代 |
| `shux-deps-lock.sh` / `verify.sh` | ✅ 原型 | 锁文件；目标输出 `shux.lock` |
| TOML 解析（`std.config`） | ✅ | `std/config/config.sx` TOML 子集，供自举后 pkg CLI 使用 |
| vendor 多根 | ✅ | `shux -L . -L vendor/foo`（见 `STBL-004`） |

**解析规则**（与 driver 一致，TOOL-007 K1）：

1. `{lib_root}/{a}/{b}/.../{last}.sx`
2. `{lib_root}/{a}/{b}/.../mod.sx`（目录模块，std 主路径）
3. 单段 `foo` → `{lib_root}/foo/foo.sx`

### 3.2 缺失能力

| 能力 | 状态 | 说明 |
|------|------|------|
| 编译器自动读 `shux.toml` | ❌ | 须手写 `-L` |
| `shux add` / `shux install` | ❌ | TOOL-007 v1 明确为非目标 |
| 网络拉包 / central registry | ❌ | K6：local-only |
| 项目模板含 manifest | ❌ | `shux-new.sh` 模板尚无 `shux.toml` |
| 外部包 tier（vendor/git） | ❌ | 锁文件原型仍为 TSV 金样 |

### 3.3 今天用户如何用外部代码

仅能 **手动 vendor + 手动 `-L`**：

```bash
git clone https://github.com/shux/shux-packages vendor/shux-packages
# 或 git submodule add ...

shux -L . -L vendor/shux-packages build main.sx
```

配合 TOOL-008 可生成锁文件（目标：`shux.lock`），但 **不算「安装」**，无 CLI 一键流程。

**结论：Shux 目前不支持安装外部库；仅支持手动放置源码并通过 `-L` 参与编译。**

## 4. 包分层与命名

### 4.1 三层模型（扩展 TOOL-007 K4）

| 层级 | import 前缀 | 来源 | 安装方式 |
|------|-------------|------|----------|
| **core** | `core.*` | 随语言仓库 bundled | 无需安装 |
| **std** | `std.*` | 随主仓库 bundled，保持精简 | 无需安装 |
| **外部包** | `shux.*`（官方）或作者自定义 dotted ID | 独立 git 仓库 / monorepo 子路径 | **自举后** `shux add` 或手动 vendor |

原则：

- `std.*` 只保留 **高频、跨场景、体积可控** 的模块（Tier-S / P1–P2）。
- P3 及 niche 协议（如 Socket.IO）、重型可选能力 → **外部官方包**。
- 社区包可用自有前缀（如 `vendor.foo.bar`），catalog 登记即可。

### 4.2 官方外部包命名

迁出 `std/` 后 **不再使用 `std.socketio`**，避免用户误以为标配 std。

推荐官方外部包统一 **`shux.*`**：

```
shux-packages/shux/socketio/mod.sx  →  import("shux.socketio")
```

与 TOOL-007 K1（包 ID = dotted import 路径）及 `std/<name>/mod.sx` 扁平布局同构。

### 4.3 require tier（v2 扩展）

在现有 `bundled` 之外扩展 tier，供锁文件与安装器使用：

| tier | 含义 | TOML 示例 |
|------|------|-----------|
| `bundled` | `core.*` / `std.*`，仓库内路径 | `"std.io" = "bundled"` |
| `vendor` | 已安装到项目 `vendor/` | 见 §4.5 嵌套表 |
| `git` | catalog 登记的 git 源 | 安装器写入 vendor 后降为 `vendor` |
| `path` | 本地路径（开发用） | `path = "../shux-packages/shux/socketio"` |

### 4.4 配置文件格式选型：TOML

#### 定版决策

| 文件 | 格式 | 谁编辑 | 说明 |
|------|------|--------|------|
| **`shux.toml`** | **TOML** | 用户 / `shux add` | 项目依赖与 `lib_roots` |
| **`shux.lock`** | **TOML** | 仅工具 | `shux deps lock` 生成，含 `sha256` |
| **`catalog.tsv`** | TSV | 官方维护 | 扁平行表，gate / `awk` 友好，**不改为 TOML** |

#### 文件名：`shux.toml`（非 `packages.toml`）

| 候选名 | 评价 |
|--------|------|
| **`shux.toml`** | ✅ **选用**：与 CLI `shux` 一致；类似 `Cargo.toml`、`go.mod`；可扩展为项目总配置（依赖、`lib_roots`、日后 fmt/target 等） |
| `packages.toml` | 不选用：过于泛化、不体现工具链；且文件内还有 `[package]`、`[lib_roots]`，不单是「包列表」 |
| `shux.pkg.toml` | 不选用：冗长；`pkg` 与表名 `[package]` 重复 |

**锁文件**：`shux.lock`（与 `shux.toml` 配对，格式仍为 TOML；类比 `Cargo.lock`；`shux deps lock` 生成，勿手改）。driver 查找顺序建议：`shux.toml` →（短期兼容）`shux.pkg.toml` → `shux.pkg.tsv`。

#### 为何选 TOML（而非继续 TSV / YAML / JSON）

| 格式 | 结论 |
|------|------|
| **TOML** | ✅ **选用**：可读性接近 Cargo.toml；支持嵌套表（`source`、`dependencies`）；`std.config` 已有 TOML 子集解析 |
| TSV | 仅保留为 TOOL-007 **v1 原型**与 **catalog / gate manifest**；扁平行表适合机器校验，不适合用户手写复杂依赖 |
| YAML | 不选用：规范歧义多；包管理场景易踩缩进坑 |
| JSON | 不选用为主配置：无注释、手写体验差 |

#### 与仓库现状的关系

- `tests/fixtures/pkgmgr/shux.pkg.tsv`、`scripts/shux-deps-*.sh` 为 **v1 原型**，自举后迁移到 TOML 解析，可短期只读兼容 TSV。
- 解析实现：自举后用 **`.sx` + `std.config` TOML 子集**，shell 脚本仅 bootstrap。

#### TOML 书写约定

- **package id 含点号**（如 `shux.socketio`）须用引号：`"shux.socketio"`，或使用 `[dependencies."shux.socketio"]` 嵌套表。
- **路径** 为相对 `shux.toml` 所在目录。
- **简单 bundled 依赖** 可用字符串：`"std.io" = "bundled"`。
- **外部包** 用内联表或子表：`{ tier = "vendor", version = "1.0.0" }`。

### 4.5 `shux.toml` 示例配置

清单文件 **`shux.toml`** 放在 **项目根目录**（与 `main.sx` 同级）；自举后 driver 从入口向上查找。

**顶层结构约定**：

| 表 / 键 | 含义 |
|---------|------|
| `[package]` | 项目名、版本、兼容 std 范围等元数据 |
| `[lib_roots]` | import 解析库根（键为逻辑名，值为相对路径） |
| `[dependencies]` | package_id → tier 字符串或依赖描述表 |
| `[dependencies."<id>"]` | 嵌套表写法（id 含 `.` 时用引号） |

`lib_roots` 顺序即探测顺序；与 `-L` 多根一致。

---

#### 示例 1：最小应用（仅 bundled，无外部包）

```toml
# shux.toml — hello-app（仅 bundled）

[package]
name = "hello-app"
version = "0.1.0"

[lib_roots]
# 相对本文件：指向 shux 主仓根（含 core/ std/）
shux = "../shux"

[dependencies]
"core.types" = "bundled"
"std.io" = "bundled"
```

编译（**今天**须手写 `-L`）：

```bash
shux -L . -L ../shux build main.sx
```

#### 示例 2：单仓多包（monorepo）— 一个 `lib_root` 装多个 `shux.*`

**vendor 目录布局**：

```
my-app/
├── shux.toml
├── main.sx
└── vendor/
    └── shux-packages/          ← git clone shux/shux-packages
        ├── shux/
        │   ├── socketio/mod.sx → import("shux.socketio")
        │   └── protobuf/mod.sx → import("shux.protobuf")
        └── catalog.tsv
```

**`shux.toml`**：

```toml
# shux.toml — chat-gateway（monorepo 外部包）

[package]
name = "chat-gateway"
version = "0.3.0"
shux_std = "0.1.0"  # 兼容的主仓 std ABI 范围（可选）

[lib_roots]
"." = "."                              # 项目根（main.sx）
shux = "../shux"                       # 主仓 core/ + std/
packages = "vendor/shux-packages"    # monorepo 根；覆盖全部 shux.*

[dependencies]
"std.http" = "bundled"
"std.net" = "bundled"

[dependencies."shux.socketio"]
tier = "vendor"
version = "1.0.0"

[dependencies."shux.protobuf"]
tier = "vendor"
version = "0.2.0"
```

应用代码：

```sx
const http = import("std.http");
const sio = import("shux.socketio");
const pb = import("shux.protobuf");
```

编译（自举前）：

```bash
shux -L . -L ../shux -L vendor/shux-packages build main.sx
```

自举后（目标）：

```bash
shux build main.sx    # driver 自动读 shux.toml 中的 lib_roots
```

#### 示例 3：混合源 — monorepo + 每包一仓（多个 `lib_root`）

```toml
# shux.toml — mixed-deps（monorepo + 独立仓）

[package]
name = "mixed-deps"
version = "1.0.0"

[lib_roots]
"." = "."
shux = "../shux"
packages = "vendor/shux-packages"      # monorepo
community = "vendor/community-foo"   # 独立 git 仓根

[dependencies]
"std.json" = "bundled"

[dependencies."shux.socketio"]
tier = "vendor"
version = "1.0.0"
source = { kind = "monorepo", url = "https://github.com/shux/shux-packages", tag = "v1.0.0", subpath = "shux/socketio" }

[dependencies."foo.bar"]
tier = "vendor"
version = "0.1.0"
source = { kind = "git", url = "https://github.com/foo/shux-foo", tag = "v0.1.0", subpath = "." }
```

`shux add` 写入 vendor 后，`tier` 为 `vendor`；`source` 块记录安装来源，供 lock 与复现。

## 5. 仓库组织：单仓多包 与 每包一仓（双模式均须支持）

安装依赖 **不要求** 每库一个 git 仓库。两种组织方式均为一等公民，由 **catalog 条目** 区分，对用户命令保持一致。

### 5.1 模式 A：单仓多包（monorepo，推荐起步）

独立仓库示例：`shux-packages`

```
shux-packages/                 ← 一个 git 仓库
├── catalog.tsv                ← 官方包目录
├── shux/
│   ├── socketio/
│   │   ├── shux.toml
│   │   ├── socketio.sx
│   │   └── mod.sx          → import("shux.socketio")
│   ├── protobuf/
│   │   └── mod.sx          → import("shux.protobuf")
│   └── ...
└── README.md
```

| 优点 | 缺点 |
|------|------|
| 一个 clone / submodule 管理全部官方包 | 仓库体积随包数量增长 |
| CI、catalog、文档集中 | 仅用小包时可能多拉代码（可用 sparse checkout 缓解） |
| 适合早期官方生态 | |

用户项目 **一个 lib_root** 即可覆盖多个包（`shux.toml` 片段）：

```toml
[lib_roots]
packages = "vendor/shux-packages"

[dependencies."shux.socketio"]
tier = "vendor"

[dependencies."shux.protobuf"]
tier = "vendor"
```

### 5.2 模式 B：每包一仓（per-package repo）

示例：`github.com/shux/socketio` 独立仓库，根目录即包根：

```
socketio/                      ← 独立 git 仓库
├── shux.toml
├── socketio.sx
└── shux/socketio/mod.sx
```

| 优点 | 缺点 |
|------|------|
| 发布节奏独立 | catalog 与 CI 分散 |
| 适合大型包、第三方社区包 | 官方早期维护成本略高 |
| 克隆体积最小 | |

### 5.3 catalog 如何统一两种模式

`catalog.tsv` 每条包记录 **源类型**，安装器按类型分支，对用户命令无差别：

```tsv
# 列：package_id	version	source_kind	source_ref	subpath	notes
shux.socketio	1.0.0	monorepo	https://github.com/shux/shux-packages	v1.0.0	shux/socketio	官方 Socket.IO
shux.protobuf	0.2.0	monorepo	https://github.com/shux/shux-packages	v1.0.0	shux/protobuf	
community.foo	0.1.0	git	https://github.com/foo/shux-foo	v0.1.0	.	社区独立仓
```

| `source_kind` | 行为 |
|---------------|------|
| `monorepo` | clone/fetch 指定 tag，`subpath` 指向仓内子目录 → 复制或 sparse checkout 到 `vendor/` |
| `git` | clone 独立仓库根（`subpath` 为 `.` 或子目录） |
| `path` | 仅开发/测试，不经过网络 |

**关键**：无论 monorepo 还是独立仓，安装结果均落在项目 `vendor/<规范化目录>/`，`shux.toml` 声明 `lib_roots` + `dependencies`，编译器行为一致。

### 5.4 与主仓库的关系

| 仓库 | 职责 |
|------|------|
| `shux/shux`（主仓） | `core/` + 精简 `std/`；编译器；TOOL-007 v2 CLI |
| `shux/shux-packages`（建议） | 官方外部包 monorepo + catalog |
| 第三方 | 独立仓，catalog 可选登记 |

主仓 **不应** 再膨胀 `std/` 塞 niche 模块；socketio 测试、npm live、大 fixture 随包迁出。

## 6. 安装与依赖工具设计（TOOL-007 v2 目标）

> **实现时机：自举完成后。** 自举前继续用手动 vendor + `-L` + 现有 shell 原型即可。

### 6.1 用户命令（目标）

```bash
# 安装单个包（背后可能是 monorepo 子路径或独立仓）
shux add shux.socketio@1.0.0

# 依赖工作流（将现有脚本并入 shux CLI）
shux deps resolve          # 等价 shux-deps-resolve.sh
shux deps lock             # 生成 shux.lock
shux deps verify           # 校验锁与磁盘一致

# 编译（driver 自动读 manifest lib_root，无需手写每个 -L）
shux build main.sx
```

### 6.2 `shux add` 流程（单包安装）

1. 读取 catalog（P3：本地/仓内 `catalog.tsv`；**P5+：Registry API**，默认 `https://packages.shux.dev` 或 `SHUX_REGISTRY`）
2. 解析 `package_id` + semver → catalog 行
3. 按 `source_kind` 获取源码：
   - **monorepo**：`git clone --depth 1` + `git sparse-checkout` 仅 `subpath`；或 fetch tag 后提取子目录
   - **git（独立仓）**：`git clone` 指定 tag 到 `vendor/<pkg-dir>/`
4. 更新项目 `shux.toml`：`dependencies` 中 `tier = "vendor"`
5. 追加/合并 `[lib_roots]`（指向 vendor 根或包根，与物理布局一致）
6. 运行 `shux deps lock` 更新 `shux.lock`
7. （可选）校验 `shux.toml` 中 `std.*` 依赖与主仓 ABI 兼容

### 6.3 driver 集成

自举后 driver / pipeline 应：

1. 从入口文件向上查找 `shux.toml`（短期可回退识别 `shux.pkg.tsv`）
2. 合并 `[lib_roots]` 到编译器 lib_roots（与 `-L` 叠加，优先级在 RFC 中固定）
3. 链接阶段按 import 闭包按需链入外部包的 `.o`（与 std 按需链接同原则，见 STBL-004 §5）

### 6.4 与 TOOL-008 联动

锁文件片段（`shux.lock`）：

```toml
[[locked]]
package_id = "shux.socketio"
tier = "vendor"
relpath = "vendor/shux-packages/shux/socketio/mod.sx"
sha256 = "<hex>"
source_kind = "monorepo"
source_ref = "v1.0.0@shux/socketio"
```

`shux deps verify` 保证可重复构建；CI 与 `run-tool-deps-lock-gate.sh` 对齐。

### 6.5 非目标（v2 仍坚持）

与 `需求分析.md` 及 TOOL-007 宗旨一致：

- 不引入 JVM/Node 式大型运行时依赖
- 依赖以 **源码或预编译 `.o`** 参与编译，体积可控
- v2 可先不实现完整 semver 求解器（精确版本即可）
- **Registry 网站**为 P5+ 长远目标（§10）；P3 用 **git + catalog.tsv** 即可跑通 `shux add`

## 9. 决策摘要

| 决策项 | 结论 |
|--------|------|
| 现在能否安装外部库？ | **不能**（仅手动 vendor + `-L`） |
| `std.socketio` 是否保留在 std？ | **否**，迁为官方外部包 `shux.socketio` |
| 官方外部包放哪？ | 独立生态仓（建议 `shux-packages`），不塞主仓 `std/` |
| 单仓多包 vs 每包一仓？ | **两种都支持**；catalog `source_kind` 区分 |
| 能否 `shux` 单独安装某一包？ | **自举后实现** `shux add`；monorepo 用 subpath/sparse checkout |
| 项目配置文件格式？ | **`shux.toml`**；锁文件 **`shux.lock`** |
| 长远 Registry / 网站？ | **P5+** 建设；`shux add` 从 Registry 安装；见 §10 |
| `shux add` 最终体验？ | 类似 `cargo add`：网站查包 → CLI 一条命令 → `vendor/` + lock |
| 发包安全检测？ | **必须**；§10.9 流水线：完整性、后门、漏洞、能力、合规 |
| 实现优先级 | **自举完成后** P1–P3；Registry 网站不阻塞自举 |

## 10. 长远规划：开源库网站与 Registry

> **愿景**：建设 Shux 开源库 **网站 + Registry API**，用户执行 `shux add shux.socketio` 时从官方（或可配置）Registry 解析元数据，自动拉取 git 源码到 `vendor/`，写入 `shux.toml` 与 `shux.lock`——体验对标 `cargo add` / `npm install`，但分发模型保持 **轻量、源码优先**。

### 10.1 定位（与项目宗旨一致）

| 原则 | Registry / 网站怎么做 |
|------|----------------------|
| **轻量级** | Registry **只存元数据**（包 id、版本、git 坐标、checksum、兼容 std 版本）；**不托管** JVM/Node 式运行时或大体积预编译包 |
| **源码参与编译** | 默认从 **git tag**（monorepo subpath 或独立仓）拉 `.sx` 源码；可选登记预编译 `.o` 的 **checksum**，仍按需下载小文件 |
| **按需链接** | 网站展示依赖体积与 std 依赖；安装结果进 `vendor/`，未 import 的不进二进制 |
| **官方 + 社区** | 网站分 **官方认证包**（`shux.*`）与 **社区包**；列表可筛 tier |

类比：**crates.io 的索引 + git 源**，而非 npm 的巨型 tarball 仓库。

### 10.2 演进路线（catalog → Registry → 网站）

与 §8 路线图衔接，Registry 不是第一步，而是 catalog 的 **线上化**：

```
阶段 A（P0–P3）  catalog.tsv + git URL     shux add 读本地/仓内 catalog
阶段 B（P4）     catalog.json 托管于 git   静态 JSON，CI 定时同步 catalog.tsv
阶段 C（P5）     Registry API              REST：/v1/packages/{id}/versions/{ver}
阶段 D（P6）     开源库网站                浏览、搜索、文档、发包入口
阶段 E（P7+）    镜像 / 私有 Registry      企业内网、地域 CDN 缓存元数据
```

**关键**：P3 的 `source_kind`（`monorepo` / `git`）与 `catalog.tsv` 列 **原样成为 Registry 记录字段**；CLI 只换「catalog 从哪读」，安装逻辑不变。

### 10.3 Registry API（最小契约，草案）

基址示例：`https://packages.shux.dev`（域名待定）。

| 端点 | 说明 |
|------|------|
| `GET /v1/packages/{package_id}` | 包元数据、最新版本、官方/社区标记 |
| `GET /v1/packages/{package_id}/versions/{version}` | 单版本：**source_kind**、url、tag、subpath、sha256、shux_std 兼容范围 |
| `GET /v1/search?q=socketio` | 网站搜索用（可延后） |
| `GET /v1/catalog` | 全量或分页 catalog（兼容静态 JSON 导出） |

单版本响应字段与 `catalog.tsv` 对齐（示例 JSON）：

```json
{
  "package_id": "shux.socketio",
  "version": "1.0.0",
  "source_kind": "monorepo",
  "url": "https://github.com/shux/shux-packages",
  "tag": "v1.0.0",
  "subpath": "shux/socketio",
  "sha256": null,
  "shux_std": ">=0.1.0",
  "tier": "official"
}
```

CLI：`shux add` 请求 Registry → 得到上述记录 → 执行与 §6.2 相同的 git sparse checkout / clone → 更新 `shux.toml` + `shux.lock`。

### 10.4 `shux add` 与 Registry 的用户体验（目标）

```bash
# 默认官方 Registry（P5+ 默认行为）
shux add shux.socketio@1.0.0

# 指定 Registry（环境变量或 shux.toml）
export SHUX_REGISTRY=https://packages.shux.dev
shux add shux.socketio

# 搜索（P6 网站成熟后）
shux search socketio
```

`shux.toml` 可选配置：

```toml
[registry]
default = "https://packages.shux.dev"
# mirror = "https://mirror.example.com/shux"   # P7 私有/镜像
```

**`shux.lock` 记录 Registry 来源**，保证 CI 可复现：

```toml
[[locked]]
package_id = "shux.socketio"
tier = "vendor"
relpath = "vendor/shux-packages/shux/socketio/mod.sx"
sha256 = "789abc...64hex"
source_kind = "monorepo"
source_ref = "v1.0.0@shux/socketio"
registry = "https://packages.shux.dev"
```

### 10.5 开源库网站（P6）功能范围

| 功能 | 说明 |
|------|------|
| **包浏览 / 搜索** | 按 `package_id`、关键词、官方/社区筛选 |
| **版本列表** | semver、发布说明、兼容 `shux` / std 版本 |
| **文档页** | 链到包仓 README 或托管生成的 API 摘要 |
| **安装命令** | 展示 `shux add shux.socketio@1.0.0` 一键复制 |
| **发包指南** | 社区包如何 `shux publish` 登记（见 §10.6） |
| **安全报告页** | 展示 §10.9 检测结果：通过 / 警告 / 待审 / 拒绝 |
| **非目标** | 不做在线 IDE、不做包内广告；**不登录也可 `shux add`** |

网站与 API **同源数据**；初期可用 **GitHub Pages + catalog.json** 过渡，再迁独立 Registry 服务。

### 10.6 发包：`shux publish`（P6+，草案）

社区/官方维护者将新版本 **登记到 Registry**（非上传整个 git 仓）：

1. 包仓已有 `shux.toml`（包元数据 + std 依赖声明）
2. 打 git tag（如 `v1.0.1`）
3. `shux publish` 提交：`package_id`、`version`、git url、tag、subpath、checksum
4. **Registry 安全流水线**（§10.9）：元数据校验 → 静态分析 → 依赖审计 → 可选沙箱烟测
5. 根据结果：**发布** / **带警告发布** / **人工审核队列** / **拒绝**
6. 网站列表更新；用户 `shux add` 可见新版本（展示安全徽章与报告链接）

官方 `shux.*` 包可走 **CI 自动 publish**（合并 tag 后写入 Registry）。

### 10.7 安全与可复现（传输与锁）

| 机制 | 说明 |
|------|------|
| **`shux.lock` + sha256** | 安装后锁定模块入口文件摘要；`shux deps verify` 防篡改 |
| **Registry TLS** | 元数据 HTTPS；git 源 HTTPS |
| **官方包审核** | `shux.*` 仅核心团队 publish；社区包标记 `community` |
| **私有 Registry** | P7：`SHUX_REGISTRY` 指向内网，同一套 API |

**发包内容安全**（后门、漏洞、恶意行为）见 §10.9。

### 10.9 发包安全检测（`shux publish` 流水线）

用户发布社区包时，Registry **不能**只校验 git tag 是否存在；须跑 **分层安全流水线**。原则：**默认不信任**、**源码可分析**（Shux 优势）、**官方包与用户包策略分级**。

#### 10.9.1 检测维度总览

| 类别 | 检测什么 | 典型手段 | 严重级别 |
|------|----------|----------|----------|
| **供应链完整性** | 版本是否可被替换、元数据是否造假 | tag 不可变、commit 与登记一致、`sha256` 与 git 快照一致 | 🔴 拒绝发布 |
| **后门 / 恶意行为** | 偷数据、远控、挖矿、持久化 | 静态规则 + 能力清单 + 沙箱行为 | 🔴 拒绝 / 🟡 人工 |
| **已知漏洞** | 依赖链上的已知问题 | 依赖树扫描、CVE/公告库（随生态成长） | 🟡 警告 / 🔴 视情况 |
| **内存与 unsafe** | 越界、UAF、过度 `unsafe` | `shux check`、unsafe 审计规则 | 🟡 警告 |
| **权限与能力** | 是否超出声明能力 | `shux.toml` 能力声明 vs 实际 `import`/`extern` | 🔴 拒绝 / 🟡 警告 |
| **混淆与隐蔽** | 故意难审计的代码 | 异常编码、巨大单行、隐藏二进制 | 🟡 人工审核 |
| **元数据与合规** | 许可证、包名冒充、描述钓鱼 | 必填 `license`、typosquat 检测、id 相似度 | 🟡 警告 / 人工 |
| **构建与发布诚信** | 登记路径与真实模块不一致 | `mod.sx` 存在、import 路径与 `package_id` 对齐 | 🔴 拒绝 |
| **质量与可用性** | 能否编译、是否破坏用户工程 | `shux check` 全包、`shux deps resolve` | 🔴 拒绝（无法编译） |

图例：🔴 阻断发布　🟡 可发布但带警告徽章　🟢 通过

#### 10.9.2 后门与恶意行为（重点）

Shux 包为 **源码 + 可选 `.o`**，检测应围绕「编译前后行为」与「敏感 API 使用」：

| 检查项 | 说明 |
|--------|------|
| **敏感 std 调用** | `std.net` / `std.http` 出站连接、`std.fs` 读写、`std.process` 执行子进程、`std.dynlib` 动态加载、`std.ffi` 调 C |
| **未声明 extern** | 调用 `*_c` / 原生符号但未在包文档或能力表声明 |
| **环境变量与路径** | 读取 `std.env`、访问 home/ssh、`.env`、密钥路径模式 |
| **数据外泄模式** | 读本地文件后立即网络发送（静态关联启发式） |
| **反调试 / 自修改** | 动态代码生成、可疑 `dynlib`、反射式加载（随语言演进补充规则） |
| **隐藏资产** | 仓库内非 `.sx` 可执行文件、加密 blob、与声明不符的 `.o` |
| **依赖投毒** | 依赖突然指向未知 git 仓或变更 `lib_roots` 到恶意路径 |
| **发布者身份** | 首次发布社区包、包名与知名包相似（typosquat）→ 人工队列 |

**官方 `shux.*`**：核心团队 CI + 全量 gate；**社区包**：自动流水线 + 首次发布人工复核。

#### 10.9.3 漏洞检测（不仅后门）

| 类型 | 说明 |
|------|------|
| **依赖漏洞** | 扫描 `shux.toml` / 传递依赖的已知问题版本（P6+ 维护 advisory 库） |
| **不安全 API 使用** | 裸指针、未检查边界、`unsafe` 块密度超阈值 |
| **Shux 语义漏洞** | 类型绕过、错误处理缺失导致 panic 路径（`shux check` 严格模式） |
| **重复历史漏洞** | 同一 `package_id` 曾被标记恶意后换版本重发 → 账号/包冻结 |

漏洞与后门区分：**漏洞**多为无意缺陷（警告 + 建议升级）；**后门**为故意恶意（拒绝 + 下架）。

#### 10.9.4 其他必须覆盖的方面

| 方面 | 说明 |
|------|------|
| **许可证** | `shux.toml` 必填 `license`；与仓库 LICENSE 文件一致性抽检 |
| **包名与 import 一致性** | `package_id` 必须能解析到登记的 `mod.sx` |
| **版本不可篡改** | 已发布版本的 git 坐标 + `sha256` 不可被发布者静默修改 |
| **SBOM / 依赖透明** | 网站展示完整 `dependencies` 树与 tier（bundled/vendor） |
| **审计日志** | 谁、何时 publish、哪条 git ref、流水线结果（供追溯） |
| **用户举报** | 网站「报告安全问题」→ 下架 / 警告 / 复测 |
| **安装侧提示** | `shux add` 对 `community` + 未审版本打印警告；`shux audit`（可选 CLI）本地复跑轻量规则 |

#### 10.9.5 流水线阶段（建议实现顺序）

```
publish 请求
  → S1 元数据与 git 真实性（tag、路径、sha256、id 冲突）
  → S2 静态：shux check / typeck 全包通过
  → S3 规则引擎：敏感 import/extern、能力声明对比（§10.9.2）
  → S4 依赖：resolve + 已知 advisory 匹配（可延后）
  → S5 沙箱烟测（可选）：无网络容器内编译 smoke；禁止出站默认
  → S6 分级决策 + 写 Registry 安全字段 + 网站徽章
```

Registry 版本记录扩展字段（示例）：

```json
{
  "security": {
    "status": "warn",
    "checks_passed": ["S1", "S2", "S3"],
    "checks_warn": ["S4-advisory"],
    "report_url": "https://packages.shux.dev/shux.socketio/1.0.0/security",
    "scanned_at": "2026-06-25T12:00:00Z"
  }
}
```

网站徽章：`official` / `verified` / `community-warn` / `quarantined`。

#### 10.9.6 能力声明（`shux.toml` 扩展，草案）

减少「包偷偷联网」类后门：发布时声明能力，静态扫描 **实际使用 ⊆ 声明**。

```toml
[package]
name = "shux.socketio"
version = "1.0.0"
license = "MIT"

[capabilities]
network = true          # std.net / std.http
filesystem = false
process = false
ffi = false
dynlib = false
```

未声明却使用 → **发布拒绝** 或 **强制降级为 `community-warn`**（RFC 定稿）。

#### 10.9.7 现实边界（诚实预期）

| 事实 | 应对 |
|------|------|
| **静态分析无法证明无后门** | 分层检测 + 人工首审 + 用户 `shux.lock` + 举报 |
| **零日漏洞无法预知** | advisory 库持续更新；警告而非假装「绝对安全」 |
| **恶意 git 源在 Registry 外** | 仅登记可信 git 托管；`path` tier 不允许 publish 到公共 Registry |
| **性能与成本** | 全量沙箱可仅社区首版 + 随机复检；官方包走自有 CI |

**非目标**：不做「100% 安全」营销；网站展示 **检测项与结果**，由用户结合官方/社区 tier 决策。

### 10.8 与现有设计的关系（小结）

| 组件 | 现在 / 近期 | 长远（Registry + 网站） |
|------|-------------|-------------------------|
| 包元数据 | `catalog.tsv` | Registry API + 网站展示 |
| 用户配置 | `shux.toml` | 不变；可增加 `[registry]` |
| 锁文件 | `shux.lock` | 不变；增加 `registry` 字段 |
| 安装命令 | 手动 vendor → P3 catalog | **P5+ `shux add` ← Registry** |
| 源码托管 | GitHub git | **仍为 git**；Registry 不替代 git |
