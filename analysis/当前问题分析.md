## 1. 已根源修复（归档）

### 1.1 R2 真迁 target_cpu_pure 跨平台 prove IDENTICAL 失败

- **commit**：`e55adc65`
- **根因**：codegen ABI 属性层 — [target_cpu_pure.x](file:///home/shu/shux/compiler/src/driver/target_cpu_pure.x) 的 3 个 export function 缺 `#[no_mangle]`，导致 macOS `./shux` 不加前缀、Ubuntu `./shux` 加 `driver_` 前缀，符号不匹配。
- **修复**：给 `tcp_eq_at` / `tcp_set_u32` / `tcp_parse_named` 加 `#[no_mangle]`。
- **PLATFORM: SHARED** — codegen ABI 契约跨平台一致。

### 1.2 codegen Vec4f 类型映射 bug

- **commit**：`cc79393c`
- **根因**：[seeds/codegen_gen.linux.x86_64.c](file:///home/shu/shux/compiler/seeds/codegen_gen.linux.x86_64.c) 是旧版 seed，缺 F32 分支，Vec4f 走 fallback 到 `int32_t`，导致 `conflicting types for 'std_simd_add'`。
- **修复**：seed 加 F32 分支（`f32x4_t` / `f32x8_t` / `f32x16_t`）。
- **遗留**：`SHUX_FORCE_REGEN_GEN=1` 在 macOS/Ubuntu 均不可用（macOS 挂死，Ubuntu 空输出），seed 只能手动编辑。源码 [src/codegen/codegen.x](file:///home/shu/shux/compiler/src/codegen/codegen.x) 的 `emit_vector_c_type_out` 已有 F32 处理，但 regen 路径不通。

### 1.3 net-context gate 重复符号 bug

- **commit**：`d45ee9a4`
- **症状**：`tests/run-std-net-context-gate.sh` cc 报 8 个 duplicate symbol（`_shux_io_submit_write` / `_std_context_deadline_ns` 等），但 io-context gate 链同样 .o 却通过。
- **根因**（两条平行缺陷）：
  1. [runtime_asm_io_stubs.from_x.c](file:///home/shu/shux/compiler/seeds/runtime_asm_io_stubs.from_x.c#L143-L183) L144/L159/L168 的 `shux_io_register`/`shux_io_submit_read`/`shux_io_submit_write` 是**强符号 T**，与 [std/io/core.x](file:///home/shu/shux/std/io/core.x) export function body 冲突。
  2. `std.context` 不在 `pipeline_codegen_std_dep_link_only` 列表，codegen co-emit body 与预编 `std/context/context.o` 强符号冲突（`std_context_deadline_ns` 等）。
  3. io-context 通过原因：macOS `-dead_strip` + `-ffunction-sections` 丢弃不可达 body；net-context 引用 `std/net/net.o`（U `_std_context_deadline_ns`）让符号可达，触发 duplicate。
- **修复**：
  1. 三个函数改为 `SHUX_WEAK`（G.7 单一权威：`std/io/core.x` 是强实现，本桩仅 fallback）。
  2. [runtime_link_abi.from_x.c](file:///home/shu/shux/compiler/seeds/runtime_link_abi.from_x.c#L3351-L3353) `pipeline_codegen_std_dep_link_only` 加入 `std.context`，由预编 `context.o` 提供符号。
- **PLATFORM: SHARED** — SHUX_WEAK 在 POSIX 为 `__attribute__((weak))`，PE/MinGW 为空（用 `--allow-multiple-definition`）。

### 1.4 shux-c 链接时丢失用户 .o 文件 bug

- **commit**：`d45ee9a4`
- **症状**：io/net-context gate cc 报 undefined symbols（`_atomic_load_i32_c` / `_time_now_monotonic_ns_c` 等）。
- **根因**：`driver_parsed_invoke_cc` → `shux_invoke_cc` 接收硬编码的 ~55 个 std/core .o 路径参数，**用户命令行传入的 .o 文件被丢弃**（如 `runtime_atomic_glue.o`、`runtime_time_os.o`）。
- **修复**（G.7 单一权威模式）：全局 setter/clearer `shux_invoke_cc_set_user_o_files_from_argv` / `shux_invoke_cc_clear_user_o_files`，`shux_invoke_cc_impl` L5036-5047 在所有 std/core .o 推入后、execvp 前推入 `g_shux_user_extra_o_files[]`。
- **修改文件**：`runtime_driver_abi.from_x.c` / `runtime_driver_abi.h` / `rt_run_compiler_parsed.from_x.c` / `runtime.from_x.c` / `rt_run_compiler_parsed.x` / `rt_run_compiler_parsed_surface.from_x.c`。
- **PLATFORM: SHARED** — C 后端链接路径跨平台一致。

### 1.5 parser C-style cast 挂死 bug

- **commit**：`d45ee9a4`
- **症状**：shux-c/shux-x 跑 `codegen.x -E-extern` 时挂死（macOS）或空输出（Ubuntu）。
- **根因**：SHUX parser 不支持 C-style cast `(void)xxx;`，遇之死循环 → 内存耗尽。`codegen.x` L126 `(void)is_method;` + `std/sys/mod.x` 4 处 `(void)xxx;`。
- **修复**：删除这些 C-style cast。
- **PLATFORM: SHARED** — parser 行为跨平台一致。

### 3.3 【P1·已收口】libc extern 再声明与系统头冲突（getenv 等）

- **症状（历史）**：`.x` 写 `export extern "C" function getenv(name: *u8): *u8;` 时，codegen 再声明 `extern uint8_t * getenv(...)`（或更早曾落成错误类型），与 `#include` 的 `char *getenv` **冲突**。
- **真根因（2026-07-21 复核）**：不是「`*u8` 发不成指针」——产品链已发 `uint8_t *`。冲突来自 **用 SHUX 类型复述 libc 符号**。`read`/`write` 早已进 `codegen_is_libc_conflicting_extern_name` 跳过再声明；**getenv 等未入表** → g05 sed 下游删行糊绿。
- **根修（`fda84704`）**：在权威谓词 `codegen_is_libc_conflicting_extern_name` 补全 g05 历史 sed 集合（getenv/getcwd/malloc/free/memcpy/…）；`emit_func_extern_declaration` 已只走该谓词。seed 同 commit。
- **验证**：`-E` 不再出现 `extern * getenv(`；调用仍在；rv=42 / hello OK。
- **遗留**：g05 sed 可作防御层暂留。
- **PLATFORM: SHARED**

### 3.4 【P1·已收口】源末无换行 → num_funcs=0（真因在 preprocess）

- **症状**：源文件末尾无 `\n` 时 `-o`/`-E` 报 XP003、`num_funcs=0`（`SHUX_DEBUG_PIPE`：`post-parse_into_buf num_funcs=0`）。历史 workaround：测试/std 末尾补 `\n`。
- **误判层**：原地图指向 parser `collect_imports_buf` / `parse_into_buf` EOF 收尾。
- **真根因（2026-07-21）**：`preprocess_x` / `preprocess_x_buf` 按行缓冲；**仅在遇 LF 时 flush**。源以 `}` 等非 LF 结束时，**最后一行（常含闭括号）被丢弃** → parser 看到残缺源 → skip 函数 → `num_funcs=0`。
- **根修（`157f1dc1`）**：扫描结束后若 `line_len > 0`，按与 LF 分支相同语义刷出缓冲并补写 LF。权威：`compiler/src/preprocess/preprocess.x` + `seeds/preprocess_gen.linux.x86_64.c` 同 commit。
- **验证（mac L2）**：`/tmp/eof_no_nl.x` / oneline / import-no-nl 全部 `-o` rv=42；矩阵 rv/hello OK。
- **PLATFORM: SHARED**

### 5.2 bstrict cooldown 机制（`tests/run-all-bstrict.sh` L313-322）

```bash
case "$(uname -s)" in
  Darwin)
    case "$script" in
      run-types-gate.sh) sleep "${SHUX_BSTRICT_DARWIN_TYPES_COOLDOWN:-5}" ;;
      run-ub.sh|run-io.sh|run-crypto.sh|run-vector.sh|run-thread.sh|run-net.sh)
        sleep "${SHUX_BSTRICT_DARWIN_HEAVY_COOLDOWN:-8}" ;;
      *) sleep "${SHUX_BSTRICT_DARWIN_COOLDOWN:-1}" ;;
    esac
    ;;
esac
```

- `run-json.sh` 不在 heavy 列表，只 sleep 1s → 可能不足。
- 解决 §3.1 的方案之一：把 `run-json.sh` 加入 heavy 列表 + 调高 cooldown。

### 5.3 bstrict 提前 exit 1 的根因

`run-all-bstrict.sh` L11 `set -e` + L462 附近 `exit 1`（run-json 失败后）。导致后续 ~30 个测试未跑到。解决 §3.1 的方案之二：改 bstrict 容错（`set +e` 包裹单个脚本，失败记录但继续）。

### 5.4 best-effort 模式（W3 fast）

- **不是产品链验证**：FAIL 可能是 W3 fast 假红，需产品链甄别

### 5.5 G.7 单一权威模式

预编 .o 或 codegen co-emit body 作为唯一权威路径，避免重复符号。`pipeline_codegen_std_dep_link_only` 列表判断 dep 是否有预编 .o，有则跳过 co-emit body。

### 5.6 SHUX_FORCE_REGEN_GEN=1 不可用

macOS 挂死、Ubuntu 空输出。seed（`codegen_gen.linux.x86_64.c`）只能手动编辑，无法从源码 regen。这是 §1.2 / §3.3 根源修复的障碍——修了 codegen.x 也要手动同步 seed。
