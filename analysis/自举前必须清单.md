## 四、自举前核心路径（关键节点）

以下三步**按顺序**完成，每步的产出是下一步的前提。完成三步后，再进入阶段 9 的「用 .sx 重写编译器」。

---

### 第一步：锁定「内存契约」

**目标**：与 C 的互操作与 ABI 有明确、可验证的约定，避免自举后布局变化导致 FFI 或运行时崩溃。

**动作**：

1. **完成 `analysis/与C互操作.md`**，至少包含：
   - FFI 语法与调用约定（如 cdecl / 系统默认）。
   - **struct 默认对齐**：与 C 一致，无 padding 需求时由 ABI 决定；**Packed**（无 padding）需**显式标注**，不得默认。
   - **Slice 内存布局固定为 (ptr, len)**：与当前 codegen 一致，文档化并承诺稳定。
   - 可选：枚举、数组、字符串的布局说明。

2. **实现或验证**：
   - 若有 Packed 语法或属性，确保仅显式启用；若无，在文档中说明「当前无 Packed，将来若增加则显式」。
   - 写一个最小用例：.sx 中 struct 与 C 侧同布局，双向传递，编译运行通过。

**验收**：`与C互操作.md` 存在且包含上述内容；内存契约在代码或测试中可验证。

---

### 第二步：定义「I/O 接口协议」

**目标**：std.io 的抽象满足「超越 io_uring」与高并发网络的扩展需求，自举后不需推翻重来。

**动作**：

1. **在 std.io 中定义 trait Reader 和 trait Writer**（或等价接口）：
   - 方法至少包含：读/写缓冲区、关闭（若适用）。
   - **硬性约束**：所有读写函数必须支持**超时（Timeout）**参数，或**在签名/文档中预留位置**。理由：高并发网络中，无超时的 I/O 易导致挂死与资源泄漏。

2. **ReadOnlySlice 与 WriteOnlySlice**：
   - 在接口或类型层面区分「只读入参」与「可写出参」，便于后端做零拷贝与 io_uring fixed_buffers 对接。
   - 若当前仅为文档约定，也需在 `高并发IO与多平台.md` 或 std.io 注释中写清。

3. **std.mem 中预留 Buffer 类型**：
   - 字段：ptr、len、可选的 handle（用于内核注册，如 io_uring buffer group）。
   - 自举前可为占位或最小实现，保证自举后网络库可直接基于该类型扩展，而无需改 API。

**验收**：Reader/Writer 可编译；超时在签名或文档中明确；Buffer(ptr, len, handle?) 存在；与 `高并发IO与多平台.md` 中「零拷贝 / fixed_buffers」一节对应。更细的「自举前为极致 IO/网络性能做的准备」见 `analysis/自举前-极致IO与网络性能准备.md`。

---

### 第三步：完善「编译器诊断」

**目标**：自举时将编写数万行 .sx 代码，若错误信息只有「类型错误」而无位置与期望/实际类型，排错成本极高。必须实现带「面包屑」的错误追踪。

**动作**：

1. **实现带面包屑的错误输出**，至少覆盖**类型不匹配**场景，格式可参考：
   ```
   Error: Type mismatch at main.sx:45:12
      | let x: i32 = get_data();
      |              ^^^^^^^^^^ Expected i32, found Option<i32>
   ```
   - 必须包含：**文件、行、列**；**出错代码行**（带上下文）；**期望类型 vs 实际类型**；**出错子表达式**的标注（如 `^^^^^^^^^^`）。

2. **逐步扩展到**：未定义名、导入失败、参数个数/类型不匹配、trait 未实现等，至少保证「自举常用路径」上的错误都有面包屑。

**验收**：对若干典型错误用例（含类型不匹配、未定义名、导入失败），编译器输出包含上述四要素；自举前测试或文档中保留示例输出。
