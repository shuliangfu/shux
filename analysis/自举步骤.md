## 0.A 先读：自举「完成」与「假绿」的定义（铁律）

| 说法 | 是否算数 | 真正需要的证据 |
|------|----------|----------------|
| `return-value` → exit **42** | **不够** | 只证明「无 import 的最小 main」能过 |
| `prove_module_selfhost` N++ / IDENTICAL | **不够** | 证明 thin/hybrid 模块 surface，**不**证明用户程序 `-o` |
| Stage2 / `bootstrap-verify` 语义自举 | **不够** | 证明两代编译器对**约定子集**一致，不等于 bstrict 全绿 |
| R2 full H=0 / rest T 下降 | 有价值 | 仍须叠加 **产品用户路径** 闸门（§0.D） |
| **Ubuntu `tests/run-all-bstrict.sh` exit 0** | **才算产品可进自举下一阶段** | **禁止** `BEST_EFFORT` / `SKIP_BUILD` / `VIS=compat` / soft typeck 当绿 |
| `shux_v2 == shux_v3` / D-03 哈希 | 完全自举终局之一 | **前置**：bstrict 全绿 + 无双权威 pin 糊弄 |

**一句话**：自举进度可以报 R2/Cap，但**任何人（含 AI）不得说「自举过了 / 可以自举了」**，除非 Ubuntu 上 **严格 bstrict 全绿** 且产品闸门矩阵（§0.D）全绿。

## 0.B 今晚（2026-07-15）问题是怎么出现的？（因果链）

> 细节与验收命令见 [问题分析文档.md](问题分析文档.md)。此处只写**怎么滚雪球**，避免下一波再犯。

### B.1 背景：14 号已经埋了「看起来能自举」的坑

14 号前后仓库里同时存在：

1. **R2 / prove / Stage2 / bootstrap-verify 大量绿点**（模块 surface、两代烟测、return-value 42）。  
2. **产品用户路径并未真正健康**：`examples/hello.x`、`tests/option`、`stdlib-import` 等「**import + 调用 + `-o`**」仍反复炸。  
3. **产品权威在 seed pin 的 `codegen_x.o` / `parser_x.o`**，`.x` 修了未必进产品；seed 与 `.x` 已有漂移（mangle、`out` 启发式、STRING 序等）。  
4. **typeck 对 dep 的 XT001 常被当噪声**：`-E` 仍吐出大 C → 心理上「能编译」；真正 `-o` 过 gcc 时才暴露 ABI/struct/body 错误。  
5. 验收习惯偏向 **「host 能跑 return-value」** 或 **「stage1 某次能 -o」**，而不是 **「当前 g05 链出来的 `shux_asm` + 全矩阵」**。

于是 14 号「自举一天」的验证**结构上就假绿**：测的是自举**工程进度**，不是自举**产品能力**。今晚所有「修了又坏、坏了再修」都是在还这笔债。

### B.2 今晚滚雪球的时间线（压缩）

```
既有债
  · dual-authority seed vs .x
  · print 重载 mangle 多路径
  · STRING 未解码、Context=0、字段 . vs ->
  · dep typeck soft-fail 被忽略
        │
        ▼
继续修 hello（以为还是 print 名）
  · 发现 call site 已是 print_u8_ptr_i32（F 基本好了）
  · 真实 blocker 变成：libc redecl / 生成 C 非法 / typeck XP003
        │
        ▼
把「本地未提交的一堆修」打成 0d717273 一次推
  · parser STRING_LIT 大改 + enum
  · std.io.sync → shux_sys_*
  · preamble / runtime
  · std.fmt / core.fmt / core.result
        │
        ▼
Ubuntu pull + g05 relink
  · return-value 仍 42  → 假安全感
  · option / hello / stdlib-import → XP003 硬失败，-E 0 字节
        │
        ▼
错误操作放大
  · 裸 gcc -c parser_gen.c → parser_x.o 残缺 →「no functions」
  · 大 commit 无法单点 bisect
  · seed 与 .x 字段启发式继续漂移（out→）
```

### B.3 根因分层（不是「一个 bug」）

| 层 | 今晚暴露的问题 | 为何在自举语境下致命 |
|----|----------------|----------------------|
| **验收层** | 用 return-value / prove / soft typeck 当「编译器 OK」 | 自举的主体是 **吃全仓库 + 用户 import 路径**，最小 main 永远不够 |
| **权威层** | 产品 `*_x.o` 来自 seed pin；`.x` 修法未进 seed 或进了半截 | 双权威 → 修一处绿一处、产品仍红 |
| **数据流层** | STRING 只扫边界不解码；field 用名字猜 `->`；struct `=0` | 生成 C「能看不能链」或语义错 |
| **操作层** | scp 源码（已禁）、裸 gcc 编 mega、大杂烩 commit | 一次污染主机状态，难回滚 |
| **策略层** | 未绿 bstrict 却持续「自举」叙事 | 债务利滚利，AI/人越修越乱 |

### B.4 当前产品真实水位（写步骤时必须认）

| 路径 | 水位（2026-07-15 晚） |
|------|------------------------|
| `tests/return-value` `-o` | 通常仍 **42** |
| `tests/option` / `stdlib-import` / `examples/hello` | **常 XP003 或 -o 被 gcc 打回**（见问题分析文档） |
| `run-all-bstrict` | **未全绿 → 禁止开自举 Stage2 庆祝** |
| print mangle call site | 在能 `-E` 的版本上 **已大致正确** |
| parser seed `0d717273` | **高风险**；实验曾用 f49ec6aa parser 恢复 hello `-E` |

## 0.C 14 号自举验证为什么有问题？（必须正面认）

下面不是甩锅，是**验收设计缺陷清单**——后面杜绝项全部对应这里。

| # | 14 号（及前后）的验收习惯 | 为什么错 | 正确闸门 |
|---|---------------------------|----------|----------|
| 1 | **return-value=42 = 编译器可用** | 无 import、无 std、无 overload、无 dep typeck | §0.D 矩阵必含 option / hello / stdlib-import |
| 2 | **`-E` 出大 C = 过了** | typeck 可 soft 失败仍 emit；C 里可有非法初始化/错符号 | 必须 **`-o` 过 host gcc + 运行 exit/stdout** |
| 3 | **XT001 当噪声跳过** | dep 失败可污染 entry；XP003 一硬失败整路径塌 | entry **硬失败必须红**；dep 噪声要单独记账，不能当绿 |
| 4 | **prove N / R2 IDENTICAL = 自举前进** | 测的是模块 hybrid surface，不是产品用户 ABI | R2 报进度可以；**自举放行**只看 bstrict + 用户矩阵 |
| 5 | **Stage2 / bootstrap-verify 通过 = 能自举** | 子集一致 ≠ 全测试矩阵；且 host 可能不是当前 g05 产品 | 放行前强制 **当前 `shux_asm` + run-all-bstrict** |
| 6 | **BEST_EFFORT / 旧 stage1 host 顶着用** | 绿的是「某次产物」不是「当前源码链」 | 只认 **本 SHA pull + make + g05 relink 后的 shux_asm** |
| 7 | **seed 与 .x 不同步仍标修完** | 产品跑 seed，文档写 .x | 改产品路径必须 **seed 与 .x 同 commit 验收** |
| 8 | **一次 commit 塞 parser+std+runtime+fmt** | 红了无法定位 | 一 commit 一主题；parser 与 stdlib 拆开 |
| 9 | **mac 测过或「逻辑上对」就算** | 金标是 Ubuntu x86_64 | 无 Ubuntu 证据不得改 KPI 完成 |
| 10 | **假绿写进进度文** | 后续 AI/人默认「已过」继续加压 | 进度文必须写 **未过 bstrict / 用户矩阵红** |

**结论**：14 号不是「没干活」，而是 **验收测错了对象**——用工程轨（R2/prove）的绿，覆盖了产品轨（用户 `-o` / bstrict）的红。今晚的爆炸是 **产品轨债务被一次大 commit + 错误重建操作引爆**。

## 0.E 后面自举如何杜绝（操作清单 + 流程改造）

### E.1 组织流程（强制）

```
改代码
  → 本地冒烟（可选）
  → commit **单主题**（parser / codegen / 一个 std 模块 拆开）
  → push
  → Ubuntu pull
  → make + g05 relink（正确方式；日常 = L2/L3 可）
  → §0.D 产品矩阵（动产品面时强制）= 日常真测
  → 矩阵绿才允许：改进度 KPI / 开下一微步
  → **整体功能收口 / Cap·R2·M 闭环 / 谈自举**：L4 真冷 + 全量 bstrict（主闸门）
  → **当天动过产品面却还没 L4**：日终兜底再跑一次真冷全测
  （节奏权威：自举方法.md §0.2 · skill §3.3；禁止每微 commit 全量 L4；禁止只靠晚上、白天假绿）
```

### E.2 绝对禁止

1. **禁止** mac→Ubuntu `scp` 源码（只 git）。  
2. **禁止** `gcc -c parser_gen.c -o parser_x.o` 等绕过 Makefile 的 mega 编译。  
3. **禁止** 用 return-value 单点代替产品矩阵。  
4. **禁止** BEST_EFFORT / SKIP_BUILD 写进验收结论。  
5. **禁止** seed 与 `.x` 只改一边就关单。  
6. **禁止** 未绿 bstrict 时推进「自举完成 / 可自举」叙事。  
7. **禁止** 把 XT001 soft 噪声当「没事」。  
8. **禁止** 超大杂烩 commit（parser+runtime+三个 std 一起推）。

### E.3 改产品路径时的「权威清单」

| 你改了 | 必须同步验收的权威 |
|--------|-------------------|
| `codegen.x` | `seeds/codegen_gen.linux.x86_64.c` → `make codegen_x.o` + g05 |
| `parser.x` | `seeds/parser_gen.linux.x86_64.c` → `make parser_x.o` + g05 |
| `typeck.x` | pin/patch/`typeck_x.o` 是否真被链进 `shux_asm`（查 nm / 时间戳） |
| `std/**/*.x` | 当前 `shux_asm` 直接 `-E/-o` 用户程序（不靠旧 .o 缓存） |
| `pipeline_glue.c` / `ast_pool.c` | Ubuntu 重建 `pipeline_x.o`（见 AGENTS.md） |

### E.4 假绿红队（每次宣称「过了」前自问）

1. 证据是不是 **本 SHA + 本机 g05 链出的 shux_asm**？  
2. 有没有 **import + 调用 + `-o` + 运行**？  
3. 有没有 **看生成 C / gcc 报错**，而不只是 shux 的 typeck 警告？  
4. seed 和 .x 是否 **同 commit**？  
5. bstrict 是否 **严格全绿**？（仅当谈「可自举」时必须问）

### E.5 与 R2 模板的关系

## 2. R2 模板（默认主线，把 `<MOD>` 换成模块名）

### 2.1 改哪些文件

| 文件 | 做什么 |

1. 目标：`typeck` / `pipeline` 等 pin 的 `*_x.o` 能否被真实 `.x` 产物替换。  
2. 先 smoke 再产品；失败写阻塞（能力缺口 → 转 Cap）。  
3. 与 R2 **不互等**：M 绿 ≠ rest H 下降。
