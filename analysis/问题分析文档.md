## 1. 当前阻塞问题（按优先级）

### 1.1 【P0·阻塞】codegen Vec4f 类型映射 bug — `codegen_gen.c` seed 未同步

**症状**：bstrict 跑到 `run-perf-simd-shuxffle-select.sh` 失败，cc 报 `conflicting types for 'std_simd_add'`：
```c
extern i32x8_t std_simd_add(i32x8_t a, i32x8_t b);   // Vec8i 正确
extern int32_t std_simd_add(int32_t a, int32_t b);   // Vec4f 被错误翻译成 int32_t
```

**根因（已定位）**：
- 源码 [src/codegen/codegen.x](file:///home/shu/shux/compiler/src/codegen/codegen.x#L2954-L2969) 的 `emit_vector_c_type_out` 函数 **已有** F32 处理（L2954-2969，返回 `f32x4_t` / `f32x8_t` / `f32x16_t`）。
- 源码 [ast_pool.c](file:///home/shu/shux/compiler/ast_pool.c#L10246-L10259) 的 `pipeline_codegen_vector_type_cstr` **已有** F32 处理（elem_kind==14 → f32x4_t）。
- **但** 实际编译使用的 [codegen_gen.c](file:///home/shu/shux/compiler/codegen_gen.c#L4074-L4105) 是从 `seeds/codegen_gen.linux.x86_64.c` 复制的**旧版**（Makefile L1741-1747 默认路径），**没有** F32 分支，Vec4f 走 fallback 到 `int32_t`。
- 验证：`grep -c 'f32x4_t' codegen_gen.c` = 0；`grep -c 'f32x4_t' src/codegen/codegen.x` = 2。

**Makefile 默认路径（L1736-1753）**：
```makefile
codegen_gen.c: src/codegen/codegen.x
	@if [ "$(SHUX_FORCE_REGEN_GEN)" = "1" ]; then \
	  ./$(SHUX_C) -L .. -L src ... src/codegen/codegen.x -E-extern > "$$tmp" && mv "$$tmp" $@; \
	elif [ -f seeds/codegen_gen.linux.x86_64.c ]; then \
	  cp -f seeds/codegen_gen.linux.x86_64.c codegen_gen.c; \  # ← 默认走这里，用旧 seed
	  echo "codegen_gen.c: synced from tip seed pin"; \
	else \
	  ./$(SHUX_C) ... -E-extern > "$$tmp" && mv "$$tmp" $@; \
	fi
```

**已尝试的修复（失败）**：
```bash
cd /home/shu/shux/compiler && SHUX_FORCE_REGEN_GEN=1 make codegen_gen.c codegen_x.o
# 输出：make: `codegen_gen.c' is up to date.
# 原因：make 检测到 codegen_gen.c (19:33) 比 src/codegen/codegen.x (19:20) 新，认为不需要重建
```

**下一步修复方案**：
```bash
# 方案 A：删除 codegen_gen.c 强制重建
cd /home/shu/shux/compiler && rm -f codegen_gen.c && SHUX_FORCE_REGEN_GEN=1 make codegen_gen.c codegen_x.o

# 方案 B：用 -B 标志强制重建
cd /home/shu/shux/compiler && SHUX_FORCE_REGEN_GEN=1 make -B codegen_gen.c codegen_x.o
```

**修复后验证**：
```bash
grep -c 'f32x4_t' codegen_gen.c  # 应该 >= 1
```

**修复后重链 shux + 重跑 bstrict**：
```bash
cd /home/shu/shux/compiler && rm -f shux shux-c bootstrap_shuxc && make bootstrap-driver-seed SHUX_SKIP_SUBSCRIPT_MAKE=1 SHUX_RUN_ALL_BOOTSTRAP_SHUX=1
cd /home/shu/shux && SHUX_BSTRICT_SKIP_BUILD=1 SHUX_BSTRICT_RUN_ALL=1 SHUX=./compiler/shux SHUX_LINK_SHUX=./compiler/shux_asm SHUX_FORCE_LINK_BACKEND=c SHUX_RUN_ALL_BOOTSTRAP_SHUX=1 SHUX_SKIP_SUBSCRIPT_MAKE=1 SHUX_CI_NO_SKIP=1 SHUX_BSTRICT_DARWIN_COOLDOWN=3 SHUX_BSTRICT_DARWIN_HEAVY_COOLDOWN=10 ./tests/run-all-bstrict.sh
```

## 4. 关键技术上下文

### 4.1 L4 真冷测流程（自举方法.md §12.6）

1. **删除全部编译物**：
   ```bash
   cd compiler && find . -name '*.o' -delete
   rm -f shux shux-c bootstrap_shuxc shux-x shux_asm
   ```
2. **重编 shux / shux_asm / shux-c / bootstrap_shuxc**：
   ```bash
   make bootstrap-driver-seed SHUX_SKIP_SUBSCRIPT_MAKE=1 SHUX_RUN_ALL_BOOTSTRAP_SHUX=1
   ```
3. **g05_prepare_and_relink**（如需）：
   ```bash
   make relink-shux
   ```
4. **产品矩阵**：bootstrap-driver-seed 已包含。
5. **bstrict 全量**：
   ```bash
   SHUX_BSTRICT_SKIP_BUILD=1 SHUX_BSTRICT_RUN_ALL=1 ... ./tests/run-all-bstrict.sh
   ```

### 4.2 新 CLI 子命令强制（commit dccfba43）

- `shux file.x` 不再工作。
- 必须用显式子命令：`shux build file.x` / `shux check file.x` / `shux run file.x`。
- `build` vs `check` 输出差异：`build` 路径输出 `typeck error[XT001]: ...` 前缀；`check` 路径输出 `error[XT001]`（无 `typeck` 前缀）。

### 4.3 codegen_gen.c 生成路径（Makefile L1736-1753）

- `SHUX_FORCE_REGEN_GEN=1`：用 `shux-c -E-extern` 从 `src/codegen/codegen.x` 重新生成。
- 默认：从 `seeds/codegen_gen.linux.x86_64.c` 复制（seed pin）。
- 无 seed：用 `shux-c -E-extern` 生成。

**坑**：seed pin 可能是旧版，不包含最新 codegen.x 的修复。修改 codegen.x 后必须 `SHUX_FORCE_REGEN_GEN=1` 或删除 codegen_gen.c 强制重建。

### 4.5 Vec4f / Vec8i 类型映射

- `Vec8i`：8-lane i32 vector → `i32x8_t`
- `Vec4f`：4-lane f32 vector → `f32x4_t`
- `Vec4f` 在 typeck 里被识别为 NAMED 类型（name="Vec4f"），但 codegen 通过 `asm_type_is_simd_vector_spelling` 识别为 SIMD vector。
- codegen 路径：`emit_type` → 检测 NAMED 名 "Vec4f" → 调用 `emit_vector_c_type_out(elem_kind_ord=F32, lanes=4)` → 输出 `f32x4_t`。

### 6.3 net-context gate 重复符号 bug（2026-07-21 根源修复）

- **症状**：`tests/run-std-net-context-gate.sh` 失败，cc 报 8 个 duplicate symbol：
  ```
  duplicate symbol '_shux_io_submit_write' in:
      /tmp/shux_shux_x-*.o
      compiler/runtime_asm_io_stubs.o
  duplicate symbol '_std_context_deadline_ns' in:
      /tmp/shux_shux_x-*.o
      std/context/context.o
  ```
  但 io-context gate (STD-091) 链同样的 .o 却通过。

- **根因分析**（上帝视角）：
  1. **`runtime_asm_io_stubs.o` 推入路径**：`shux_invoke_cc_impl` L5004-5014（page_mmap 路径）在 `std/heap/page_mmap.o` 推入后，无条件推入 `runtime_asm_io_stubs.o` 提供 `shux_sys_mmap/munmap` weak 桩。**非 Linux-only**（之前的错误判断）。
  2. **强符号冲突**：`runtime_asm_io_stubs.from_x.c` L144/L159/L168 中 `shux_io_register`/`shux_io_submit_read`/`shux_io_submit_write` 是**强符号 T**，与 `std/io/core.x` L30/L113/L179 的 export function body 冲突。
  3. **`std.context` 未在 link_only 列表**：`std/context/mod.x` L126 `export function deadline_ns` 生成强符号 `std_context_deadline_ns`，codegen co-emit body 与预编 `std/context/context.o` 的强符号冲突。
  4. **io-context 通过原因**：macOS `-dead_strip` + `-ffunction-sections` 把不可达的 body 丢弃，不参与 duplicate 检测；net-context 引用 `std/net/net.o`（U `_std_context_deadline_ns`），让这些符号可达，触发 duplicate。

- **根源修复**（两条改动，同一 commit）：
  1. [compiler/seeds/runtime_asm_io_stubs.from_x.c](file:///home/shu/shux/compiler/seeds/runtime_asm_io_stubs.from_x.c#L143-L183) L144/L159/L168 三个函数改为 `SHUX_WEAK`，与文件中其他 io 桩一致。G.7 单一权威：`std/io/core.x` 是强实现，本桩仅 fallback。
  2. [compiler/seeds/runtime_link_abi.from_x.c](file:///home/shu/shux/compiler/seeds/runtime_link_abi.from_x.c#L3351-L3353) `pipeline_codegen_std_dep_link_only` 列表加入 `std.context`，让 codegen 不 co-emit body，由预编 `std/context/context.o` 提供符号。
