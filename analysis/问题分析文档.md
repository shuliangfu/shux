**现象**：

**修复方向**：在 `g05_relink_env.sh` 的 `case "$UNAME_S"` 中添加 MINGW 分支：

```bash
MINGW*|MSYS*|CYGWIN*)
```

### 2.2 ❌ P0：`asm_full_link_stubs` 增量构建时间戳竞态（阻塞 shux-c 链接）

**现象**：
```
undefined reference to `platform_coff_write_coff_o_to_buf'
collect2.exe: error: ld returned 1 exit status
make: *** [makefile:2501: shux-c] Error 1
```

**根因**：
1. `src/asm/user_asm_seed_bridge.o` 重新编译后引入新 U 符号 `platform_coff_write_coff_o_to_buf`（SHUX_WEAK 弱符号，Windows PE 不支持 weak）
2. `build_asm/seed_host/asm_full_link_stubs.o` 时间戳比 `user_asm_seed_bridge.o` 新（因其他依赖触发重新编译），make 不重新生成 stubs.c
3. stubs.c 不含 `platform_coff_write_coff_o_to_buf` stub → shux-c 链接失败

**手动临时修复**（已验证有效）：
```bash
rm -f build_asm/seed_host/asm_full_link_stubs.c build_asm/seed_host/asm_full_link_stubs.o
make shux-c SHUX_LEGACY_C_FRONTEND=1
```

**根治方向**（codex 决定）：
- 方案 A：让 `asm_full_link_stubs.o` 依赖 `.x` 源文件（而非 .o），这样 .x 改了就重新生成 stubs
- 方案 B：用 `.PHONY` 强制每次重新生成 stubs（性能成本）
- 方案 C：让 stubs 生成步骤扫描所有 U 符号，对比已有 stubs，不一致就重新生成
- 方案 D：在 `gen_asm_full_link_stubs.pl` 中加一个 sentinel 文件记录上次扫描的符号集，对比变化

**关键文件**：
- `compiler/makefile` L1503-1509：`$(USER_ASM_SEED_HOST_STUBS)` 依赖定义
- `compiler/scripts/gen_asm_full_link_stubs.pl`：stub 生成脚本（已正确处理 Windows non-weak）
- `compiler/seeds/user_asm_seed_bridge.from_x.c` L275：`extern int32_t platform_coff_write_coff_o_to_buf(...) SHUX_WEAK;`

**验证**：
```bash

touch src/asm/user_asm_seed_bridge.x
```

## 6. Windows 平台硬限制速查（codex 写代码必读）

详见 `analysis/Windows平台限制与测试指南.md`，这里列最关键的 5 条：

1. **`__attribute__((weak))` 不被 PE 支持**：stub 文件必须提供强符号定义，用 `--allow-multiple-definition` 解决冲突
2. **Win32 `CreateFileA` 不识别 `/tmp/`、`/c/...`**：传给 SHUX 的路径必须用 `C:/...` 或相对路径，用 `cygpath -m` 转换
3. **Smart App Control 阻止未签名 exe**：需 self-signed cert（已配置）+ `Set-AuthenticodeSignature` 签名
4. **Windows std handle 常量是负数**（-10/-11/-12）：错误检查用 `== -1`，非 `< 0`
5. **MinGW gcc 自动给 `-o` 加 `.exe` 后缀**：检查/执行时需加后缀

- Cygwin/MSYS weak 别名不可靠，bootstrap 链须强符号转发

- Windows 无 `cc` 命令（必须 `CC=gcc`）

- TEMP 路径必须用短路径（`TEMP=C:/shux_tmp`）

### 9.4 最终真根因详解（commit 9f68cda6）

**数据流**：
1. gate 调 `make bootstrap-driver-bstrict` → 调 `build_shux_asm.sh`（experimental 链接）
2. MSYS 上 experimental 链接失败（`cannot find -lc`）→ 回退 `gen_driver` 路径
3. 回退路径调 `ensure_runtime_cc_stubs()` → **用残缺 flags**（仅 `-DSHUX_USE_X_PREPROCESS`，缺 `-DSHUX_NO_C_FRONTEND`/`-DSHUX_RT_*_FROM_X`/`-DSHUX_ASM_USE_COMPILER_IMPL_C`）**覆盖** `src/runtime_driver.o`
4. 之后 `refresh-shux-asm-gate` → `relink-shux`（g05）链接这个被污染的 `runtime_driver.o`
5. g05 产物的 X pipeline 静默返回 -1（XP001/XP003），连 `function main(): i32 { return 42; }` 都失败

**实证**：md5 对比 — `build_shux_asm` 跑完后 `runtime_driver.o` md5 变了；g05 产物从 rv=42 变成 NO OUTPUT。

**修复**：`ensure_runtime_cc_stubs` 不覆盖已存在的 `src/runtime_driver.o`（make/g05 是权威），只在冷启动缺失时才建。
