## 1. 术语（读进度前先对齐）

| 层次 | 名称 | 判据（摘要） |
|------|------|--------------|
| 语义自举 | Stage 0 烟测 | `make bootstrap-verify`：两代 `shux` 约定测试 outcome 一致 |
| 黄金自举 | 阶段 D | Stage1 再编同一 tree → Stage2；**SHA256(stage1)==SHA256(stage2)** |
| 编译器无 C | 阶段 E | 默认 bootstrap **不链、不 cc -c** 编译器前端 C（E-soft：文件可保留） |
| std 无 C | 阶段 F | `std/` 下 **0 个手写 `.c`/`.h`**（F-ZC） |
| **完全自举** | **D + E + F** | 上述三者齐备；允许 seed **仅冷启动** |
| **终局清场** | 阶段 G | 物理删除仓库内全部 `.c/.h`；Linux bootstrap **`-nostdlib` 硬绿**；`build.x` 替代 Makefile |

**两层进度口径**（下文模块表均区分）：

- **运行时自举**：默认 `bootstrap-driver-bstrict` 已用 `.x` 产物，不链对应 C 前端。
- **物理零 C**：仓库内对应 `.c`/`.h` 已删除且 **git 已提交**；终局以 `SHUX_NO_HANDWRITTEN_C_STRICT=1` 硬绿为准。

**验收平台**：Linux x86_64 为金标准；macOS arm64 开发机用 Docker `linux/amd64` 跑 gate（`./tests/run-linux-a09-a11-gate.sh`）。

### 5.1 防塌陷（鸡生蛋链不可静默断裂）

自举循环：**Stage0（冷启动 seed）→ gen1（shux_asm_stage1）→ gen2（shux_asm2）→ SHA256(gen1)==SHA256(gen2)**。下列情况视为**塌陷**（表面绿、实际未自举）：

| 塌陷模式 | 表现 | 检测 |
|----------|------|------|
| **seed 静默回退** | `make bootstrap-driver-seed` 失败 → `cp seeds/bootstrap_shuxc.* ./shux` 无审计 | `logs/bootstrap-seed.pinned-fallback` + gate FAIL（除非 `SHUX_BOOTSTRAP_COLD_START_PINNED=1`） |
| **postlink 复制** | strict `shux_asm` 烟测失败 → `cp ./shux` 或 `experimental` 覆盖 | `logs/bootstrap-postlink.compiler-fallback` → gate **硬 FAIL** |
| **假 fixed point** | stage1==stage2==pinned 同 SHA256 | gate FAIL（triple equal） |
| **假 gen1** | stage1==seed（gen1 未编译新一代） | gate FAIL |

**允许的唯一「外部蛋」**：Git 中 **`seeds/bootstrap_shuxc.<os>.<arch>`** 作 Stage0 冷启动（须显式 `SHUX_BOOTSTRAP_COLD_START_PINNED=1`），但 **gen1 必须与 pinned 不同字节/哈希**，且 gen2 须与 gen1 一致。禁止在 gate 内复制 prebuilt 冒充 stage1/2。

# 审计日志
cat logs/bootstrap-audit.log
./tests/run-bootstrap-anti-collapse-gate.sh
```

**实测验证**：`SHUX_NO_C_SEED_LINK=1` 构建的 shux-x 对 `return_operand_type_mismatch.x` 报 `codegen error[CG002]`（typeck 失效），而默认构建正确报 `typeck error[T001]`。**2026-07-08 修复后**：`SHUX_NO_C_SEED_LINK=1` shux-x 正确报 `typeck error: return expression type mismatch: expected i32, found bool`（EXIT=1），与默认构建行为一致。

**根因定位与修复（2026-07-08，commit 1790d12f）**：`token.x` 的 `TokenKind` 枚举比 `include/token.h` 少 8 个 `TOKEN_ATTR_*` 条目（GLOBAL_ALLOCATOR/COLD/INLINE_NEVER/INLINE_ALWAYS/EXPORT_NAME/PANIC_HANDLER/THREAD_LOCAL/PERCPU）。`lexer_gen.c` 枚举经 `sync_lexer_gen_token_enum.pl` 从 `token.h` 同步（TOKEN_IDENT=59），而 `parser_gen.c` 枚举从 `token.x` 生成（TOKEN_IDENT=51）。lexer 正确 lex "main" 为 IDENT(kind=59) 但 parser 将 59 解释为 TOKEN_ISIZE，导致所有函数解析失败（num_funcs=0）、typeck 被跳过、check 静默成功。修复：token.x 补齐 8 个条目 + Makefile 为 `parser_x.o`/`pipeline_x.o` 添加枚举同步步骤。修复后 `SHUX_NO_C_SEED_LINK=1` shux-x 正确报 `typeck error: return expression type mismatch: expected i32, found bool`（EXIT=1）。

**解决路径（2026-07-08 God-View 修订）**：原计划"迁移 `typeck_module`/`codegen_module_to_c`/`lexer_new` 入口到 `.x`"**不再必要**。God-View 全局分析发现：

1. **SHUX_NO_C_FRONTEND 模式下不调用这些 C API** — runtime.c 走 `shux_pipeline_run_x_pipeline_large_stack`（[runtime.c:5062](file:///home/shu/shux/compiler/src/runtime.c#L5062)），不经过 `typeck_module`/`codegen_module_to_c`/`lexer_new`。这些 C API 的 weak stubs（返回 -1/NULL）根本不会被调用。
2. **X pipeline 在 Linux x86_64 上已完全工作** — 2026-07-08 ubuntu-server 真机验证：`SHUX_NO_C_SEED_LINK=1` 构建的 shux-x，check ✅（typeck error 正确报告）、compile ✅（`function main(): i32 { return 42; }` 编译运行返回 42）、hello ✅、binary-expr ✅、return-value ✅、types gate check+link+run ✅。仅 1 个 typeck 测试有差异（`import_const_bare_fail.x`）。
3. **macOS compile 失败是 arm64 后端 stub 导致** — `arch_arm64_enc_enc_label` 在 `seed_link_compat.c:846` 是 weak stub（返回 -1），真正的 arm64 后端实现未链入。这是平台限制，非 X pipeline 问题。

**G-02c 第一步（X pipeline parser 枚举修复）已完成** ✅（commit 1790d12f）。**G-02c 后续步骤（迁移 C API 入口）不再必要**，可直接进入 G-02a（物理删除 LEGACY C 前端 .c 文件），前提是将默认构建也切换到 `SHUX_NO_C_SEED_LINK=1`（no_c）模式。

### 10.5 G-02b runtime ABI 收薄

> **God-View 分析结论（2026-07-10）**：9 个 runtime ABI 文件（~16000 行 C）**全部含语言限制**，不可简单 .x 迁移。分类为"语言限制 C 桩"（与 G-01 `runtime_slice_glue.c` 同类），需策略调整。

**9 文件语言限制统计**：

| 文件 | 行数 | argv | #if | weak | 语言限制 |
|------|------|------|-----|------|----------|
| `runtime_abi.c` | 108 | ✓ | ✓ | ✓ | argv 双指针, weak, #if, C main 转发 |
| `main.c` | 14 | ✓ | — | — | C main 入口 `int main(int argc, char **argv)` |
| `runtime_proc_abi.c` | 62 | — | — | — | POSIX syscall (waitpid/stat) |
| `runtime.c` | 6859 | 299 | 88 | 25 | argv, #if, weak |
| `runtime_link_abi.c` | 5344 | 237 | 80 | — | argv, #if |
| `runtime_driver_abi.c` | 624 | 10 | 4 | 6 | argv, #if, weak |
| `runtime_pipeline_abi.c` | 1918 | 0 | 5 | — | #if, extern |
| `runtime_driver_diagnostic.c` | 854 | — | — | — | va_list/vsnprintf/getenv |
| `runtime_io_abi.c` | 222 | 0 | 4 | — | #if |
| **合计** | **~16000** | **546** | **181** | **31** | — |

**6 类语言限制（.x 无法表达）**：

1. **`char **argv`（双指针）**：.x 无法安全从 `*u8` 读取 `argv[i]`，须调 `driver_get_argv_i` C 桩
2. **`__attribute__((weak))`**：.x 无 weak 属性；POSIX 用 weak 实现 stub 覆盖，Windows/MinGW 不支持 weak 函数符号
3. **`#if` 条件编译**：.x 无预处理器；平台差异（`__APPLE__`/`__linux__`/`_WIN32`）需 C 层 `#if` 处理
4. **`va_list`/`vsnprintf`**：.x 无可变参数函数；诊断格式化输出依赖 C va_list
5. **POSIX syscall（waitpid/stat/open）**：.x 无 libc 依赖；系统调用需 C 层封装
6. **C main 入口**：`.x` `main_entry` 由 C `main()` 调用；crt0 `.s` 调 C main

**策略调整**：

- 这 9 个文件是**语言限制 C 桩**，非"待迁移 .x 的业务代码"

### 10.9 G-08 延后

> **延后理由更新（2026-07-09）**：原延后理由是"无 Windows 机器"。现本地已有 Windows 真机（`ssh windows-server`），但延后仍成立 —— 理由改为"G 主链未闭合 + codegen 缺 Windows ABI"。关键结论：**延后不会因种子问题死锁**，自举完成后走交叉编译路径，无需 Windows seed。

**两条 Windows 落地路径对比**：

| 路径 | 需要 Windows seed？ | 种子不兼容风险？ | 可行性 |
|------|-------------------|----------------|--------|
| **A. Windows 自举**（Windows seed 冷启动 → 编译 .x → shux.exe） | 是 | **是**（.x 语法演进后旧 seed 无法解析新语法） | ❌ 不采用 |
| **B. 交叉编译**（Linux 已自举 shux → `-target x86_64-windows -backend asm` → shux.exe） | **否** | **否** | ✅ 采用 |

**路径 B 已就绪**（查证 2026-07-09）：
- ✅ COFF 后端已实现：[`compiler/src/asm/platform/coff.x`](../compiler/src/asm/platform/coff.x) `write_coff_o_to_buf`（x86_64，与 elf.x/macho.x 三平台对等）
- ✅ `-target windows` 已接入：[`ast_pool.c`](../compiler/ast_pool.c) `use_coff_o`（Windows -target 且 -o .obj 时为 1）；[`条件编译.md`](条件编译.md) `-target windows → OS_WINDOWS`
- ✅ runtime 层 Windows 分支已有：`runtime_process_os_glue.c` / `runtime_sync_os.c` / `runtime_path_fast.c` 均有 `#if defined(_WIN32)` 分支
- ✅ 交叉编译有先例：`platform-freebsd-v1.md` 已在 Linux runner 交叉编译出 FreeBSD `bootstrap_shuxc`

**唯一功能缺口（非种子问题）**：
- ❌ codegen 仅有 System V ABI，**缺 Windows MS x64 ABI**（grep `compiler/src/asm/` 全是 "SysV" 注释，无 shadow space / rcx/rdx/r8/r9 / win64 处理）
- 后果：交叉编译出的 .obj 机器码用 rdi/rsi 传参，Windows 期望 rcx/rdx + 32B shadow space → 调 Windows API 会崩
- **性质**：codegen 层功能扩展（改 [`backend.x`](../compiler/src/asm/backend.x) / `backend_x86_64_enc_c.c` 参数寄存器分派 + shadow space prologue + MS ABI struct 返回），属 .x 源码修改

**为什么自举后补来得及**：
1. Windows ABI 实现在 .x codegen 源码（backend.x 等），自举后 Linux shux 能编译它
2. 补 ABI = 改 codegen 参数寄存器分派（rdi→rcx 等）+ shadow space prologue + MS ABI struct 返回规则
3. 改完用 Linux shux 交叉编译 → 产出的 shux.exe 自动 Windows ABI 兼容
4. **全程不需要 Windows seed，不需要 Windows 自举** —— seed 在自举完成后退役（§1：seed 仅冷启动）

**G-08 启动条件**：G-02a/b/d + G-03 + G-04 + G-05 全部 ✅（G 主链除 G-07 发布外全绿）。此时 Linux 自举链完全稳定，Windows 移植可纯粹按平台差异定位。

**预防性探针（低成本，不阻塞主线）**：现已有 Windows 真机，建议跑一次探针记录真实状态，把 §4.4 Windows 关卡二的 "⬜" 填上：

```
