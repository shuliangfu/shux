## 0. 阅读约定

- **自举（本文件口径）**：指"内核自举"——用 shux 写出一个能在 QEMU/真机点亮、跑通 `hello kernel` 的最小内核镜像，并由此为起点增量扩展。区别于编译器自身的自举（D+E+F 已闭合，见 [`自举进度.md`](自举进度.md)）。
- **自举前必须**：缺了就无法用 shux 写出"可点亮的最小内核"的硬前置项；属第一波交付。
- **自举后可缓**：内核已点亮、能跑之后，再按需补齐的高级/横向能力；属第二波及以后。
- 难度标记：⭐（轻）～ ⭐⭐⭐⭐⭐（极高），下文统一。

## 2. 缺口总览（按"自举前 / 自举后"分流）

### 2.1 自举前必须完成（第一波 — 让 shux 能点亮一个最小内核）

| # | 特性 | 为什么必须 | 涉及层 | 难度 |
|---|------|-----------|--------|------|
| K1 | **内联汇编 `asm!`**（含 in/out/clobber 约束） | `cr3/msr/lgdt/lidt/wrmsr/hlt/in/out` 没有任何高级语法能表达；这是写内核的命脉 | lexer/parser/typeck/codegen(asm 后端) | ⭐⭐⭐⭐ |
| K2 | **语言级 `volatile`**（指针/左值限定符，优化器不得消除） | MMIO 读必须真正发出；当前 `core.mem` 运行时函数会被 DCE 掉，硬件失效 | typeck + codegen(标记 vol、跳过 DCE) | ⭐⭐⭐ |
| K3 | **naked 函数 `#[naked]`** + **自定义调用约定** | 中断/异常入口必须无 prologue/epilogue；syscall 入口需控制寄存器传参 | codegen(asm 后端 prologue/epilogue 开关) | ⭐⭐⭐⭐ |
| K4 | **`link_section` 属性 + 多段 ELF 发射**（`.text/.data/.rodata/.bss` + 任意段如 `.boot`） | 内核须把 multiboot/stivale 头放进固定段；`.bss` 须单独可清零；当前只发 `.text` | codegen(ELF writer) + parser(属性) | ⭐⭐⭐⭐ |
| K5 | **内核入口符号 / 启动接管**（`#[entry]` 或 `-entry` 指定 `_start`，naked） | bootloader 跳进的 `_start` 须由 shux 生成、naked、负责设栈后 `call kmain` | driver/link + codegen | ⭐⭐⭐ |
| K6 | **boot 协议头宏**（multiboot2 / stivale2 头节） | QEMU/真机 bootloader 要识别头部；必须能放进指定段、指定物理地址对齐 | std(K0 模块) + K4 | ⭐⭐⭐ |
| K7 | **完整 `core.mem` volatile + 最小原子**（`AtomicU64` load/store/CAS + Ordering） | 自旋锁、per-CPU 计数、二级页表项更新都需要；否则 SMP/多核无法起步 | core + 依赖 K1(asm) | ⭐⭐⭐ |
| K8 | **全局可变变量落 `.data/.bss`**（含零初值 `.bss` 不占文件） | 内核大量全局状态；当前后端以代码为主，全局 mutable 支持需补 | codegen(ELF writer) | ⭐⭐⭐ |
| K9 | **内核构建链**（`build.x` 接受 `kernel` 目标 + linker script + QEMU 烟测） | 端到端可验收；无此则特性无法验证 | tools/build + scripts | ⭐⭐⭐ |
| K10 | **中断栈帧 struct + ISR 桥**（CPU 压栈布局 struct + `#[naked]` handler 装/拆帧） | 没有它接不了中断/缺页/系统调用 | std + 依赖 K3/K1 | ⭐⭐⭐⭐ |
| K11 | **地址空间布局与重定位策略**（RIP 相对取址、高半区基址、`.boot` 物理低位段、绝对/PC 相对重定位） | 内核常驻高半区（`0xFFFF...`）、boot 头须在低物理地址；早期无重定位加载器时 RIP 相对省掉大量麻烦 | codegen(ELF writer + 取址模式) + 链接器脚本 | ⭐⭐⭐⭐ |

> K1→K4 是原文档的四个原子特性的"语言级"升级版（不是外置 `.s`，而是 `.x` 内可写）。K5–K10 是补齐"内核工程链"的必要拼图。完成 K1–K11，shux 就具备写一个"点亮 + 串口/帧缓冲输出 + 接中断 + 开页表"的最小内核的全部物理骨骼。

### 2.3 写完整 OS 的语言面缺口（文档原表未覆盖）

> 本节是 2026-07-02 补充的完整度评估。结论先行：**§2.1（K1–K11）覆盖「点亮最小内核」约七成、路线清晰；但「用纯 shux 写一个完整 OS」还差一批被原表误判为「纯 `.x` 工程量 / 库」、实则必须落到语言与编译器层的能力。** 原表把 percpu、分配器、panic、栈上限、动态分派等都丢进 A5「`std.kernel.*` 纯库」，但这些项里有若干**没有语言/后端支持就写不出来**，须从 A5 里拆回语言级。下列 L1–L13 即这部分缺口。

| # | 特性 | 为什么是语言级（非纯库） | 涉及层 | 难度 | 波次 |
|---|------|------------------------|--------|------|------|
| L1 | **全局分配器挂钩**（`#[global_allocator]` 类：注册内核自己的 alloc/free） | 内核无宿主堆；`with_arena`/`Vec`/`String` 等「带分配」类型在内核里若无分配器钩子就不可用——这不是写个 `kmalloc` 就完事，而是**语言分配点须能路由到用户注册的分配器** | typeck(分配点识别) + codegen(调用钩子) + std | ⭐⭐⭐⭐ | 自举后、写真实 OS 前 |
| L2 | **freestanding 契约 + 纯净度 gate**（core 全可用、`std/sys` 在无 OS 目标禁用或条件可用；镜像扫描证明无 host-libc/隐藏堆/OS 符号依赖） | 内核只能依赖 `core` + 自提供的极小运行时；当前只有 `-freestanding` 标志，**未定义「哪些 API 在 no-OS 下可用」契约，也无「镜像无宿主依赖」的自动 gate** | driver + std + gate 脚本 | ⭐⭐⭐ | 自举后 |
| L3 | **freestanding panic 策略（abort-only、不穿越 unsafe/原生帧 unwind）** | 内核 panic 不能 unwind 进 ISR / naked / `asm!` 原生帧（会拆坏 CPU 压栈状态）；须语言级 `panic=abort`（freestanding 默认）+ 不解构跨越 unsafe 帧的语义 | typeck + codegen(unwind) + std.runtime | ⭐⭐⭐⭐ | 自举后 |
| L4 | **栈用量分析与上限保证**（编译期栈深分析 / `#[max_stack(N)]` / 禁递归属性） | 内核栈小且定长（如 16KiB）；ISR / 热路径爆栈即整机死。**没有栈深分析就没法证明「这条中断路径不爆栈」**——这是内核安全刚需，纯库无法提供 | codegen(栈帧统计) + typeck(调用图深) + driver(gate) | ⭐⭐⭐⭐ | 自举后 |
| L5 | **per-CPU 数据模型**（`#[percpu]` + 段相对寻址 / x86_64 GS base、arm64 TPIDR） | SMP 内核每 CPU 数据访问靠段相对寻址，**这是后端寻址模式 + 段控制，不是写个 `percpu!` 宏就行**；原 A5 把它当库是低估 | codegen(寻址) + K4(段) + typeck | ⭐⭐⭐⭐ | 自举后 |
| L6 | **线程安全契约**（`Send`/`Sync` 类，或等价别名检查） | SMP 内核共享数据安全；shux 当前无此模型，跨核共享只靠人盯不安全。可作长期，但应明确「shux 是否要引入并发安全契约」的设计决策 | typeck + trait | ⭐⭐⭐⭐⭐ | 长期 |
| L7 | **freestanding 动态分派 / trait object（vtable，无 RTTI/GC）** | 内核驱动按 trait 分派（block/net/FS 驱动 trait）；**须确认 fat-pointer vtable 在无堆分配下可构造、无运行时类型注册** | typeck + codegen(vtable) | ⭐⭐⭐⭐ | 自举后 |
| L8 | **内联汇编操作数补全**（`sym` 符号操作数、`label` 跳转操作数、clobber memory vs reg、`nomem/readonly/pure` 语义） | K1 只给了 `r/a/c/d` + 部分 options；内核要引用 IDT 表地址（`sym`）、在 asm 内跳转（`label`），缺这两个写 IDT 装载 / 跳转表就别扭 | parser/typeck/codegen(asm) | ⭐⭐⭐ | 随 K1 一起做 |
| L9 | **段属性细控 + 符号控制**（逐段 `SHF_WRITE/ALLOC/EXECINSTR`、`weak` 符号、tentative、linker-defined 符号如 `__bss_start/__bss_end`、`#[no_mangle]`/`#[export_name]`） | 清 `.bss` 要 `__bss_start/__bss_end`；可选驱动要 `weak`；bootloader 符号要 `#[no_mangle]` 不被改名——**当前 parser 无 `no_mangle`/`export_name`/`link_name`**（已核实），K4 只给命名段不给标志位 | parser(属性) + ELF writer + 符号表 | ⭐⭐⭐⭐ | 随 K4 一起做 |
| L10 | **内核 code model + 栈/ABI 约束**（x86_64 `kernel`/`large` code model、`-mno-red-zone`、soft-float/早期 boot 禁 SIMD ABI） | 高半区内核地址 >2GB 必须用 kernel/large code model，否则 rel32 跨段跳转溢出；ISR 栈对齐要求 `-mno-red-zone`（128B 红区会撞中断压栈）。K11 只提地址空间，**未提 code model 与 red-zone** | codegen(编码模型) + driver(target triple) | ⭐⭐⭐⭐ | 随 M2/K11 |
| L11 | **外部汇编 / 外部符号链接视图**（`extern "asm"` / 引用手写 `.s` 符号、符号名 mangling 规则公开） | 内核早期入口、boot trampoline 常是手写 `.s` 与 `.x` 互调；**语言须有稳定的符号命名规则 + extern asm 声明**，否则链接对不上 | parser(extern) + codegen(符号) + driver(link) | ⭐⭐⭐ | 随 K5 |
| L12 | **freestanding 纯净度 gate**（扫描产物 ELF 的未定义符号，拒绝任何 host-libc/OS/隐藏堆依赖） | K9 有 QEMU gate，**缺「镜像纯净度」gate**——内核镜像一旦误链 libc 符号会在真机加载时炸；须自动卡 | gate 脚本 + driver | ⭐⭐⭐ | 随 K9 |
| L13 | **const-eval 能力评估与内核常量构造**（boot 头 magic、IDT 项布局、初始页表项编译期算） | A9 一句带过；内核重度依赖编译期常量布局，**须评估 shux 当前 const-eval 是否够算 IDT/页表常量**，不够则补 | typeck(const) + codegen | ⭐⭐⭐ | 自举后 |

**重新分类（对原 §2.2 的修正）**：
- A5 里的 **per-CPU** → 拆出为 **L5**（语言级），不是纯库。
- A5 里的 **自旋锁/页表/IDT/GDT 表** 仍是纯库 ✅（依赖 K7/L5/K10）。
- A6 **panic 内核版** → 拆出 unwind 策略为 **L3**（语言级）；backtrace 本身可缓。
- A2 **完整内存序 + DMA barrier** 保留库级，但 **L10 的 red-zone/code model** 是其前置语言项。
- **A9 常量求值** → 升为 **L13** 并要求做能力评估，而非一句带过。

**方法论缺口（非语言特性，但写 OS 必备）**：
- **M-host**：内核模块的 host 端单测方法论——内核代码能用 mock OS 接口在宿主编译跑单测（不进 QEMU）；当前 `tests/` 无此模式。
- **M-repro**：可复现构建（OS 镜像须可 bit-identical 复现），便于验证与供应链可信；当前未涉及。
- **M-syz**：长期看，OS 级 syscall fuzzing / 形式化接口契约；远期。

## 3. 第一波详细设计（K1–K10）

### K1 内联汇编 `asm!`（命脉）

**语法提案**（与现有属性风格一致）：

```sx
unsafe {
  asm!("mov %0, %%cr3" : "+r"(cr3_val));            // out/in
  asm!("wrmsr" : : "c"(msr), "a"(lo), "d"(hi));      // in only
  asm!("hlt");                                       // 无操作数
  asm!("in %1, %0" : "=a"(v) : "n"(port));           // 立即数端口
}
```

要素：
- 模板字符串（AT&T/Intel 可配，默认 AT&T 与现有 `.s` 一致）。
- 输出操作数（`=`/`+`）、输入操作数、clobber 列表。
- 可选 options：`pure`（可被消除）、`nostack`、`preserves_flags`、`noreturn`。
- **必须**在 `unsafe { }` 内。

**实现要点**：
1. lexer：新增 `asm!` 关键字与模板字符串词法（复用现有字符串字面量）。
2. parser：新增 `AsmExpr` AST 节点，解析操作数约束串（`"r"/"a"/"c"/"d"/"+r"/"=a"...`）。
3. typeck：把约束映射到寄存器类别，登记 clobber，禁止在 unsafe 外出现。
4. codegen(asm 后端)：**最短路径**——把模板字符串原样塞进当前 `.text` 流，并把 in/out 操作数按约束分配到指定物理寄存器（与现有寄存器分配器协同，clobber 寄存器在块内视为占用）。这是"先塞汇编字符串"的折中，能最快支持硬件开发，不必大改后端。
5. 后续：把模板内占位（`%0`）解析成重定位/操作数绑定，避免纯字符串拼接错位。

**难度 ⭐⭐⭐⭐**：难点在约束匹配与寄存器分配的协同，而不是模板本身。

**验收**：`tests/kernel/asm_*.x` —— `mov %cr3` 读出、`out` 串口输出、`hlt` 不死机，QEMU 跑通。

### K2 语言级 `volatile`（修复 MMIO 硬伤）

**问题**：当前 `core.mem.volatile_load_u32` 是普通函数调用，DCE/autovec 看不到副作用，可能把"读后丢弃"的 MMIO 探测读删掉。

**提案**：类型限定符 + 操作语义双管齐下。

```sx
let mmio: *volatile u32 = 0xB8000 as *volatile u32;
let v: u32 = *mmio;        // volatile 读，不可消除、不可合并、不可重排到序列点外
*mmio = 0x0720;            // volatile 写，同理
```

- `*volatile T` / `*mut volatile T`：指针所指为 volatile。
- 对 volatile 左值的读/写，codegen 标记不可 DCE、不可 GVN、不可 autovec；emit 真实 load/store。
- 与 K1 互补：volatile 管"内存映射的常规读写"，asm! 管"指令集直通"。

**实现要点**：
1. typeck：识别 `volatile` 限定符，传播到解引用点。
2. codegen：对 volatile 访问打标，DCE/autovec 跳过；emit 单条 load/store，禁止合并。
3. 兼容：保留 `core.mem.volatile_*` 作为底层实现，但语义上语言级 volatile 才是"保证"。

**难度 ⭐⭐⭐**。

### K3 naked 函数 + 自定义调用约定

**语法提案**：

```sx
#[naked]
#[calling_conv("kernel_interrupt")]   // 或寄存器参数列表
function irq_handler_14(): void {
  unsafe {
    asm!("push %rax; ...; call page_fault; ...; iret");
  }
}
```

- `#[naked]`：codegen 不生成 prologue/epilogue（无 `push rbp; sub rsp,N`），函数体须自行管理栈与返回。
- `#[calling_conv(...)]`：指定参数寄存器、被调用者/调用者保存集；用于 syscall 入口、ISR。
- naked 函数体只允许 `asm!`（安全子集内无 C 调用语义），typeck 强制。

**实现要点**：在 asm 后端的 `enc_prologue`/`enc_epilogue`（见 `backend.x`）按属性跳过；调用约定表驱动参数寄存器选择。

**难度 ⭐⭐⭐⭐**：需要后端精确控制指令"原始程度"。

### K4 段控制 + 多段 ELF

**现状**：`platform/elf.x` 只发 `.text`/`.text.hot`，符号 `st_shndx` 仅这二者。

**目标**：

```sx
#[link_section(".boot")]
const multiboot_header: MultiBoot2Header = ...;        // .boot 段，bootloader 找它

#[link_section(".data")]
let boot_info: *BootInfo = 0 as *BootInfo;             // 可变全局 .data

#[link_section(".bss")]
let stack: u8[16384];                                  // 零初值，不占文件
```

**实现要点**：
1. parser/typeck：解析 `#[link_section("name")]`，可加 `#[align(N)]`。
2. ELF writer：扩展为多段（`.text/.data/.rodata/.bss` + 任意命名段），每段独立 `SHT_*`/`SHF_*` 标志；`.bss` 用 `SHT_NOBITS`，文件零长。
3. 符号表：`st_shndx` 指向正确段。
4. 重定位：跨段地址引用（`call .text 里的函数`、`.data 取指针`）须记 `R_*` 重定位。
5. 与链接器脚本协同：`build.x` 支持 `-T linker.ld`，把段按物理地址排布。

**难度 ⭐⭐⭐⭐**：ELF writer 改造面较大，但是内核布局的硬前提。

### K5 内核入口接管

- 现有 `crt0_user_x86_64.s` 是用户态 `main`；内核需要 `_start`：naked、设栈、必要时清 `.bss`、`call kmain`、`kmain` 不返回则 `hlt`。
- 提案：`#[entry]` 标注的函数生成裸名 `_start`，并配合 K3 的 naked；或 `shux build --kernel --entry _start --linker-script linker.ld`。
- `kmain` 可用普通函数，编译器正常生成帧。

**难度 ⭐⭐⭐**。

### K6 boot 协议头

- multiboot2 / stivale2 头是固定布局常量结构，须在 `.boot` 段、4 字节对齐、物理地址可达。
- 作为 std 提供：`std.kernel.boot.multiboot2` / `stivale2`，导出 `Header` 结构与构造宏；借助 K4 落段、K9 的 const-eval（或编译期常量）算 magic。
- 烟测：QEMU `-kernel` 识别头并跳进 `_start`。

**难度 ⭐⭐⭐**。

### K7 volatile + 最小原子

- 完整 `AtomicU64/AtomicUsize`：`load/store/swap/cmpxchg/fetch_add` + `Ordering`。
- `cmpxchg`/`lock` 前缀走 K1 的 `asm!`（`lock cmpxchg`）。
- 自旋锁、per-CPU 计数、二级页表项原子更新都依赖它。
- 单核内核可先用 `Relaxed/Acquire/Release`，`SeqCst` 留到多核。

**难度 ⭐⭐⭐**（依赖 K1）。

### K8 全局可变变量落段

- `let` 顶层全局 → `.data`（有初值）/ `.bss`（零初值）；`const` → `.rodata`。
- 后端需支持"非函数符号"写入与段归属，配合 K4。

**难度 ⭐⭐⭐**。

### K9 内核构建链

- `build.x` 增加 `kernel` target：产出 ELF + 指定 linker script + entry。
- QEMU 烟测脚本：`tests/run-kernel-gate.sh`，串口/帧缓冲回显 `hello kernel`。
- 每加一个内核特性，都要有对应 QEMU 烟测（验证难度极高，原文档已点出）。

**难度 ⭐⭐⭐**。

### K10 中断栈帧 + ISR 桥

- CPU 压栈布局结构体（x86_64 `InterruptFrame`：`rip/cs/rflags/rsp/ss` + errcode）。
- `#[naked]` ISR 入口：压寄存器、`call` C 风格 handler、拆帧、`iret`。
- IDT 表项结构 + `lidt` 加载（K1 asm）。
- 缺页 / 时钟中断 / syscall 三类起步。

**难度 ⭐⭐⭐⭐**。

> **任务切换（context switch）是 K10 的衍生用法，不是新的语言特性**：`switch_to(old, new)` 只需 `#[naked]`（K3）+ `asm!`（K1）手动保存/恢复 callee-saved 寄存器（x86_64: rbx/rbp/r12–r15）并切换栈。只要 K3/K1 到位，它就是纯 `.x` 工程量，`kthread` 结构体与栈帧设计须与 shux 调用约定完全匹配即可。

### K11 地址空间布局与重定位策略

**问题**：内核常驻高半区（如 `0xFFFF800000000000`），而 boot 头须在低物理地址被 bootloader 识别。若后端只产基址为 0 的代码、且数据访问用绝对地址，则分页开启前需要手写重定位加载器。

**现状**：控制流已用 rel32（PC 相对，`R_X86_64_PC32`）做 `jmp/call`，跨段跳转天然可重定位。缺口是**全局/数据取址**——目前只有 `movabs imm64` 取字面立即数，没有 `lea rip+sym` 形式的 PC 相对数据取址，也没有把 `.boot` 段钉到低物理地址的能力。

**目标**：
1. **RIP 相对数据取址**：取全局/常量地址统一走 `lea r,[rip+sym]`（`R_X86_64_PC32` / `R_X86_64_GOTPCREL` 类），代码无论加载到哪个虚拟地址都能正确引用数据，省掉早期重定位加载器。
2. **段地址锚定**：配合 K4 段控制 + 链接器脚本，把 `.boot`（multiboot/stivale 头）钉在低物理地址、`.text/.data` 钉在高半区虚拟地址。
3. **绝对地址逃生口**：对确需绝对地址的场合（页表项、物理地址直填），提供 `#[link_section] + 绝对重定位（R_X86_64_64）` 并在早期启动手动 fixup。

**实现要点**：x86_64_enc 增加 `enc_lea_rip_sym`；ELF writer 对数据符号记 PC 相对/绝对重定位；driver 接受 `-T linker.ld` 与基址设定。

**难度 ⭐⭐⭐⭐**：与 K4 强耦合，建议同 M2 一起做。

## 4. 实施顺序与里程碑

| 阶段 | 内容 | 出口标志 | 预计 |
|------|------|---------|------|
| **M1 硬件直通** | K1 内联汇编 + K2 语言级 volatile | `asm!` 写串口、`*volatile` 改 VGA 字符，QEMU 显字 | ⭐⭐⭐⭐ |
| **M2 段与入口** | K4 段控制 + K5 入口 + K6 boot 头 + K8 全局 + K11 地址空间 | multiboot2 头被识别、`_start` 跳进 `kmain`、`hello kernel` 输出 | ⭐⭐⭐⭐ |
| **M3 中断与原子** | K3 naked + K10 ISR(+ 任务切换) + K7 原子 | 接时钟中断 + 缺页 handler、自旋锁 + `switch_to` 可用 | ⭐⭐⭐⭐ |
| **M4 工程化** | K9 构建链 + QEMU gate + 内核 panic 串口版 | `make kernel` 一键出镜像并过 gate | ⭐⭐⭐ |
| **M5+ 自举后** | A1–A10 增量 | 高级 ISR、UEFI、多架构、std.kernel.* | 长期 |

> 建议严格按 M1→M2→M3→M4 推进：原文档"先内联汇编、再 volatile+段、最后 section"的速成建议在升级为语言级特性后仍然成立，只是把 K4 段控制提前到 M2，因为 boot 头强依赖它。

## 5. 难度与风险

- **架构实现难度 ⭐⭐⭐⭐**：要改寄存器分配器（volatile 不被消除）、ELF writer（多段）、prologue/epilogue（naked）。这是对编译器原理的深度实践。
- **验证难度 ⭐⭐⭐⭐⭐（极高）**：每个内核特性都需要 QEMU 小环境验证机器码在物理内存中的表现；这是最容易积债的环节，必须 gate 化。
- **技术债风险**：volatile 与段控制若不在设计早期内置，后期改后端成本指数上升。这正是现在做 K1–K4 的理由——趁自举刚闭合、后端结构尚清晰。
- **安全边界**：内核特性全部落在 `unsafe { }` 与 naked/asm 内，安全子集不外泄硬件危险面，与 shux"默认安全"宗旨一致。

- 验证难度极高 → 落到 K9 的 QEMU gate。
- 自举后可缓 → §2.2 的 A1–A10。

## 7. 战略结论与下一步

**完整度结论（2026-07-02 评估）**：原 K1–K11 覆盖「点亮最小内核」约七成、路线清晰可行；但「用纯 shux 写一个完整 OS」还差 §2.3 的 L1–L13 一批**必须落到语言/编译器层**的能力（分配器挂钩、freestanding 契约+纯净度 gate、panic 不穿越原生帧 unwind、栈用量上限、per-CPU 段相对寻址、freestanding 动态分派、段属性/符号控制、内核 code model+no-red-zone、外部汇编符号视图、const-eval 评估）。这些原表多误判为「`std.kernel.*` 纯库」，实则无语言/后端支持就写不出来，已从 A5 拆回语言级。

完成 K1–K11 这十一项语言级与工程链能力，shux 即获得"纯 `.x` 写内核"的全部物理骨骼；之后用 shux 自研内核、长期替换 C 在系统软件中的位置，就是纯 `.x` 工程量问题，而非语言能力问题。

**最重要的一步：先把 K1 `asm!` 做出来。** 一旦内联汇编落地，K2–K11 中的绝大部分（volatile 读写、naked 函数体、ISR/任务切换、原子 `lock cmpxchg`、甚至重定位探测）都可以先在 `.x` 里用 `asm!` 顶上、绕过后端原生支持的缺失。这是通往内核开发的最短捷径——后端一时写不出 `mov %cr3` 的原生指令编码，就用 `asm!` 先顶。

**与编译器自举闭环的关系**：当前编译器自举（D+E+F v1）已闭合，`diag.c` 等收尾属阶段 G。建议**双线并行**：一条线推 G 收尾达成硬闭环（这样 K1–K11 可由 shux 自编译自身来维护，避免长期依赖宿主 C）；另一条线**立即开 K1**，不等 G 完全绿。两者会合时，shux 既能自编译、又能写内核，路线即闭环。

## 8. 测试可行性分析（2026-07-02 补充）

> 回答「这些内核级特性写好方便测吗?是否必须上外部硬件?能否全部本地测?」
> **结论先行**:K1–K11 + L1–L13 **几乎全部可本地测**。其中约一半走 T1 纯 host 侧(最快、无需模拟器),其余走 T2 QEMU(本地软件模拟器,`brew/apt install qemu` 即可,**不是外部硬件**)。只有 T3 一小条尾(真实设备时序/厂商固件/真实缓存一致性/CPU errata/性能精度)才必须上真机,且属**最终验证、不阻塞开发**。

### 8.1 三层测试模型

| 层 | 跑在哪 | 前置 | 覆盖什么 | 不能覆盖什么 |
|----|--------|------|----------|-------------|
| **T1 纯 host 侧** | 宿主进程 / 产物静态检查 | 无 | 语言语义、类型/约束、段表/符号/汇编编码、编译期常量、栈深、分配器钩子、vtable 构造 | 真正执行特权指令、真实设备 MMIO 副作用 |
| **T2 QEMU-guest** | 本地 QEMU 全系统模拟 | 装 QEMU | 特权指令(cr3/msr/lgdt/lidt/in/out/hlt/cli/sti)、VGA(0xB8000)、串口(0x3F8)、PIC/IOAPIC、PIT 定时器、SMP(`-smp N`)、分页/高半区加载、中断注入 | 真实设备 DMA 时序比特精度、真实厂商固件、真实多 socket 缓存/NUMA 一致性、CPU errata、周期精确性能 |
| **T3 真机** | 物理硬件 | 真机/烧录 | 上述 T2 不能覆盖的尾 | — |

> **关键澄清**:QEMU 是跑在开发机上的软件 CPU/设备模拟器,全系统模式(TCG 或 KVM/HVF 加速)能执行特权指令与模拟标准设备,本地即可,与「外部硬件」无关。macOS 上用 HVF 加速、Linux 上用 KVM,没加速也能 TCG 跑(慢但功能对)。所以「能在 QEMU 跑」= 本地可测。

### 8.3 QEMU 能/不能覆盖的边界(写清楚免得日后误判)

**QEMU 能(本地覆盖)**:
- x86_64 特权指令语义:cr0/cr2/cr3/cr4、`rdmsr/wrmsr`、`lgdt/lidt/sgdt/sidt`、`in/out`、`hlt`、`cli/sti`、`lock cmpxchg/xadd`、`iret/sysret/syscall`。
- 标准设备:VGA 文本帧缓冲(0xB8000)、串口 16550(0x3F8)、8259 PIC / IOAPIC、PIT/HPET 定时器、PS/2 键盘、PCI 配置空间、CMOS。
- SMP:`-smp N` 多 vCPU、IPI(`apic`/`x2apic`)、每 CPU 局部 APIC。
- 内存:分页/页表、高半区加载(`-kernel` + multiboot)、`.bss` 清零、MMIO。
- 中断注入:外部 IRQ、软中断(`int N`)、缺页(#PF)、通用保护(#GP)。

**QEMU 不能(才需 T3 真机)**:
- 真实设备 **DMA 时序 / 比特级寄存器语义**(QEMU 设备是行为模拟,不是寄存器级精确)。
- 真实厂商 **BIOS/UEFI 固件怪癖**(QEMU 用 SeaBIOS/edk2,非真实 OEM)。
- 真实多 socket **缓存一致性 / NUMA 行为**(TCG/KVM 不精确模拟缓存层级)。
- 真实 CPU **errata / MSR 私有位语义**。
- **周期精确性能 / 时序敏感驱动**(QEMU 时序不可信)。

### 8.5 结论

- **能本地测**:K1–K11 + L1–L13 **全部**。T1(host 侧)覆盖语言/编译器语义与产物静态属性;T2(QEMU,本地软件)覆盖特权指令与设备行为。

## 9. 可继续完善方向（2026-07-03 评估）

> K1–K11 / L1–L13 / A1–A10 已全部完成（120 项 ✅），shux 已能纯 `.x` 写完整内核。剩余项分两类：**asm 后端原生移植**（当前多为 C 前端实现，需脱离 C 中间层）与**完整功能扩展**（已有基础功能的增强）。

### 9.1 asm 后端原生移植（脱离 C 前端依赖）

> **根因**：当前 K1–K11 多数特性通过 C 前端实现（`shux-c -E` → C → `zig cc`），依赖 C 编译器折叠 / `__attribute__` / `__asm__`。真正自举闭环要求这些特性由 shux asm 后端原生 emit，不经 C 中间层。

| # | 项 | 现状 | 目标 | 阻塞 |
|---|---|------|------|------|
| N1 | K1d asm 后端原生 `asm!` emit | C 前端 `__asm__` | 模板塞进 `.text`，不经 C | asm 后端扩展 |
| N2 | K1e / K4d `.x` 前端移植 | C 前端验证 | `.x` 前端原生（须 bstrict 重编） | macOS 内存限制 |
| N3 | K2c asm 后端 volatile | C 前端 `volatile T*` | asm 后端标记 vol、跳过 DCE | asm 后端扩展 |
| N4 | K4c asm 后端原生多段 ELF | C 前端 `__attribute__((section))` | 直接出机器码 / ELF 多段 | ELF writer 扩展 |
| N5 | K5c / K6b QEMU `-kernel` 跳进 `_start`（asm 后端版） | C 前端 + zig cc 交叉编译 | asm 后端出 ELF → QEMU 直接识别 | N1/N4 完成 |

### 9.3 时机决策：全部自举后做

> **核心判据**：自举要求 asm 后端能编译 shux 编译器自身的 `.x` 源码。编译器自身用到的特性是基本类型 / struct / Arena / `Linear(T)` / X ABI / `extern "C"`——**不是**内核级特性（`asm!` / `volatile` / `naked` / `link_section`）。因此 §9.1/§9.2 的 18 项**全部不阻塞自举**。

**决策：§9 全部 18 项自举后再做。**

理由：

1. **当前 C 前端已让所有内核特性可用**（K1–K11 全 ✅ / 83 gate 全 PASS），内核开发不受阻；
2. **自举是 2026 年 7 月核心目标**，不应被功能扩展分散精力；
3. **asm 后端原生移植（N1–N6）自举后做更稳定**——自举完成 = asm 后端已稳定，再扩展内核特性不会引入自举期不稳定；
4. **功能扩展（E1–E12）自举后做更合理**——避免在自举期引入不稳定因素，与 X ABI 决策逻辑一致（自举前最小必要，自举后功能扩展）。

**自举完成后的推荐顺序**：

| 优先级 | 项 | 理由 |
|--------|-----|------|
| P1 | N1–N6（asm 后端原生移植） | 脱 C 的硬前置，自举后 asm 后端稳定即可推进 |
| P2 | E1（SMP 多核） | 真实内核调度刚需，当前仅 BSP 运行 |
| P3 | E3（native CTFE） | 内核重度依赖编译期常量布局 |
| P4 | E5（`#[global_allocator]`） | 内核 `Vec`/`String` 等带分配类型的硬前置 |
| P5 | E6（自研 linker） | 长期项，当前借 `lld` 完全可用 |

> **一句话**：语言能力已齐全，自举前不动；自举后按"脱 C → SMP → CTFE → allocator → linker"顺序推进。

## 10. 内核系统级优化方向（自举后）

> **前提**：K1–K11 / L1–L13 / A1–A10 已完成，内核已能跑通 `Hello → Interrupt → Paging → Scheduler → User Mode` 完整闭环。增强优化本质从"功能实现"转向"**系统稳健性、执行效率与架构完备性**"。
> **层次区分**：§9 是**语言层**（改 shux 编译器），§10 是**内核层**（用 `.x` 写内核代码，不改编译器）。
> **时机**：全部自举后做。大规模内核代码推荐 asm 后端稳定后编写，避免 C 前端依赖成为维护负担。

### 10.1 可观察性（P0，最先做）

> 用户建议优先做 KDebug——"黑盒运行是最危险的状态"。内核已能跑但不可观测，崩溃时无法定位根因，后续优化全是盲调。

| # | 项 | 现状 | 目标 | 难度 |
|---|---|------|------|------|
| K1 | Panic 自动堆栈回溯（Stack Unwinding） | 无 | Panic 时自动打印调用栈（帧指针回溯 / DWARF `.x` 行号） | ⭐⭐⭐ |
| K2 | 轻量级 Trace 系统 | 无 | 调度器 / 分配器关键位置埋点，串口输出实时统计（分配页数 / 调度次数 / 中断频率） | ⭐⭐ |
| K3 | 性能计数器（PMU） | 无 | 读取 CPU 周期计数器，为 §10.6 PGO 提供数据基础 | ⭐⭐⭐ |

**理由**：没有可观察性，后续内存管理 / 调度器 / I/O 优化全是盲调。先让内核"可看见"，再优化。

### 10.2 内存管理（P1）

> 当前 L1a 是 bump 分配器，适合短生命周期对象。内核级长生命周期对象（PCB / 页表）需要防腐蚀机制。

| # | 项 | 现状 | 目标 | 难度 |
|---|---|------|------|------|
| M1 | Buddy System 物理内存管理器 | L1a bump | 按 2^n 阶分配物理页，减少外部碎片 | ⭐⭐⭐ |
| M2 | Slab 分配器 | 无 | 内核对象池（PCB / inode / file descriptor），提升分配性能 | ⭐⭐⭐ |
| M3 | Guard Pages | 无 | 栈空间 / 关键数据结构间插入无效页，内核栈溢出触发 fault 而非静默 corruption | ⭐⭐ |
| M4 | `#[global_allocator]` 路由 | L1a 顶替 | 语言分配点路由到 Buddy/Slab（依赖 §9 E5） | ⭐⭐⭐⭐ |

**关联**：M4 依赖 §9 E5（`#[global_allocator]` 语言级挂钩），是语言层 + 内核层协同点。

### 10.3 调度器（P1）

> 当前 Scheduler 已跑通，但可能是协作式。真正的高性能响应需要抢占 + 多核。

| # | 项 | 现状 | 目标 | 难度 |
|---|---|------|------|------|
| S1 | 优先级抢占（Preemption） | 协作式？ | 基于时钟中断的抢占式调度，实时性 / 高并发 | ⭐⭐⭐ |

| S3 | Local APIC 管理 | 无 | per-CPU 本地中断控制器，为 SMP + 时钟抢占提供硬件基础 | ⭐⭐⭐ |
| S4 | `#[save_simd]` 自动保存 | A10a 文档化 | 抢占时自动 `fxsave/xrstor`（依赖 §9 E8） | ⭐⭐⭐ |

**关联**：S2 与 §9 E1 是同一件事的两个视角——E1 是语言层（asm 后端原生原子），S2 是内核层（AP 唤醒 + 锁竞争）。

### 10.4 I/O 子系统（P2）

### 10.6 编译器辅助优化（P3，与 x-neuron 协同）

> 这是 SHUX 创作者最大的优势——内核每一行代码都能为编译器提供信息。

| # | 项 | 现状 | 目标 | 难度 |
|---|---|------|------|------|
| P1 | Profile-Guided Optimization（PGO） | 无 | 内核运行期收集热点函数，回传 shux 编译器做针对性内联 / 分支预测优化 | ⭐⭐⭐⭐⭐ |
| P2 | 内核级热补丁（Hot Patching） | 无 | 利用 x-neuron 的 hotswap 机制，实现关键驱动无重启升级 | ⭐⭐⭐⭐⭐ |

**关联**：P2 与 x-neuron 项目的 `app/hotswap.x` 直接协同——x-neuron 的 Pass-level adaptation 机制（学习产出 → 增量自编译 → 影子预验证 → 热替换 → 回滚保护）正好可以做内核热补丁。这是 SHUX + x-neuron + 内核三位一体的协同点。

### 10.7 时机与优先级

**全部自举后做**，按以下顺序推进：

| 阶段 | 项 | 理由 |
|------|-----|------|
| **P0** | §10.1 K1–K3（可观察性） | 黑盒运行最危险，后续优化全靠可观察性指引 |
| **P1** | §10.2 M1–M4（内存管理） + §10.3 S1（抢占） | 稳健性 + 实时性基础，为后续 I/O / SMP 铺路 |
| **P2** | §10.3 S2–S4（SMP） + §10.4 I1–I3（I/O） + §10.5 C1–C3（安全） | 性能与安全并重 |
| **P3** | §10.6 P1–P2（PGO + 热补丁） | 与 x-neuron 协同的长期项 |

> **一句话**：内核已能跑，下一步从"可观察"起步，按"可观察 → 内存/调度 → SMP/I/O/安全 → 编译器协同"顺序推进。

### 11.2 ABI 边界 Panic 防护 —— 文档自举前，实现自举后

> **时机**：契约定义**自举前做**（不阻塞自举，但自举后混编时必须有）；实现**自举后做**。

**现状**：
- X ABI 设计文档（`/home/shu/shux/analysis/X-ABI-设计分析.md`）已规定"X ABI 禁止异常传播，错误用 `Result` 返回值"
- C ABI 必须 `unsafe`
- 但**缺少 Panic 在 ABI 边界的行为定义**——当 `.x` 和 C 混编 / `.x` 和 `.bs`（BaseScript）混编时，异常传播逻辑没定义

**缺口**：

| # | 场景 | 现状 | 目标 | 时机 |
|---|------|------|------|------|
| A1 | `.x` 函数 panic 跨 X ABI 边界 | 未定义 | X ABI 函数 panic 必须 `catch_unwind` 降级为 `Result`，禁止 unwind 跨边界 | 文档自举前 / 实现自举后 |
| A2 | `.x` 调用 C 函数触发 `abort` | 未定义 | C `abort` 映射为内核 `panic`，走 A1 降级路径 | 自举后 |
| A3 | BaseScript 驱动触发 panic | 未定义 | `.bs` panic 隔离在驱动沙箱内，返回 `Result` 给调度器，**不崩内核** | 自举后 |
| A4 | 信号 / `SIGSEGV` 跨 ABI 边界 | 未定义 | 内核态 fault → `ErrorTrace`；用户态 fault → 信号注入 | 自举后 |

**契约定义（自举前定稿，自举后实现）**：

> 以下规则是 X ABI / C ABI / BaseScript 跨边界 Panic 行为的**权威定义**。自举后编译器实现必须严格遵守。

**规则 P1 — X ABI Panic 降级（对应 A1）**：
- X ABI 函数（`extern function` / `extern "X"`）入口由编译器自动插入 `catch_unwind` 胶水。
- 函数体内任意 panic 不允许 unwind 跨越 X ABI 边界；必须降级为 `Result::Err(PanicPayload)` 返回。
- `PanicPayload` 布局：`{ kind: u32, msg_ptr: *u8, msg_len: usize, backtrace_off: u32 }`（16 字节，与 `Result` 的 Err 分支兼容）。
- 调用方（X ABI 调用方）必须检查返回值是否为 Err；`?` 传播在 X ABI 边界自动插入判别 + 传播。
- **Invariant**：unwind 栈展开绝不穿越 X ABI 函数帧；`catch_unwind` 胶水是唯一降级点。

**规则 P2 — C ABI 禁止 Panic（对应 A1 强化）**：
- `extern "C"` 函数**禁止 panic**——编译器静态拒绝函数体内的 `?` 传播、`panic!`、除零等可能 panic 的表达式。
- 若 C ABI 函数体内必须有可能 panic 的操作，必须用 `catch_unwind` 显式包裹并降级为 C 侧错误码（如 `i32` 返回 `-1`）。
- C `abort()` 调用映射为内核 `panic`（不可恢复），不走 P1 降级路径——`abort` 是立即终止，非 unwind。
- **Invariant**：C ABI 函数帧内零 panic 概率；`extern "C"` 是 `unsafe` 的硬约束之一。

**规则 P3 — BaseScript 驱动隔离（对应 A3）**：
- `.bs`（BaseScript）驱动运行在独立 Arena（`DriverArena`），与内核 Arena 物理隔离。
- `.bs` panic 时：①`catch_unwind` 捕获 → ②`DriverArena` 整块回收（bump pointer 归零，零开销） → ③返回 `Result::Err` 给调度器 → ④调度器记录 `ErrorTrace` 并继续调度其他驱动。
- **Invariant**：`.bs` panic 不波及内核 Arena；调度器永不因驱动 panic 而崩溃。
- **Asm/Perf**：Arena 回收是 `bump_ptr = arena_base` 单指令，无 free 链遍历。

**规则 P4 — 信号 / fault 跨边界（对应 A4）**：
- 内核态 fault（如 `#PF`、`#GP`）：转入 ISR → 构造 `ErrorTrace` → 走 P1/P3 降级路径（若 fault 点在 X ABI 函数内）或直接 `panic`（若在 naked/ISR 帧）。
- 用户态 fault：调度器注入信号（`SIGSEGV` / `SIGBUS`），用户态信号处理器在独立栈运行；若用户态无处理器则进程终止。
- **Invariant**：naked 函数 / `asm!` 块内禁止 unwind（CPU 压栈状态不可恢复）；fault 在这些帧内只能 `panic`。

**实现时机**：
- 自举前：本契约定义定稿（已完成）。
- 自举后：编译器实现 P1（`catch_unwind` 胶水生成）、P2（C ABI 静态 panic 检查）、P3（`DriverArena` 隔离）、P4（ISR fault 降级）。
