# 条件编译 — 实现说明与使用边界

> 本文档说明 Shux 编译器内**条件编译**的实现方式、符号来源与使用边界。条件编译为**编译器与标准库内部使用**，不作为面向最终开发者的正式语言特性对外文档化或承诺稳定。

---

## 一、定位与使用边界

### 1.1 用途

- **标准库按平台选实现**：在 core / std 等库源码中，根据目标 OS 或架构选择不同代码路径（如 Linux 与 Darwin 的 syscall 封装），而不把 `#ifdef` 暴露给用户代码。
- **编译器/构建内部**：若将来用 .sx 重写编译器或构建脚本，可在内部使用条件编译做平台分支。
- **不面向最终开发者**：不写入「Shux 语言用户手册」、不承诺语法与符号集的长期稳定；普通用户写业务 .sx 时**不依赖**本能力，也不建议在应用层随意使用 `#if SYMBOL`。

### 1.2 不对外「开放」的含义

- 不在语言规范或用户文档中把 `#if` / `#else` / `#endif` 列为正式语法。
- 不保证将来不增加或调整指令形式（如增加 `#ifdef`、改变符号命名）。
- 命令行 `-D` 主要供测试或内部构建使用；标准库所需符号由编译器自动注入（见下文），用户一般无需传 `-D`。

---

## 二、实现方式概览

条件编译在**词法分析（lexer）之前**对 .sx 源码做**一次预处理**：

1. **输入**：整份源码字符串 + 当前「已定义符号」列表 `defines`。
2. **按行扫描**：识别以 `#` 开头的指令行（`#if SYMBOL`、`#elseif SYMBOL`、`#else`、`#endif`），根据 `defines` 决定每一块是保留还是跳过。
3. **输出**：保留下来的内容拼成新字符串，被跳过的行用**空行占位**以保持行号与源文件一致（便于后续报错定位）。
4. **下游**：预处理后的字符串交给 lexer → parser → typeck → codegen，后续阶段**看不到**被丢弃的代码，也不会对 `#` 指令做语法解析。

实现文件：`compiler/src/preprocess.c`，接口声明在 `compiler/src/preprocess.h`。入口模块与每个 import 的 .sx 在解析前都会对读入的源码调用一次 `preprocess(source, defines, ndefines)`。

## 三、指令语法与语义

### 3.1 支持的指令

| 指令 | 格式 | 说明 |
|------|------|------|
| `#if SYMBOL` | 行首空白后 `#`，再空白后 `if`，再空白后**单个标识符** SYMBOL（字母/数字/下划线） | 若 SYMBOL 在 `defines` 中则保留本分支，否则跳过；开始一个新块，可配 `#else` / `#elseif` / `#endif`。 |
| `#elseif SYMBOL` | 同上，`elseif` + 空白 + 标识符 | 与 `#if` 同属一块；仅当前面 then/elseif 均未成立时判断本 SYMBOL，成立则保留本分支。 |
| `#else` | `#` + 空白 + `else` | 与 `#if` 同属一块；前面 then/elseif 均未成立时保留本分支。 |
| `#endif` | `#` + 空白 + `endif` | 结束当前 `#if` 块。 |

- **嵌套**：支持多层 `#if` … `#endif`，栈深度上限 32（`PREPROCESS_STACK_MAX`）。
- **未支持**：无 `#ifdef`、无 `#if defined(SYM)`、无数值或表达式，仅「符号是否在 defines 中」的布尔判断。

### 3.2 保留与跳过规则

- 若当前处于「保留」状态（栈顶表示 then 或 else 分支被选中），则本行**非指令**内容原样输出。
- 若处于「跳过」状态，则本行**非指令**内容不输出，只输出一个换行以占位。
- **指令行本身**一律不输出到结果中，只输出一个换行，以保持行号一致。

## 四、无参数时的自动检测（OS + CPU 架构）

**不传 `-D`、不传 `-target` 时**，编译器仍会做一次**运行机环境检测**，并自动注入两个符号，供「同一系统、不同 CPU 架构」的条件编译使用：

- 在调用 `preprocess()` 前，若 `defines` 未满，则调用 **`uname()`**（POSIX），取：
  - **`sysname`** → 注入 **`SHUX_OS_<SYSNAME>`**（操作系统名，**全大写**，如 `SHUX_OS_DARWIN`、`SHUX_OS_LINUX`）；
  - **`machine`** → 注入 **`SHUX_ARCH_<MACHINE>`**（当前 CPU 架构名，**全大写**，如 `SHUX_ARCH_X86_64`、`SHUX_ARCH_ARM64`）。

因此：

- **按 OS 分支**：可用 `#if SHUX_OS_LINUX`、`#if SHUX_OS_DARWIN` 等区分 Linux / macOS 等。
- **按 CPU 架构分支**：同一系统下可再细分为不同构架。例如：
  - **苹果**：Intel 机型一般为 `SHUX_ARCH_X86_64`，Apple Silicon 为 `SHUX_ARCH_ARM64`；
  - **Linux**：常见 `SHUX_ARCH_X86_64`、`SHUX_ARCH_AARCH64`、`SHUX_ARCH_ARMV7L` 等；
  - **其他**：`SHUX_ARCH_I386`、`SHUX_ARCH_I686` 等依运行机 `uname -m` 而定。

标准库中可同时判断「系统 + 架构」，例如：

```text
#if SHUX_OS_DARWIN
  #if SHUX_ARCH_ARM64
    // Apple Silicon 专用路径
  #elseif SHUX_ARCH_X86_64
    // Intel Mac 专用路径
  #else
    // 其他 Darwin 架构
  #endif
#else
  // 非 macOS
#endif
```

**注意**：`uname()` 在非 POSIX 环境（如原生 Windows）可能不可用，此时不会注入上述两个符号；跨平台编译时若需固定 OS/架构，见下节「符号来源」中的 `-target` 与 `-D`。

---

## 五、符号来源（defines）汇总

预处理使用的「已定义符号」由 driver（`main.c`）在调用 `preprocess()` 前**按顺序**组装，顺序为：

1. **命令行 `-D`**  
   - `-D SYMBOL` 或 `-DSYMBOL`：将 `SYMBOL` 加入 `defines`。  
   - 供测试、内部构建或**交叉编译时显式指定架构**（如 `-D SHUX_ARCH_ARM64`）使用。

2. **`-target` 推导的 OS 符号**（仅当传入 `-target` 且未超上限）  
   - `-target ...linux...` → `OS_LINUX`  
   - `-target ...darwin...` 或 `...apple...` → `OS_MACOS`  
   - `-target ...windows...` → `OS_WINDOWS`  
   - 当前 **`-target` 只推导 OS，不推导架构**；交叉编译时若需按架构分支，需用 `-D SHUX_ARCH_ARM64` 等传入。

3. **uname 自动注入**（只要 `defines` 还有空位就执行，与是否传 `-target` 无关；**符号中 OS/架构部分统一转为大写**）  
   - **`SHUX_OS_<SYSNAME>`**：来自 `uname()->sysname` 转大写，常见如 `SHUX_OS_LINUX`、`SHUX_OS_DARWIN`。  
   - **`SHUX_ARCH_<MACHINE>`**：来自 `uname()->machine` 转大写，常见取值见下表。

### 5.1 常见 `SHUX_ARCH_*` 取值（来自 `uname -m`，注入时整段转大写）

| 运行机 / 目标 | 典型 `machine` | 注入符号 |
|---------------|----------------|----------|
| x86_64 (Intel/AMD 64 位) | `x86_64` | `SHUX_ARCH_X86_64` |
| Apple Silicon (M1/M2/…) | `arm64` | `SHUX_ARCH_ARM64` |
| 其他 64 位 ARM (Linux 等) | `aarch64` | `SHUX_ARCH_AARCH64` |
| 32 位 ARM | `armv7l` 等 | `SHUX_ARCH_ARMV7L` 等 |
| 32 位 x86 | `i386` / `i686` | `SHUX_ARCH_I386` 等 |

同一份 `defines` 会用于**入口文件**与**所有 import 的 .sx** 的预处理，因此标准库内可按 `SHUX_OS_*`、`SHUX_ARCH_*`（以及可选的 `OS_*`、`-D`）做「系统 + 架构」的细粒度条件编译。

## 六、错误处理

- `#else` / `#elseif` 无对应 `#if`、`#endif` 无对应 `#if`、`#if` 未闭合、`#elseif` 出现在 `#else` 之后、重复 `#else`、嵌套过深等，会报错并写 stderr（如 `preprocess: #else without #if`），`preprocess()` 返回 NULL，编译中止。
- 负向用例见 `tests/preprocess/`（如 `else_without_if.sx`、`unclosed_if.sx` 等）。

---

## 七、与自举/性能文档的对应

- **自举实现分析.md §3.4** 中「条件编译与平台选择」指的就是：由编译器注入 `SHUX_OS_*` / `SHUX_ARCH_*`（及可选的 `-target` 推导），使标准库内部能按平台选实现，而不向用户暴露 `#ifdef` 风格约定。
- **自举前性能与语义基建.md §5.1** 中「条件编译（@if 或等价）在标准库内部可用」在本实现中体现为：标准库 .sx 可使用 `#if SYMBOL` + 上述自动注入与 `-target` 符号。

## 八、小结

| 项 | 说明 |
|----|------|
| **实现** | 词法前单遍扫描，按行识别 `#if` / `#elseif` / `#else` / `#endif`，依 `defines` 保留或跳过块；被跳过行以空行占位。 |
| **无参数时** | 自动用 `uname()` 注入 `SHUX_OS_<SYSNAME>`、`SHUX_ARCH_<MACHINE>`（**全大写**，如 `SHUX_OS_DARWIN`、`SHUX_ARCH_ARM64`），可同时按**系统**与 **CPU 架构**做条件编译。 |
| **符号** | 命令行 `-D`、`-target` 推导的 `OS_*`、uname 注入的 `SHUX_OS_*` / `SHUX_ARCH_*`；架构细粒度依赖 `SHUX_ARCH_*` 或 `-D`。 |
| **使用边界** | 供**编译器与标准库内部**按 OS/arch 选实现；**不**作为面向开发者的正式语言特性文档化或承诺长期稳定。 |
