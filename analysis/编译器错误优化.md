# Shux 编译器错误信息优化方案

> 目标：编译时出错，必须把**文件名、行号、列号、错误类别、涉及的源码行、错误指示线**详细打印到 stderr，让开发者一眼定位问题，无需肉眼排查。

```
  |             ^ expected ';' after return

error: aborting due to 2 previous errors
 --> tests/parser/if_missing_paren.x

# 当前：C 端 import 失败
import error: cannot open import 'std.not_found' (tried ...)
 --> /tmp/.../main_missing_import.x

# 当前：C 端 asm backend 失败
codegen error: asm_codegen_elf_o failed (elf_ec=-1, out_len=0, num_funcs=1)
 --> /tmp/.../main_ok.x
note: elf ctx code_len=32 num_labels=2 num_patches=1
 --> /tmp/.../main_ok.x
```

### 1.3 当前方案的不足

以下列表改为**当前剩余不足**，不再重复描述已经解决的问题：

1. **错误编号已基本覆盖**：错误级站点已全量编码（含新增 `CG004`）；note/warning/info/debug 按设计无编号，`aborting` 汇总行刻意 code=null。
2. **.x 路径尚未完全收平**：`shux-c` 已大面积统一，`shux_asm` / `strict_glue` 仍有部分老分支未迁移。
3. **Parser 同步恢复已落地四级边界**：顶层声明/import 段/函数体语句/聚合体成员（struct/enum/trait/impl）均能在坏成员报错后同步到下一边界继续；表达式子句中途错误经语句级恢复已能跳到下一条语句继续（已验证），表达式内部更深层任意点为 best-effort。
4. **JSON 诊断输出已实现（opt-in）**：`--diag-json` / `SHUX_DIAG_JSON=1` 输出 NDJSON；默认仍为人类可读 stderr。
5. **建议类诊断部分实现**：`--explain <未知码>` 已支持 `did you mean ...` 编码级建议；CLI JSON 诊断输出（`--diag-json`）已实现（opt-in）；仅标识符/选项级 did-you-mean 仍未做（后续增强）。

## 二、目标错误输出格式

### 2.1 标准格式（仿 Rust/Clang deno check 风格）

```
error: expected ')' after import path string
 --> tests/parser/import_bad.x:14:10
   |
14 |     import("std/io" extra)
   |                    ^^^^ expected ')'
   |
```

多错误时连续输出：

```
error: undefined name 'foo'
 --> src/main.x:23:5
   |
23 |     let x = foo + 1;
   |             ^^^ undefined name
   |

error: type mismatch: expected i32, found bool
 --> src/main.x:31:18
   |
31 |     let y: i32 = true;
   |                  ^^^^ expected i32, found bool
   |

error: aborting due to 2 previous errors
```

### 2.2 格式要素

| 要素 | 说明 | 来源 |
|------|------|------|
| `error:` / `warning:` | 严重级别前缀，error 红色 | 错误类型决定 |
| 错误简述 | 一句话说明问题 | 各阶段 fail/err 函数的 msg |
| `-->` 行 | `文件路径:行:列`，可被终端/编辑器点击跳转 | Token/ASTExpr 的 line/col + 全局文件名 |
| 行号竖线 `\|` | 对齐用 | 固定格式 |
| 源码行 | 出错那一行的原始内容 | 从源码缓冲区按行号提取 |
| 指示线 | `^^^` 指向出错列，后附说明 | 根据 col 计算 |

### 2.3 兼容性约束

- **必须保持 `error:` / `typeck error` / `parse error` 前缀**：现有测试脚本（`tests/run-typeck.sh` 等）用 `grep -q "typeck error"` 验证，新格式不能破坏这些前缀。
- **`file:line:col` 格式必须可被编辑器识别**：VS Code、Terminal.app、iTerm 均识别 `path:line:col` 格式，Cmd/Ctrl+点击可跳转。
- **ANSI 颜色可选关闭**：检测 `isatty(stderr)` 或环境变量 `SHUX_NO_COLOR`，非 TTY 或设了变量则纯文本输出（不影响管道/grep）。

## 三、实现方案

### 3.1 核心数据结构：全局错误上下文

该方案已落地，但实际接口已经比初稿更完整。当前真实接口除 `diag_set_file()` 外，还包含上下文快照/恢复与格式化输出：

```c
/* compiler/include/diag.h — 诊断上下文（全局单例） */

/**
 * 诊断上下文：记录当前编译文件名与源码缓冲区，供错误打印时提取源码行。
 * 全局单例，由 driver 在读入源文件后设置，各阶段错误输出函数引用。
 */
typedef struct DiagContext {
    const char *file_path;   /**< 当前编译的 .x 文件路径（NUL 结尾）；NULL 表示未知 */
    const char *source;      /**< 源码缓冲区起始（NUL 结尾）；NULL 表示无法提取源码行 */
    size_t source_len;       /**< 源码长度 */
    int use_color;           /**< 是否使用 ANSI 颜色（0=否，1=是） */
} DiagContext;

typedef struct DiagContextSnapshot {
    const char *file_path;
    const char *source;
    size_t source_len;
    int use_color;
} DiagContextSnapshot;

void diag_set_file(const char *path, const char *source, size_t source_len);
void diag_push_file(DiagContextSnapshot *snapshot, const char *path, const char *source, size_t source_len);
void diag_restore(const DiagContextSnapshot *snapshot);
const char *diag_get_file(void);
const char *diag_get_source(void);
size_t diag_get_source_len(void);
void diag_reportf(const char *file, int line, int col,
                  const char *kind, const char *detail,
                  const char *fmt, ...);
```

### 3.2 源码行提取函数

```c
/**
 * 从 DiagContext.source 中提取第 line_no 行的内容（不含换行符）。
 * 写入 out_buf（最多 cap-1 字节，NUL 结尾），返回行长度；失败返回 -1。
 * 实现：遍历源码，遇 \n 计数，到目标行后复制到换行前。
 * 复杂度 O(n) 但仅在报错时调用一次，对性能零影响。
 */
int diag_extract_line(int line_no, char *out_buf, size_t cap);
```

### 3.3 统一错误打印函数

```c
/**
 * 统一错误输出函数：打印完整的多行错误信息到 stderr。
 *
 * 参数：
 *   file   - 文件路径（若为 NULL 则用 diag_get_file()）
 *   line   - 行号（1-based，0 表示未知）
 *   col    - 列号（1-based，0 表示未知）
 *   kind   - 错误类别前缀："error" / "warning" / "parse error" / "typeck error"
 *   msg    - 错误简述
 *   detail - 指示线后的补充说明（可为 NULL）
 *
 * 输出格式（见 §2.1）。无源码/无位置时降级为单行输出，保持向后兼容。
 */
void diag_report(const char *file, int line, int col,
                  const char *kind, const char *msg, const char *detail);
```

### 3.4 各阶段改造点

#### 3.4.1 Parser（parser.c）

当前 `fail()` 函数（`parser.c:175-183`）：

```c
static int fail(Parser *p, const char *msg) {
    const Token *t = peek(p);
    if (lsp_diag_enabled) {
        lsp_diag_add(t->line, t->col, 1, msg);
        return -1;
    }
    fprintf(stderr, "parse error at %d:%d: %s\n", t->line, t->col, msg);
    return -1;
}
```

改造为：

```c
static int fail(Parser *p, const char *msg) {
    const Token *t = peek(p);
    if (lsp_diag_enabled) {
        lsp_diag_add(t->line, t->col, 1, msg);
        return -1;
    }
    /* 保持 "parse error" 前缀兼容测试脚本 grep */
    diag_report(diag_get_file(), t->line, t->col, "parse error", msg, NULL);
    return -1;
}
```

#### 3.4.2 Typeck（typeck.c）

当前 `lsp_diag_report_typeck()` 在非 LSP 模式下（`lsp_diag.c:312-314`）：

```c
fprintf(stderr, "typeck error: %s at %d:%d\n", buf, line, col);
```

改造为：

```c
diag_report(diag_get_file(), line, col, "typeck error", buf, NULL);
```

`TYPECK_ERR` 宏保持不变（它调 `lsp_diag_report_typeck`），只需改 `lsp_diag_report_typeck` 内部。

#### 3.4.3 Codegen（codegen.c）

当前 codegen 错误完全无位置信息（`codegen.c:3957,4073`）。改造：

1. 在 `emit_expr` 等函数中，利用 `ASTExpr->line/col` 传入错误位置。
2. 将 `fprintf(stderr, "codegen error: ...")` 改为 `diag_report(diag_get_file(), e->line, e->col, "codegen error", msg, NULL)`。
3. 若 `line==0`（位置未知），降级为无位置的单行输出。

#### 3.4.4 .x 流水线诊断（runtime_driver_diagnostic.c）

当前 `.x` pipeline typeck 失败只报 `type check failed`（`runtime_driver_diagnostic.c:78-86`）。改造：

1. `driver_diagnostic_typeck_fail()` 调用 `diag_report` 输出格式化错误。
2. `driver_diagnostic_typeck_func_fail()` 已有 `func_idx` 和 `name`，增加调用 `diag_report` 带 func 名称的详细输出。
3. `driver_diagnostic_typeck_ret_fail()` 已有 `line/col`（`runtime_driver_diagnostic.c:184`），改用 `diag_report`。

#### 3.4.5 Driver / Pipeline 入口

在 `main.x` 的 `driver_run_x_emit_x()` 和 C 侧 `driver_run_compiler_full()` 中，读入源文件后立即调用 `diag_set_file(path, source, source_len)`，确保后续各阶段都能提取文件名和源码行。

对于多依赖（import）编译，在解析每个依赖模块时更新 `DiagContext.file_path`，使依赖模块中的错误也能正确标注文件名。

### 3.5 ANSI 颜色输出

```c
/* 仅当 stderr 是 TTY 且未设 SHUX_NO_COLOR 时启用 */
#define ANSI_RED     "\x1b[31m"
#define ANSI_YELLOW  "\x1b[33m"
#define ANSI_BOLD    "\x1b[1m"
#define ANSI_RESET   "\x1b[0m"
#define ANSI_BLUE    "\x1b[34m"

int diag_should_color(void) {
    if (getenv("SHUX_NO_COLOR")) return 0;
    return isatty(fileno(stderr)) ? 1 : 0;
}
```

输出时：
- `error:` 用 `ANSI_BOLD ANSI_RED`
- `warning:` 用 `ANSI_BOLD ANSI_YELLOW`
- `-->` 行号竖线用 `ANSI_BLUE`
- 源码行正常色
- 指示线 `^^^` 用 `ANSI_RED`
- 行尾 `ANSI_RESET`

### 3.6 多错误收集（可选，建议第二阶段）

该项已**部分落地**，但仍未完成 parser recovery 的最终形态：

1. `check` / LSP 人类可读输出已支持打印多条诊断。
2. `lsp_diag_print_stderr_human()` 现已在多条 error 后追加：
   - `error: aborting due to N previous errors`
3. 真正的 parser 同步恢复（跳到 `;` / `}` 再继续）仍未实现，仍属于后续增强。

此功能不影响性能（仅报错路径触发），但提升开发体验显著。

## 四、性能影响评估

| 改动 | 热路径影响 | 说明 |
|------|-----------|------|
| `diag_set_file()` | 零 | 仅在编译开始调用一次 |
| `diag_extract_line()` | 零 | 仅在报错时调用，正常编译不触发 |
| `diag_report()` | 零 | 仅在报错时调用 |
| Token 增加 file_id | 不需要 | Token 已有 line/col，文件名通过全局 DiagContext 提供 |
| ASTExpr 改动 | 不需要 | ASTExpr 已有 line/col |
| `isatty()` 检测 | 零 | 仅在初始化时调用一次，结果缓存 |

**结论：所有改动仅在错误路径（cold path）执行，对正常编译性能零影响，符合项目「极致压榨性能」宗旨。**
