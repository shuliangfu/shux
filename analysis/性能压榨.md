## 0. 核心结论（读这一节就够）

**问：编译器能不能用向量（SIMD）提升编译速度？**  
**答：可以，但要分清两件事：**

| 维度 | 含义 | Shux 现状 | 优先级 |
|------|------|-----------|--------|
| **A. 生成代码性能** | 用户 `.x` 编译出的程序跑多快 | **已在推进**：`simd_enc` / `simd_loop`、向量类型、`@shuffle`/`@select`、loop peel | **现在**（M1 前后定点） |
| **B. 编译器自身吞吐** | `shux` 编译 `.x` 有多快 | Lexer/Parser 仍为标量扫描；无 simdjson 式批量跳过 | **M2 后**（自举链稳定后再动） |

**与 Mojo 的关系**：不宜用「35000× Python」类营销数字对标。Mojo 强在 **MLIR + GPU/Tensor 生态 + Python 互操作**；Shux 强在 **无 GC、直出 ASM、region/linear、轻量 freestanding、端侧可控体积**。底层「零代价抽象 + 硬件感知 codegen」可对齐思路，**产品形态与后端路线不同**，不存在「抄 Mojo 就赢」的捷径。

**全景地图**：除编译器 SIMD 外，性能压榨还覆盖 **生成代码、内存、标准库 IO/Net、体积、观测与门禁** — 见 **§7**；**自举前必做清单**见 **§7.7**。

## 1. 问题界定

### 1.1 两类「性能」不要混谈

原文档把 **Lexer 扫源码**、**符号表查找**、**typeck 位图** 与 **Mojo/MLIR** 写在同一叙事里，容易让人以为「自举一完成编译器就会自动 GB/s」。实际上：

- **编译器加速**：缩短 `parse_ms` / `typeck_ms` / `codegen_ms`（见 `SHUX_COMPILE_PHASE_TIMING`）。
- **程序加速**：缩短用户 `main()` 或内核 hot loop 的运行时间（见 `tests/bench/`、`perf-vs-zig-baseline`）。

Shux 仓库里 **SIMD 工程投入主要在 A→用户程序**（`compiler/src/asm/simd_*.c`、`pipeline_glue.c` 向量路径），**不在 B→编译器 Lexer**。

### 1.2 何时值得做「编译器 SIMD」

《[自举阶段进度与路线图](自举阶段进度与路线图.md)》§7.1 已写明：

> **C. 编译器自身编译速度** — **建议 M2 后**；自举链调试成本高。

理由简要：

1. 自举未完成时，改 Lexer/typeck 热路径会同时影响 seed、`.x` pipeline、LSP，回归面过大。
2. 尚无全仓库 **phase timing 基线**（OBS-001 已有开关，缺大规模 TSV 与瓶颈占比报告）。
3. 过早 SIMD 化 Lexer 会绑定平台 intrinsics，与「编译器用 `.x` 写、可移植自举」目标冲突——宜在 **纯 `.x` Lexer 第二实现** 或 **明确 C 加速层边界** 后再做。

## 2. 编译器吞吐：三个战场（务实版）

以下三项均可 SIMD 化，但 **收益、难度、前置条件** 差异很大。去掉「一枪 GB/s」式表述，改为可验收目标。

### 2.1 战场一：词法分析（Lexer）

**思路（成立）**：借鉴 [simdjson](https://github.com/simdjson/simdjson)、[re2c](https://re2c.org/) 等——用 16/32/64 字节向量加载 + compare/mask **批量** 判定空白、换行、`//` 注释起点，跳过无关字节，再回退到标量状态机处理标识符/字符串。

**现状**：

- `compiler/src/lexer/lexer.c`、`compiler/src/lexer/token.x` — **逐字符/逐规则** 扫描。
- Token 层已预留 `@`（SIMD comptime builtin），与 Lexer 加速无关。

**预期收益**：

- 大文件、注释/空白占比高时：**parse 阶段** 可显著下降。
- 小文件、typeck 重模块：Lexer 往往 **不是** 第一瓶颈（需用 OBS-001 实测）。

**落地建议（M2+）**：

| 步骤 | 内容 |
|------|------|
| L0 | `SHUX_COMPILE_PHASE_TIMING=1` 对 `compiler/src/**/*.x` 全量编译，统计 parse/typeck/codegen 占比 |
| L1 | 仅实现 **ASCII whitespace + `\n` + `//` 行注释** 的 SIMD 跳过（Neon + SSE2 双路径） |
| L2 | 保留标量 fallback；`tests/run-lexer.sh` 全绿 |
| L3 | 再考虑 block comment `/* */` 与 UTF-8（若语言规范需要） |

**不宜承诺**：「吞吐量一定 GB/s」——取决于输入分布、分支预测、是否内存带宽 bound；工业界 GB/s 级 JSON 解析是 **专用库 + 多年优化** 的结果，不是开关即用。

### 2.2 战场二：符号表 / 名字查找

**思路（部分成立）**：开放寻址哈希表中，用 SIMD 并行比较连续 slot 的 tag/key（Swiss Table / abseil 部分技巧）。对 **极热** 的 intern / 关键字 lookup 有效。

**现状**：

- 编译器符号解析分散在 parser/typeck/module 加载；无独立「SwissTable + SIMD」模块。
- 名字比较多为 **长度前缀 + memcmp** 或 arena 内短串。

**预期收益**：

- 单文件、符号量 <10⁴：提升 **有限**，常低于 parse AST 分配与 typeck。
- 全仓库 LSP / 多模块 incremental：若做 **全局 symbol index**，SIMD probe 更有价值（偏 IDE，偏编译）。

**落地建议**：

- **P2**：先做 **字符串 intern 池**（arena + 哈希），再评估 SIMD probe；无 profiling 数据前不上向量。
- 参考：`rustc` 的 `FxHash` + intern；不必为了 SIMD 而 SIMD。

### 2.3 战场三：Typeck / 借用域（Region）检查

**思路（需收窄）**：将「未初始化 / 已借用 / 可变」等状态压入 **位集（bitset）**，批量按位与。适用于 **固定上限** 的局部槽位（如每函数 ≤64 个 tracked slot）。

**现状**：

- `compiler/src/typeck.x` — region / linear / slice 域以 **语义规则 + 诊断** 为主（见 `analysis/type-region-v1-rfc.md`、`type-borrow-conflict-v1.md`）。
- 并非简单「64 变量一次 VAND」：域冲突含 **跨块逃逸、返回值域、slice stamp** 等，需约束求解或数据流，位图只是 **加速数据结构** 之一。

**落地建议**：

| 层级 | 做法 |
|------|------|
| 正确性 | 先完成 region/linear 诊断稳定、`bootstrap_shuxc -E` 全绿 |
| 结构 | 函数内 `LocalSlot` 用 `u64`/`u128` 位图表示 def/use/borrow 状态 |
| 向量 | 仅当 slot 数 > 32 且 hot path profiling 证明 typeck 占比 >40% 时，用 Neon/AVX 做 batch merge |
| 参考 | Rust Polonius / LLVM 内存 SSA — 位图是辅助，不是替代 AST 遍历 |

**结论**：位图 + SIMD **可做为 typeck 内部优化**，但不是「一枪清算全部生命周期」；与类脑算法里的「突触位图」是不同问题域，不宜混为一谈。

## 3. 生成代码性能：Shux 已在做什么

这与「编译器编译有多快」无关，但与「能否对标 Mojo 在 kernel 上的体感」直接相关。

### 3.1 已有能力（仓库可查证）

| 模块 | 作用 |
|------|------|
| `compiler/src/asm/simd_enc.c` | x86 SSE/AVX2、arm64 Neon 向量 add/sub/mul 等指令编码 |
| `compiler/src/asm/simd_loop.c` | `while i<n { a[i]=b[i] op c[i] }` 条带化 / peel |
| `compiler/include/target_cpu.h` | 宿主 CPU feature 探测，驱动向量宽度选择 |
| `compiler/src/codegen/autovec.c`（迁移中） | 自动向量化策略；自举进度待删 C 依赖 |
| `std/simd/` | 标准库向量 API |
| 语言 | `i32x4`/`f32x4`、`@shuffle`/`@select` comptime |

详见《[编译时自动内存管理和自动向量化](编译时自动内存管理和自动向量化.md)》§0–§1。

### 3.2 与 OBS/PERF 工具链配合

```bash
# 编译阶段耗时（找瓶颈）
SHUX_COMPILE_PHASE_TIMING=1 ./compiler/shux check path/to/file.x

# 生成代码 alloc 热点（Arena 路径应零 per-op malloc）
tests/lib/perf-alloc-hotspot.sh
```

先 **测量** 再 **SIMD**：避免在 non-hot 路径写 intrinsics。

## 4. 「达到 Mojo 级别」的务实对照

### 4.1 Mojo 的核心技术（摘要）

| 技术 | 作用 |
|------|------|
| **MLIR** | 多级 IR；tensor/GPU dialect；与 LLVM 生态衔接 |
| **Parametric types / comptime** | 编译期特化、shape/硬件参数 |
| **Ownership + value semantics** | 减少运行时 refcount/GC |
| **Python 互操作** | 生态与落地场景 |

### 4.2 Shux 对标矩阵（不臆测倍率）

| 能力 | Mojo | Shux 现状 | Shux 路线 |
|------|------|-----------|-----------|
| 默认后端 | LLVM/MLIR | **ASM 直出** + 可选 C/cc | 保持轻量；MLIR **非近期必选项** |
| GPU/Tensor | 一等公民 | 未定位为主战场 | 若需要：MLIR NVVM/ROCm 或专用 DSL 外挂，单独立项 |
| 编译期硬件感知 | comptime + target | `target_cpu`、`-target`、feature 位掩码 | 扩展 **comptime `@cpu()` 探针**（L1/cache line/vector width） |
| 内存安全 | 部分 ownership | **region + linear + arena** | typeck 稳定 → 零运行时检查 |
| 体积/依赖 | 较大工具链 | **freestanding、无 libc 子集** | 项目宗旨，不 swap 成 LLVM 全家桶 |
| 交互生态 | Python | 无 | 非目标 |

### 4.3 MLIR 是否要上？

**长期选项，非 Stage 2/3 阻塞项。**

| 方案 | 优点 | 缺点 |
|------|------|------|
| **维持 ASM 后端** | 体积可控、可预测、契合嵌入式 | 自动向量化/跨平台需自研 |
| **可选 MLIR 方言** | 复用 affine/vector/GPU 优化 | 依赖重、与「轻量级」张力大 |
| **混合** | 热路径 ASM，实验性 MLIR 分支 | 双后端维护成本 |

若立项，建议命名与范围克制：例如 **`shux.vector` dialect**（仿射 loop + tile）而非泛泛 `shux_brain`；类脑算法语义仍在 **语言 + 库**，不必全部塞进 IR 名字。

### 4.4 编译期硬件拓扑（Hardware Probe @ Comptime）

**方向正确，宜分阶段：**

1. **已有**：`shu_target_cpu_detect_host()` → AVX2/SSE/Neon 影响 codegen 向量宽度。
2. **下一步**：comptime 可读 `@cpu(lanes)` / `@cache(line)`，常量折叠进 **默认 tile 大小**（见 `simd_loop` peel 宽度）。
3. **避免**：编译时读 GPU 寄存器文件等——跨平台不可移植，应通过 **显式 target triple** 或 **用户 `@cfg(target)`** 约束。

树莓派 5 → 128-bit Neon、x86 AVX-512 等差异，应体现在 **codegen 与 std/simd**，而非编译器 Lexer。

### 4.5 零代价抽象：region / linear / with_arena

这与 Mojo 的 value ownership **目标一致、机制不同**：

- Shux 用 **编译期域标签 + typeck 拒逃逸**（`region`、`with_arena`），codegen 多为 **嵌套块 / 栈 promotion**，无运行时 borrow checker。
- **Linear types** 防止 alias 与 double-free；与「脉冲分发引擎独占借用」类场景对齐，但需 **`type-linear-v1-rfc` 落地证据**，不宜写「已锁死」。

**性能承诺**：安全子集内 **0 运行时 GC/RC**；不等于 **0 原子/0 锁**——若算法需要并发，仍须 `std.sync` 或显式 atomic，由类型系统区分「编译期可证无 alias」与「必须同步」。

## 5. 优先级路线图（与自举对齐）

```text
现在 ──► M1（asm 稳定）──► M2（自举闭环）──► M3（编译器吞吐 / 高级 IR）
  │              │                  │                    │
  生成代码 SIMD   loop fold/        SIMD Lexer 立项      可选 MLIR 分支
  autovec/peel   peephole 扩展      symbol intern       comptime @cpu 完善
  OBS/PERF 基线  region/typeck 全绿  phase timing 报告   incremental LSP
```

| 阶段 | 交付物 | 验收 |
|------|--------|------|
| **P0（当前）** | typeck/自举；生成代码 SIMD 补洞 | `bootstrap_shuxc -E`；bench 不回归 |
| **P1** | `SHUX_COMPILE_PHASE_TIMING` 全仓库报告；top parse/typeck 文件清单 | TSV 进 `tests/baseline/` |
| **P2** | Lexer ASCII SIMD skip（可选 C 加速层或 `.x` 双实现） | `run-lexer.sh`；parse_ms ↓≥15% 于大文件基准 |
| **P3** | typeck 函数内位图 + 热点 batch | typeck_ms ↓；region 诊断不回归 |
| **P4** | comptime hardware probe；MLIR 可行性 RFC | 独立 `analysis/` RFC，不阻塞主链 |

## 6. 常见误区（原文档修正）

| 误区 | 修正 |
|------|------|
| 自举完成 → 编译器自动 SIMD | 自举完成只保证 **用 `.x` 写编译器**；SIMD Lexer 仍需 **单独设计与测试** |
| Lexer GB/s → 整体编译 GB/s | parse 通常只占一部分；typeck/codegen/link 常占更大头 |
| typeck 一位图万能 | region/linear 需 **约束 + 逃逸分析**；位图仅加速局部状态 |
| 上 MLIR = 超越 Mojo | MLIR 是工具；优势在 **生态与 dialect**，Shux 优势在 **体积与 ASM 可控** |
| 与 Mojo「完全对齐」 | **生成代码与硬件感知**可对齐方向；**生态与后端**不对齐 |
| 状态栏/IDE 等自举 | IDE 与编译器吞吐无关（见 `editors/vscode/完善功能.md`） |

## 7. 全景：Shux 性能压榨体系（全部分析）

本节把仓库内分散的 perf 文档、工具链与代码路径收拢为 **六层模型**，便于回答：「现在该压榨哪一层、用什么测、验收标准是什么」。

### 7.1 六层性能模型

```text
                    ┌─────────────────────────────────────┐
  L6 观测/门禁      │ OBS-001 阶段计时 · PERF-004 dogfood │
                    │ PERF-005 flamegraph · weekly gate   │
                    └─────────────────┬───────────────────┘
                                      │ 数据驱动
  L5 编译器吞吐     │ parse/typeck/codegen · Lexer SIMD   │  ← §2，M2 后
                    └─────────────────┬───────────────────┘
  L4 编译器产出质量 │ loop fold · inline · peephole · DCE  │
                    │ autovec/simd_loop · -O2 cc 路径      │  ← §3，M1 前后
                    └─────────────────┬───────────────────┘
  L3 标准库/runtime │ std.net/io/thread · io_uring 批量    │
                    │ Buffer 切片化 · 零拷贝 · 绑核         │  ← 按需深化
                    └─────────────────┬───────────────────┘
  L2 内存与分配     │ region/arena/linear · 热路径零 malloc │
                    │ PERF-007 alloc hotspot · grow pool    │  ← 现在 + 持续
                    └─────────────────┬───────────────────┘
  L1 语言语义安全   │ typeck 证明 → BCE/noalias/autovec 前提 │  ← 自举主线
                    └─────────────────────────────────────┘
```

**原则**（《[安全与性能](安全与性能.md)》）：**L1 是 L4/L2 的前提** — 编译期安全证不出时，要么拒绝优化，要么靠 `unsafe` 自担风险；**L6 贯穿各层** — 无测量不优化。

### 7.2 各层详解

#### L1 — 语言语义与安全子集（性能的地基）

| 机制 | 性能作用 | 文档 / 代码 |
|------|----------|-------------|
| **region** | slice 域 compile-time 拒逃逸 → 无运行时 borrow checker | `type-region-v1-rfc.md` |
| **linear** | move 语义 → 无 double-free / alias 误优化 | `type-linear-v1-rfc.md` |
| **with_arena** | 热路径 bump、零 per-op free | `编译时自动内存管理和自动向量化.md` |
| **noalias / BCE** | 边界检查消除、向量化合法前提 | `type-zero-cost-v1.md`、`zc-semantics-v1.md` |
| **unsafe 边界** | 证不出时显式降级；误用 = C 级性能与安全双输 | `安全与性能.md` §四 |

**现状**：typeck/region/linear 仍在自举攻坚（`typeck.x`、`bootstrap_shuxc -E`）。**未完成 L1 不宜在 L4 上赌 UB 换速度**。

**验收**：`run-typeck-region.sh`、`run-typeck-linear.sh` 全绿；safe 子集内无静默 UB。

#### L2 — 内存与分配（零隐藏成本）

| 方向 | 目标 | 工具 / 门禁 |
|------|------|-------------|
| Arena / grow pool | 解析、typeck、LSP 热路径少 malloc | `perf-alloc-hotspot-v1.md` |
| 字符串 arena | concat/format 零 heap | ZC-4、`string_arena_concat` bench |
| `with_arena` 用户代码 | vec/map 等同块内零追踪 | `with_arena_vec_push` bench |
| 编译器自身 | ast pool、pipeline arena 池化 | `X侧车grow池与动态上限清单.md` |

```bash
tests/lib/perf-alloc-hotspot.sh
SHUX_ALLOC_HOTSPOT_FAIL=1 tests/lib/perf-alloc-hotspot.sh   # 超 cap 硬失败
```

**压榨要点**：先 strace 计数，再改数据结构；**禁止**在已证 Arena 路径偷偷 `malloc`。

#### L3 — 标准库与运行时（IO / Net / Thread）

| 模块 | 已做 | 可选深化 | 文档 |
|------|------|----------|------|
| **std.net** | io_uring accept/connect 批量、Buffer[]、fixed 注册、UDP mmsg | multishot accept；阶段 6 DPDK 仅预留 | `std.net性能压榨与超越Zig.md` |
| **std.io** | readv/writev 切片 API、timeout、driver 抽象 | 舒 IO 自研 ring（自举后） | `自举前-极致IO与网络性能准备.md` |
| **std.thread** | 零成本抽象、绑核、栈大小、macOS QoS | 文档化最佳实践 | `std.thread性能压榨.md` |
| **切片化** | `*_buf(ptr, n)` 去魔数 | 余下写死参数见清单 | `写死参数与切片化优化清单.md` |

**对标**：`perf-io-zig-v1.md`、`perf-net-zig-v1.md`、`SHU与ZIG性能对比测试.md`。

**策略**（《[接下来做什么-性能压榨与新std](接下来做什么-性能压榨与新std.md)》）：**必做项已基本完成**；multishot/DPDK/内核 bypass 为 **benchmark 触发后再做**，避免空转。

#### L4 — 生成代码与后端（用户程序跑多快）

| 技术 | 位置 | 状态 |
|------|------|------|
| ASM 直出 | `compiler/src/asm/` | 默认 `-backend asm` |
| loop 计数折叠 | `try_fold_count_up_while` | `loop_i32` 等 bench 受益 |
| 参数 struct 内联 | `try_inline_param0_field_sum_call_elf` | `struct_param` bench |
| SIMD 指令编码 | `simd_enc.c` | i32x4/f32x4 add/sub/mul |
| loop peel / 条带化 | `simd_loop.c` | 固定数组 index-add while |
| 自动向量化 | `autovec.c`（迁移中） | 自举后纯 `.x` |
| 向量 comptime | `@shuffle` / `@select` | parser + codegen |
| C/cc 回退 | `-backend c` + `-O2` | 对比与 bootstrap 用 |

**P0 基准**（PERF-001）：

| 用例 | 测什么 |
|------|--------|
| `loop_i32` | 10⁹ 循环、fold/peephole |
| `mem_copy` | 内存带宽 + alias |
| `struct_param` | 按值传递、字段访问 |
| `call_boundary` | 5 层调用链、内联深度 |

**门禁目标**：Shux asm median **≤ C -O2**，再追 **≤ Zig -O2**（`perf-vs-zig-baseline.md`）。

**阶段 8 长期**（《[阶段8-优化与体积分析](阶段8-优化与体积分析.md)》）：更多常量传播、DCE、`-Os`/strip、可选 IR/SSA — **M1 稳定后立项**。

### 7.3 决策树：现在该压榨什么？

```text
                    是否有 benchmark / CI 回退 / 用户瓶颈报告？
                                    │
                    ┌───────────────┴───────────────┐
                   否                              是
                    │                               │
         自举/typeck 全绿？              瓶颈在编译还是运行？
                    │                               │
         ┌──────────┴──────────┐          ┌─────────┴─────────┐
        否                    是         编译                运行
         │                     │           │                   │
    优先 L1 typeck      新 std 或 L4     L5/L6              L3/L4
    不赌 UB 优化        定点 peephole   dogfood+flamegraph  net/io/simd
```

| 场景 | 建议 |
|------|------|
| 自举 / `bootstrap_shuxc -E` 未绿 | **只做 L1**，L4 单点 fold 可以，不大改后端 |
| 测试全绿、无 perf 回归 | **P0 新 std 小步** + 维持 bench（见《接下来做什么》） |
| CI dogfood 变慢 | PERF-005 找热点 → 针对性减 alloc 或算法 |
| 业务 net 瓶颈 | multishot、批大小、绑核；勿先动编译器 Lexer |
| 嵌入式 / 体积敏感 | L4 `-Os`、按需链接 std、strip（阶段 8） |
| 要对标 Mojo/GPU | **单独立项** MLIR/GPU dialect，不阻塞主链 |

### 7.4 与项目宗旨的对照

| 宗旨 / 目标 | 性能压榨落点 |
|-------------|--------------|
| 性能比肩 C 甚至超越 C | L4 asm + SIMD + fold；PERF-001 门禁 |
| 内存安全、无泄漏 | L1 region/linear/arena；L2 零热路径 malloc |
| 轻量级、体积小 | 无 LLVM 默认；按需链接；阶段 8 `-Os`/strip |
| 标准库丰富 | L3 模块 benchmark；与「空转优化」平衡 |
| 编译自举 | L5 让位于正确性；M2 后再 Lexer SIMD |
| 开发简单易上手 | L6 门禁防回退；文档写清「何时该优化」 |
