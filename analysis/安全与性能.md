# shux 安全与性能

> **语言**：shux（源文件 `.sx`）  
> **定位**：系统级语言，参考 C 的执行模型，默认安全子集 + 显式 `unsafe` 边界。  
> **关联**：[语法与类型设计-高性能与内存安全.md](./语法与类型设计-高性能与内存安全.md)、[UB与未定义行为.md](./UB与未定义行为.md)、[safe-unsafe-audit-v1.md](./safe-unsafe-audit-v1.md)

---

## 一、先给结论

| 问题 | 简短回答 |
|------|----------|
| 写 shux 要注意什么才少漏洞？ | **待在安全子集内**：用切片带长度、显式处理错误、不滥用 `unsafe`；边界操作走 typeck 能证明的路径。 |
| 写坏了会不会被利用攻击系统？ | **会**。内存破坏、逻辑漏洞、编译器被恶意源码打穿，都可能升级为 RCE、提权、数据泄露——与 C/Rust `unsafe` 写错时一样。 |
| 能做到「绝对安全」吗？ | **不能**。通用 OS + 硬件 + 人写代码 + FFI，不存在数学意义上的绝对安全；目标是 **可证明的安全子集 + 可审计边界 + 纵深防御**。 |
| 安全与性能冲突吗？ | **在 shux 的设计里，默认不冲突**：安全语义在 **编译期（typeck）** 证明，运行期零额外开销；冲突只出现在 **证不出** 或 **人为 `unsafe`** 时。 |
| 极致性能会不会伤安全？ | **会**，若靠猜 alias、强删边界、依赖 UB 换优化；**不会**，若性能来自 BCE / noalias / arena 等 **有证明的优化**。 |
| 极致安全会不会伤性能？ | **运行时安全**（GC、每下标检查）会慢；**编译期安全**（shux 路线）不但不慢，还是 SIMD、BCE、arena 的前提。 |

**一句话**：shux 的安全是 **骨架**（typeck + 门禁），性能是 **肌肉**（codegen 在证明成立后压榨）。骨架不牢，肌肉越大越危险。

## 二、威胁模型：写坏了会发生什么

### 2.1 应用层（你的 `.sx` 业务代码）

| 缺陷类型 | 典型后果 | 是否可被远程利用 |
|----------|----------|------------------|
| 缓冲区 / 切片越界 | 崩溃、堆栈破坏、信息泄露 | 常见于解析网络/文件输入时 |
| Use-after-free / 悬垂引用 | 崩溃或任意代码执行 | 高（经典内存漏洞） |
| 整数溢出导致错误分配 | 堆溢出 | 高（编译器、解码器里常见） |
| 除零、空指针解引用 | 崩溃（DoS） | 中 |
| 逻辑错误（权限、校验遗漏） | 越权、注入 | 高（与语言无关） |
| `unsafe` 块内约定被破坏 | 等同 C 级漏洞 | 高 |

**结论**：shux **不是魔法**。在安全子集外、或在 `unsafe` 里写错，效果和 C 写错一样——进程可能被攻破；若进程以 root / 高权限运行，**宿主系统也可能被利用**。

### 2.2 编译器与工具链（`shux` / `shux-c` 自身）

恶意或畸形 `.sx` 可能触发：

- Lexer/Parser **缓冲区溢出**（超长标识符、畸形 token）
- Typeck **整数截断/溢出**（数组长度、布局计算错误 → 生成错误代码）
- Codegen **错误代码**（静默错误比崩溃更危险）

这类问题属于 **供应链 / 编译器安全**，需要通过 Arena、溢出检查、ASAN 门禁、fuzz 覆盖。

### 2.3 系统边界（`std.sys`、FFI、syscall）

- 错误 syscall 参数、错误 mmap 权限 → 内核交互风险  
- 链接不安全 C 库（`strcpy` 等）→ 经典 C 漏洞回归  
- 目标：**硬件操作收拢在 `std/sys/`，FFI 显式、可审计**（阶段 B/F）

## 三、为什么没有「绝对安全」

在通用计算机上，以下因素使「绝对安全」不成立：

1. **图灵完备 + 系统调用**：程序总能请求 OS 能力；逻辑漏洞无法被类型系统完全消除。  
2. **硬件与微架构**：侧信道（Spectre 等）不在语言内存安全模型内。  
3. **依赖链**：编译器、链接器、libc、驱动 — 任何一层有洞，语言保证可被绕过。  
4. **人为 `unsafe` 与 FFI**：程序员断言「我保证安全」时，类型系统 intentionally 停止证明。  
5. **实现缺陷**：typeck/codegen 自身的 bug 可产生「看起来合法、运行有毒」的二进制。

shux 项目宗旨中的「绝对安全」应理解为：

> **在安全子集内**：无泄漏、不因内存错误崩溃、访问有定义；  
> **在子集外**：行为文档化、边界可 grep、变更可审计。

详见 [.cursor/rules/00-项目目标与宗旨.mdc](../.cursor/rules/00-项目目标与宗旨.mdc)。

## 四、开发者怎么写 shux 才安全

### 4.1 默认待在安全子集（不写 `unsafe`）

| 做法 | 原因 |
|------|------|
| 用 **`T[]` 切片**（ptr + len），不要手写裸指针遍历 | 长度显式，便于 BCE 与边界检查 |
| **`let` 默认不可变**；需要改再用可变绑定 | 减少意外别名与数据竞争面 |
| **显式初始化**；不读未赋值变量 | 排除未初始化 UB（见 [UB与未定义行为.md](./UB与未定义行为.md)） |
| **错误用返回值 / Result**，不要 silent ignore | 避免逻辑漏洞与 panic 路径失控 |
| **整数尺寸/偏移**用足够宽类型（如 `i64`），注意字面量范围 | 避免编译期截断（如 INT64_MIN 写法） |
| **不要假设溢出 wrap** | 有符号溢出在子集内仍为 UB，除非显式 wrapping API |
| **并发共享可变**须走 `std.sync` / 原子，不要裸共享 | 数据竞争 = UB |

### 4.2 必须碰底层时：显式 `unsafe`

规则（与 [safe-unsafe-audit-v1.md](./safe-unsafe-audit-v1.md) 一致）：

- 只有 **`unsafe { ... }` 内** 才允许：裸指针算术、未证明边界的解引用、特定 FFI、直接 syscall 等。  
- **新增 unsafe 面**必须更新 `tests/baseline/safe-unsafe-api.tsv` / 审计 ledger，并跑 `run-safe-unsafe-audit-gate.sh`。  
- 业务代码应 **零 unsafe**；底层集中在 `std/sys/`、`std/ffi/` 等少数模块。

### 4.3 处理不可信输入（网络、文件、用户文本）

- 解析器：**长度上限 + 有界缓冲区**（`with_arena` / 固定 cap），拒绝超长 token。  
- 所有索引：在 **typeck 可证明** 或 **显式运行时检查** 后再访问。  
- 不要相信「这次不会越界」——要么证明，要么检查，要么 `panic` 终止（缩小攻击面为 DoS 而非内存破坏）。

### 4.4 反例（高风险写法）

```sx
// 危险：裸指针 + 未证明的下标（即使在非 unsafe 块，若语义允许也可能 UB）
function bad_write(p: *u8, i: i32) {
  p[i] = 0;  // i 是否越界？p 是否有效？
}

// 危险：把不可信长度直接用于分配或拷贝
function bad_copy(len: i32, src: u8[]) {
  // 若 len 来自网络且未校验，可能溢出或 OOM 攻击
}
```

安全写法方向：用 `u8[]` 携带真实长度、在边界内循环、失败返回 `-1` / `Result`，或在契约违反时 `panic`（明确终止，不 silent corrupt）。

## 五、编译器与标准库作者额外要注意什么

### 5.1 编译器（`compiler/src/**/*.sx`）

| 项 | 要求 |
|----|------|
| 解析存储 | **Arena**（`with_arena` / MEM-C1），避免固定小栈数组被超长输入撑爆 |
| 大小计算 | 布局、数组长度、偏移用 **溢出检查**（C 侧 `__builtin_*_overflow` 或等价） |
| 常量折叠 | **i64 等大整数**不可截断为 i32（CTFE 与 lexer 须一致） |
| 优化 | noalias / BCE / 向量化：**证不出就回退**，禁止 silent wrong code |
| 测试 | `run-scope-borrow-gate.sh`、`run-ub.sh`、ASAN 夜跑（SAFE-005） |

### 5.2 标准库

- **禁止**链接无界 C 字符串 API（`strcpy`、`sprintf` 等）。  
- 拷贝/格式化：走 **带长度的** `core.fmt`、`std.mem` 有界 API。  
- 平台代码：`#[cfg(target_os = "...")]` 分流，Linux syscall 等收拢在 `std/sys/`（阶段 B）。

### 5.3 安全不能等自举后再补

性能优化可以后置；**typeck 拒错规则不能后置**：

```
借用/域检查 ──→ noalias 证明 ──→ BCE 消除边界 ──→ 自动向量化 (SIMD)
```

自举前若放水，后果包括：

- 编译器 `.sx` 源码充满「肉眼安全」的隐患，Stage2 硬化时全盘报错（**语义溃疡**）  
- UB 导致 Stage1/Stage2 **行为漂移、SHA256 对不齐**（自举门禁失败）  
- 优化证明失败 → 永久标量回退 → **性能目标落空**

## 六、安全与性能：shux 的因果链

### 6.1 与传统语言的对比

| 模型 | 安全放在哪 | 典型性能代价 |
|------|------------|--------------|
| C/C++ | 程序员脑子 | 无检查，快，但漏洞多 |
| Java / Go | 运行时 GC + 检查 | 延迟、边界检查、STW |
| **shux** | **编译期 typeck** | 安全子集内 **零运行时税** |

shux ** deliberately 不用 GC**：性能比肩 C 的底线之一；内存安全靠 **所有权/借用/region/linear/arena**，不是运行时扫描。

### 6.2 编译期安全如何变成性能

| 安全机制 | 编译器得到的信息 | 性能收益 |
|----------|------------------|----------|
| MEM-A1 **noalias** / 借用 | 两指针不重叠 | 自动向量化（VEC-V1）、指令重排 |
| MEM-A2 **BCE** | 下标恒在 `[0, len)` | 删除循环内边界分支 |
| MEM-C1 **`with_arena`** | 分配域与块同生命周期 | 热路径 `malloc=0`、无 free |
| **Linear(T)** move | 无 double-free / 无别名 | 更激进的销毁与布局优化 |
| **显式类型**（切片、Option） | 无隐式 UB | 常量折叠、死代码消除更可靠 |

因此：**在 shux 里，可证明的安全不是性能的敌人，而是极致优化的许可证。**

### 6.3 唯一真正的冲突：「证不出」时

当 typeck **无法静态证明** 安全（例如外部传入的两个裸指针是否 alias、下标是否越界）：

| 策略 | 性能 | 安全 | shux v1 默认 |
|------|------|------|--------------|
| 激进优化（猜 noalias / 删检查） | 最快 | 可能崩溃或被利用 | **禁止** |
| 保守回退（保留检查 + 标量） | 较慢 | 安全 | **默认** |
| 程序员 `unsafe` 断言 | 可快 | 依赖人工 | **显式 opt-in** |

**极致性能会不会影响安全？**  
- 靠 **证明后的优化** → 不影响。  
- 靠 **猜、UB、未审计 unsafe** → 严重影响。

**极致安全会不会影响性能？**  
- **编译期安全** → 不拖慢，常加速。  
- **运行时堆检查/GC** → shux **不选这条路**；若每下标都动态检查且证不出，会慢——这是刻意的 **安全保底**，不是最终态。

## 七、纵深防御（最后一层）

即使 typeck 与 codegen 都正确，仍建议默认启用 OS/链接器防护（release 链路逐步收紧）：

| 机制 | 作用 |
|------|------|
| **NX / DEP** | 数据页不可执行，缓解 shellcode |
| **ASLR + PIE** | 地址随机，提高 ROP 成本 |
| **栈保护 / 金丝雀** | 检测栈溢出（链接器选项） |
| **最小权限** | 服务不以 root 跑；capability 隔离 |

这些是 **进程被攻破后的减速带**，不能替代语言层内存安全。

## 九、自举前要不要补安全？

**要，而且不能等 Stage2 闭环后再补**——但不必从零造一套「安全语言」：shux 已有 **SAFE-002～007**、**MEM-A/B/C**、**TYPE-003** 等门禁与 RFC；自举前要做的是 **把已有骨架焊进日常验收**，并补齐 **编译器自身** 与 **CTFE/布局** 上的少数缺口。

### 9.1 现在必须做的是什么（不是「重写 typeck」）

| 层次 | 自举前必须达到 | 原因 |
|------|----------------|------|
| **语言拒错** | 借用逃逸、域逃逸、UB 收窄 **有 gate 且硬失败** | 否则 Stage1/Stage2 行为漂移、SHA256 对不齐 |
| **unsafe 治理** | 清单 + 审计 ledger 不漂移 | 编译器/std 膨胀 `extern` 时可控 |
| **编译器实现** | 解析有界、布局算术防溢出、CTFE 不截断 i64 | 恶意/畸形输入不能打穿 `shux`；常量不能 silently wrong |
| **优化默认** | noalias/BCE **证不出就回退** | 禁止为性能生成带毒代码 |
| **纵深防御** | PIE/NX/ASAN 逐步进 release 链 | 最后一层；**不替代** typeck |

### 9.2 可以 Stage2 闭环后再加强的是什么

| 项 | 说明 |
|----|------|
| 编译器源码 fuzz / 持续模糊测试 | 提高供应链上限；P1 可先做 smoke |
| 全仓库 `unsafe` 清零（业务层） | 阶段 F；底层仍集中在 `std/sys` |
| TSAN 全量 CI（SAFE-006） | 实验线已定版；默认可 track-only |
| 侧信道 / 微架构攻击 | 超出语言模型 |
| Windows/Alpine 安全变体 | A-08/A-13，自举闭环后 |

### 9.3 与《自举前性能与语义基建》的关系

- [自举前性能与语义基建.md](./自举前性能与语义基建.md) 管 **noalias / BCE / defer / CTFE** 等「性能语义闸门」。  
- **本文 §十一** 管 **安全门禁挂接、编译器自身硬化、unsafe/FFI 治理**。  
- 二者 **同一因果链**：安全证明失败 → 优化回退 → 自举性能与哈希双双失败。  
- 评审「能否推进 B/C/D」时，**两份文档的 P0 项应一起看**。

## 十三、总结

1. **shux 默认安全子集** 能在编译期消掉大部分内存漏洞，但 **不是绝对安全**；`unsafe`、FFI、逻辑 bug、实现 bug 仍可能伤系统。  
2. **写对 shux** = 少碰 `unsafe`、用切片与显式错误、不可信输入有界、并发走同步原语。  
3. **安全与性能在 shux 中同向**：typeck 证明 → BCE / noalias / SIMD / arena；证不出则回退，不赌。  
4. **不要等自举再补安全**；门禁与 typeck 是自举与极致性能的共同前提。  
5. **自举前不必重写 typeck**，但必须完成 **§十一 P0**：把已有 borrow/unsafe/UB gate **挂进 bstrict 或 Docker 硬门禁**，并补上 **i64/CTFE** 等编译器自身回归。

**安全是骨架，性能是肌肉——骨架先焊死在 typeck 与 gate 上，再谈压榨 codegen。**
