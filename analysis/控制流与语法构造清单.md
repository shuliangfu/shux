# Shux 控制流与语法构造清单（分析文档）

本文档列出**所有控制流及与之同类的语法构造**（如 if、else、else if、while、for、break、continue、return 等），与《内置函数清单》同风格：按类别与阶段整理，标明**是表达式还是语句**、实现状态，便于**不遗漏**、不重复。开发语法/控制流前先在此表勾选并补充。

---

## 一、概念区分：表达式 vs 语句

| 类型 | 说明 | 示例 |
|------|------|------|
| **表达式（expression）** | 有值，可放在需要值的位置（如赋值的右边、函数实参、return 后、另一表达式的子项） | `1 + 2`、`if (c) { a } else { b }`、`f(x)` |
| **语句（statement）** | 无值或值为 unit/void，用于副作用或控制流；不能当“值”用 | `while true { }`、`break;`、`let x = 1;` |
| **兼有** | 同一关键字在不同语法位置既可当表达式也可当语句（按语言设计） | 如 `return expr` 是语句，但 `return` 本身不产生值 |

本清单中每一项会标明：**仅表达式 / 仅语句 / 兼有（说明用法）**。

### 1.1 分号规则（当前实现，类 TypeScript）

块内**语句**以分号 `;` 结尾；**`}` 后可不写分号**（与 TS 一致：大括号后分号可选）；**块的最后一个表达式**后不加分号：

| 位置 | 分号 | 示例 |
|------|------|------|
| **const / let 声明** | 必须加 `;`，**除非**下一 token 是 `}` 或下一句开头（const/let/while/for/if 等） | `let x = 1;` 或 `let x = if (c) { 1 } else { 2 }` 后直接换行写 `}` |
| **return / goto** | 必须加 `;`，**除非**下一 token 是 `}` | `return x;` 或块尾 `return x` 后直接 `}` |
| **块的最后一个表达式（final_expr）** | 不加 `;` | 块尾写 `x` 或 `if (c) { a } else { b }` 后直接跟 `}` |

因此：若整块只有一个表达式，该表达式即 final_expr，**不要**在末尾加分号；若 const/let/return 后紧跟 `}` 或下一句关键字，可省略分号。

## 二、条件分支（if / else / else if）

| 构造 | 语法形式 | 表达式 or 语句 | 说明 | 状态 |
|------|----------|----------------|------|------|
| **if** | `if (cond) { then_block }` | 表达式 | 条件须带括号（C/Java 风格）；条件为真时求值 then_block，否则无 else 时按 0（或 unit） | ✅ 已实现（AST_EXPR_IF，无 else 时生成 0） |
| **else** | `if (cond) { then } else { else_block }` | 表达式 | 条件为假时求值 else_block 的最终表达式 | ✅ 已实现 |
| **else if** | `if (c1) { a } else if (c2) { b } else { c }` | 表达式 | 链式条件，等价于嵌套 if-else；可有多级；每个条件均须带括号 | ✅ 已实现（else 后允许 `if (cond) { then }` 递归解析） |

说明：当前实现中，**if/else 整体是一个表达式**（三元运算 `cond ? then : else`），不是“if 语句 + else 子句”两条语句。

## 三、循环（while / for / loop）

| 构造 | 语法形式 | 表达式 or 语句 | 说明 | 状态 |
|------|----------|----------------|------|------|
| **while** | `while (cond) { body }` | 语句 | 条件为真时重复执行 body；条件须用括号包裹；body 为块（const/let + 最终式或 break/continue） | ✅ 已实现（块内 loops 列表，body 不包含嵌套 while） |
| **for** | `for (init; cond; step) { body }`（init/cond/step 均可省略，cond 空表示恒真） | 语句 | C 风格 for；init/cond/step 为可选表达式 | ✅ 已实现（ASTForLoop，codegen 生成 for(;;){}） |
| **loop** | `loop { body }` | 语句 | 无限循环，等价于 `while (1) { body }`；靠 break 退出 | ✅ 已实现（解析为 cond=1 的 while） |

说明：**while 的 body 当前是“块”**，块内最后一项可为普通表达式（求值丢弃）或 **break/continue**，整体是语句，不产生值。

## 四、循环内控制（break / continue）

| 构造 | 语法形式 | 表达式 or 语句 | 说明 | 状态 |
|------|----------|----------------|------|------|
| **break** | `break` 或 `break;` | 语句 | 跳出当前循环（仅允许在 while/for/loop 体内） | ✅ 已实现（作为块 final_expr，生成 C break;） |
| **continue** | `continue` 或 `continue;` | 语句 | 进入当前循环下一轮（仅允许在循环体内） | ✅ 已实现 |

说明：当前实现里 break/continue 是**块的最后一项“表达式”位置**，但在语义与 codegen 上当作**语句**（生成 `break;`/`continue;`），不参与求值。

## 五、返回与退出（return）

| 构造 | 语法形式 | 表达式 or 语句 | 说明 | 状态 |
|------|----------|----------------|------|------|
| **隐式 return** | 块的最后一条表达式即返回值 | 表达式 | 当前设计：块体 `(const/let)* (while)* final_expr`，final_expr 即函数返回值 | ✅ 已实现 |
| **显式 return** | `return expr;` 或 `return expr` | 语句 | 立即从当前函数返回 expr；可出现在块中任意位置（含循环体内、if 分支内） | ✅ 已实现（作为块 final_expr，AST_EXPR_RETURN） |

说明：若引入**显式 return**，需约定与隐式 final_expr 的互斥或兼容（如“有 return 则不允许块尾再写裸表达式”或“return 与 final_expr 二选一”）。

## 六、其它可能构造（不遗漏清单）

以下为常见语言中与“控制流/语句/表达式”同类的构造，一并列出，避免遗漏。

| 构造 | 语法形式 | 表达式 or 语句 | 说明 | 状态 |
|------|----------|----------------|------|------|
| **match** | `match expr { lit \| _ => expr ; ... }` | 表达式 | 多分支匹配，最小形式仅整数字面量与 `_` 通配 | ✅ 已实现 |
| **try / catch** | `try { e } catch { handler }` 或 `e?` | 表达式/语句 | 错误传播；依赖 Result/Option 与语法设计 | ⬜ 未实现 |
| **defer** | `defer { block }` | 语句 | 当前块退出时逆序执行（含 return/goto 出口） | ✅ 已实现 |
| **goto / label** | `label:` 与 `goto label;` | 语句 | 块内显式跳转，生成 C label/goto | ✅ 已实现 |
| **throw / panic** | `panic` 或 `panic(expr)` | 表达式 | 终止程序；无参生成 abort()，有参生成 fprintf+abort | ✅ 已实现 |
| **block 表达式** | `{ stmt1; stmt2; expr }` | 表达式 | 块的值 = 最后一条表达式；当前块已有此语义 | ✅ 已实现（块即 final_expr） |
| **条件编译** | `#[cfg(...)]` 或 `$if const { }` | 编译期 | 按条件包含/排除代码 | ⬜ 未实现 |

## 八、极致压榨：控制流与性能

在「极致压榨性能」准则下，控制流不仅是逻辑跳转，更是**编译器优化、分支预测（Branch Prediction）和指令流水线（Pipeline）**的主战场。本节给出各构造的性能压榨点与优化方案，供 typeck/codegen 阶段落地。

### 8.1 条件分支：预测与消除（if / else / match）

分支跳转是流水线的敌人；分支预测失败时 CPU 需清空流水线，损失约 10–20 个时钟周期。

- **编译期分支预测 Hint（阶段 6）**  
  - 引入 `likely(cond)` 与 `unlikely(cond)` 内置宏（映射到 C 的 `__builtin_expect`）。  
  - **压榨点**：告诉编译器哪一分支更可能发生，使热点代码落在指令流的直线路径上，减少 jmp。  
  - 示例：`if unlikely(error_count > 0) { panic }`。

- **条件分支的性能映射（Typeck → IR/Codegen）**  
  - 为压榨分支预测，typeck 应将 `likely`/`unlikely` 携带到中间表示（IR），codegen 时映射为 C 的 `__builtin_expect(cond, 0)` 或 `__builtin_expect(cond, 1)`。  
  - **If 转换示例**：  
    - 源码：`if unlikely(err != 0) { return err }`  
    - 生成 C：`if (__builtin_expect(err != 0, 0)) { return err; }`  
  - **压榨点**：`__builtin_expect(cond, 0)` 表示“cond 通常为假”，GCC/LLVM 会将 then 分支（return err）放在汇编的远端，保证**正常路径**（err == 0）在流水线中是**线性取指**，最大化指令预取效率。

- **条件移动（CMOV）与无分支化**  
  - **压榨点**：对简单 `if (c) { a } else { b }`，若 a、b 均为简单运算且无侧作用（Side Effect），应优先生成 `cmov` 而非 jmp。  
  - **Shux 优化**：在 typeck 阶段识别「if 表达式两支均无侧作用」时，可标记该 if 为“可无分支化”，codegen 生成无分支代码。

- **Match 的跳转表（Jump Table）**  
  - **压榨点**：当 match 处理**连续或稠密**的整数字面量时，必须生成跳转表（Jump Table），实现 O(1) 多路分支，而非一串 if-else。  
  - **进阶**：对稀疏但有规律的 match，可考虑完美哈希（Perfect Hashing）或二分搜索树，在 codegen/后端选择策略。

### 8.2 循环：向量化与展开（while / for）

循环通常占据程序绝大部分执行时间。

- **循环展开（Loop Unrolling）（阶段 8）**  
  - **压榨点**：减少循环计数器与跳转次数。  
  - **方案**：支持 `#[unroll(N)]` 或等价属性；对 §11.4 的 **Const-Size Slice**，在长度已知时编译器应自动全展开。

- **循环对齐（Loop Alignment）（阶段 6）**  
  - **压榨点**：取指缓存（ICache）对齐，提升取指吞吐。  
  - **方案**：在生成循环的机器码/汇编前，在循环入口插入对齐（如 `.p2align 4` 实现 16 字节对齐），使循环入口落在 Cache Line 头部。

### 8.3 退出机制：零开销错误处理（return / panic / defer）

- **Defer 的零开销实现**  
  - **性能坑点**：若 defer 用运行时栈或动态分配维护，开销会很大。  
  - **Shux 压榨方案**：defer **必须在编译期静态内联**；**禁止**生成任何 `runtime_push_defer` 等运行时调用，仅在 **AST 变换阶段** 做代码注入。  
  - **极致压榨**：若 defer 块内仅为简单资源释放（如 `close(fd)`），确保不破坏当前函数的寄存器分配。  

  **Defer 编译期静态内联算法（Stage 5 硬约束）**  

  1. **收集阶段**  
     - Parser 进入一个块（Block）时，为该作用域创建一个空的 **defer_stack**（可挂在块节点的元数据上）。  
     - 遇到 `defer { <stmt> }`：将该代码块**压入**当前作用域的 defer_stack，**不生成任何代码**，不产生运行时调用。  

  2. **出口点注入**  
     - **Return 语句**：在“执行 return 表达式求值之后、真正跳转/返回之前”，按**逆序**遍历当前块（及外层块）的 defer_stack，将对应 defer 的代码块**拷贝并插入**到此处。  
     - **Break / Continue**：若它们**跨越**了带有 defer 的作用域，在跳出前先执行该作用域内 defer_stack 中的代码（逆序），再执行 break/continue。  
     - **块自然结束**：在块的右大括号 `}` 前（即块的最后一条有效语句/表达式之后），按**逆序**插入当前 defer_stack 中的所有代码块。  

  3. **Codegen 优化（可选）**  
     - 若多个出口点导致 defer 代码**重复过多**，后端可将公共的 defer 序列提取为一段代码，各出口点用 **goto 到统一 cleanup 标签**（类似 Linux 内核风格），以减小二进制体积，同时保持零运行时开销。

  **Defer 静态内联的「作用域陷阱」（Typeck/Codegen 必读）**  

  实现算法时必须正确处理**嵌套作用域**，避免注入错位或遗漏：

  - **局部出口规则**：当 **break** 或 **continue** 跳出**当前**带有 defer 的 `{ }` 块时，必须在生成跳转指令**之前**，**原地**插入该块内已收集的所有 defer 代码（逆序）。循环内、if 分支内的 defer 同理：只在该块出口处注入**该块**的 defer，再执行 break/continue。
  - **重复代码抑制**：若函数内有大量 return（如 10 个）且多个 defer（如 5 个），直接在每个出口展开会导致二进制体积膨胀。**Stage 5 建议采用 "Goto Cleanup" 模式**：
    - 在函数末尾生成统一标签 `cleanup:`；
    - 将所有 defer 代码按**逆序**排列在 `cleanup:` 之后（含 `return _ret_val;` 或函数 epilogue）；
    - 所有原本的 `return expr` 统一改写为 `_ret_val = expr; goto cleanup;`。  
  这样既满足零运行时开销，又控制代码膨胀。

- **Panic 的冷代码移动（Cold Path）**  
  - **压榨点**：panic 分支几乎永不执行，不应污染热点代码的指令缓存。  
  - **方案**：codegen 时将 panic 相关基本块（Basic Block）放到二进制**冷路径**（如 Section `.text.unlikely`），或为对应函数/块打上 `__attribute__((cold))`。  
  - **效果**：热点代码在 ICache 中保持紧凑，减少指令缓存污染。

  **Panic 与 ICache 局部性（Codegen 约束）**  

  为充分利用 L1 指令缓存与 CPU 分支预测：

  - **Codegen 约束**：不仅给 panic 辅助函数加 `cold` 属性，还要在生成 C 时，将 `if (err) { panic(...) }` 中的**条件**包在 **unlikely** 中，即生成 `if (__builtin_expect(!!(err), 0)) { panic(...); }`。  
  - **现代 CPU 行为**：前向分支预测器（Static Predictor）会默认认为该分支**不执行**，从而预取 **if 之后的正常业务指令**，保证热点路径线性取指，减少 ICache 污染与误预测代价。

### 8.5 总结：控制流能优化吗？能，而且提升巨大

当前控制流实现偏「功能性」。要达到「压榨性能」目标，下一步重点放在 **Codegen 阶段的 Hint 映射**：

- **Stage 5–6**：为 panic 增加 cold 属性；为 defer 实现**编译期静态内联**（按 §8.3 算法，无运行时 defer 栈）。
- **Stage 8**：配合 §11.3 的 `*restrict` 指针，让 for 循环支持**自动向量化**（消除别名顾虑后生成 SIMD 宽指令）。

### 8.6 下一步：Stage 5 的核心交付

在 if/else、for/while 已实现的前提下，Stage 5 的**性能重点**建议放在以下三项，作为硬约束落地：

1. **实现 defer 的静态内联**  
   - 按 §8.3 的 **Defer 编译期静态内联算法** 实现：收集阶段（defer_stack）、出口点注入（return / break·continue / 块自然结束）、可选 goto cleanup 优化。  
   - **目标**：彻底消除运行时 defer 栈，零运行时开销。

2. **引入 panic 的冷路径属性**  
   - 在生成的 C 代码中，为 panic 对应路径或辅助函数标记 `__attribute__((noreturn, cold))`。  
   - **目标**：让热点路径在 ICache 中保持紧凑，避免 panic 分支污染取指与预取。

3. **跳转表探测（Match Codegen）**  
   - 在 match 的 codegen 中，根据 case 的**密度**做策略选择，**目标**：多路分支 O(1) 或 O(log N)，减少分支预测失败与流水线清空。  

   **密度计算公式**：`Density = Count_cases / (Value_max - Value_min + 1)`（即 case 值在 [Value_min, Value_max] 范围内的覆盖比例）。  

   **压榨策略**：  
   - **Density > 0.7**：生成 **switch**，促使 C 编译器产生 TBB/TBH（ARM）或 Jump Table（x86），实现 O(1) 多路分支。  
   - **Density 极低**：生成**二分查找树**（Binary Search Tree）结构的 if-else，将线性 O(N) 压至 O(log N)。  
   - **Case 数 ≤ 3**：直接用 **CMOV**（条件移动）实现无分支化，避免跳转对流水线的破坏。

4. **寄存器压力感知（§8.7）**  
   - 静态内联 defer 时需考虑**寄存器保存策略**，避免过多注入导致 spill；详见 §8.7。

### 8.7 寄存器保存策略（基于 Defer 注入）

**问题**：在按 §8.3 做 defer 静态内联时，若在每个 return 出口**原地**展开的 defer 代码过长，会打乱原有寄存器分配，导致 return 前出现大量**溢出（Spill）**——即被迫将寄存器写回内存，抵消零开销收益。

**策略（Typeck + Codegen）**：  
- 在注入 defer 时，**typeck 需标记 defer 块内涉及的变量**（读/写集合）。  
- 若涉及的变量较多、或 defer 块总“体积”超过阈值，**强制走 §8.3 的 Goto Cleanup 模式**：不在各出口重复展开，而是统一跳到 `cleanup:` 标签，由**集中一段清理逻辑**执行所有 defer 并 return。  
- **效果**：后端只需在单一 cleanup 路径上做一次寄存器分配与保存，避免多出口重复展开带来的寄存器压力与 spill，使 Stage 5 的 defer 实现既零运行时开销，又不过度牺牲代码质量。

**建议**：将“是否走 Goto Cleanup”作为 Stage 5 核心交付的**硬约束**之一：在实现 defer 内联时，根据 defer 块复杂度与变量数量自动选择“出口原地展开”或“Goto Cleanup”，并优先保证热点路径无多余 spill。

### 8.8 进一步压榨：三项高级特性

以下特性通常面向**高性能网络库或大数据计算引擎**，可在 Stage 6–8 之后按需引入，作为控制流“极致性能”的进阶选项：

1. **静态可证明的边界检查消除（Range Analysis）**  
   - **现状**：for 循环与数组访问往往依赖运行时边界检查。  
   - **压榨点**：在 **typeck** 阶段利用循环的 init、cond、step 做**值域分析**；若能**静态证明**索引 `i` 恒在 `0..len` 内，则在 **codegen** 时**物理删除**边界检查指令。  
   - **目标**：消除分支跳转，使循环体进入纯净线性流水线，为 §8.2/8.5 的**自动向量化**扫清障碍。

2. **尾调用优化（Tail Call Optimization, TCO）**  
   - **压榨点**：当 `return f(args)` 是函数的**最后一步**时，不生成 `call`，而是**复用当前栈帧**并生成 **jmp**（尾调用变为尾跳转）。  
   - **目标**：将 O(N) 空间的递归转化为 O(1) 空间的循环，消除函数调用的栈帧压栈与回退开销；适用于深度优先搜索、递归解析器等场景。

3. **指令融合与逻辑短路优化**  
   - **压榨点**：针对 `if (a && b)`，若 **a** 是**热点冷分支**（经常为假），在 codegen 时调整生成顺序与分支布局，利用 CPU 的**短路求值**，使 **b** 的指令在 a 为假时**甚至不会被载入 L1 Cache**。  
   - **目标**：最大化内存带宽利用率，减少无效取指。
