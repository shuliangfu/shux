# SHUX 性能基准测试规划

> **文档状态**：草案 | **日期**：2026-07-08
> **定位**：SHUX 性能基准测试的四层框架与首个测试套件建议。
> **目标**：从"微观指令效率"到"宏观系统交互"系统性验证 SHUX 在系统编程核心维度上的竞争力。

---

## 1. 测试维度总览

| 层面 | 验证目标 | 关键指标 |
| --- | --- | --- |
| 微观计算性能 | 编译器代码生成质量 + 零开销抽象 | 指令效率、Cache 命中、分配开销 |
| 语言特性基准 | SHUX 语义设计是否产生预期性能优势 | 调用开销、抽象开销、所有权转移 |
| 系统级交互性能 | SHUX 适合写 OS / 底层库的能力 | 切换开销、I/O 吞吐、零拷贝 |
| 编译器质量指标 | 开发者体验 + 产物纯粹性 | 编译吞吐、二进制体积、冷启动 |

**对比基线**：所有 benchmark 必须对比 C 语言（`clang -O2` / `gcc -O2`）。若微观测试无法逼近 C，说明 IR 后端有优化空间。

---

## 2. 微观计算性能（Micro-benchmarks）

验证编译器代码生成质量与零开销抽象能力。

### 2.1 算术运算与循环

- **Fibonacci / Pi 计算**：测试循环优化、寄存器分配、内联（Inlining）效果。
- **矩阵乘法（Matrix Multiplication）**：测试数组内存布局、缓存友好性（Cache Locality）、SIMD 优化（后续支持时）。

### 2.2 排序与搜索

- **QuickSort / Binary Search**：测试分支预测（Branch Prediction）与函数调用开销。

### 2.3 内存分配与销毁

- **Arena Allocation Test**：SHUX 核心优势。对比 SHUX Arena 与传统 `malloc/free` / GC 的分配速度，展示零开销回收的威力。
  - 测试方式：循环分配 / 销毁成千上万个小对象，证明 SHUX 不产生碎片且无需 GC。

---

## 3. 语言特性基准（Language-level Benchmarks）

验证 SHUX 语义设计是否产生预期的性能优势。

### 3.1 函数调用开销

- **递归调用深度测试**：对比 X ABI（SHUX 默认，`extern function`）与 C ABI（`extern "C"`）在深层函数调用时的寄存器压栈损耗。
- **X ABI 跨模块内联**：验证 LTO / WPO 阶段能否跨 X ABI 边界内联（C ABI 禁止内联，这是 X ABI 的关键优势点）。

### 3.2 抽象开销

- **零成本抽象验证**：编写高阶逻辑结构，测试 `struct` + `impl` 封装后的运行性能是否与直接编写原生逻辑完全一致。

### 3.3 内存拷贝与转移

- **`Linear(T)` 所有权转移测试**：验证编译器在无任何运行时检查的情况下，正确管理内存生命周期。
  - 测试 move 语义是否零开销（编译期解析，无运行期 refcount）。

---

## 4. 系统级交互性能（System-level Benchmarks）

验证 SHUX 适合写 OS 与底层库的能力。

### 4.1 上下文切换 / 任务切换

- **轻量级任务模型**：若 SHUX 支持协程 / 任务模型，测试切换开销是否远低于系统线程。

### 4.2 I/O 与系统调用

- **Buffer I/O 吞吐量**：文件读写中测试从缓冲区读取数据的吞吐量（Throughput），展示 SHUX 最小化内存拷贝的能力。

### 4.3 解析器对比

- **极简解析器**：用 SHUX 写一个极简 JSON / 配置解析器，对比同样逻辑下原生 SHUX 比解释型语言快多少倍。

---

## 5. 编译器质量指标（Compiler Metrics）

验证开发者体验与产物纯粹性。

- **编译吞吐量**：测试 `shux build` 编译 10k 行代码文件的秒级响应。
- **二进制体积（Binary Size）**：对比同样逻辑，SHUX 生成的二进制应显著小于引入 runtime 的 Go / Rust（衡量 SHUX 是否"纯粹"）。
- **冷启动延迟（Cold Start Time）**：测量 SHUX 产出的二进制从加载到执行第一行代码的时间（系统级语言关键指标）。

---

## 6. 测量方法论

为保证测试结果可信度，遵循以下原则：

- **控制变量**：测试时禁用所有调试符号（`--release` 模式），确保编译器开启所有优化 Pass。
- **基准对比（Baseline）**：必须对比 C 语言（`clang -O2` / `gcc -O2`）。微观测试无法逼近 C 则说明 IR 后端有优化空间。
- **冷启动测量**：系统级语言启动速度极重要，单独测量二进制加载到首行代码执行的时间。
- **多次采样**：每个 benchmark 取中位数 + P99，避免单次抖动误导。

### 6.1 反作弊防御（开发规范 v1 §4 硬约束）

> 性能测试最大的谎言是"Wall Time = 0"——编译器常量折叠把整个循环优化掉了。必须主动防御。

**C 侧防御**：
- 用 `__asm__ volatile("" : "+r"(var) : : "memory")` 阻止 gcc/clang -O2/O3 常量折叠。
- 参考实现：`tests/bench/loop_i32.c` 第 13 行——`i` 和 `s` 被 `__asm__` 标记为 clobber，迫使编译器真跑 1e8 次循环。

**SHUX 侧防御**：
- 禁止依赖"未用变量被 DCE"的副作用。必须 `return` 聚合结果作为 exit code，Runner 级校验返回值非零。
- 若 benchmark 循环结果被 DCE 抹除（exit code = 0），门禁脚本报 `[WARN] 采样为零` 并拒绝通过。
- 参考实现：`tests/bench/loop_i32.x` —— `return s;` 强制编译器保留整个循环计算。

**噪声控制**：
- 3 次 Warmup 预热（填充 cache / 分支预测器）。
- 重复采样 ≥ 20 轮，报告中位数 + P99 分布。
- 禁止用统计平均值注水——平均值受离群点拉偏，中位数才是系统语言性能的真实刻画。

### 6.2 PERF-001 门禁脚本

- 脚本位置：`tests/bench/gate/perf_001_gate.sh`
- 硬门禁：SHUX 自研后端生成代码的 wall time 中位数 ≤ C -O2 中位数。
- 用法：`./tests/bench/gate/perf_001_gate.sh [bench_name]`（默认 `loop_i32`）。
- 时机：自举后激活硬门禁；自举前可运行供参考（ASM 后端 bug ①② 未修时结果不准）。

## 7. 首个测试套件建议

不要试图一次做完所有 benchmark。从以下两个最硬核的点切入：

### 7.1 Arena 分配速度

- **测什么**：循环分配 / 销毁成千上万个小对象。
- **证明什么**：SHUX 不产生碎片且无需 GC。
- **对比基线**：C 的 `malloc/free` + Go 的 GC。

### 7.2 函数内联效果

- **测什么**：深度递归算法。
- **证明什么**：编译器成功消除不必要的栈操作。
- **对比基线**：C 的 `-O2` 内联效果。

> **里程碑**：若能在"内存分配"和"递归处理"上证明 SHUX 优于或等于 C，即拿到系统编程领域的入场券。

---

## 8. SHUX 特有优势验证清单

以下优势点应在对应 benchmark 中明确验证，作为 SHUX 区别于 C / Rust / Go 的核心竞争力：

| 优势点 | 验证 benchmark | 预期结果 |
| --- | --- | --- |
| Arena 零开销回收 | §2.3 Arena Allocation Test | 分配 / 回收速度远超 `malloc/free` + GC |
| `Linear(T)` 零运行期检查 | §3.3 所有权转移测试 | 无 refcount，move 语义编译期解析 |
| X ABI 跨模块内联 | §3.1 X ABI 内联测试 | LTO / WPO 跨 ABI 边界内联，消除调用开销 |
| 零拷贝 I/O | §4.2 Buffer I/O 吞吐量 | 索引引用传递，无数据拷贝 |
| 无 runtime 二进制 | §5 二进制体积 | 显著小于 Go / Rust（无 GC / 无 panic runtime） |
