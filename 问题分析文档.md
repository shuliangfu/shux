## 2. 剩余问题

### 问题 A：hello -o 重载函数名未 mangle（最高优先级）

**现象**：`hello.x` 的 `fmt.print(msg, 12)` 生成 C 中为 `std_fmt_print(msg, 12)`（裸名），应为 `std_fmt_print_u8_ptr_i32(msg, 12)`（mangle 名）。cc 报 `implicit declaration of function 'std_fmt_print'`。

**根因分析**：

1. `codegen_resolve_binding_import_path_for_method_call` 从 `ctx->current_codegen_module` 读取 import path。当 codegen 处理 dep 模块函数体时，`current_codegen_module` 是 dep 模块（如 std.fmt 有 `import("core.fmt")`），返回 "core.fmt" 而非入口模块的 "std.fmt"。

2. `codegen_find_dep_index_by_path(ctx, "core.fmt", 8)` 在 pctx dep 路径中找不到 "core.fmt"（pctx 中是 "std.fmt"），返回 -1，`fb_dep_mod = NULL`。

3. `codegen_emit_call_func_name` 收到 `current_module = NULL`，fallback 搜索也找不到函数（或 `codegen_emit_func_link_name` 因 `search_arena` 错误返回裸名）。

4. `codegen_emit_func_link_name` 中 `codegen_type_ref_to_suffix(search_arena, param_type_ref, ...)` 使用了入口 arena 而非 dep arena 查找参数类型后缀，导致后缀为空，函数名退化为裸名。

**已尝试的修复**：
- `a0d45ceb`：`codegen_emit_skipped_dep_type_definitions` 后恢复 `current_codegen_module`
- `96c9f8af`：每个 `codegen_emit_func` 调用前恢复 `current_codegen_module`
- `49e88485`：当 `search_mod` 是 dep 模块时从 pctx 查找正确的 dep arena
- `ce84eea6`：method call 路径直接在 dep 模块中搜索函数名+arity

**仍未解决**：debug 显示 `dep_path_fb=core.fmt` 对所有 `method=print nargs=2` 调用。说明 `current_codegen_module` 仍非入口模块，或 `codegen_resolve_binding_import_path_for_method_call` 读取了错误模块的 import。

**-E 与 -o 差异**：-E 输出（167936B）不含入口模块 `main` 函数（仅 dep 代码）；-o 输出（3437 行）含 `main` 函数且有裸名问题。两者走不同 codegen 路径。

**下一步建议**：
- 在 `codegen_resolve_binding_import_path_for_method_call` 中增加按 binding 名匹配 pctx dep 路径的 fallback（不依赖 `current_codegen_module` 的 import path）
- 或在 method call codegen 路径中，当 `codegen_emit_call_func_name` 返回裸名时，搜索所有 pctx dep 模块并用正确 dep arena 调用 `codegen_emit_func_link_name`
- 关键验证点：确认 `ctx->current_codegen_module` 在入口模块 `main` 函数 codegen 时确实为入口模块（可能需要在 `codegen_emit_func` 入口处加安全 debug）

### 问题 B：stdlib-import -o struct 不完整类型

**现象**：`stdlib-import/main.x` -o 失败，cc 报 `return type is an incomplete type`（`core_result_ok` 返回 `struct core_result_Result_i32` 但 struct 只有前向声明无完整定义）。

**根因**：codegen 的类型定义顺序问题——dep 模块的函数定义在 struct 完整定义之前。`core_result_Result_i32` 在 `core_result_ok` 函数定义处只有前向声明。

**修复方向**：确保 codegen 在 emit dep 模块函数体之前先 emit 所有 dep 模块的 struct 完整定义（forward declaration → full definition → function bodies）。

### 问题 C：bstrict libc 符号冲突

**现象**：`make bootstrap-driver-bstrict` 在编译 `std/process/mod.x` 时失败：`conflicting types for 'getcwd'`。

**根因**：`shux_compile_std_module.sh` 的符号重命名仅覆盖 `free/open/close/malloc/realloc/calloc`，未覆盖 `getcwd` 等 libc 函数。

**修复方向**：在 `shux_compile_std_module.sh` 的 `objcopy --redefine-sym` 列表中添加 `getcwd`、`read`、`write`、`lseek`、`fcntl`、`stat`、`fstat`、`unlink` 等 libc 符号。或从根源修复 codegen 对 std 模块函数应用正确的 path lib_name 前缀。

### 问题 D：io_driver 类型不匹配

**现象**：生成 C 中 `std_io_driver_submit_read` 和 `std_io_driver_submit_write` 的参数类型不匹配。

**根因**：codegen 对 `std.io.driver` 模块函数参数类型的 C 表示与实际不匹配（可能是 slice/pointer 类型转换问题）。

## 6. 问题 A 修复要点（详细）

### 数据流

```
hello.x: const fmt = import("std.fmt");
  → parser: module.imports[0] = { binding="fmt", path="std.fmt" }

std/fmt/mod.x: const fmt = import("core.fmt"); const io = import("std.io");
  → parser: std.fmt module.imports[0] = { binding="fmt", path="core.fmt" }

codegen 处理 std.fmt dep 模块函数体时:
  ctx->current_codegen_module = std.fmt module
  → codegen_resolve_binding_import_path_for_method_call 读 std.fmt 的 import("core.fmt")
  → dep_path_fb = "core.fmt" (8 bytes)
  → codegen_find_dep_index_by_path("core.fmt") 在 pctx 中找不到（pctx 有 "std.fmt"）
  → fb_dep_ix = -1, fb_dep_mod = NULL
  → codegen_emit_call_func_name(current_module=NULL) 搜索失败
  → 发出裸名 "print" → cc 报 implicit declaration

codegen 处理 hello 入口模块 main 函数时:
  ctx->current_codegen_module 应该 = hello module
  → codegen_resolve_binding_import_path_for_method_call 读 hello 的 import("std.fmt")
  → dep_path_fb = "std.fmt" (7 bytes) → 应该能找到 dep[0]
  → 但 debug 显示仍为 "core.fmt" → 说明 current_codegen_module 仍非 hello module
```

## 7. 一句话总结

**当前真正挡住「hello -o + bstrict」的根因是：codegen 处理入口模块函数体时，`ctx->current_codegen_module` 可能仍指向最后一个 dep 模块（std.fmt），导致 import path 解析读取了 dep 模块的 `import("core.fmt")` 而非入口模块的 `import("std.fmt")`，最终 `codegen_emit_func_link_name` 因找不到正确 dep 模块和 arena 而发出裸函数名。需要确认 `current_codegen_module` 在入口模块 `codegen_emit_func` 时的实际值，并确保它指向入口模块。**
