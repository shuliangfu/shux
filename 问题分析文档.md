## 3. 问题 A — 带 import 的主程序 typeck hard-fail（XP003）

### 3.1 现象

在 **`0d717273` + 新 parser seed（含 STRING_LIT 直建/转义）+ g05 relink** 后：

| 用例 | 命令 | 结果 |
|------|------|------|
| `tests/return-value/main.x` | `-o` | **仍绿**，运行返回 42 |
| `tests/option/main.x` | `-E` / `-o` | **失败**：`typeck error[XT001] ... main (check_block failed)` + `pipeline error[XP003]`，**无 C 输出** |
| `examples/hello.x` | 同上 | 同上（常伴随 dep 模块 typeck 噪声） |
| `tests/stdlib-import/main.x` | `-o` | 同上 |

stderr 典型形态：

```text
typeck error[XT001]: .x type check failed in function #0 main (check_block failed)
 --> tests/option/main.x
pipeline error[XP003]: .x pipeline failed
 --> tests/option/main.x
```

hello 还会连带一批 **dep 模块** 的 soft-looking 失败（历史上 -E 时常仍继续）：

```text
typeck error[XT001]: ... register  --> ./std/io/driver.x
typeck error[XT001]: ... format    --> ./std/fmt/mod.x
...
typeck error[XT001]: ... main      --> examples/hello.x
pipeline error[XP003]: ...
```

**关键对比（会话早期，f49ec6aa 产品、未动 parser seed）**：

- hello `-E` 可出 ~150KB C，call site 已是 `std_fmt_print_u8_ptr_i32(msg, 12)`  
- option/stdlib-import 曾 `-o` 返回 42 / 0  
- dep typeck 报错多为 **软噪声**，**不**挡 entry 出 C  

**当前**：entry `main` 的 `check_block failed` 会触发 **XP003 硬失败**，`-E` 文件 0 字节。

### 3.2 涉及数据流

```
源 .x
  → lexer
  → parser (产品：parser_x.o ← seeds/parser_gen.linux.x86_64.c)
  → AST pool（Expr.kind / call / import binding）
  → typeck（产品：typeck_x.o）
       · 模块 dep 按 import 拉取 std/core
       · 对每个 function 做 check_block
       · entry 失败 → pipeline 记 XP003，中止 emit
  → codegen（仅 typeck 过关或 soft 策略允许时）
```

### 3.3 根因判断（尚未最终钉死，按证据排序）

| 假设 | 证据 | 反证 / 备注 |
|------|------|-------------|
| **A1. 新 parser seed 破坏 AST** | 新 seed 引入 `EXPR_STRING_LIT` 直建 + 转义 + enum 末尾补 `TRY_PROPAGATE/STRING_LIT`；曾出现 enum 一度截断到 `SPAWN` 的失误 | 把 **parser seed 整文件回退到 f49ec6aa** 并 `make parser_x.o` + relink 后，**hello -E 恢复**；但 **option 仍 XP003** → parser 不是唯一根因 |
| **A2. 0d717273 中 std/core 改动改变 typeck 结果** | 同 commit 改了 `core/fmt`、`core/result`、`std/fmt`、`std/io/sync` | option **只** `import("core.option")`，不直接依赖上述文件；临时 checkout 旧 core 后现象仍混杂，需干净 A/B |
| **A3. g05 relink 后「硬失败策略」或周边 .o 漂移** | typeck_x.o 时间戳可仍为旧文件，但 `shux_asm` 整体 relink；runtime/preamble 大改 | 需对比「仅 relink 不改 seed」是否复现 |
| **A4. dep typeck 失败向上污染 entry** | hello 日志里 format/register 先失败再 main | import-only 小程序（`const fmt=import("std.fmt"); function main(){return 0}`）曾能 e=0；**一加 `fmt.print(...)` 就 XP003** → 更像 **call 路径** 而非「只要有 import」 |

**结论（工作假设）**：

1. **Parser seed 的 STRING_LIT 大改风险高**，已验证会单独搞挂 hello 的 hard-fail 路径；**产品侧应用 f49ec6aa 的 parser seed 作为安全基线，直到 STRING 修法重做**。  
2. **option/stdlib-import 的 XP003 另有根因**（与 STRING 无直接关系），必须单独做「从 f49ec6aa 干净树 + 逐 commit  bisect」或「对比 typeck 对 call_resolved / Option 的行为」。

### 3.4 已做尝试（避免重复踩坑）

| 尝试 | 结果 |
|------|------|
| 裸 `gcc -c` 编 `parser_x.o` | **灾难**：对象过小，全项目 `no functions` |
| `0d717273` 整包 pull + g05 | return-value 绿；option/hello hard-fail |
| parser seed 回退 f49ec6aa + make + g05 | hello `-E` 恢复；option 仍 fail |
| `SHUX_TYPECK_BEST_EFFORT=1` | **禁止当验收**；只可用于对照，不能当绿 |

### 3.5 建议修法（根源层，不写补丁）

1. **冻结 parser 产品 seed** 在「hello -E 能出 C」的版本（f49ec6aa 或修好后的新版）。  
2. **STRING_LIT 重做**（见问题 D）：转义应在 **parser 建 AST 时** 解码，但必须：
   - 与 `ast.x` 的 `ExprKind` 序一致（STRING_LIT=59）
   - seed 内 `enum ast_ExprKind` **完整**（含 `TRY_PROPAGATE` + `STRING_LIT`）
   - 本地用 **最小用例矩阵** 测（§6.2），再动 hello  
3. **option XP003**：  
   - 打开 typeck 对 `main` 的细日志（若有 `SHUX_TYPECK_DEBUG` 等）  
   - 对比 f49ec6aa 与 0d717273 的 `typeck_x.o` / `pipeline_glue` / `ast_pool` 是否被 g05 静默换掉  
   - 确认 `check_block failed` 是 **第一条真实错误** 还是 **二次报错**

## 4. 问题 B — hello `-o`：生成 C 无法过 gcc

### 4.1 现象

在 **parser 回退、hello `-E` 能出 C** 时，`-o` 仍失败，gcc 典型错误（保留文件如 `/tmp/shux_shux_x.*.c`）：

#### B1. Context 用 `= 0` 初始化 struct

```c
struct std_context_Context _rc = 0;  /* invalid initializer */
```

对应源（修复**已写在 mac 工作区未提交文件** `std/context/mod.x`）：

```text
// 错：let _rc: Context = 0;
// 对：let _rc: Context = Context { handle: 0 };
```

函数：`background` / `with_cancel` / `with_deadline` / `with_timeout`。

#### B2. 值类型字段赋值误用 `->`

```c
struct shux_slice_uint8_t out = { 0 };
(void)(((out->data) = (s.data)));   /* invalid type argument of '->' */
(void)(((out->length) = (s.length)));
```

对应 `std/io/backend.x`：

```text
export function io_read_ptr_slice(...): u8[]<io_read_ptr> {
  let s: ShuxSliceU8 = ...;
  let out: u8[]<io_read_ptr>;
  out.data = s.data;
  out.length = s.length;
  return out;
}
```

`out` 是 **值类型**，C 必须 `out.data`，不能 `out->data`。

### 4.2 数据流（. vs ->）

```
EXPR_FIELD_ACCESS / assign to field
  → codegen_emit_expr（FIELD_ACCESS 分支）
  → is_ptr_base =
        field_access_base_is_pointer_ref(resolved_type)
     || field_access_base_is_pointer_param(...)
     || (未解析类型时) field_access_base_is_slice_param_name(名字启发式)
  → is_ptr_base ? 发射 "->" : 发射 "."
```

权威实现：

- seed：`compiler/seeds/codegen_gen.linux.x86_64.c` ~4136–4155  
- `.x`：`compiler/src/codegen/codegen.x` ~5258+ 与 `field_access_base_is_slice_param_name`

**根因（已定位）**：

- seed 里 `codegen_field_access_base_is_slice_param_name` **仍把名字 `"out"`（3 字节）当成指针启发式**（约 1681 行：`var_name_len == 3 && 'o','u','t'` → return 1）。  
- `.x` 侧注释已写明：**勿把裸名 out 当指针**（`codegen.x` ~1745–1746），且表中已去掉 `"out"`。  
- **seed 与 .x 双权威漂移** → 产品 seed 仍误发 `out->`。

这是典型的 **双权威 + 名字启发式** 问题，修法应是：

1. **seed 与 .x 对齐**：删除 `"out"` 启发式；  
2. 更根本：局部 let 的类型应靠 **resolved_type_ref / 声明类型** 决定 `.`/`->`，名字表只作 typeck 未解析时的窄回退。

### 4.3 Context `= 0`

- 根因在 **stdlib 源** 写了 `let x: Struct = 0`，codegen 忠实翻译成非法 C。  
- mac 上 `std/context/mod.x` 已有 struct 字面量修法，**尚未进入已推送 commit 的可靠验收**（工作区仍脏）。  
- 另：`std/context/context.x` 有 FNV 字面量 `as u32` 修法，同属未提交。

## 5. 问题 C — libc 符号冲突（read/write/poll）— 本轮已部分收敛

### 5.1 现象（修复前 / 旧 std.io.sync）

```text
error: conflicting types for 'write'; have 'ssize_t(int32_t, uint8_t *, size_t)'
note: previous declaration in unistd.h ...
```

同类：`read` / `readv` / `writev` / `poll`。

### 5.2 根因

- `std/io/sync.x` 曾 `extern "C" function write/read/...`，与 preamble 注入的 `#include <unistd.h>` 冲突。  
- **0d717273** 改为 `shux_sys_*`，preamble/runtime 注入：

```c
static inline ssize_t shux_sys_write(...) { return write((int)fd, (const void *)buf, count); }
// read / readv / writev / poll 同理
```

### 5.3 当前状态

在 hello 能生成 C 的实验中，可见：

- preamble 内 `shux_sys_write`  
- 模块侧 `extern ssize_t shux_sys_write(...)` + `return shux_sys_write(...)`  
- **不再**与 unistd 的 `write` 直接 redecl 冲突  

仍需在 **完整 `-o` 绿** 后确认：多重 `extern shux_sys_*` 与 `static inline` 同 TU 是否在某些 gcc 版本 warning-as-error。

### 5.4 验证

```bash

# 期望：无 bare write/read extern；有 shux_sys_*
```

## 6. 问题 D — 字符串字面量转义 `\n` 未进语义字节

### 6.1 现象

hello 主函数里曾见（compound 或 hex 路径）：

```c
// 错误语义：92= '\\', 110= 'n'，而不是 10= LF
uint8_t * msg = (uint8_t[]){..., 92, 110, 0};
// 或
((uint8_t*)"\x48\x65...\x5c\x6e")
```

源：

```text
let msg: *u8 = "Hello World\n";
fmt.print(msg, 12);  // 12 按「含 1 字节换行」计数
```

若 AST 存了 `\`+`n` 两字节，则长度为 13（+NUL 更乱），打印内容错误。

### 6.2 数据流

```
源 "Hello World\n"
  → lexer：TOKEN_STRING，token_start 在开引号后；扫描时跳过 \X 占两字符，但 **不解码**
  → parser：把 [token_start, token_start+ident_len) **原样** 拷进 EXPR_STRING_LIT.var_name
  → codegen：按 var_name 字节发 \xHH 或 historically compound
```

### 6.3 修法原则

| 层 | 是否应解码 | 说明 |
|----|------------|------|
| lexer | 只定界 | 已跳过 escape，保持 token 长度正确 |
| **parser 建 AST** | **应解码** | 唯一权威：语义字节进 `var_name`，`var_name_len` 为解码后长度 |
| codegen | 只忠实输出 | 对 AST 字节做 `\xHH`；**不要**在 codegen 再猜 escape（防双权威） |

**0d717273 的教训**：

- 在 seed 里改 STRING_LIT + 动 `enum ast_ExprKind` 时，曾出现 **enum 丢 TRY_PROPAGATE** 等事故；  
- 新 parser 与 typeck hard-fail 强相关；**必须用最小矩阵单测 parser 后再合产品**。

### 6.4 最小验证矩阵（parser/codegen）

```bash

  let msg: *u8 = "A\n";
  return 0;
}
EOF

echo s1:$?; cat /tmp/s1.err
sed -n '/^int32_t main/,/^}/p' /tmp/s1.c
# 期望：\x41\x0a 或 65, 10；禁止 \x5c\x6e / 92, 110

cat > /tmp/s2.x <<'EOF'
function main(): i32 {
  let msg: *u8 = "Hi";
  return 0;
```
