# SHUX 编译器全量测试问题分析文档

## 一、架构概览

### 1.1 双编译器架构

| 编译器 | 路径 | 用途 | 特点 |
|--------|------|------|------|
| `shux` | bootstrap-driver-seed | .x pipeline | 支持 `-backend asm` |
| `shux-c` | C 前端 | 纯 C pipeline | 不支持 `-backend asm`，用 `-E` 生成 C 后 `cc -c` |

### 1.2 编译脚本职责

**`shux_compile_std_x.sh`**：编译 std/*.x 为 .o 的核心脚本
- 自动选择逻辑：优先 `shux_asm` → `shux` → `shux-c`
- `shux_asm`/`shux` 使用 `-backend asm`
- `shux-c` 使用 `-E + cc -c`

**`run-all.sh`**：全量测试入口
- 在 `SHUX` 未设置时走 `else` 分支，构建 C-only `shux`（无 ASM 后端）

## 二、已解决问题

### 2.1 shux_compile_std_x.sh 回退导致失败数从 29 升至 54

**根因**：`shux-c` 分支被改为 `-E + cc -c` 生成 .o，导致符号不兼容

**修复**：恢复 `shux-c` 分支为原始的 `exec "$shux_bin" -L .. "$x_path" -o "$out_o"`，仅保留 `*)` 分支的 `-backend asm` 失败回退逻辑

**文件**：[shux_compile_std_x.sh](file:///home/shu/shux/compiler/scripts/shux_compile_std_x.sh)

### 2.2 runtime_*.o 缺失导致 80 个失败

**根因**：`make all` 不构建 `std-objs`，自动构建函数未触发

**修复**：手动构建所有 runtime_*.o，失败数降至 27

### 2.3 core_types_placeholder 重复符号

**根因**：`core.types.placeholder()` 被多个模块编译为强符号
- `core/types/mod.x` 定义无参数的 `placeholder()`
- `base64.x` import `core.types`，函数被内联到 `base64.o` 中
- 用户程序再 import `core.types` 时，同样的函数又被编译为强符号
- `runtime.c` 中定义的是两个参数的 weak 版本，签名不匹配

**修复**：
1. 在 `codegen_dep_module_skip_emit` 中添加对 `core.types` 的跳过逻辑
2. 修改 `runtime.c` 中的 `core_types_placeholder` 签名为无参数
3. 在 `codegen.c` 中将 `placeholder` 函数标记为 `__attribute__((weak)) inline`

**文件**：
- [codegen.c](file:///home/shu/shux/compiler/src/codegen/codegen.c)
- [runtime.c](file:///home/shu/shux/compiler/src/runtime.c)

### 2.4 io-driver undefined _shux_io_register

**根因**：
- `shux_io_register` 函数在 `std/io/core.x` 中定义，并调用 `io_backend.io_register_buffer`
- `shux_invoke_cc` 函数（C 前端链接路径）没有链入 `runtime_asm_io_stubs.o`
- `runtime_asm_io_stubs.c` 中缺少 `shux_io_register`、`io_register_buffer`、`shux_io_register_buf` 的定义

**修复**：
1. 在 `runtime_asm_io_stubs.c` 中添加 `shux_io_register`、`io_register_buffer`、`shux_io_register_buf` 的桩定义
2. 在 `runtime_link_abi.c` 中添加检测生成 C 是否引用 io 符号的函数 `link_abi_generated_c_needs_io`
3. 在 `shux_invoke_cc` 中添加按需链入 `runtime_asm_io_stubs.o` 的逻辑

**文件**：
- [runtime_asm_io_stubs.c](file:///home/shu/shux/compiler/src/asm/runtime_asm_io_stubs.c)
- [runtime_link_abi.c](file:///home/shu/shux/compiler/src/runtime_link_abi.c)

### 2.5 生成的 C 代码中缺少 main 函数

**根因**：`codegen_module_to_c` 中未初始化 `codegen_library_prefix = NULL`，导致 `!codegen_library_prefix` 条件不满足，main 函数不生成

**修复**：在 `codegen_module_to_c` 中初始化 `codegen_library_prefix = NULL`

**文件**：[codegen.c](file:///home/shu/shux/compiler/src/codegen/codegen.c)

### 2.6 std_io_write_stdout 重复符号

**根因**：`runtime_asm_io_stubs.o` 中定义了 `std_io_write_stdout` 等强符号，与生成的 C 代码冲突

**修复**：将 `runtime_asm_io_stubs.c` 中的 `std_io_write_stdout`、`std_io_write_with_timeout`、`std_io_print_u8_ptr_usize`、`std_io_print_str` 改为 `__attribute__((weak))` 定义

**文件**：[runtime_asm_io_stubs.c](file:///home/shu/shux/compiler/src/asm/runtime_asm_io_stubs.c)

### 2.7 crypto_mem_eq_c 符号未定义

**根因**：`std/crypto/core.x` 编译时符号被加上 `core_` 前缀（如 `core_crypto_mem_eq_c`），但 `mod.x` 中的 extern 声明期望不带前缀的符号（`crypto_mem_eq_c`）

**修复**：在 `runtime_crypto_inc_glue.c` 中添加包装函数，将 `core_` 前缀符号重导出为无前缀符号

**文件**：[runtime_crypto_inc_glue.c](file:///home/shu/shux/compiler/src/asm/runtime_crypto_inc_glue.c)

### 2.8 diag_reportf 未定义（math test）

**根因**：`runtime_math_libm.o` 中的 `math_fenv_emit_cap_report` 调用 `diag_reportf`，但独立编译时该符号未定义

**修复**：在 `runtime_math_libm.c` 中添加 `diag_reportf` 的 weak 桩定义

**文件**：[runtime_math_libm.c](file:///home/shu/shux/compiler/src/asm/runtime_math_libm.c)

### 2.9 random_fill_bytes_c 未定义

**根因**：`shux_invoke_cc` 的 `needs_random` 分支只链接了 `random_o`，但没有链接 `runtime_random_fill.o`

**修复**：在 `needs_random` 分支中添加链接 `runtime_random_fill.o` 的逻辑

**文件**：[runtime_link_abi.c](file:///home/shu/shux/compiler/src/runtime_link_abi.c)

### 2.10 std.encoding 重复符号

**修复**：在 `codegen_dep_module_skip_emit` 中添加对 `std.encoding` 的跳过逻辑

**文件**：[codegen.c](file:///home/shu/shux/compiler/src/codegen/codegen.c)

### 2.11 std.time 重复符号

### 3.1 typeck 类失败（3 个）

| 测试脚本 | 失败原因 |
|----------|----------|
| run-typeck.sh | typeck error[T001]: unsupported expression kind |
| run-check.sh | typeck 失败 |
| run-types-gate.sh | typeck 失败 |

**根因**：typeck 阶段对 `?` 运算符的传播处理存在缺陷，以及其他类型检查问题

### 3.2 fmt-std string literal 特化失败（1 个）

| 测试脚本 | 失败原因 |
|----------|----------|
| run-fmt-std.sh | typeck error[T001]: method call base has no type |

**现象**：`fmt.println("Hello World")` 需要编译器对字符串字面量做特化（单参 → ptr+len），当前 typeck 报 "method call base has no type"

**根因**：typeck 和 codegen 中缺少对字符串字面量的特化处理

### 3.3 编译失败类（14 个）

| 测试脚本 | 失败原因 |
|----------|----------|
| run-backtrace.sh | 链接失败，undefined symbols |
| run-hash.sh | 编译失败（hash.x 生成的 C 代码有类型错误） |
| run-ffi.sh | `-backend asm not available`，ffi_core.o 缺失 |
| run-json.sh | 链接失败，undefined symbols |
| run-csv.sh | 链接失败，undefined symbols |
| run-unicode.sh | 链接失败，undefined symbols |
| run-compress.sh | 链接失败，undefined symbols |
| run-http.sh | 链接失败，undefined symbols |
| run-tar.sh | 链接失败，undefined symbols |
| run-time.sh | bench_timer 编译失败，符号不匹配（`_std_time_elapsed_ms` vs `_std_time_timer_elapsed_ms`） |
| run-io.sh | 链接失败，undefined symbols |
| run-slice.sh | typeck error[IMP004]: typeck failed for import 'core.slice' |
| run-vector.sh | 链接失败，undefined symbols |
| run-net.sh | 编译失败（udp.x 生成的 C 代码有 `poll` 函数未声明） |

**共同根因分析**：

1. **符号命名不一致**：部分模块的 .x 文件中函数名与编译后生成的符号名不一致（如 `std.time` 的 `elapsed_ms` vs `timer_elapsed_ms`）

2. **std .o 文件缺失或构建失败**：
   - `std/hash/hash.o` 构建失败（类型错误）
   - `std/ffi/ffi_core.o` 需要 ASM 后端
   - `std/net/net.o` 构建失败（`poll` 未声明）

3. **传递依赖链入不完整**：部分 std 模块依赖 runtime glue 文件（如 `runtime_backtrace_platform.o`、`runtime_net_udp_batch.o` 等），但 `shux_invoke_cc` 未正确链入

4. **typeck 对 core.slice 处理失败**：`run-slice.sh` 报 `typeck failed for import 'core.slice'`

## 四、关键技术细节

### 4.1 Darwin safety guard

`build_seed_asm_host.sh` 在 Darwin 上跳过 `asm_seed_full.x -E`，保留已有的 `asm_backend_partial.o`

- 设置 `SHUX_DARWIN_ALLOW_ASM_SEED_FULL_E=1` 可强制重新生成
- `has_real_partial_seed_mega` 仅检查 `backend_asm_codegen_ast_seed_mega` 符号是否存在
- fallback 版只有 14 个弱符号（5952 字节），真实指令发射器（`backend_emit_block_body` 等）未被编译入

### 4.2 asm_backend_partial.o seed_mega 机制

- 所有 `arch_arm64_emit_*` 函数为 `__attribute__((weak))` 返回 -1，等待被 `asm_backend_partial.o` 的真实实现覆盖
- Darwin 上真实实现不存在 → 产生 code_len=0

### 4.3 runtime_link_abi.c 无条件链入

line 3202-3227 将多个 std 模块设为 1，无视 on-demand 检测结果：
- needs_encoding, needs_base64, needs_crypto, needs_log, needs_atomic
- needs_channel, needs_backtrace, needs_hash, needs_sort, needs_ffi
- needs_elf, needs_json, needs_csv, needs_regex, needs_unicode
- needs_tar, needs_simd, needs_datetime, needs_uuid, needs_url
- needs_cli, needs_security, needs_config, needs_cache, needs_trace
- needs_schema

### 4.4 codegen_dep_module_skip_emit 逻辑

跳过以下模块的函数体 emit（符号由预编 .o 提供）：
- `std.process` 族（process.o 提供强符号）
- `std.net` 族（net.o 提供强符号）
- `std.context` 族（context.o 提供强符号）
- `core.types` 族（base64.o 提供 core_types_* 强符号）
- `std.encoding` 族（encoding.o 提供 utf8 相关强符号）
- `std.time` 族（time.o 提供时间相关强符号）
