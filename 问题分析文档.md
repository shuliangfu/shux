| 项 | 状态 | 验证 |

| METHOD_CALL 盲信错 `call_resolved` | **根治** | 名+arity 校验后回退 binding 重搜 |
| 产品矩阵 | **绿** | rv=42 / opt=102 / si=0 / hello OK / mem OK |

### 本轮 hello 根因（勿再误判为「仅 mangle 后缀」）

```
typeck 错写 call_resolved_dep/func
  → METHOD_CALL 路径盲信 → 体错乱（uring_ok→print_u8_ptr_usize，writev→std_fmt_append_*）
  → 另：CALL 路径 mod_arena 错 → print mangle 成 print_u8
修：
  1) codegen_emit_call_func_name：名+arity 校验 resolved；arena 按 Module* 回填；arity/全 dep 搜索
  2) METHOD_CALL 快路径：resolved 必须名==method_call_name 且 arity 一致，否则 binding 回退
  3) seeds/codegen_gen.linux.x86_64.c 同向
  4) rt_preamble：co-emit 未覆盖的 shux_io_*/driver_*/ctx_* weak 桩（同 TU 勿与强定义重复）
```

| 项 | 结果 |
|----|------|
| rv / option / stdlib-import / hello / mem | **绿**（hello 含 `0a`） |

### multi-error recovery（mac 已通 · 待 Ubuntu 金标 bstrict）

**根因（2026-07-15）**：
1. `PREFER_X_O=0` 时 `runtime_driver_no_c.o` 只编 `runtime.from_x.c`，**未 merge** `rt_parse_diag` → 符号 `runtime_report_parse_recovery_diagnostics` 不进二进制。
2. `runtime.from_x.c` 的 `check_only` 在 pipeline 成功后直接 `driver_print_check_ok`，**未调用** recovery；而 hybrid 切片 `rt_run_compiler_parsed.from_x.c` 已有 recovery → 双路径漂移。
3. parser 宽容：`pr.ok=0` 且 `num_funcs=0` 仍当 parse 成功。

**修法**：
- g05：`PREFER_X_O=0` fallback 将 `rt_parse_diag.from_x.c`（`-DSHUX_RT_PARSE_DIAG_FROM_X` recovery-only）`ld -r` 并入 `runtime_driver_no_c.o`。
- hybrid rest：`-DSHUX_RT_PARSE_DIAG_PRECISE_BRIDGE` 桥接 thin `.x` 模块前缀符号。
- `runtime.from_x.c` check 路径与 slice 同形：recovery + `nfuncs<=0` 兜底。

**mac 验证**：`shux_asm check` multi/control/top/unsafe/import recovery 均 exit 1 且文案命中；`SHUX_G05_PREFER_X_O=0`。

## 1. 核心阻塞 A：`core_mem_placeholder`（已修）

### 1.1 现象

```
Undefined symbols: _core_mem_placeholder
ld: symbol(s) not found
```

`tests/mem/main.x` 经 `import("core.mem")` 调用 `mem.placeholder()` 等；链接期望 `core_mem_*`。

### 1.2 错误修复路径（已证伪）

| 尝试 | 为何无效 |
|------|----------|
| 仅在 `rt_run_compiler_parsed.from_x.c` 设 `current_codegen_prefix` | 产品 `-x -E` **不走** `driver_run_compiler_parsed`，走 `driver_run_x_emit_c` |
| 假定 `driver_c_frontend_smoke` 做 codegen | `SHUX_NO_C_FRONTEND` 下 smoke 被 `#if` 裁掉；且 smoke 本身不做 codegen |
| 入口 inherit `current_codegen_prefix` | pipeline **先** codegen 各 dep，会把 `current_*` 写成最后 dep 前缀，入口被污染 |

### 1.3 真正数据流（权威）

```
main.x (-x -E) 
  → driver_run_x_emit_c          # seeds/runtime.from_x.c
  → pctx_e 钉 entry 前缀
  → shux_pipeline_run_x_pipeline
  → codegen_x_ast(dep_i)         # 各 dep 写 current_codegen_prefix = dep 前缀
  → codegen_x_ast(entry, -1)     # 必须读 entry_module_import_path_*，不能读 current_*
  → emit_func / 同模块 CALL
```

库模块 `.o`：`shux_compile_std_module.sh` 优先 `-o`，失败回退 `-x -E`；符号面须与 import 调用端 `core_mem_*` 一致。

用户程序（`tests/mem/main.x`）：**co-emit** 会把 `core.mem` 函数体写进用户 C；若 dep 侧 skip 掉 `placeholder`，用户 `.o` 缺该符号且 on-demand 因其余 `core_mem_*` 已定义而不链 `mem.o` → 仅 `placeholder` undefined。

### 1.4 根因列表（已修）

1. **`-x -E` 未设入口前缀**  
   修：`driver_run_x_emit_c` 与 `driver_run_compiler_parsed` 均写：
   - `current_codegen_prefix_mirror/len`
   - **`entry_module_import_path_mirror/len`**（钉死，dep 不可覆写）

2. **`codegen_x_ast(entry)` 清空/误用前缀**  
   修：`dep_index < 0` 时用 `entry_module_import_path_*` 填 `prefix_buf`。

3. **同模块 CALL 只认 dep 池路径**  
   修：`codegen_emit_prefix_len_from_ctx` + 同模块 CALL 路径统一走该函数；入口优先 entry pin。

4. **`placeholder` 全局 skip**  
   修：`prefix` 以 `core_mem_` 开头时 **不** skip 裸名 `placeholder` / `string_new`（仍 skip `core_types_` / `std_string_` 的弱桩场景）。

5. **局部类型 `struct String` 无前缀**  
   修：user struct 的 `emit_type` 入口也走 `codegen_emit_prefix_len_from_ctx`（入口读 pin）。

### 1.5 修改文件

| 文件 | 作用 |
|------|------|
| `compiler/seeds/runtime.from_x.c` | `driver_run_x_emit_c` + `driver_run_compiler_parsed` 钉入口前缀 |
| `compiler/seeds/rt_run_compiler_parsed.from_x.c` | 与 mega 对称（FROM_X 切片时） |
| `compiler/src/codegen/codegen.x` | entry pin、prefix_from_ctx、同模块 CALL、placeholder skip、user struct 前缀 |
| `compiler/codegen_gen.c` | 本地 pin 与 `.x` 对齐 |
| `compiler/seeds/codegen_gen.linux.x86_64.c` | Ubuntu seed 对齐 |

## 2. 阻塞 B：`std/string/string.o` 重定义（未修）

### 现象

```
error: redefinition of 'std_string_String'
```

同一生成 C 中完整定义出现两次（约 L202 单行体 + L946 多行体）。`pipeline_codegen_struct_tag_try_claim` 未挡住第二条路径。

### 排查方向

1. `codegen_emit_skipped_dep_type_definitions` 与 entry `codegen_emit_module_struct_definitions` 是否对同一 mangled tag 都 emit 完整体。  
2. claim 键是否 prefix 不一致（空 vs `std_string_`）。  
3. 修点：单一权威路径 emit 完整 struct；第二次只允许 forward 或 skip。

### 验证

```bash

# 期望：OK，nm 有 std_string_* 符号
```

## 3. 阻塞 C：hello `std_io_print_u8`（未修）

### 现象

```
error: call to undeclared function 'std_io_print_u8'
```

定义端为重载 mangle：`std_io_print_u8_ptr_usize`；`std.fmt` 内 `io.print(ptr, len)` 生成未完整 mangle 的 `std_io_print_u8`。

### 数据流

```
std.fmt: io.print(ptr, len as usize)
  → import field + CALL
  → 应解析到 std.io 的 print(*u8, usize)
  → 定义: std_io_print_u8_ptr_usize
  → 实际调用: std_io_print_u8   # 缺 overload 后缀
```

### 验证

```bash
./compiler/shux -backend c -E -L . examples/hello.x 2>/dev/null \
  | grep -E 'std_io_print_u8'
# 期望调用与定义同为 std_io_print_u8_ptr_usize
bash tests/run-hello.sh
```

## 6. 纪律与同步

- 全量 bstrict 未绿 → **禁止**下一步自举  
- git commit **仅 macOS**；Ubuntu 只 `git pull` 验证  
- 改 `codegen.x` 必须 seed `codegen_gen.linux.x86_64.c` 同向  
- 禁止双权威：入口前缀只认 `entry_module_import_path_*` + `shux_entry_lib_name_from_path`  
- 禁止 BEST_EFFORT / SKIP_BUILD 当放行  
- 汇报须带：**构建级别**（L2/L3）、**SHUX 路径**、**矩阵结果**
