# 去 C/H/.inc 收口路线（自举阶段执行版）

> **编写时间**：2026-07-12  
> **适用阶段**：当前自举阶段  
> **目标**：在不引入隐藏回归、不制造 ABI 漂移、不把系统拖回多权威实现的前提下，持续收口手写 `C/H/.inc`，让产品构建链稳定走向纯 `.x`

---

## 0. 先给结论

原文档的**大方向是对的**：

1. 真正目标不是“把 C 机械翻译成 `.x`”。
2. 真正目标是“让产品构建链以 `.x -> -E -> .o` 为主路径，并最终成为唯一权威实现”。
3. 当前仓库确实已经具备这条路的基础设施：
   - `g05_try_x_to_o()`
   - `G05_X_O_SYM_RENAME`
   - `thin + rest`
   - `SHUX_G05_PREFER_X_O`

但原文档还有 4 个关键问题：

1. **把自动化转译工具看得太重**，把“证明正确”的迁移闭环看得还不够重。  
2. **把部分已过期的 `-E / parser` 阻塞继续当作总前提**，和当前仓库状态不完全一致。  
3. **把 `H` 文件硬编码**当作低成本方案，这条路长期风险很高。  
4. **缺少“什么时候允许物理删除 seed”的硬门槛**，这会导致删得太早、回归太深。

因此，当前阶段不应把主线定义为：

> “用工具把 C/H/.inc 快速转成 `.x`。”

而应定义为：

> “用工具把 `.x` 主路径的正确性证明、符号/ABI 对齐、thin+rest 收口、以及 seed 删除前验证全部标准化。”

再往前推进一步，当前阶段还应补上 4 个增强点：

1. **TU Contract 不能只停留在文档层，必须机器化。**
2. **`prove_x_o.sh` 必须分层，先过对象等价，再过行为等价。**
3. **bootstrap-safe `.x` 子集不能只靠自觉，必须 gate 化。**
4. **`.inc` 黑盒保留只能是阶段性策略，必须带退出机制。**

---

## 1. 当前阶段的真实目标

“去掉 C/H/.inc” 必须拆成 3 层，不能混成一句话。

### L1：产品零 C

产品构建时，不再以手写 seed C 作为主实现，而是优先使用 `.x -> -E -> .o` 的产物。

### L2：权威零 C

逻辑唯一权威实现收敛到 `.x`。  
这时 seed C 只允许作为：

- fallback
- 对照 oracle
- 过渡期 rest

而不能再承担“真实功能演化”的职责。

### L3：仓库物理零 C/H/.inc

在 L1 和 L2 稳定后，才允许物理删除对应的 `C/H/.inc` 文件。

**当前阶段主目标：L1 -> L2。**  
**当前阶段不应直接追求 L3。**

---

## 2. 当前仓库现状（按今天的工程状态重新表述）

### 2.1 已经成立的事实

1. 仓库已经不是“从零开始设计纯 `.x` 替代路径”。  
   `g05` 体系、`PREFER_X_O`、`thin + rest` 已经存在并且在产品路径上真实落地。

2. `PREFER_X_O` 已经推进到 **50/50**。  
   当前基线是 **44 个 runtime TU + 6 个 frontend TU** 已接入。

3. `seed .from_x.c` 仍然很多，但其中大量文件已经不是“原始手写 C 唯一实现”，而是：
   - 过渡期 seed
   - thin + rest 的 rest
   - 或 `.x` 逻辑的同语义副本

4. `.inc` 的最大风险区仍然是 `parser_asm` 系列。  
   这部分不适合用“整包自动转译器”一把梭。

### 2.2 不应继续沿用的旧判断

以下判断已经不能再作为总前提：

1. “先把所有 `-E parser bug` 修完，自动化才有意义。”  
   现在更准确的说法是：  
   **仍有局部 `-E / parser / bootstrap` gate，但不是全局不可用。**

2. “`unsafe parent_block_ref` 仍是当前总阻塞。”  
   该类问题已不再是唯一前排，不能继续用旧结论指导整体路线。

3. “把 macOS 的 `-E` 产物 scp 到 Ubuntu” 可作为正式路径。  
   这只能作为**临时取证手段**，不能作为正式迁移链的一部分。

## 3. 原方案里保留、修正、删除的部分

## 3.1 可以保留的判断

### 方案 1：从 C 自动转 `.x`

保留其中一个子问题：

- **扫描 seed C，自动生成 `.x` 需要的声明/前向声明**

但删除“把原始 C 自动转成最终 `.x` 实现”这一主张。  
对当前仓库而言，这不是主线。

### 方案 2：符号映射

这个方向应保留，而且要加强。  
`nm`/`ld -r`/符号集合对齐，是当前阶段最重要的“安全护栏”之一。

### 方案 4：二进制预编译产物

只保留“预编译 pin 可作为过渡期稳定器”这层认识。  
不能把它误当成“纯 `.x` 路线”的最终答案。

## 3.2 必须修正的判断

### “forward declaration 生成器”不是当前 P0

它有价值，但不是第一优先。  
当前第一优先是：

1. 建立**迁移证明闭环**
2. 建立**删除前硬门槛**
3. 建立**TU 级 contract**

### “thin + rest 自动分割”不能只看符号交集

仅靠：

- `seed.o` 导出符号
- `.x -E` 导出符号

求交集，并不足以安全决定切割位置。  
还必须考虑：

- ABI 签名是否一致
- 是否依赖 seed 内部 helper
- 是否包含宏副作用
- 是否只是 wrapper，而非真实现

所以这个工具应该生成“**候选切割点**”，不能直接自动提交为最终答案。

## 3.3 当前阶段不建议作为主线的方案

### 不建议：H 文件硬编码

这会把“声明收口”变成“复制粘贴再维护一份”，长期风险很高：

- 容易制造 ABI 漂移
- 容易制造双权威
- 容易让 `.x`、seed C、硬编码头三边失配

正确方向不是“把 `.h` 内容硬编码到别处”，而是：

- 收敛唯一 ABI 面
- 用统一的 accessors / thin ABI / 生成声明维持一致性

### 不建议：全自动 `.inc -> .x` 最终转译器

它可以做**初稿生成器**，不能做主线迁移器。  
原因不是语法，而是语义：

- 宏展开副作用
- 指针与数组语义
- bootstrap-safe 子集约束
- 控制流扁平化要求
- ABI 与 helper 依赖

当前阶段不能把“自动转出来的 `.x`”直接当成可信成品。

## 4. 当前阶段推荐总策略：证明驱动迁移

### 4.1 核心原则

当前阶段的核心不是“转得快”，而是：

1. **每前进一步，都能证明没有引入错误**
2. **每删掉一个 seed，都能证明删得对**
3. **任何时候都只有一个逻辑权威在演化**

### 4.2 标准迁移闭环

每个待迁 TU 都必须走同一套闭环：

1. `.x -> -E -> cc -c`
2. `nm` 对齐导出符号
3. `ld -r` 验证 thin + rest 合并结果
4. 最小 smoke / probe 验证真实行为
5. 与 seed 路做对象级或行为级比对
6. 对关键状态做 canonical snapshot 比对（只比稳定字段，不比原始内存垃圾位）
7. 连续通过后，才允许进入“可删除 seed”候选状态

也就是说，当前真正要自动化的不是“翻译”，而是这条闭环。

### 4.3 为什么“证明驱动”比“翻译驱动”更快

在自举期，真正昂贵的不是代码书写，而是定位“行为漂移到底来自哪里”。

如果走“翻译驱动”：

1. 脚本或 AI 先产出 `.x`
2. 编译通过
3. 行为漂移或深层 gate 崩溃
4. 排查时无法快速判断问题来自：
   - 迁移代码本身
   - parser / typeck
   - codegen / backend
   - 链接与 ABI

如果走“证明驱动”：

1. 先做对象级合同与 ABI 对齐
2. 先过 `nm` / `ld -r`
3. 再做最小 probe
4. 再做关键状态快照比对

这样一旦出问题，排查范围会被压缩到很小的边界层，而不是重新怀疑整条迁移链。

因此，当前阶段的速度来源只有两个：

1. **不返工**
2. **可证明**

## 5. 推荐工具链重排

下面是更适合当前阶段的优先级。

| 优先级 | 工具 | 作用 | 为什么现在就该做 |
|--------|------|------|------------------|
| **P0** | `prove_x_o.sh` | 分层执行 `-E`、`cc -c`、`nm`、`ld -r`、probe、snapshot compare | 没有这个，后面所有自动化都不稳 |
| **P0** | `gen_tu_contract.sh` | 为每个 TU 生成迁移合同（符号、ABI、rest、验证命令、manifest） | 没有合同，就会盲迁 |
| **P1** | `gen_x_decls.sh` | 从 seed / ABI 面提取 `.x` 所需声明 | 能减手工同步成本，但不应先于证明闭环 |
| **P1** | `auto_thin_rest_candidates.sh` | 生成 thin + rest 候选切割点 | 只能产出候选，不能直接自动落盘 |
| **P2** | `inc_to_x_skeleton.sh` | 只生成 `.inc -> .x` 初稿 skeleton | 只做辅助，不做最终迁移 |
| **P2** | `abi_snapshot_diff.sh` | 比对 C 侧和 `.x` 侧 struct / symbol contract | 用于删除前验收 |
| **P2** | `x_bootstrap_safe_lint.sh` | 检查是否违反 bootstrap-safe `.x` 子集 | 用于 gate fail-fast |

## 6. 迁移合同（TU Contract）应成为唯一事实源

每个待迁 TU 必须先有一份合同，再允许推进。

建议模板如下：

```md
# TU：runtime_link_abi / parser_asm_xxx / ...

## 1. 当前权威源
- 逻辑权威：xxx.x / seed xxx.from_x.c / mixed

## 2. 目标
- 本次目标：pure .x / thin + rest / 仅收口 ABI / 不删 seed

## 3. 导出符号合同
- 必须由 thin/.x 提供：
- 必须仍由 rest 提供：
- 禁止重复导出：

## 4. ABI Manifest
- symbol:
- arg_count:
- arg_shapes:
- ret_shape:
- pointer_or_value:
- struct_size_snapshot:
- critical_field_offsets:
- weak/strong expectation:

## 5. ABI 敏感点
- char** / *u8
- struct 布局
- weak / strong 符号覆盖
- 平台条件分支

## 6. 验证命令
- shux -E
- cc -c
- nm
- ld -r
- smoke / probe
- canonical snapshot compare

## 7. 删除 seed 的门槛
- 连续 N 次通过
- fallback 未命中
- probe 与 seed 路一致
```

**重要原则**：  
文档、脚本、构建逻辑必须围绕这份合同收敛，而不是每次靠人脑临场判断。

### 6.1 合同必须机器化

`TU Contract` 不能只是人读的 markdown，还必须对应一份机器可校验的 manifest。

建议一对一生成两类产物：

1. `tu_name.contract.md`
   - 给人读
   - 记录目标、风险、删除门槛
2. `tu_name.contract.json`
   - 给脚本验
   - 记录 symbol / ABI / provider / snapshot 规则

这样 `gen_tu_contract.sh` 产出的就不再是说明文档，而是：

- 迁移合同
- ABI 合同
- gate 输入

### 6.2 “ABI 签名哈希”应实现为 ABI fingerprint，而不是单纯哈希

当前阶段不能只对函数名做哈希。  
更稳的做法是为每个关键 TU 生成 **ABI fingerprint**，内容至少包括：

1. 导出 symbol 集
2. 参数个数
3. 参数宽度 / 指针性
4. 返回值宽度 / 指针性
5. 关键 struct 的 size snapshot
6. 关键 field offset snapshot

也就是说，真正要比对的是：

> **ABI contract manifest + fingerprint**

而不是一个不可解释的黑盒 hash。

这样一旦漂移，报错可以直接落到：

- 哪个 symbol 变了
- 哪个参数形态变了
- 哪个 struct 布局变了

而不是等到运行期才变成段错误或静默错。

### 6.3 `prove_x_o.sh` 应分层

`prove_x_o.sh` 不应只是“跑一串命令”，而应有清晰的 4 层结构：

#### L1：构建层

- `.x -> -E`
- `cc -c`

目标：证明 `.x` 至少能稳定落到 `.o`

#### L2：对象层

- `nm` 导出符号对齐
- `ld -r` 合并验证
- 重复 / 缺失 / 弱覆盖检查

目标：证明对象级 contract 成立

#### L3：行为层

- smoke
- probe
- seed / `.x` 路最小行为对照

目标：证明对外行为没有明显漂移

#### L4：状态层

- canonical snapshot compare

目标：对关键结构体或关键状态做稳定视图比对

这里必须强调：

> 不直接比整块原始内存镜像，而是比 canonical snapshot。

因为原始内存镜像会受到以下因素污染：

- padding
- 未初始化字节
- 指针地址波动
- 平台差异

所以 `snapshot compare` 只允许比：

- 已定义字段
- 关键状态字段
- 经过 canonicalize 的稳定视图

例如：

- parser 状态机关键字段
- AST block / stmt / ref 关系
- codegen 关键中间结果
- dep slot / symbol table

## 7. 删除前硬门槛

如果目标是“不能出错不能出问题”，那就必须明确：**什么情况下允许物理删除 C/H/.inc**。

### 7.1 删除 seed C 的必要条件

必须同时满足：

1. 对应 `.x` 已成为产品主路径
2. `.x -> -E -> .o` 稳定通过
3. `nm` 导出符号与合同一致
4. `ld -r` 合并无重复 / 无缺失 / 无弱桩误覆盖
5. 最小 probe 与产品 smoke 通过
6. 连续若干次构建未回退 seed
7. 不再承担 rest 职责

缺一不可。

### 7.2 删除 `.h` 的必要条件

必须同时满足：

1. 头文件只剩声明，不再承载独立逻辑
2. ABI 面已经由统一 accessors / 统一生成声明覆盖
3. 不再存在第二份手写布局定义
4. 所有引用侧已切到统一来源

### 7.3 删除 `.inc` 的必要条件

必须同时满足：

1. 对应切片 `.x` 已具备明确权威实现
2. 原 `.inc` 的宏语义已被显式处理
3. 引用链已从 `#include .inc` 收口到 `.x` 产物
4. 关联 parser / runtime / asm probe 全部通过

### 7.4 `.inc` 黑盒保留的退出机制

对于 `parser_asm` 一类超高风险 `.inc`，当前阶段可以采用“黑盒保留法”，即：

1. 先将 `.inc` 视为阶段性不可变基底
2. 通过 thin + rest / 独立 `.o` 保持产品稳定
3. 等 `.x` 路真正具备等效实现后再整切

但这里必须有一条硬约束：

> 黑盒保留只能是阶段性策略，不能演化成永久核心依赖。

因此，每个被黑盒保留的 `.inc` 都必须登记：

1. `why_kept`
2. `what_replaces_it`
3. `proof_still_missing`
4. `delete_condition`

没有退出条件的黑盒，不允许进入长期方案。

## 8. 当前阶段真正的主风险

## 8.1 不是“能不能转”，而是“会不会静默错”

最危险的不是直接编不过，而是：

- 编得过
- 链得上
- 行为 subtly 偏
- 直到 bootstrap 深处才炸

因此，本阶段的第一风险是：

> **隐性行为漂移**

而不是“工具写得够不够炫”。

## 8.2 仍存在局部 `-E / parser` 风险

要承认这点，但不能把它夸大成“所有工作都要停下来等它修完”。  
更稳的做法是：

1. 冻结一套 **bootstrap-safe `.x` 子集**
2. 所有新迁移都遵守这套子集
3. 对已知危险写法做脚本审计
4. 将这套规则做成 lint gate，而不是只写文档

例如重点限制：

- 深层嵌套 `if`
- 易触发 `-E` 丢体的控制流形态
- `unsafe` 中复杂组合语句
- 大函数里过密表达式堆积

### 8.2.1 bootstrap-safe `.x` 子集必须 gate 化

规范如果只写在文档里，最终一定会被绕开。  
所以这条规则必须落成一个最小 lint gate。

建议第一版 `x_bootstrap_safe_lint.sh` 或等价 checker 至少检查：

1. 深层嵌套 `if`
2. 高风险 `unsafe` 形态
3. 已知会触发 `-E` 丢体的控制流组合
4. 明显超大的单函数体
5. 明知会导致 parser / typeck 不稳的局部 pattern

执行策略：

- 在迁移前跑一次
- 在 `prove_x_o.sh` 中再跑一次
- 一旦命中，直接 fail-fast

这里的原则不是“提醒开发者注意”，而是：

> **规则必须强制执行，不能只靠自觉遵守。**

## 8.3 ABI 漂移风险仍是核心风险

当前阶段最不该犯的错是：

- `.x` 一套声明
- seed 一套声明
- 文档或硬编码再来一套声明

所以必须坚持：

> **所有 ABI 面只能收敛，不能复制。**

## 9. 推荐执行顺序

### Phase 0：冻结规则，不再盲迁

先补齐以下基线：

1. bootstrap-safe `.x` 子集规范
2. bootstrap-safe lint gate
3. TU contract 模板 + ABI manifest
4. `prove_x_o.sh`
5. 删除前门槛文档

在这 5 件事没有成形前，不再推进大规模“机械迁移”。

### Phase 1：把已有 `PREFER_X_O` 路径做成可证明系统

针对当前已接入的 50/50 路径：

1. 为每个 TU 补合同
2. 自动化 `nm / ld -r / probe / snapshot compare`
3. 标记哪些 TU 已达到“可删 seed 候选”

### Phase 2：继续推进对象级收口，而不是全文级收口

优先顺序：

1. 已有 thin + rest 基础的 runtime / link_abi / frontend TU
2. 再推进 parser_asm 切片
3. 最后才考虑更大块 `.inc`

### Phase 3：只对 `.inc` 做“辅助生成”，不做全自动定稿

`inc_to_x_skeleton.sh` 的定位必须明确：

- 生成 skeleton
- 预处理宏
- 标注危险段
- 生成候选 helper 拆分建议

但最终 `.x` 仍需按合同和验证闭环收口。

### Phase 4：达到门槛后，物理删除

只有当某个 TU 连续通过合同验证、构建验证、probe 验证之后，才允许真正删除对应 seed。

## 10. 当前阶段不该做的事

以下事情当前阶段应明确禁止：

1. **禁止把 `.h` 内容硬编码到多个实现文件中当“去 H”方案**
2. **禁止把全自动 `.inc -> .x` 转译器当主线**
3. **禁止未过合同验证就提前删除 seed**
4. **禁止为了“纯 `.x`”而引入新的双权威实现**
5. **禁止把临时 scp / 手工拷贝产物当正式构建链**
6. **禁止把没有退出条件的 `.inc` 黑盒保留写进长期方案**

## 12. 最终结论

当前阶段最稳、最高效、最符合“不能出错不能出问题”的路线，不是：

> “快速把 C/H/.inc 自动转成 `.x`。”

而是：

> “先把 `.x` 主路径的正确性证明、符号/ABI 对齐、thin + rest 收口、以及删除前门槛标准化，再逐步把 seed 从产品主路径和仓库里物理消灭。”

换句话说：

1. **主线是证明驱动迁移，不是转译驱动迁移。**
2. **先收口权威，再删除文件。**
3. **先把自动化用在验证和护栏上，再把自动化用在生成初稿上。**
4. **先把合同做成机器可验，再让迁移规模化。**
5. **先允许阶段性黑盒保留，但必须附带退出机制。**

这才是当前自举阶段最稳的去 `C/H/.inc` 路线。
